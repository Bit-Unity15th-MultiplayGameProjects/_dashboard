# Rougelike-EndWalker — 변경 없음 재검토 스냅샷

## 1. 주요 변경사항
- 이번 리뷰 범위의 커밋 수는 0개다. 실제 diff가 없으므로 새 구현 진전이나 이전 결함 해결 근거는 확인되지 않는다.
- 현재 문서 스냅샷에는 Concept, GDD, TDD, Client Network Guide, Server Architecture가 존재한다. 문서 기반은 이전보다 넓지만 검증 절차와 수치 사양은 여전히 약하다.
- 네트워크 구현 경로는 고수준 프레임워크가 아니라 서버·패킷·룸 상태를 직접 다루는 custom-network 성격이다. 그만큼 패킷 계약과 2인 happy path 검증이 핵심 리스크다.
- 이전 backlog 중 해결로 판정할 만한 명시 근거가 없어 resolved 항목은 없다.

## 2. 코드 품질 리뷰
이번 diff가 비어 있어 신규 코드의 가독성·구조·Unity API 사용을 직접 지적할 대상은 없다. 그러나 남아 있는 open item 기준으로 보면 위험의 중심은 로비 상태와 서버 권위가 클라이언트 재전송 타이밍에 기대는 구조다. 왜 문제냐면 마스터 위임, 늦은 입장, 씬 전환처럼 순서가 흔들리는 순간 선택 상태와 버튼 권한이 클라이언트마다 달라질 수 있기 때문이다. 개선은 RoomService가 선택 상태와 마스터 권한을 보존하고, 클라이언트는 서버 room snapshot을 렌더링하는 방향이어야 한다.

RoomService disconnect 처리도 계속 위험 신호다. ConcurrentDictionary와 일반 컬렉션이 섞인 상태에서 퇴장, master 갱신, object despawn, 패킷 flush가 동시에 접근하면 재현이 어려운 간헐 버그가 된다. 단일 lock, room actor, 또는 명령 큐 중 하나를 선택해 룸 상태 변경 경로를 하나로 묶어야 한다. 이번 diff 범위에서는 자료구조/복잡도 관련 명확한 신규 지적은 없다. 다만 이전 ledger의 ToList/ToArray, 송신 주기, 선형 조회 항목은 해결 근거 없이 사라졌으므로 네트워크 재검증 표에서 다시 판정해야 한다.

## 3. 진행도 평가
진행도는 55로 본다. custom-network 경로에서 서버, 패킷, 네트워크 문서, 룸 흐름까지 구축한 점은 분명히 인정할 수 있다. 고수준 프레임워크 프로젝트보다 콘텐츠 분량 기대치는 낮춰야 하며, 직접 네트워크 스택을 만든 난이도도 반영된다.

하지만 제출 가능한 수직 슬라이스 관점에서는 아직 high risk다. 2인 로비에서 Game 씬 진입, S_ObjectSpawned 수신, 마스터 위임, 재접속·퇴장 처리가 자동 테스트나 체크리스트로 고정되지 않았다. 지금 단계에서 “가끔 된다”는 제출 안정성이 아니다.

## 4. 다음 권장사항
- 2인 로비 입장부터 Game 씬 진입까지 PlayMode 테스트 또는 수동 체크리스트로 고정해야 한다.
- 마스터 사전 선택은 클라이언트 재전송이 아니라 RoomService의 서버 상태로 보존해야 한다.
- 마스터 위임 뒤 버튼 활성화와 권한 UI가 서버 snapshot 기준으로 갱신되는지 검증해야 한다.
- 패킷 version, count 상한, 호환 실패 응답을 Packet 코드와 네트워크 문서에 동시에 고정해야 한다.
- RoomService disconnect 경로는 master 갱신과 컬렉션 접근을 하나의 동기화 규칙으로 묶어야 한다.

## 5. 문서화 상태
design은 7점이다. Concept와 GDD가 코어 루프, 세계관, 시스템 구성을 갖추고 있고 멀티플레이와 엔딩 구조도 설명한다. 다만 핵심 인물의 플레이 특성, 보스 역할, 밸런스 의도가 아직 사양 수준은 아니다.

technical은 7점이다. TDD, Client Network Guide, Server Architecture가 있어 신규 유입자가 큰 구조를 따라갈 기반은 있다. 그러나 패킷 버전 호환성, 실패 응답, 재접속·퇴장 상태 전이 규칙이 테스트 기준으로 닫히지 않았다.

spec은 5점이다. stage_config, Stage Nod Visualizer, GDD의 템플릿은 있으나 `_sample/docs` 수준의 필드 정의, 검증 체크리스트, 수치 근거에는 못 미친다. 확률 노드 가중치와 BO 쿨다운 2 스테이지 근거가 특히 비어 있다.

- 확률 노드 가중치·BO 쿨다운 2 스테이지의 근거 수치 부재는 사양 리스크다.

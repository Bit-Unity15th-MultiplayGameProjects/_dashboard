# Rougelike-EndWalker — 몬스터 FSM/데모 루프 스냅샷

## 1. 주요 변경사항
- 싱글/멀티 몬스터 FSM, `NetworkMonsterState/Transform`, 서버 `MonsterLoop`, `C_SpawnMonsters`, `C_MonsterHit` 패킷이 추가됐다.
- `NetworkObjectManager.Spawn()` 순서를 문서상 `S_ObjectSpawned` 선전송 후 `MarkClean`으로 정리한 점은 의미 있는 개선이다.
- `dimension_shard` 프리팹, 몬스터 데이터, Soul 스프라이트, 데모 종료 팝업, TestScene이 추가되어 시연 범위가 넓어졌다.
- TDD가 v1.8로 갱신되어 몬스터 스폰, 서버 권위, 패킷 흐름을 비교적 자세히 설명한다.

## 2. 코드 품질 리뷰
이번 변경은 “몬스터가 보이고 움직인다”는 수직 슬라이스를 만든 점이 좋다. 특히 FSM 순수 로직과 Unity/서버 래퍼를 분리하려는 방향은 맞다. 다만 실제 프리팹에는 `MonsterController`와 네트워크 컴포넌트가 함께 붙어 있다. 멀티에서 로컬 FSM이 살아 있으면 서버 위치 보정과 클라이언트 이동이 충돌한다. 싱글 전용 컴포넌트 비활성화 조건을 코드로 보장하거나, 싱글/네트워크 프리팹을 분리해야 한다.

가장 큰 위험은 권한 검증이다. 문서는 `C_MonsterHit`를 서버가 사거리 재검증한다고 말하지만, 패킷 자체가 `Damage`를 클라이언트에서 받는 구조다. 이 값이 그대로 반영되면 전투 결과가 클라이언트 권위가 된다. 왜 문제인지 명확하다. 데모에서는 “피격이 된다”가 아니라 “조작 불가능한 피격만 된다”가 기준이어야 한다.

자료구조/복잡도 관점에서는 `_monsters`를 `ConcurrentDictionary<int, MonsterEntry>`로 둔 선택은 object id 기반 조회에 적절하다. 반대로 `C_SpawnMonsters`의 배열 count 상한과 패킷 reader 상한이 코드 계약으로 보이지 않는다. 현재 구조는 잘못된 count 하나가 배열 할당과 루프 비용을 서버에 전가한다. `MaxMonsterSpawnCount`, `MaxPacketBytes` 같은 상수를 두고 deserialize 단계에서 먼저 차단해야 한다.

## 3. 진행도 평가
이 프로젝트는 명백한 `custom-network` 경로다. 서버, 룸, 패킷, NetworkObject, dirty flush를 직접 구현하고 있으므로 같은 콘텐츠 양의 프레임워크 프로젝트보다 네트워크 기반 작업을 더 크게 인정한다. 이번 커밋은 몬스터 멀티 루프까지 붙였으므로 진행도는 상승했다.

다만 제출 가능한 안정성은 아직 부족하다. 2인 NodeSelect→Game→몬스터 스폰→전투 클리어를 자동으로 반복 검증하는 테스트가 없고, 서버 권한 검증도 빈틈이 크다. 그래서 진행도는 62로 보되, 리스크는 high가 맞다.

## 4. 다음 권장사항
- `C_MonsterHit`는 클라이언트 Damage를 버리고 서버가 공격자, 무기, 거리, 대상 생존 상태로 피해를 재계산해야 한다.
- `C_BattleClear`는 `_nodeEntered`만 보지 말고 현재 열, 입장자 목록, 활성 전투 상태를 모두 검증해야 한다.
- 몬스터 프리팹의 싱글 FSM과 네트워크 수신 경로를 분리해 멀티 위치 권한 충돌을 제거해야 한다.
- 2인 NodeSelect→Game→몬스터→클리어 PlayMode 테스트를 먼저 고정해야 한다.
- 패킷 header version, count 상한, 호환 실패 응답을 코드와 문서 양쪽에 같은 규칙으로 박아야 한다.

## 5. 문서화 상태
design은 7점이다. 컨셉 문서와 GDD가 코어 루프, 멀티 엔딩, 주요 인물, 시스템 구성을 설명한다. 다만 핵심 인물의 플레이 수치와 히든 루트 조건은 아직 기획 설명 수준이다.

technical은 8점이다. TDD v1.8은 몬스터 FSM, NetworkObject spawn 순서, 서버/클라이언트 책임을 이번 구현에 맞춰 잘 갱신했다. 단, 문서의 “서버 재검증”이 코드 계약으로 닫히지 않으면 기술 문서가 안전성을 과장하게 된다.

spec은 5점이다. 데이터 파일과 일부 시각화 문서는 있으나 Stage 5 relic placeholder, 확률 노드 가중치 근거, BO 쿨다운 근거가 남아 있다. 테스트 체크리스트로 바로 쓰기에는 수치 표가 부족하다.

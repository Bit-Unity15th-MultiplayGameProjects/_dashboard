# Rougelike-EndWalker — 변경 없음 재검토 스냅샷

## 1. 주요 변경사항
- 이번 범위의 커밋 수는 0개이며 diff/stat 모두 비어 있다. 코드 기준 전진은 확인되지 않는다.
- 직전 TODO 6개는 현재 스냅샷에서 해결 근거가 없어 모두 유지한다. 특히 2인 로비 happy path 검증 부재는 여전히 쇼스토퍼다.
- 문서는 Concept, GDD, TDD, Network Guide, Server Architecture까지 폭이 넓다. 다만 테스트 체크리스트와 프로토콜 실패 규칙은 아직 사양으로 고정되지 않았다.
- 과거 사라진 네트워크 open item은 해결로 보지 않는다. 현재 파일 기준으로 재분류하는 작업을 다음 TODO로 유지한다.
- 멀티플레이 스택 자체는 TCP Dedicated Server + 자체 NetworkObject 경로로 문서화되어, 오래된 스택 결정 backlog는 해결 처리한다.

## 2. 코드 품질 리뷰
이번 diff가 비어 있으므로 새 코드의 가독성·중복·에러 처리 변화는 평가할 수 없다. 그러나 누적 항목 기준으로 보면 `LobbySceneController`가 마스터 선택 상태를 클라이언트 재전송 타이밍에 기대는 구조는 위험하다. 왜 문제냐면 로비 권한의 원천이 서버인지 클라이언트인지 흐려지고, 마스터 위임·재입장·지연 패킷 상황에서 UI와 실제 서버 상태가 갈라질 수 있기 때문이다. 개선은 `RoomService`가 선택 상태를 보존하고, 클라이언트는 서버 스냅샷만 렌더링하는 방향이 맞다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 새 hot path 할당이나 LINQ 반복 생성 지적은 없다. 다만 기존 open item인 `RoomService` disconnect 경로는 `ConcurrentDictionary`와 일반 컬렉션 접근이 섞인 것으로 기록되어 있다. 현재 비용은 퇴장 중 master 갱신과 룸 컬렉션 순회가 경합 조건을 만들 수 있다는 점이며, 단일 lock 경계 또는 불변 스냅샷 기반 갱신으로 끊어야 한다. custom-network 프로젝트에서는 이 문제가 단순 성능 문제가 아니라 세션 상태 불일치로 이어진다.

## 3. 진행도 평가
구현 경로는 명확히 `custom-network`다. 서버 프로세스, 패킷 정의, 바이트 직렬화, 룸 관리, 클라이언트 NetworkObject 동기화를 직접 구현하는 경로이므로 컨텐츠 양은 일부 보정한다. 그럼에도 2인 로비 입장부터 Game 씬 진입까지 반복 검증 가능한 증거가 없고, 패킷 version/count/호환 실패 규칙도 고정되지 않았다.

진행도는 52로 본다. 직접 네트워크 스택을 세운 점은 진척이지만, 이번 스냅샷은 커밋이 없고 직전 위험이 그대로다. 마감 시연 기준으로는 “될 때도 있음”이 아니라 “반복 재현 가능함”이 필요하므로 risk는 high다.

## 4. 다음 권장사항
- 2인 로비 입장부터 Game 씬 진입까지 PlayMode 테스트 1개와 수동 체크리스트를 먼저 고정한다.
- 마스터 사전 선택은 클라이언트 재전송이 아니라 `RoomService`의 서버 상태로 보존한다.
- 마스터 위임 뒤 선택 버튼·시작 권한 UI가 서버 스냅샷으로만 갱신되는지 검증한다.
- `RoomService` disconnect 경로의 master 갱신과 컬렉션 접근 경계를 동기화한다.
- 패킷 헤더 version, 리스트 count 상한, 호환 실패 응답을 코드 상수와 네트워크 문서에 동시에 박는다.
- 과거 사라진 네트워크 open item을 현재 파일 기준으로 해결·무효·재등장으로 재분류한다.

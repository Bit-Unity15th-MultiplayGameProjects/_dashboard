# Rougelike-EndWalker — 변경 없음 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개라 코드 diff 기준의 신규 전진은 없다. 이번 평가는 현재 HEAD 문서와 누적 open item 재판정 중심이다.
- 현재 문서는 컨셉, GDD, TDD, 네트워크 가이드, 서버 아키텍처까지 존재해 구조 설명의 폭은 넓다.
- 반대로 2인 로비, 마스터 위임, 패킷 버전·count 상한, RoomService disconnect 동기화는 여전히 제출 안정성을 흔드는 영역이다.
- README 작성자 혼용과 아트/UI placeholder는 현재 스냅샷 기준 해결로 판정한다.

## 2. 코드 품질 리뷰
이번 diff에는 코드 변경이 없으므로 새로 유입된 코드 품질 회귀는 없다. 그러나 남아 있는 open item의 성격은 가볍지 않다. `LobbySceneController`와 `RoomService` 사이에서 마스터 선택 상태를 클라이언트 재전송 타이밍에 의존하면, 마스터 위임 직후 게스트와 신규 마스터의 권한 UI가 서로 다른 상태를 볼 수 있다. 이 문제는 UI 보정으로 덮을 사안이 아니라 서버 스냅샷을 단일 진실 공급원으로 고정해야 한다.

패킷 계층도 위험하다. `Packet.cs`에 count 계열 필드는 보이지만 version, count 상한, 호환 실패 응답이 코드 계약으로 고정됐다는 근거가 없다. 직접 TCP 서버와 바이트 직렬화를 다루는 custom-network 경로에서는 이 부재가 단순 문서 누락이 아니라 수신 루프 파손, 잘못된 패킷 해석, 재현 어려운 2인 플레이 실패로 이어진다. 최소한 헤더 버전, 리스트 최대 길이, 실패 응답 packet id를 상수와 문서 양쪽에 박아야 한다.

자료구조·복잡도·할당 관점에서는 이번 diff 범위에서 명확한 신규 지적은 없다. 다만 `RoomService` disconnect 경로의 컬렉션 접근 동기화가 남아 있으므로, 최적화보다 먼저 단일 lock 또는 스냅샷 기반 순회로 경쟁 상태를 제거하는 것이 우선이다.

## 3. 진행도 평가
이 프로젝트는 Netcode/Mirror류 프레임워크가 아니라 서버 프로세스, 패킷, 직렬화, 룸 관리를 직접 구현하는 custom-network 경로다. 따라서 컨텐츠 양만으로는 낮게 보지 않고, 서버·클라이언트 연결과 NetworkObject 동기화 문서화가 존재하는 점은 진척으로 인정한다.

그럼에도 제출 가능한 수직 슬라이스 기준으로는 52% 수준이다. 2인 happy path를 자동 또는 수동 체크리스트로 반복 검증하지 못하고, 패킷 실패 처리와 마스터 위임 상태가 닫히지 않았다. 지금 상태로는 시연 중 한 번 성공해도 같은 절차가 다시 성공한다고 말하기 어렵다.

## 4. 다음 권장사항
- 2인 로비 입장부터 Game 씬 진입까지 체크리스트와 PlayMode 1개 추가.
- 마스터 사전 선택을 클라이언트 재전송이 아니라 RoomService 상태로 보존.
- 마스터 위임 후 선택 버튼·권한 UI 갱신을 서버 스냅샷 기준으로 검증.
- 패킷 헤더 version·count 상한·호환 실패 응답을 코드와 문서에 고정.
- RoomService disconnect 경로의 master 갱신과 컬렉션 접근을 동기화.

## 5. 문서화 상태
design은 7점이다. 컨셉 문서와 GDD가 코어 루프, 핵심 인물, 노드·스테이지·멀티 엔딩 구조를 설명하고 있어 방향성 파악은 가능하다. 다만 일부 항목은 “왜 이 선택인가”보다 기능 나열에 가깝다.

technical은 7점이다. TDD, Client Network Guide, Server Architecture 문서가 있어 신규 유입자가 큰 구조를 따라갈 수 있다. 하지만 패킷 스키마, 버전 호환성, 실패 응답 규칙이 코드 계약 수준으로 닫히지 않았다.

spec은 5점이다. 조작, FSM, 일부 데이터 규칙은 있으나 Stage 5 분기 유물 placeholder, 확률 노드 가중치 근거, 핵심 인물 밸런싱 수치가 비어 있다. QA 체크리스트로 바로 쓰기에는 아직 부족하다.

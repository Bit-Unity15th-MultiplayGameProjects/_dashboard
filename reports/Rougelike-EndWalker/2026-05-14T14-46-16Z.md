# Rougelike-EndWalker — 2026-05-14 스냅샷

## 1. 주요 변경사항
- `NodeSelect ⇄ Game` 흐름을 Additive 로드/언로드 구조로 문서화하고, `BattleSceneController`에 `StageClearPopup` 참조와 `S_BattleCleared` 수신 처리를 추가했다.
- 게스트 로비 대기 패널과 준비 토글 UI를 추가했다. 다만 서버 준비 상태가 아니라 로컬 표시 토글에 가깝다.
- 이벤트 노드를 양 클라이언트에서 같은 항목으로 보이게 하려고 `column` 기반 결정론적 이벤트 선택을 추가했다.
- TDD, Network Service, Screen Flow 문서가 현재 네트워크 흐름을 크게 따라잡았다. 이 점은 분명한 전진이다.
- 서버 파일 변경 없이 클라이언트와 문서 중심으로 “완료” 상태를 선언한 부분이 있어, 구현 근거와 문서 표현 사이 간극이 남아 있다.

## 2. 코드 품질 리뷰
`BattleSceneController`의 `_clearSent`는 “내가 클리어 패킷을 보냈는가”와 “원격 클리어를 이미 확인했는가”를 동시에 표현한다. 왜 문제냐면 같은 bool이 로컬 클리어, 원격 클리어, 팝업 확인 후 복귀라는 서로 다른 상태를 압축해 디버깅 난도를 높인다. `ClearState.None / Submitted / ConfirmedRemote / Returning`처럼 상태를 분리하면 중복 클릭과 지연 패킷 처리가 명확해진다.

가장 위험한 부분은 `NodeSelectSceneController.HandleNodeEntered()`에 게스트 `ClearTracker.MarkCleared(packet.Column)`이 여전히 남아 있다는 점이다. 전투를 실제로 클리어하기 전에 노드를 클리어 처리하면 실패한 전투, 연결 끊김, 서버 거부 상황에서도 다음 열이 열릴 수 있다. 진행 상태는 반드시 `S_BattleCleared` 또는 Game 언로드 완료 후 서버가 승인한 column으로만 갱신해야 한다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 hot path의 명확한 할당 문제는 크지 않다. `EventInfo.GetEventForColumn()`의 O(category+event) 순회는 이벤트 선택 시 단발성이라 성능 문제는 아니다. 다만 `new System.Random(column)`은 stage, run seed, node type을 무시하므로 같은 column이 매 런·매 스테이지 같은 이벤트로 고정될 수 있다. 결정론은 필요하지만 seed 입력은 `runSeed + stage + column + nodeType` 조합으로 확장하는 편이 맞다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. TCP 서버, 패킷, 룸, NetworkObject, Additive 씬 전환까지 직접 구현하고 있으므로 같은 컨텐츠 양의 프레임워크 프로젝트보다 네트워크 기반 진척을 더 인정할 수 있다. 이번 커밋은 NodeSelect 복귀와 배틀 클리어 동기화 방향을 잡았다는 점에서 수직 슬라이스 완성도에 기여했다.

다만 제출 가능한 반복 시연 기준으로는 아직 위험하다. 2인 happy path 자동 검증이 없고, 클리어 권한 검증·패킷 호환성·disconnect 동기화가 코드로 고정되지 않았다. 진행도는 62%로 본다. 전진은 분명하지만, 현재 상태를 “완료”로 부르면 이후 회귀를 발견하기 어렵다.

## 4. 다음 권장사항
- `C_BattleClear`는 서버의 active battle, 입장 완료 client set, 현재 column과 대조해 거부 경로까지 구현해야 한다.
- 게스트 `ClearTracker` 갱신을 `S_NodeEntered`에서 제거하고 `S_BattleCleared` 승인 이후로 옮겨야 한다.
- 2인 로비→NodeSelect→Game→클리어→NodeSelect 복귀 PlayMode 테스트 1개를 먼저 고정해야 한다.
- `S_PlayerLeft` 이후 Title 복귀, Master 승격, 시작 버튼 권한 정책 중 하나를 선택하고 코드·문서를 일치시켜야 한다.
- 패킷 header version, count 상한, 호환 실패 응답을 `Packet.cs`와 네트워크 가이드에 같은 계약으로 박아야 한다.

## 5. 문서화 상태
design은 7점이다. 컨셉 문서와 GDD에 코어 루프, 멀티 엔딩, 핵심 경험, 시스템 관계가 정리돼 있다. 다만 메타 루프와 일부 레퍼런스가 placeholder라 9점권 문서로 보긴 어렵다.

technical은 8점이다. TDD v1.4와 Network Service 문서가 Additive 씬 전환, 패킷 범위, NetworkObject 흐름을 꽤 구체적으로 설명한다. 단, 문서의 “완료” 선언이 실제 diff의 서버 검증 부재를 가리는 부분은 위험하다.

spec은 5점이다. 레벨·노드·캐릭터·밸런싱 수치가 테스트 체크리스트로 바로 쓸 정도는 아니다. `stage5_branch_relics` placeholder, M3~M5 일정 미정, 확률·쿨다운 근거 부재가 계속 남아 있다.

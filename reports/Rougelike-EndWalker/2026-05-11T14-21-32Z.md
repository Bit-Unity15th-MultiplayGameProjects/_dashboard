# Rougelike-EndWalker — 멀티플레이 동기화 스냅샷

## 1. 주요 변경사항
- 로비와 게임 씬에 `NetworkObjectManager`를 배치하고, Knight/Veteran 프리팹에 네트워크 동기화 컴포넌트를 추가했다.
- 캐릭터 선택, 테마 선택, 인원 표시가 서버 패킷과 연결되었다. 단순 UI 상태에서 실제 룸 상태 반영으로 한 단계 전진했다.
- `C_EnterNode`를 통해 Game 씬 진입 후 서버가 선택 캐릭터를 스폰하도록 흐름을 바꿨다.
- `PacketIO`에 최대 패킷 크기 상수를 두고, 알 수 없는 패킷을 스킵하도록 변경했다.
- 멀티플레이 디버깅 기록 문서를 추가했으나, 문서 내용 일부가 현재 코드와 어긋난다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 좋다. 로비 선택값을 서버에 저장하고 Game 씬에서 다시 스폰하는 구조는, 선택 UI와 실제 전투 오브젝트를 분리하려는 의도가 보인다. 그러나 구현은 아직 디버그 로그와 런타임 메시지 호출에 강하게 기대고 있다. `NetworkObjectManager`의 `BroadcastMessage("StopRotating")`, `NetworkPlayerTransform`의 `SendMessage("Enable", 5f)`는 컴파일 타임 검증이 되지 않는다. 왜 문제냐면 메서드명 변경, 컴포넌트 제거, 프리팹 구조 변경이 런타임 침묵 실패로 이어진다. 개선은 `INetworkSpawnView`, `IPlayerControlGate` 같은 작은 인터페이스나 명시 컴포넌트 참조로 바꾸는 쪽이 맞다.

서버 쪽은 더 위험하다. `RoomService.RemoveSession`은 새 마스터 후보를 계산하지만 `_masterClientId`를 갱신하지 않는다. 또한 `_characterSelections`, `_lobbySpawnedObjects`, `_enteredClients`, 실제 NetworkObject despawn 정리가 함께 보이지 않는다. 왜 문제냐면 2인 happy path는 통과해도 재접속, 퇴장, 씬 재진입 순간에 ghost object와 권한 불일치가 발생한다. 이 부분은 기능 추가보다 먼저 정리해야 한다.

자료구조/복잡도 관점에서는 `NetworkPlayerTransform.Update()`가 핵심 지적이다. 현재는 owner 위치가 `Vector3` 단위로 조금이라도 바뀌면 매 프레임 `SetAndSend`가 호출된다. 비용은 이동 중 매 프레임 TCP 패킷 생성·직렬화·브로드캐스트이며, 원격 보간도 없어 화면 흔들림으로 이어진다. 추천 흐름은 송신 tick, 위치 변화 임계값, 마지막 전송값 캐시, 원격 보간 버퍼를 두는 것이다. 커스텀 네트워크 프로젝트에서는 이 비용이 곧 시연 안정성이다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. Unity NGO 패키지가 존재하지만 실제 핵심은 자체 서버, 패킷 ID, `NetworkStream`, 룸 서비스, 직접 직렬화로 구성되어 있다. 따라서 네트워크 기반 공사는 진척으로 인정한다. 캐릭터 선택부터 transform sync까지 잇는 시도는 낮은 수준의 수직 슬라이스에 가까워졌다.

다만 제출 가능한 반복 시연 단계로 보기는 어렵다. 현재 스폰 성공 여부가 로그 확인에 의존하고, 퇴장·재진입·버전 불일치·패킷 호환성 대응이 약하다. 진행도는 56으로 본다. 클라이언트 선택, 서버 스폰, 로비와 Game 씬 연결, 네트워크 오류 수정 기록까지 포함되어 이전보다 실제 멀티플레이 수직 슬라이스에 가까워졌다. 그래도 custom-network 경로의 불안정성 때문에 `risk_level`은 high가 타당하다.

## 4. 다음 권장사항
- 2인 Game 씬 진입부터 `S_ObjectSpawned` 수신, owner 이동, remote 반영까지 한 번에 검증하는 절차를 고정해야 한다.
- `RoomService`에서 퇴장 시 master 갱신, 선택 상태 제거, 로비/게임 스폰 오브젝트 despawn을 한 트랜잭션처럼 처리해야 한다.
- `NetworkPlayerTransform`은 매 프레임 송신을 멈추고 tick rate, 임계값, 원격 보간을 명시해야 한다.
- `SendMessage`와 `BroadcastMessage`는 명시 인터페이스 호출로 바꿔 런타임 침묵 실패를 제거해야 한다.
- `PacketIO`의 unknown packet skip은 버전 불일치 응답과 연결해 프로토콜 계약으로 승격해야 한다.

## 5. 문서화 상태
design 점수는 7이다. `Doc/1.EndWalker_Concept.md`, `Doc/2.EndWalker_GDD.md`, README가 사서, 기록자, 테마, 노드, 보스 흐름을 설명하고 Visualizer 문서도 플레이 구조를 보강한다. 다만 일부 캐릭터·엔딩·히든 루트의 수치와 조건은 아직 데이터와 완전히 닫히지 않았다.

technical 점수는 8이다. `Doc/3.EndWalker_TDD.md`, `Server/Server_Architecture.md`, `Visualizer Doc/4.Client_Network_Guide.md`, `Visualizer Doc/6.networkerror fixed.md`가 서버 권위, 클라이언트 네트워크, 버그 수정 흐름을 넓게 다룬다. 남은 약점은 실제 코드 변경과 문서의 일부 표현이 어긋나는 지점, 그리고 패킷 실패 처리 계약이 더 명확해야 한다는 점이다.

spec 점수는 7이다. 노드 가중치, BO 쿨다운, Stage 구성, 캐릭터 데이터, 네트워크 시퀀스가 문서와 JSON에 상당히 내려와 있다. 다만 패킷 버전 호환성, count 상한, PlayMode 검증, `stage5_branch_relics` 같은 placeholder는 아직 테스트 체크리스트 수준으로 닫히지 않았다.

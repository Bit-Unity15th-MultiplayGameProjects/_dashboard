# Rougelike-EndWalker — 커스텀 네트워크 도입 스냅샷

## 1. 주요 변경사항
- `ClientNetworkManager`, `PacketIO`, `NetworkObjectManager` 등 클라이언트 TCP 수신·송신·오브젝트 동기화 뼈대가 추가됐다.
- 로비 입장 실패 UI와 `LobbyErrorPopup`이 붙었고, 타이틀 씬에 방 ID 입력과 에러 팝업 참조가 연결됐다.
- `NetworkObject`, `NetworkBehaviour`, `NetworkPlayerTransform/State` 계층이 생겨 인게임 동기화 구조를 직접 구현하기 시작했다.
- 타이틀 이미지 에셋과 README·TDD·Visualizer 문서가 갱신됐다. 단, 문서 변경의 검증 근거는 diff 일부만으로는 제한적이다.

## 2. 코드 품질 리뷰
가장 큰 전진은 네트워크 계층을 별도 `Network` 어셈블리와 런타임/인게임 구조로 나눈 점이다. 역할 분리는 이전보다 명확해졌다. 그러나 `ClientNetworkManager`는 연결 생성, 수신 루프, 큐 디스패치, 클라이언트 ID 반영, 종료 처리를 모두 들고 있다. 왜 문제인지 분명하다. 커스텀 네트워크에서 이 클래스가 비대해지면 접속 실패, 씬 전환, 재접속, 서버 종료가 모두 같은 지점에서 얽혀 디버깅 비용이 급격히 오른다. 연결 상태 모델을 `Disconnected/Connecting/Connected/Closing` 정도로 명시하고, 실패 이벤트를 UI 계층으로 전달하는 구조가 필요하다.

에러 처리도 아직 조용히 실패하는 패턴이 많다. `Send`는 `_stream == null`이면 그냥 return 하고, 송신 예외는 로그 없이 `Disconnect`만 호출한다. `NetworkObjectManager.ApplyDelta`도 대상 오브젝트, behaviour index, variable index가 맞지 않으면 대부분 return 한다. 왜 문제인지: 서버와 클라이언트의 상태가 이미 갈라졌는데 로그가 없으면 원인 패킷을 찾을 수 없다. 개선은 단순하다. 패킷 ID, objectId, behaviour/variable index, payload 길이를 포함한 warning/error 로그를 최소한 개발 빌드에서 남겨야 한다.

자료구조/복잡도 관점에서는 `_spawnedObjects`를 `Dictionary<int, NetworkObject>`로 둔 선택은 맞다. objectId 기반 델타 적용은 반복 조회가 핵심이므로 선형 탐색보다 안전하다. 반면 `_prefabs`는 서버 registry 순서와 배열 인덱스가 암묵적으로 결합돼 있다. 현재 비용은 O(1)이지만, 서버와 클라이언트 prefab 순서가 어긋나면 잘못된 캐릭터가 스폰되는 논리 오류가 된다. `PrefabIndex` 계약을 문서화하거나 id→prefab 매핑 테이블을 ScriptableObject로 빼는 편이 유지보수에 낫다.

## 3. 진행도 평가
구현 경로는 명확히 `custom-network`다. `TcpClient`, 패킷 직렬화, 수신 루프, 룸 서비스, 네트워크 오브젝트 동기화를 직접 만드는 방향이므로 프레임워크 프로젝트와 같은 컨텐츠 밀도를 요구하긴 어렵다. 이번 변경은 직접 구현 경로에서 의미 있는 기반 작업으로 인정된다.

다만 제출 가능한 수직 슬라이스 관점에서는 아직 40%대다. 2인 happy path가 자동 또는 체크리스트로 고정됐다는 증거가 없고, 패킷 계약·실패 응답·재접속 처리도 불안정하다. 특히 네트워크와 씬 싱글톤이 같이 커지고 있어 risk는 high로 둔다.

## 4. 다음 권장사항
- 2인 접속, 방 생성, 게스트 입장 실패, 정상 입장, 씬 전환을 한 번에 재현하는 검증 절차를 고정해야 한다.
- `PacketId`, `Packet`, `PacketIO` 기준으로 프레이밍, 길이 제한, 버전, 알 수 없는 패킷 처리 규칙을 문서화해야 한다.
- `ClientNetworkManager`의 종료·재접속·이벤트 해제를 정리해 씬 재로드 후 중복 수신 가능성을 제거해야 한다.
- `NetworkObjectManager`의 무시 return 지점에 개발 로그를 넣어 동기화 불일치 원인을 추적 가능하게 만들어야 한다.
- Unity Test Runner 기준으로 패킷 직렬화 EditMode 테스트와 로비 PlayMode smoke test를 추가해야 한다.

## 5. 문서화 상태
design은 6점이다. 컨셉 문서와 README가 갱신됐고 세계관·방향성은 누적되고 있다. 그러나 핵심 인물 특성, 보스 밸런스, 수치 근거가 아직 사양서 수준으로 닫히지 않았다.

technical은 5점이다. Visualizer 문서와 네트워크 가이드가 갱신된 점은 좋다. 하지만 이번 코드가 직접 패킷·TCP 수신을 다루는 만큼, 권한 모델과 실패 처리 규칙이 문서의 중심에 있어야 하는데 아직 확정 근거가 부족하다.

spec은 4점이다. 룸 입장 실패, 오브젝트 스폰, 네트워크 변수 델타 같은 기능은 구현됐지만 입력·상태·응답·에러 코드 표가 테스트 체크리스트로 쓰일 정도는 아니다. `_sample/docs` 수준과 비교하면 필드 정의와 검증 항목이 많이 비어 있다.

## 6. Backlog
- 테스트 코드·플레이모드 검증 절차 부재는 계속 남는다. 네트워크가 들어온 지금은 단순 품질 문제가 아니라 시연 안정성 문제다.
- 패킷 프로토콜 스키마와 버전 호환성 규칙 부재도 남는다. 직접 TCP를 쓰는 이상 이 항목은 우선순위가 높다.
- 문서 위치 이동 후 기존 참조 경로 검증 부재는 계속 추적해야 한다. Visualizer 문서가 늘어난 만큼 깨진 링크가 생기기 쉽다.
- 확률 노드 가중치, BO 쿨다운 2 스테이지 근거 수치와 핵심 인물·보스 밸런싱 문서는 아직 사양서로 닫혔다고 보기 어렵다.
- README placeholder와 작성자 표기 혼용은 이번 diff 일부만으로 해결을 확인할 수 없어 이월한다.

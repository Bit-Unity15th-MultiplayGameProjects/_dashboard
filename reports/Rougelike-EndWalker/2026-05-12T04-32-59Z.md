# Rougelike-EndWalker — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개이므로, 이번 범위에서 코드나 문서의 실질 변경은 없다.
- 직전 TODO의 핵심인 2인 로비-게임 진입 검증, 마스터 위임, 패킷 규칙 고정은 해결 근거가 없다.
- 현재 문서 스냅샷은 양이 충분하지만, 일부 링크가 `[링크]` placeholder이고 일정·수치 사양이 비어 있다.
- `stage_config.json`에는 `stage5_branch_relics`가 `!$데이터작성요망!$`로 남아 있어 분기 데이터 공백이 확인된다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 신규 코드 품질 저하는 없다. 그러나 이전부터 이어진 커스텀 네트워크 경로의 위험은 그대로다. `Packet.cs`, `PacketId.cs` 기준으로 version, 호환 실패 응답, 리스트 count 상한이 명시됐다는 근거가 없고, 이는 수신 버퍼나 직렬화 오류가 발생했을 때 “왜 실패했는지”를 추적하기 어렵게 만든다. 직접 TCP 서버와 바이트 직렬화를 구현하는 프로젝트에서는 이 영역이 게임 로직보다 먼저 안정화돼야 한다.

`RoomService`와 `LobbySceneController` 관련 TODO도 해결 근거가 없다. 마스터 권한과 선택 상태가 클라이언트 재전송에 기대면, disconnect나 master delegation 이후 UI가 우연히 맞아 보이는 상태가 된다. 서버 스냅샷을 단일 진실 공급원으로 삼고, 클라이언트는 그 결과만 렌더링하는 구조로 좁히는 편이 안전하다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 신규 지적은 없다. 다만 이전 ledger의 disconnect 경로 동기화 항목은 단순 최적화가 아니라 공유 컬렉션 접근 안정성 문제다. 네트워크 세션·룸 컬렉션은 lock 또는 단일 스레드 큐 처리 중 하나로 정책을 고정해야 재현 불가능한 퇴장 버그를 줄일 수 있다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷 정의, 룸 관리, 클라이언트 동기화 문서가 존재하므로 단순 스캐폴드 단계는 넘었다. 이 난이도는 감안해야 하지만, 2인 happy path 검증과 프로토콜 실패 처리 부재는 제출 가능한 수직 슬라이스의 신뢰도를 크게 깎는다.

진행도는 52로 본다. 문서와 구조는 중간 이상이나, 신규 커밋이 없고 직전 고위험 TODO가 그대로 남았다. 다음 진척은 기능 추가보다 “2인 입장-씬 전환-스폰 수신”을 반복 재현 가능하게 만드는 데서 나와야 한다.

## 4. 다음 권장사항
- 2인 로비 입장부터 Game 씬 진입까지 PlayMode 검증 1개를 먼저 고정한다.
- 마스터 사전 선택은 클라이언트 재전송이 아니라 `RoomService` 상태로 보존한다.
- 마스터 위임 뒤 버튼·권한 UI를 서버 스냅샷 기준으로만 갱신하는지 검증한다.
- 패킷 header version, count 상한, 호환 실패 응답을 코드와 문서에 함께 고정한다.
- `RoomService` disconnect 경로의 master 갱신과 컬렉션 접근 정책을 동기화한다.

## 5. 문서화 상태
design은 7점이다. `Doc/1.EndWalker_Concept.md`와 GDD가 코어 루프, 시스템 구성, 멀티 엔딩 의도를 설명하고 있어 방향성은 분명하다. 다만 M3-M5 일정과 일부 경험 목표가 아직 실행 계획으로 닫히지 않았다.

technical은 7점이다. TDD, 서버 아키텍처, 클라이언트 네트워크 가이드가 있어 신규 유입자가 구조를 따라갈 수 있다. 하지만 패킷 버전·호환성 규칙이 코드 계약으로 고정됐다는 근거가 약하다.

spec은 5점이다. 씬 흐름, 스테이지 생성, 네트워크 학습 문서는 있지만 `stage5_branch_relics` placeholder, 확률 가중치 근거, 캐릭터·보스 수치가 비어 있어 테스트 체크리스트로 바로 쓰기에는 부족하다.

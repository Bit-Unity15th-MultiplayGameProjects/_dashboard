# Rougelike-EndWalker — 멀티플레이 루프 안정화 스냅샷

## 1. 주요 변경사항
- RoomService를 방 생성 시점에 즉시 만들고, 로비 진입 후 `C_RequestLobbySync`로 선택·스폰 상태를 다시 받는 구조로 바뀌었다.
- `NodeSelectSceneController`가 추가되어 NodeSelect의 노드 선택, 추천, 서버 확정, Game 씬 전환 책임을 UIManager에서 분리했다.
- `S_NodeEntered.Column`, `C_BattleClear`, `S_BattleCleared`, `C_RequestLobbySync`가 추가됐고 서버 `S_GameStarted` 직렬화 불일치도 수정됐다.
- 멀티 노드맵 결정론을 위해 공유 seed와 `SortedDictionary` 기반 선택 순서 고정이 들어갔다.
- PR 템플릿과 네트워크 오류 문서에 2P smoke test가 추가되어 수동 회귀 확인 기준은 좋아졌다.

## 2. 코드 품질 리뷰
이번 변경은 방향 자체는 좋다. 특히 UIManager가 씬 additive 로드·언로드를 맡고, NodeSelectSceneController가 노드/패킷 흐름을 맡도록 나눈 점은 이전보다 디버깅 지점이 명확하다. 로비 패킷 유실을 `C_RequestLobbySync`로 보정한 것도 “구독 전 패킷 소비”라는 실제 Unity 씬 타이밍 문제를 정확히 잡은 결정이다.

다만 서버 권위 경계가 아직 위험하다. `RoomService.HandleBattleClear`는 현재 클라이언트가 보낸 `Column`을 거의 그대로 신뢰한다. 왜 문제냐면 버그가 난 클라이언트나 악의적 클라이언트가 배틀 미진입 상태에서도 클리어를 발생시키고 NetworkObject despawn을 유발할 수 있기 때문이다. 서버가 active node, active column, entered client 여부를 저장하고 `C_BattleClear`를 그 상태와 대조해야 한다. 또한 `NodeSelectSceneController.HandleNodeEntered`가 게스트에서 `S_NodeEntered` 수신 즉시 ClearTracker를 갱신하는 것은 의미가 틀렸다. 입장 확정과 클리어 확정은 다른 사건이며, 전투 실패나 중단 시 진행 상태가 앞서갈 수 있다.

자료구조/복잡도 관점에서는 `CreateNodesBlueprintProvider`의 `Dictionary`를 `SortedDictionary`로 바꾼 선택이 적절하다. 비용 최적화보다 결정론 보장이 더 중요한 경로이며, 같은 seed에서도 순회 순서가 달라지는 문제를 직접 줄인다. `SpawnedObjects.Keys.ToList()`는 컬렉션 수정 중 순회를 피하기 위한 클리어 시점 스냅샷이라 hot path 최적화 대상으로 보지 않는다.

## 3. 진행도 평가
구현 경로는 명확히 `custom-network`다. 서버 프로세스, 패킷 ID, 직렬화, 룸 상태, NetworkObject 스폰까지 직접 만들고 있으므로 같은 컨텐츠 분량의 framework 프로젝트보다 네트워크 수직 슬라이스 진척은 더 크게 인정한다. 이번 범위에서 로비 선택, NodeSelect 동기화, Game 씬 진입, 배틀 클리어 복귀까지 한 줄로 연결하려는 전진이 있다.

그럼에도 진행도는 58%로 본다. 이유는 2인 happy path가 문서상으로는 매우 구체화됐지만 자동 PlayMode 검증이 없고, 서버가 클리어 권한을 충분히 검증하지 않으며, disconnect와 room 상태 동기화가 아직 불안정하기 때문이다. 시연은 가능권에 들어왔지만 반복 시연 안정성은 아직 높다고 보기 어렵다.

## 4. 다음 권장사항
- `C_BattleClear`는 서버가 저장한 active battle 상태, entered client, active column과 대조한 뒤 처리해야 한다.
- 게스트 ClearTracker는 `S_NodeEntered`가 아니라 `S_BattleCleared`에서만 갱신하도록 옮겨야 한다.
- 2인 로비, NodeSelect, Game 진입, 클리어 복귀를 PlayMode 테스트 1개로 고정해야 한다.
- 마스터 위임 후 시작 버튼과 캐릭터 잠금 UI는 서버 snapshot 기준으로 회귀 검증해야 한다.
- 패킷 header version, count 상한, 호환 실패 응답을 PacketIO/Packet 계약에 넣어야 한다.
- RoomService의 disconnect와 상태 딕셔너리 접근은 lock 또는 단일 처리 루프로 통일해야 한다.

## 5. 문서화 상태
design은 7점이다. Concept와 GDD에 코어 루프, 멀티 엔딩, 주요 시스템 의도는 잡혀 있다. 다만 M3~M5 일정이 미정이고, 핵심 인물의 플레이 수치가 부족해 scope 관리 문서로는 아직 약하다.

technical은 7점이다. TDD, Client Network Guide, Server Architecture, networkerror 문서는 신규 유입자가 흐름을 따라갈 만큼 양이 있다. 그러나 패킷 버전, 실패 응답, 호환성 규칙이 코드 계약과 함께 고정되지 않아 운영 가능한 기술 사양으로는 빈틈이 남는다.

spec은 5점이다. Stage, Story Branch, Visualizer 문서가 존재하지만 `stage5_branch_relics` placeholder, 확률 노드 가중치 근거, BO 쿨다운 수치, 캐릭터·보스 밸런싱 표가 부족하다. 테스트 체크리스트로 바로 쓰기에는 아직 추상도가 높다.

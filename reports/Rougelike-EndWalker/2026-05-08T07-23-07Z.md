# Rougelike-EndWalker — TCP 서버 도입 스냅샷

## 1. 주요 변경사항
- Unity 프로젝트가 `Client/` 아래로 이동했고, 별도 `Server/` 솔루션이 추가됐다. 클라이언트·서버 분리 방향은 좋다.
- TCP 기반 세션, 로비, 방 이관, RoomService, 패킷 직렬화 계층이 새로 작성됐다.
- NetworkObject/NetworkVariable 구조를 서버에 미러링하려는 시도가 들어갔다. 의도 자체는 명확하다.
- 서버 아키텍처와 NetworkObject 학습 문서가 추가되어 신규 진입자가 흐름을 따라가기 쉬워졌다.
- 반대로 `.vs`, `bin`, `obj` 산출물이 커밋됐고, 루트 ignore 체계가 깨진 흔적이 있다.

## 2. 코드 품질 리뷰
가장 위험한 문제는 `Packet.cs`의 `LobbyInfo.Serialize()`가 `PlayerList.Count`를 기록하지 않는데 `Deserialize()`는 count를 먼저 읽는다는 점이다. 이 패킷은 방 생성, 참가, 게임 시작 흐름의 핵심 데이터라서 실제 클라이언트가 붙는 순간 스트림이 어긋난다. 이는 단순 실수가 아니라 프로토콜 신뢰성을 깨는 버그다. 모든 패킷은 serialize/deserialize 왕복 테스트를 먼저 고정해야 한다.

네트워크 권한 모델도 아직 위험하다. `NetworkObjectManager.ApplyVariableUpdate()`는 보낸 클라이언트가 해당 오브젝트의 owner인지, 해당 variable을 클라이언트가 수정해도 되는지 검증하지 않는다. 이 구조에서는 클라이언트가 다른 플레이어의 위치나 HP 계열 상태까지 조작할 여지가 생긴다. 서버 권위 게임으로 갈 생각이면 owner 검증과 variable allowlist가 최소 기준이다.

자료구조/복잡도/할당 관점에서는 네트워크 sync hot path에서 불필요한 복사가 반복된다. `FlushDirtyObjects()`와 `ApplyVariableUpdate()`는 `ToArray().ToList()`를 통해 payload를 매번 복제하고, `Packet.cs`는 byte payload를 한 바이트씩 쓴다. 현재 구조의 비용은 30ms 루프마다 복사·할당·반복 write가 누적되는 것이다. `List<byte>` 대신 `byte[]` 기반 payload로 고정하고 `BinaryWriter.Write(byte[])` 흐름으로 바꾸는 편이 이 프로젝트의 실시간 동기화에 더 맞다.

## 3. 진행도 평가
이번 변경은 양적으로는 크고 방향도 전진이다. 특히 클라이언트와 서버를 분리하고 TCP 서버 골격을 세운 점, 아키텍처 문서를 추가한 점은 좋은 변화다. 다만 현재 서버는 “붙여서 디버깅 가능한 프로토타입”보다 “설계 실험 코드”에 가깝다.

진행도는 41%로 본다. 핵심 루프와 데이터는 존재하지만, 멀티플레이 안정성·권한·테스트·빌드 위생이 동시에 비어 있다. 지금 단계에서 직렬화와 권한 검증을 미루면 이후 버그 재현 비용이 급격히 커진다.

## 4. 다음 권장사항
- `LobbyInfo`를 포함한 모든 패킷의 serialize/deserialize 왕복 테스트를 추가한다.
- `C_NetworkVarUpdate`에 owner 검증과 variable별 수정 가능 정책을 넣는다.
- `FlushLoop`와 패킷 처리 루프가 공유 상태를 동시에 만지지 않도록 구조를 정리한다.
- 루트 `.gitignore`를 복구하고 `.vs`, `bin`, `obj` 산출물을 repo에서 제거한다.
- `NetworkVariableDelta.Payload`를 `byte[]`로 바꾸고 반복 복사를 줄인다.

## 5. 문서화 상태
design은 6점이다. 컨셉/GDD 문서는 유지되고 있으나 사서, 기록자, 최종 보스 같은 핵심 인물의 수치와 플레이상 역할이 아직 사양으로 닫히지 않았다.

technical은 7점이다. `Server_Architecture.md`, `NetworkObject_Architecture.md`는 이번 변경의 가장 좋은 산출물이다. 다만 권한 모델, 실패 케이스, 재접속, 패킷 호환성 규칙이 문서 기준으로는 아직 부족하다.

spec은 4점이다. 데이터 파일은 있지만 밸런싱 근거, 검증 절차, 패킷 스키마 테스트 표가 부족하다. `_sample/docs` 수준의 사양서처럼 구현·QA 기준으로 바로 쓰기에는 아직 멀다.

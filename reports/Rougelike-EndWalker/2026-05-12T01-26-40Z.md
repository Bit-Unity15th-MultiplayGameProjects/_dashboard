# Rougelike-EndWalker — 로비 네트워크 안정화 스냅샷

## 1. 주요 변경사항
- `LobbySceneController`에 자신의 선택 prefab index를 저장하고, 방이 찼을 때 `C_SelectCharacter`를 재전송하는 우회 처리가 추가됐다.
- `RoomService._masterClientId`가 변경 가능해졌고, 기존 마스터 이탈 시 새 마스터를 서버 내부 상태에도 반영한다.
- 접속 해제 시 로비 스폰 오브젝트, 캐릭터 선택, 씬 진입 상태를 정리하도록 서버 상태 누수를 줄였다.
- TDD와 네트워크 가이드가 로비 프리뷰 스폰, Game 씬 진입 후 `C_EnterNode`, 중복 스폰 방어 흐름까지 갱신됐다.
- 버그 수정 기록 문서가 실제 원인-수정-흐름 중심으로 보강된 점은 명확한 개선이다.

## 2. 코드 품질 리뷰
이번 변경은 실제 멀티플레이 happy path를 막던 문제를 정확히 찌르고 있다. 특히 마스터 이탈 후 `_masterClientId`가 갱신되지 않던 문제는 서버 권한 모델을 깨는 명백한 결함이었고, 이를 서버 필드 갱신까지 연결한 것은 올바른 방향이다. 퇴장 플레이어의 `_characterSelections`, `_lobbySpawnedObjects`, `_enteredClients`를 정리한 것도 재접속과 씬 재진입 버그를 줄이는 실질적 개선이다.

다만 마스터의 사전 캐릭터 선택 문제를 클라이언트 재전송으로 푼 결정은 위험하다. 왜 문제냐면 서버가 패킷을 드롭하는 구조적 원인을 남겨둔 채, `S_PlayerJoined` 수신 타이밍과 `PlayerCount >= MaxPlayers` 조건에 기대기 때문이다. 개선하려면 `LobbyService` 단계에서 선택 상태를 보존해 `RoomService` 생성 시 넘기거나, 방 생성 시점부터 `RoomService`가 선택 패킷을 받을 수 있게 수명주기를 정리하는 편이 낫다.

자료구조/복잡도 관점에서는 이번 diff에 `Update` hot path의 LINQ나 반복 할당 증가는 보이지 않는다. 다만 `RoomService`는 `ConcurrentDictionary`와 일반 `Dictionary`/`HashSet`을 동시에 수정한다. 현재 비용 문제라기보다 스레드 안전성 문제다. disconnect, packet handler, broadcast가 서로 다른 스레드에서 들어올 수 있다면 일반 컬렉션 정리가 경합 지점이 되므로 lock 경계나 단일 room event queue로 통일해야 한다.

## 3. 진행도 평가
이 프로젝트는 서버, 패킷, 룸, 오브젝트 동기화를 직접 구현하는 `custom-network` 경로다. 고수준 프레임워크 프로젝트보다 컨텐츠 양은 적게 보더라도, 2인 로비-선택-시작-씬 진입-스폰이 이어지는 경로가 실제로 연결되는 점은 큰 진전이다. 진행도는 58로 본다.

위험도는 여전히 high다. 이유는 기능이 전진하지 않아서가 아니라, 핵심 네트워크 결함을 발견할 때마다 코드와 문서로 수동 보정하는 흐름이 반복되고 있기 때문이다. 자동 테스트, 재현 체크리스트, 패킷 계약 문서가 없으면 다음 수정이 기존 happy path를 다시 깨뜨릴 가능성이 높다.

## 4. 다음 권장사항
- 마스터 사전 선택 재전송을 서버 상태 보존 방식으로 재설계해야 한다. 클라이언트 타이밍 의존은 데모 직전 불안정성을 만든다.
- 마스터 위임 후 시작 버튼, 테마 선택, UI 표시 상태까지 한 번의 시나리오로 검증해야 한다.
- 2인 로비 시나리오 체크리스트를 문서화하고 매 커밋마다 같은 순서로 재현해야 한다.
- `RoomService`의 disconnect 처리와 packet 처리 사이 스레드 경계를 명확히 해야 한다.
- `PacketId`별 payload, 버전, 호환성 실패 처리 규칙을 별도 표로 고정해야 한다.

## 5. 문서화 상태
design은 6점이다. 로그라이트 배치 철학과 플레이 경험 설명은 일부 보강됐지만, 핵심 인물·보스·밸런스 의도는 아직 사양으로 고정되지 않았다.

technical은 7점이다. 네트워크 흐름, 로비 프리뷰 스폰, `C_EnterNode`, 중복 ObjectId 방어까지 최신 구현이 반영됐다. 신규 합류자가 전체 흐름을 따라가기에는 좋아졌으나, 패킷 스키마와 실패 처리 규칙은 아직 부족하다.

spec은 5점이다. 버그 수정 기록은 상세하지만 테스트 체크리스트와 밸런싱 수치 표가 사양서 수준으로 닫히지 않았다. `_sample/docs` 기준의 “검증 가능한 표”에는 아직 못 미친다.

## 6. Backlog
- 테스트 코드·플레이모드 검증 절차 부재는 계속 남아 있다. 이번 변경은 네트워크 회귀 위험이 커서 우선순위가 더 높다.
- 패킷 프로토콜 스키마와 버전 호환성 규칙 부재도 미해결이다. 문서에 흐름은 늘었지만 계약은 아직 없다.
- 문서 위치 이동 후 기존 참조 경로 검증 부재는 이번 diff에서 해결 근거가 없다.
- 확률 노드 가중치와 BO 쿨다운 2 스테이지의 근거 수치도 여전히 확인되지 않는다.
- 핵심 인물 특성, README placeholder, 작성자 표기 혼용은 이번 커밋 범위에서 수정되지 않았다.

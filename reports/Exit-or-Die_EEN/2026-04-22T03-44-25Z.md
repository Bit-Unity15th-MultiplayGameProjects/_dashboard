# Exit-or-Die_EEN — 프로젝트 부트스트랩 스냅샷

## 1. 주요 변경사항
- Unity URP Blank 템플릿으로 프로젝트 초기 생성 (Unity Editor 산출물 일괄 커밋, 7,501 라인 추가).
- `Assets/InputSystem_Actions.inputactions` 가 템플릿 기본값으로 포함됨 (Player/UI 두 액션맵, Move·Look·Attack·Interact·Crouch·Jump·Sprint 등).
- `Assets/Settings/` 하위에 PC/Mobile 두 종 RP Asset, Renderer, Volume Profile 자동 생성물 커밋.
- `Assets/TutorialInfo/` 의 URP 튜토리얼 README 에셋과 `ReadmeEditor.cs`, `Readme.cs` 가 정리되지 않은 채 그대로 남아 있음.
- `docs/` 폴더에 기획서·컨셉·기술설계서 3종 마크다운 신규 작성 후 다회 갱신 (총 253 라인 규모).
- `README.md` 가 6회 업데이트되었으나 변경 내용은 git log 상으로는 알 수 없음 (메시지 일괄 "Update README.md").
- `ProjectSettings/ProjectSettings.asset` 의 `productName: Exit-or-Die`, `companyName: DefaultCompany` 로 식별자 미정리.

## 2. 코드 품질 리뷰
현 시점에서 **평가 가능한 게임 코드가 존재하지 않는다**. `Assets/` 하위 C# 파일은 URP 템플릿이 동봉한 `TutorialInfo/Scripts/Readme.cs` 와 `Editor/ReadmeEditor.cs` 두 개뿐이며, 둘 다 팀이 작성한 것이 아니다. 따라서 가독성·구조·SRP·매직 넘버 같은 일반 코드 품질 축은 이번 리뷰에서 판정 보류한다. 다만 그 사실 자체가 곧 위험 신호다. 첫 커밋(`8537055`)으로부터 이번 스냅샷까지 23 커밋이 누적됐는데 그중 22 커밋이 `README.md`/`docs/*.md` 갱신이고 1 커밋이 Unity 프로젝트 자동생성물이다. 즉 **실제 구현 사이클이 아직 한 번도 돌지 않았다**.

Unity/C# 모범사례 측면에서는 긍정적인 출발점이 두 가지 있다. 첫째, 새 Input System 기반 Action Asset 이 사전에 준비되어 있어 향후 `Input.GetAxis` 같은 구식 API 잔존 위험이 낮다. 둘째, URP + PC/Mobile 렌더러 분리 설정이 템플릿에 의해 미리 들어와 있어 그래픽 파이프라인 결정을 다시 할 필요가 없다. 다만 **InputSystem_Actions 가 템플릿 기본값 그대로**라는 점에 주의해야 한다. 'Attack'/'Sprint'/'Crouch' 같은 액션이 게임 컨셉(탈출 호러 추정)과 맞는지 검증 없이 남아 있고, 코드와 연결되는 `.inputactions.cs` 자동 생성 옵션 활성화 흔적도 보이지 않는다. 액션을 실제로 쓰기 전에 액션 목록을 게임에 맞게 정리하고 C# Wrapper 생성을 켜야 한다.

위생 측면에서 두 가지 부채가 즉시 보인다. 하나는 `Assets/TutorialInfo/` 가 통째로 남아 있는 점 — Unity 가 첫 실행 시 자동으로 띄우는 안내 에셋이라 학습용 외에는 가치가 없고, 빌드 사이즈와 리뷰 노이즈를 키운다. 다른 하나는 `ProjectSettings.asset` 의 `applicationIdentifier` 가 `com.UnityTechnologies.com.unity.template.urpblank` 인 채로 남아 있다는 점 — 모바일/스토어 배포 단계에서 반드시 충돌하므로 지금 정리하는 비용이 가장 싸다. 커밋 단위로도 `Update X.md` 형식이 반복돼 변경 의도가 드러나지 않는다. 이 단계에서 메시지 컨벤션(예: `docs(컨셉): 코어루프 추가`, `feat(player): 이동 입력 처리`)을 잡지 않으면 코드가 들어오기 시작한 뒤에 history 가 역추적 불가능해진다.

네트워킹 코드는 아직 없으므로 평가 기준 3 (멀티플레이/네트워킹) 은 **해당 없음**. 다만 organization 명칭이 `MultiplayGameProjects` 인 만큼 본 프로젝트의 멀티플레이 의도 여부를 기획서/기술설계서가 명시해야 하며, 현재 문서에는 그 결정 자체가 빠져 있다(기술설계서가 비어 있어 확인 불가).

## 3. 진행도 평가
첫 리포트라 비교 기준은 없다. 절대치로는 **스캐폴드 단계 초입**으로 본다 (`progress_estimate: 8`). 근거는 ① 게임 로직 코드 0줄, ② SampleScene 이 URP 템플릿 기본 씬 그대로, ③ 기획 문서 3종이 신규 작성되었으나 사양서 수준에는 못 미친다는 점. 23 커밋 중 코드 변경이 0건이라는 비율은 정상적인 게임 프로젝트 사이클에서 비정상적이다. 학습/문서 정리 단계로는 자연스럽지만 "빌드 가능한 게임"까지 거리가 멀다는 사실은 명확히 인지해야 한다.

남은 기간(과제 마감일 미상이므로 일반적인 학기 프로젝트 가정)을 고려하면 다음 2주 안에 **플레이어 이동 + 상호작용 + 최소 1개 씬 구성**까지 가야 코드 리뷰 가능한 단계로 진입한다. 지금 시점에서 가장 큰 risk 는 기술 부채가 아니라 **시작 지연**이다. 그래서 risk_level 은 `medium` 으로 둔다 — 부채는 작지만, 코드 진입이 늦어질수록 medium 이 high 로 빠르게 전이될 수 있다.

## 5. 문서화 상태
**design (4/10)**: `docs/컨셉.md` 와 `docs/기획서.md` 가 신규 작성되어 출발점은 마련되었다. 8회 갱신된 컨셉 문서는 적어도 진지하게 다루고 있다는 신호다. 다만 본 리뷰는 diff 의 마크다운 본문 내용이 첨부 범위에서 잘려 직접 검증할 수 없었으며, 파일 크기(컨셉 91 라인, 기획서 97 라인)로 추정컨대 코어 루프·승패 조건·플레이어 경험 목표가 결정된 수준은 아니라고 판단한다. "왜 이 선택인가"가 드러나는 의사결정 문서로 다듬는 작업이 필요하다.

**technical (3/10)**: `docs/기술설계서.md` 가 65 라인 규모로 가장 얇다. 이 분량으로 아키텍처 개요·모듈 분리·데이터 흐름·빌드 절차를 모두 담기는 어렵다. 신규 합류자가 이 문서만 보고 코드 구조를 짐작할 수 없을 가능성이 높고, 멀티플레이 도입 여부 같은 핵심 결정이 빠져 있을 것으로 보인다. 다이어그램 1장(플레이어→인풋→매니저→씬)부터 시작해 살을 붙여야 한다.

**spec (2/10)**: 사양서로 분류할 만한 문서가 없다. `_sample/docs` 의 레시피 사양서·레벨 디자인 사양서 수준의 "테스트 체크리스트로 쓸 수 있는" 표·수치는 현재 repo 어디에도 없다. 게임 장르가 정해지면(탈출/호러/퍼즐 추정) 레벨 사양서 1개부터 시작해 적/아이템/상호작용 객체 스펙으로 확장해야 한다.

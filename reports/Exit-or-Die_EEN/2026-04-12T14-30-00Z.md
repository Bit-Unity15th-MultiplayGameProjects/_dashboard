# Exit-or-Die_EEN — 2주차 리뷰

## 1. 주요 변경사항
- `EnemyFSM` 도입: `Idle → Patrol → Chase → Attack` 4상태 전환.
- 스테이지 프리팹 시스템 구축 (`Stage_01`, `Stage_02` 템플릿화).
- `PlayerController`가 Input System으로 전환됨 — 지난 리뷰 권장사항 반영.
- UI: HP 바, 미니맵 스켈레톤 작성.
- `GameManager` 싱글톤 신설 — 씬 전환과 스코어 관리 담당.

## 2. 코드 품질 리뷰
FSM 전환은 잘 이루어졌지만, `EnemyFSM`의 `Update()` 분기문이 이미 100줄을 넘겼다.
지금 상태 클래스별로 분리(Strategy 패턴 혹은 `IState` 인터페이스)하지 않으면
상태가 추가될 때마다 기하급수적으로 복잡해진다.

`GameManager` 싱글톤은 정적 참조가 여러 스크립트에 퍼져있다. 테스트 가능성과
씬 재로드 시의 라이프사이클 이슈가 걱정된다. `DontDestroyOnLoad` 의존을
줄이는 방향을 검토할 것.

매직 스트링 (`"Enemy"`, `"Player"` 태그, `"Attack"` 트리거)이 곳곳에 산재.
상수 클래스 또는 `ScriptableObject` 카탈로그로 모아둘 시점.

## 3. 진행도 평가
계획 대비 살짝 앞서있다. 다만 설계 부채가 쌓이기 시작했다. 지금 1일 리팩터에
쓰면 이후 2주를 아낄 수 있는 수준.

## 4. 다음 권장사항
- `EnemyFSM`을 `IEnemyState` 인터페이스 기반으로 분리 (우선순위 높음).
- 매직 스트링 → `GameConstants.cs` 또는 `TagSO` 구조로 이동.
- 플레이 테스트 1회 진행 — 현재 난이도 감각 파악 필요.
- 브랜치 전략 정리: 현재 `main` 직접 푸시가 많다. feature 브랜치 권장.

# Exit-or-Die_EEN — 1주차 문서 정비 스냅샷

## 1. 주요 변경사항
- README.md 전면 개편: Unity 6000.0.38f1 / URP / NGO / Addressables 명시, 빌드 절차·브랜치 전략·커밋 컨벤션·Assets 구조 추가.
- 기술설계서 v1.3: Mermaid 상태 다이어그램, 씬(Initial/Lobby/GameSession)·매니저(Global/Inventory) 구조, RPC 명세(`RequestPickupServerRpc`, `NotifyResultClientRpc`) 추가.
- GDD v1.3: 코어 루프를 1분 단위 시간축(0–10s/10–40s/40–60s)으로 재구성, 적 NPC HP·이속·공격력 수치 테이블 신설, 승/패 조건 명문화.
- 컨셉 문서 v1.3: 핵심 가치(High Risk High Return) 중심으로 재구성, 코어 루프와 패배 시 `CurrentInventory.json` 초기화 규칙 명시.
- 코드/씬/프로젝트 설정 변경 0건 — 4개 문서 파일만 수정.

## 2. 코드 품질 리뷰
이번 커밋 범위는 `.md` 4개에 국한되어 평가할 C# 코드가 한 줄도 없다. 이는 단순한 "코드 리뷰 생략" 문제가 아니라 **프로젝트 자체의 위험 신호**다. README가 `Assets/Scripts/`와 `InitialScene.unity`를 빌드 진입점으로 명시했지만, repo 어디에도 해당 산출물이 존재하지 않는다. 즉, 문서가 가공의 코드베이스를 묘사하고 있는 상태로, 신규 합류자가 README 절차를 따르면 4단계(`InitialScene.unity` 오픈)에서 즉시 막힌다.

기술설계서에는 `GlobalManager`(DontDestroyOnLoad 싱글톤), `InventoryManager`, `SessionManager` 같은 핵심 객체가 명시됐으나 — 이 단계에서 가장 비용이 낮은 작업은 **빈 `MonoBehaviour` 클래스라도 먼저 커밋**해 문서-코드 격차를 좁히는 것이다. 또한 RPC 시그니처는 `RequestPickupServerRpc` / `NotifyResultClientRpc` 두 개만 의사코드로 적혀 있는데, 권한 모델("호스트가 모든 대미지 판정")을 구현 단계에서 일관되게 강제하려면 NetworkVariable 의 `Read/WritePermission` 매트릭스가 표로 정리되어야 한다. 지금은 "Host-Authoritative" 라는 레이블만 있고 어떤 상태가 누구 소유인지 코드 기준이 없다.

긍정적 변화로, README 의 커밋 컨벤션(feat/fix/docs/chore/refactor) 정의와 브랜치 전략(`main`/`develop`/`feature/*`)이 합의 가능한 수준으로 적혔다. 다만 같은 PR 의 5개 커밋이 전부 `Update X.md` 형식으로 들어왔다는 점은 **컨벤션을 적은 직후 본인이 위반한** 형태라 신뢰도를 떨어뜨린다.

## 3. 진행도 평가
이전 리포트(2시간 전)와 동일한 날짜의 후속 작업이라 코드 진척은 사실상 0이지만, **문서 품질은 의미 있는 도약**을 했다. 특히 NGO 권한 모델·씬 구조·적 수치표가 들어오면서 technical 축이 placeholder 단계를 벗어났다. 그러나 4주 일정(5/14 마감)의 1주차 종료 시점에 "네트워크 및 로비 기초 구축" 마일스톤이 있는데 코드가 0줄이라는 것은 **2주차 마일스톤("좀비 AI, RPC, 인벤토리 로직")이 사실상 불가능**해진다는 뜻이다. 진도율을 18%로 잡은 것은 문서만의 점수이며, 실행 리스크는 medium이 아닌 **high**로 판정한다.

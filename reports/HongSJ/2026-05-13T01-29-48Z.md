# HongSJ — README 정비 스냅샷

## 1. 주요 변경사항
- `README.md`가 새로 추가되어 게임 컨셉, 기술 스택, 씬 흐름, 핵심 시스템, 로컬 실행 방법이 한곳에 정리됐다.
- Server Authority, `CMD_`, `RPC_`, `SyncVar` 사용 원칙과 클라이언트 임시 배치 상태가 문서에 명시됐다.
- WebGL, SimpleWebTransport, AWS EC2, Nginx, WSS 배포 구성이 추가되어 실행 환경 이해도가 올라갔다.
- 이번 커밋의 실제 변경은 문서 1개뿐이다. GameSession, MatchmakingQueue, Scene 전환 코드의 기존 위험은 아직 코드로 해소되지 않았다.

## 2. 코드 품질 리뷰
이번 변경은 README 추가라서 코드 구조 자체를 악화시키지는 않았다. 이 점은 명확한 전진이다. 특히 서버 권위, 클라이언트 임시 상태, 씬 흐름을 README에 적은 것은 신규 참여자가 “무엇이 서버 책임인가”를 추측하지 않게 만든다는 점에서 좋다. 다만 README가 말하는 서버 검증과 실제 `SubmitTurnMessage` 처리 경로가 일치하는지는 아직 코드 변경으로 증명되지 않았다. 문서가 구현보다 앞서가면 더 위험하다. 리뷰 기준은 “문서에 적힌 계약이 코드와 테스트로 강제되는가”로 바꿔야 한다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 새로 생긴 명확한 성능 지적은 없다. 기존 `CardRegistry`는 `Dictionary<int, CardData>` 조회 자체는 적절하지만, `static` 전역 상태와 `Resources.LoadAll<CardData>("Cards")` 초기화가 결합되어 테스트 격리와 런타임 카탈로그 교체가 어렵다. 해결 방향은 조회 자료구조를 바꾸는 것이 아니라, 안정적인 카드 ID 카탈로그를 명시 데이터로 만들고 부트스트랩에서 한 번 주입하는 것이다.

## 3. 진행도 평가
구현 경로는 Mirror를 쓰지만 드래프트, 매칭, 서버 권위 검증, 반복 아레나 상태 전이를 직접 설계하는 `hybrid`에 가깝다. Mirror가 연결과 RPC 비용을 줄여주므로, 이제부터는 “2인 happy path가 반복 재현되는가”와 “클라이언트 입력을 서버가 검증하는가”가 진행도의 핵심이다.

README 추가로 온보딩과 배포 설명은 크게 좋아졌지만, 이번 범위에는 런타임 안정성을 높이는 코드 수정이 없다. 따라서 진행도는 소폭만 인정한다. 현재는 수직 슬라이스의 윤곽은 있으나, 결과 기록·재매칭·씬 전환·제출 검증이 동시에 흔들릴 수 있어 `high` 위험으로 본다.

## 4. 다음 권장사항
- `GameSession`의 정상 종료, 항복, 연결 해제 결과 기록을 `FinishGameResult` 하나로 모아야 한다.
- `SubmitTurnMessage`에서 카드 소유권, 에너지, 구역 인덱스, 슬롯 제한을 서버에서 최종 검증해야 한다.
- `GoToSceneMessage` 수신 등록은 `SceneRouter` 하나로 모으고, sceneName은 enum 또는 상수 매핑으로 제한해야 한다.
- `MatchmakingQueue`의 반복 매칭은 matchId별 `Pending/Ready` 상태머신으로 고정해야 한다.
- README의 아키텍처 설명을 `docs/technical.md`로 분리하고, 권한·씬·재매칭 계약을 테스트 기준으로 작성해야 한다.

## 5. 문서화 상태
design 점수는 7점이다. `game_design_document.md`와 README가 게임 컨셉, 코어 루프, 씬 흐름, 드래프트와 대전 구조를 설명하고 있어 GDD 성격은 실무적으로 읽을 수 있다.

technical 점수는 5점이다. README에 서버 권위, 인프라, 실행 방법이 들어갔지만 `docs/technical.md`가 없고, 실제 클래스 간 책임·메시지 흐름·재매칭 상태 전이가 문서 계약으로 고정되어 있지 않다.

spec 점수는 4점이다. 승패 판정, 턴 수, 손패, 구역 제한 같은 규칙 요약은 있으나 카드/장소/상태/테스트 체크리스트가 사양서 수준으로 분리되어 있지는 않다.

## 6. Backlog
- 프로젝트 문서 backlog는 부분 해결이다. GDD와 README는 생겼지만 기술 계약과 사양 문서가 없어 신규 유입자가 여전히 코드와 README를 교차 추론해야 한다.
- `CardRegistry`의 `static + Resources.LoadAll` 구조는 남아 있다. 테스트용 카드 풀 주입, 실패 재현, 런타임 교체가 어려운 구조다.
- `cardId` 안정성 문제도 아직 닫기 어렵다. 네트워크 payload에 쓰이는 ID는 에셋 정렬이나 자동 생성 순서와 분리된 명시 매핑으로 관리해야 한다.

## 7. 이전 Backlog 해결

# HongSJ — 무변경 재점검 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개이며 실제 diff도 비어 있다. 이번 기간의 코드 전진은 확인되지 않는다.
- 직전 high TODO인 `GameSession` 결과 기록 중복과 `MatchmakingQueue` 전환 상태 문제는 해결 근거가 없다.
- 현재 문서 스냅샷에는 `game_design_document.md`와 위치 목록 문서가 있으나, 프로젝트 기술 문서와 네트워크 권한 문서는 여전히 보이지 않는다.
- `CardRegistry`의 static `Resources.LoadAll` 의존과 자동 증가 `cardId` 문제도 해결 기록이 없어 그대로 위험 항목으로 유지된다.
- 2인 Draft-Match-Result 반복 전환을 검증하는 체크리스트가 확인되지 않아 회귀 판정 기준이 아직 약하다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 새로 추가된 코드의 가독성·구조 문제를 평가할 대상은 없다. 다만 기존 open item 기준으로 보면 위험은 낮아지지 않았다. `GameSession`의 정상 종료와 항복 처리가 각각 승패 초기화와 메시지 전송을 처리하는 구조라면, 한쪽만 수정되는 순간 결과 화면·점수·네트워크 알림이 서로 어긋난다. 이 로직은 “경기 종료 결과를 서버에서 한 번 확정하고, UI/클라이언트 통지는 그 결과를 구독한다”는 단일 흐름으로 묶는 것이 맞다.

`MatchmakingQueue`도 `ReadyForNext` 이후 같은 연결 상태를 제거했다가 다시 삽입하는 흐름이 남아 있다면 의도가 불명확하다. 왜 문제냐면 씬 전환, 재매칭, 연결 해제 타이밍이 겹칠 때 같은 connection이 어느 match에 속하는지 추적하기 어려워진다. `Waiting`, `Drafting`, `InMatch`, `ReturningResult` 같은 phase와 `matchId`를 명시한 상태머신으로 바꾸는 것이 개선 방향이다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 신규 hot path LINQ, 반복 할당, O(n²) 검색 문제는 확인되지 않는다. 대신 근거 있는 기존 이슈는 `CardRegistry`의 ID 매핑 방식이다. 현재 비용은 성능보다 계약 안정성 문제에 가깝다. `Resources.LoadAll` 결과 순서와 자동 증가 ID에 의존하면 카드 추가·삭제만으로 네트워크 페이로드의 의미가 바뀔 수 있으므로, 명시 ID 또는 GUID 기반 카탈로그로 조회 계약을 고정해야 한다.

## 3. 진행도 평가
이 프로젝트는 Mirror를 쓰지만 매칭, 씬 전환, 카드 draft/배치/점수 권한을 직접 설계하는 부분이 커서 `hybrid` 경로로 보는 것이 타당하다. 프레임워크가 RPC와 동기화 기반을 제공하므로, 제출 가능한 수직 슬라이스 기준에서는 2인 반복 플레이와 상태 복구 안정성이 더 엄격하게 요구된다.

현재는 draft-match-result 축의 골격은 있는 것으로 보이나, 커밋이 없고 핵심 TODO가 유지되므로 진행도는 54로 본다. 플레이 루프가 존재하더라도 결과 확정, 재전환, 권한 문서가 흔들리면 시연 중 재현 어려운 버그가 발생할 가능성이 높다.

## 4. 다음 권장사항
- `GameSession`의 종료·항복 처리를 단일 서버 함수로 통합한다.
- `MatchmakingQueue`의 ReadyForNext/Pending 흐름을 `matchId` 기반 상태머신으로 고정한다.
- `docs/technical.md`에 카드 draft, 배치, 점수, 씬 전환의 서버 권위 계약을 먼저 쓴다.
- `GoToSceneMessage.sceneName`은 문자열 자유 입력이 아니라 enum 또는 상수 매핑으로 제한한다.
- 2인 Draft-Match-Result 반복 전환 체크리스트를 호스트·클라이언트 순서별로 작성한다.
- `CardRegistry`의 자동 순번 ID를 명시 ID 또는 GUID 매핑으로 전환한다.

## 5. 문서화 상태
design 점수는 6이다. `game_design_document.md`가 18KB 규모로 존재하므로 컨셉과 코어 루프 문서의 기반은 인정된다. 다만 `_sample/docs`처럼 시스템 의도, 의사결정 근거, 변경 이력까지 체계화된 수준으로 보기는 어렵다.

technical 점수는 2이다. Mirror 패키지 README들은 외부 라이브러리 설명이지 HongSJ의 아키텍처 문서가 아니다. 특히 `docs/technical.md`와 서버 권위, 씬 생명주기, 재접속 흐름 문서가 확인되지 않는 점은 네트워크 프로젝트에서 직접적인 위험이다.

spec 점수는 4이다. `marvel_snap_locations_all.md`는 위치 데이터 후보나 콘텐츠 목록으로는 의미가 있으나, 카드 ID 계약, 씬별 상태, 입력·결과 매핑, QA 체크리스트로 바로 쓰기에는 부족하다. 사양서는 “구현자가 그대로 검증할 수 있는 표”까지 내려가야 한다.

## 6. Backlog
- 프로젝트 문서 backlog는 부분만 해소됐다. GDD는 있으나 기술 문서와 사양서가 분리되지 않아 신규 유입자는 여전히 코드에서 설계 의도를 역추적해야 한다.
- Mirror 권한 모델은 계속 high risk다. 카드 draft, 배치, 점수 계산, 씬 전환의 주인이 서버인지 클라이언트인지 문서와 코드에서 동시에 드러나야 한다.
- `CardRegistry`의 static `Resources.LoadAll` 의존은 테스트 격리와 런타임 교체를 어렵게 만든다. 장기적으로 ScriptableObject 카탈로그 또는 DI 가능한 registry가 필요하다.
- 자동 증가 `cardId`는 네트워크와 저장 계약에 직접 닿는 부채다. 카드 에셋 정렬 변경만으로 기존 ID가 재할당될 수 있으므로 명시 키로 고정해야 한다.

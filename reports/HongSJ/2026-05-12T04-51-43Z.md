# HongSJ — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 범위가 같은 해시 하나를 가리키고 커밋 수가 0이므로, 이번 스냅샷에는 코드 diff 기준의 전진이 없다.
- 현재 문서 스냅샷에는 `game_design_document.md`가 존재하지만 `docs/technical.md`는 없다. 기술·사양 문서 축은 여전히 비어 있다.
- Mirror 기반 코드와 `GameSession`, `Player`, `MatchmakingQueue` 관련 open item은 해결 근거가 없다. 네트워크 권한과 씬 전환 부채가 그대로 남았다.
- `CardRegistry`는 여전히 static + `Resources.LoadAll` 패턴 증거가 확인된다. 조회 비용보다 테스트 격리와 안정 ID 계약이 핵심 위험이다.

## 2. 코드 품질 리뷰
이번 범위는 실제 변경이 없으므로 새로 들어온 버그를 지적할 수는 없다. 하지만 이전에 지적된 핵심 구조 문제가 해결됐다는 증거도 없다. `GameSession`의 종료 처리와 항복 처리 결과 기록이 별도 흐름으로 남아 있으면, 승패 계산·보상·결과 씬 전환 중 하나만 수정되는 회귀가 쉽게 발생한다. 개선 방향은 결과 기록을 `RecordMatchResult` 같은 서버 전용 함수 하나로 모으고, 항복과 정상 종료가 같은 경로를 타게 만드는 것이다.

네트워킹 측면에서는 위험 신호가 크다. Mirror를 쓰더라도 카드 선택, 배치, 점수 계산, 씬 이동의 최종 권한이 서버에 있다는 계약이 코드 주석이나 기술 문서에 명시되어 있지 않다. 이 상태에서는 클라이언트 Command 검증 누락이 곧 점수 조작이나 재매칭 꼬임으로 이어진다. `Player` 입력은 소유권, 에너지, 존 범위, 턴 상태를 서버에서 재검증해야 한다.

자료구조/복잡도/할당 관점에서는 `CardRegistry`의 `Dictionary<int, CardData>` 조회 자체는 O(1)이므로 lookup 비용이 문제는 아니다. 문제는 초기화가 static 전역 상태와 `Resources.LoadAll("Cards")`에 묶여 테스트별 카탈로그 교체가 어렵고, cardId 안정성이 에셋 생성·정렬 흐름에 의존할 가능성이 있다는 점이다. 이번 diff 범위에서는 매 프레임 LINQ, 반복 할당, O(n²) hot path에 대한 신규 근거는 없다.

## 3. 진행도 평가
구현 경로는 `hybrid`로 보는 것이 맞다. Mirror가 연결, RPC, SyncVar 기반을 맡지만 매칭 큐, 게임 세션 결과, 씬 전환, 권한 검증 규칙은 프로젝트 코드가 직접 설계해야 한다. 따라서 단순 framework 프로젝트보다 네트워크 설계 부담이 크고, 현재 미문서화 상태는 디버깅 비용을 크게 키운다.

진행도는 52로 평가한다. 낮은 수준의 수직 슬라이스는 존재하는 것으로 보이나, 2인 Draft-Match-Result 반복 전환과 서버 권위 검증이 고정되지 않았다. 커밋 수 0인 이번 스냅샷에서는 이전 리스크가 줄었다고 볼 근거가 없다.

## 4. 다음 권장사항
- GameSession 승패·항복 기록을 단일 서버 결과 함수로 통합한다. 먼저 중복 결과 기록 경로를 없애야 후속 씬 전환 디버깅이 가능하다.
- MatchmakingQueue 재매칭을 matchId별 Pending/Ready 상태머신으로 고정한다. 전역 슬롯 재삽입 흐름은 반복 플레이에서 가장 먼저 깨질 가능성이 높다.
- docs/technical.md에 Mirror 서버 권위와 씬 전환 계약을 작성한다. 코드보다 먼저 판정 기준을 세워야 한다.
- Player Command 입력에서 카드 소유권·에너지·존 범위를 서버 검증한다. 클라이언트 입력은 요청일 뿐 결과 권위가 아니다.
- CardRegistry cardId 발급을 기존 값 보존형 명시 ID/GUID 매핑으로 전환한다.
- GoToSceneMessage 씬명을 상수화하고 2인 반복 전환 체크리스트를 작성한다.

## 5. 문서화 상태
Design은 6/10이다. `game_design_document.md`가 18KB 규모로 존재하고 프로젝트 개요와 기술 스택 섹션이 확인되므로 placeholder 수준은 아니다. 다만 현재 스냅샷만으로 시스템 간 상호작용과 선택 이유가 충분히 체계화됐다고 보기는 어렵다.

Technical은 2/10이다. Mirror 예제 README는 외부 패키지 문서일 뿐 프로젝트 기술 문서가 아니다. `docs/technical.md`가 없고, 서버 권위, 씬 생명주기, 매칭 상태 전이가 프로젝트 기준으로 설명되어 있지 않다.

Spec은 3/10이다. `marvel_snap_locations_all.md`와 GDD 일부가 컨텐츠 후보를 담고 있을 가능성은 있으나, 기능별 입력·상태·결과 매핑이나 QA 체크리스트로 바로 쓸 수 있는 사양서 근거는 부족하다.

## 6. Backlog
- Mirror 기반 네트워킹 코드의 권한 모델(카드 draft/배치/점수 계산의 서버 권위) 이 코드·문서 어디에도 명시되지 않음: 서버 권위 기준이 없으면 버그와 악용 가능성을 구분할 수 없다.
- 프로젝트 문서(GDD/기술/사양)가 repo 에 존재하지 않아 신규 유입자가 코드만으로 설계 의도를 복원해야 함: GDD는 있으나 기술 문서와 사양서 축이 아직 부족하다.
- CardRegistry 가 static + Resources.LoadAll 기반이라 테스트 격리/런타임 교체가 어렵다 (장기적으로 DI 또는 ScriptableObject 카탈로그 고려): 장기적으로 카탈로그 주입 구조가 필요하다.
- cardId 가 자동 증가 순번이라 에셋 정렬 순서에 의존 — 카드 추가/삭제 시 ID 충돌 또는 재할당 위험: 네트워크와 저장 데이터의 안정 키로 쓰기에는 위험하다.

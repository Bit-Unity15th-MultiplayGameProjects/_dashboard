# HongSJ — week2 game-session 스냅샷

## 1. 주요 변경사항
- `MatchmakingQueue.Pending` 전역 슬롯을 제거하고, 매칭 시 `GameSession` 프리팹을 직접 생성·초기화한 뒤 두 클라이언트에만 `GoToSceneMessage`를 보내도록 바뀌었다.
- `SnapNetworkManager`, `SnapMatchInterestManagement`가 추가되어 연결 직후 Ready/AddPlayer와 매치별 관전 대상 필터링을 담당한다.
- Draft/Game 입력 경로가 `Player.Command` 의존에서 직접 `NetworkMessage` 전송 방식으로 일부 이동했다.
- `DraftManager`, `MatchmakingQueue`가 Main 씬 상주 오브젝트로 옮겨져 씬 전환 중 서버 오브젝트 소실 문제를 줄이려는 방향은 명확하다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 이전보다 낫다. 전역 `Pending` 하나에 매치 데이터를 걸어두고 `ServerChangeScene`에 의존하던 구조를 끊은 것은 중요한 진전이다. 특히 `NetworkServer.Spawn` 전에 `GameSession.Init()`을 호출해 observer rebuild 시점의 초기화 순서를 의식한 점은 좋다. 다만 지금은 네트워크 메시지 라우팅 책임이 `MatchmakingQueue`에 과도하게 몰렸다. 드래프트 시작, 픽, 매칭, 게임 준비, 턴 제출까지 한 컴포넌트가 모두 받으므로 이름과 책임이 어긋난다.

가장 위험한 부분은 결과 기록 경로가 계속 갈라진 점이다. 정상 종료, 항복, 연결 해제에서 wins/losses와 `RecordGameResult`, `GameEndMessage`, `NetworkServer.Destroy`가 각각 따로 처리된다. 왜 문제냐면 한 경로만 수정해도 다른 종료 경로의 보상·전환·아레나 종료 조건이 쉽게 어긋난다. `FinishGameResult(winner, loser, reason)` 같은 서버 전용 함수 하나로 승패 반영, 재매칭 기록, 종료 메시지, 세션 파괴를 묶는 편이 맞다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 hot path O(n²) 문제는 뚜렷하지 않다. 다만 `SubmitTurnMessage`에서 매 제출마다 배열을 `new List<CardPlacement>`로 복사한다. 턴제라 비용 자체는 작지만, 이 복사 지점이 서버 검증의 관문이므로 단순 성능보다 계약 문제가 크다. 배열을 받은 즉시 소유 카드 set, 남은 에너지, zone index 범위를 검증하고, 유효한 placement만 서버 상태로 확정해야 한다.

## 3. 진행도 평가
구현 경로는 Mirror를 쓰되 매칭, 씬 전환, 권한 라우팅을 직접 설계하는 `hybrid`에 가깝다. 이번 커밋은 2인 Draft-Match-Game 흐름을 실제 시연 가능 쪽으로 밀어낸 전진으로 평가한다. 전역 씬 전환 대신 개별 클라이언트 전환을 도입한 점도 현재 게임 구조에 맞다.

다만 제출 가능한 수직 슬라이스로 보기에는 리스크가 높다. 씬명 문자열, UI별 메시지 핸들러 등록, 클라이언트 제출 검증 부재, 종료 처리 중복이 모두 네트워킹 흐름 한가운데 있다. 그래서 진행도는 58%로 본다. 동작 가능성은 올랐지만 반복 플레이 안정성은 아직 검증 전이다.

## 4. 다음 권장사항
- `GameSession`의 정상 종료·항복·연결 해제 결과 기록을 단일 서버 함수로 통합한다.
- `GoToSceneMessage` 수신 등록을 UI마다 두지 말고 `SceneRouter` 같은 상주 컴포넌트 하나로 옮긴다.
- `SubmitTurnMessage` 처리 전에 카드 소유권, 에너지, zone index, 중복 제출을 서버에서 검증한다.
- `docs/technical.md`에 서버 권위, 씬 전환, 재매칭 상태 흐름을 먼저 작성한다.
- `"Draft"`, `"Game"` 문자열은 상수 또는 enum 매핑으로 제한한다.

## 5. 문서화 상태
design 점수는 7점이다. `game_design_document.md`가 18KB 규모로 존재하고 프로젝트 개요와 기술 스택을 포함하므로 기본 GDD 역할은 한다. 다만 시스템별 의사결정 근거와 최신 네트워크 구조 반영 여부는 스냅샷만으로 충분하지 않다.

technical 점수는 2점이다. `docs/technical.md`가 없고, Mirror 권한 모델과 씬 생명주기 계약이 코드 주석에 흩어져 있다. 신규 인원이 문서만 보고 서버 소유 상태와 클라이언트 제출 범위를 판단할 수 없다.

spec 점수는 3점이다. `marvel_snap_locations_all.md` 같은 데이터성 문서는 있으나, 현재 게임의 씬/입력/상태/결과 매핑을 테스트 체크리스트로 쓸 수준의 사양서는 확인되지 않는다.

## 6. Backlog
- Mirror 서버 권위 모델은 아직 backlog로 남긴다. 메시지 구조는 개선됐지만 draft, 배치, 점수 계산의 최종 권한 계약이 문서와 코드 양쪽에서 명확하지 않다.
- 프로젝트 문서 backlog도 유지한다. GDD는 있으나 기술 문서와 기능 사양서가 없어 설계 추적성이 부족하다.
- `CardRegistry`의 static `Resources.LoadAll` 구조는 그대로 남아 테스트 격리와 런타임 카탈로그 교체가 어렵다.
- `cardId` 안정화도 해결 근거가 없다. 카드 추가·삭제 후 기존 네트워크 ID가 보존되는지 검증 가능한 매핑이 필요하다.

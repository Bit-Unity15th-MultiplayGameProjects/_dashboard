# HongSJ — 무변경 재점검 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개이고 diff 가 비어 있다. 이번 리포트는 신규 구현 평가가 아니라 기존 위험 항목 재판정이다.
- 직전 high TODO 인 `GameSession` 결과 기록 통합과 `MatchmakingQueue` 상태머신 정리는 해결 근거가 없다.
- `game_design_document.md` 는 존재하지만 `docs/technical.md` 수준의 서버 권위·씬 전환 계약 문서는 확인되지 않는다.
- `CardRegistry` 의 static `Resources.LoadAll` 구조와 자동 증가 `cardId` 위험은 계속 남아 있다.

## 2. 코드 품질 리뷰
이번 diff 에서는 새 코드가 없으므로 신규 가독성·구조 결함을 지적할 근거는 없다. 다만 기존 open item 기준으로 보면 핵심 위험은 그대로다. `GameSession` 에 정상 종료와 항복 종료의 결과 기록 경로가 갈라져 있으면 승패 초기화, 결과 전송, 매치메이킹 기록 중 하나만 수정되는 순간 화면과 서버 상태가 어긋난다. 종료 결과는 서버 전용 함수 하나에서 확정하고 UI/RPC 는 그 결과를 구독하는 흐름으로 접어야 한다.

`MatchmakingQueue` 의 ReadyForNext/Pending 계열도 명시적 matchId 상태머신이 필요하다. 연결 단위 상태를 제거했다가 재삽입하는 식이면 같은 connection 이 어느 match 의 어느 phase 에 속하는지 추적하기 어렵고, 반복 매칭에서 중복 전환이나 누락이 런타임에만 드러난다. `Waiting`, `Drafting`, `InMatch`, `Result` 같은 phase 를 matchId 와 함께 보관하는 쪽이 디버깅 기준을 만든다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 신규 hot path LINQ, 반복 할당, O(n²) 탐색 근거는 없다. 대신 `CardRegistry` 는 성능보다 계약 안정성이 문제다. `Resources.LoadAll` 결과 순서에 자동 증가 ID 를 묶으면 카드 추가·삭제만으로 네트워크 페이로드의 의미가 바뀐다. 명시 ID 또는 GUID 기반 카탈로그로 조회 계약을 고정해야 한다.

## 3. 진행도 평가
구현 경로는 Mirror 를 쓰지만 매칭, 씬 전환, draft/배치/점수 권한을 직접 설계해야 하므로 `hybrid` 로 보는 것이 맞다. 프레임워크가 RPC 와 동기화 기반은 제공하지만, 서버 권위와 반복 매치 복구는 프로젝트 코드의 책임이다.

진행도는 54로 유지한다. 플레이 루프의 축은 있는 것으로 보이나 이번 범위에서 전진이 없고, 결과 기록·ReadyForNext·문서화 위험이 모두 남았다. 특히 2인 반복 Draft-Match-Result 가 안정적으로 재현된 근거가 없으므로 마감 시연 리스크는 high 다.

## 4. 다음 권장사항
- `GameSession` 종료·항복 결과 기록을 단일 서버 함수로 통합한다.
- `MatchmakingQueue` ReadyForNext/Pending 흐름을 matchId 상태머신으로 고정한다.
- `docs/technical.md` 에 서버 권위와 씬 전환 계약을 먼저 작성한다.
- `GoToSceneMessage` 상수화와 2인 반복 전환 체크리스트를 같은 기준으로 고정한다.
- `CardRegistry` 의 자동 순번 ID 를 명시 ID 또는 GUID 매핑으로 전환한다.

## 5. 문서화 상태
design 은 6점이다. `game_design_document.md` 가 18KB 규모로 존재하므로 컨셉과 기본 루프 문서의 기반은 인정된다. 다만 변경 이력, 의사결정 근거, 시스템 간 상호작용 추적은 `_sample/docs` 수준에는 못 미친다.

technical 은 2점이다. Mirror 패키지 README 는 외부 라이브러리 문서일 뿐 HongSJ 의 아키텍처 문서가 아니다. 서버 권위, 씬 생명주기, 매치 상태 전이, 실패 처리 기준이 별도 문서로 확인되지 않는다.

spec 은 4점이다. `marvel_snap_locations_all.md` 같은 콘텐츠 목록은 사양 초안으로 활용 가능하지만, 카드 ID 계약, 입력·상태·결과 매핑, QA 체크리스트로 바로 쓰기에는 부족하다.

## 6. Backlog
- 프로젝트 문서 backlog 는 부분 잔존이다. GDD 는 있으나 기술·사양 문서가 분리되지 않아 신규 유입자는 여전히 코드 독해에 의존한다.
- Mirror 권한 모델 backlog 는 high 로 유지한다. 카드 draft, 배치, 점수 계산, 씬 전환의 최종 주인이 코드·문서에서 동시에 드러나야 한다.
- `CardRegistry` static `Resources.LoadAll` 장기 항목은 여전히 유효하다. 테스트 격리와 런타임 카탈로그 교체가 어려운 구조다.
- 동일한 `CardRegistry` static backlog 의 단기 항목도 남긴다. 다음 리포트에서 해결 여부를 파일 기준으로 판정해야 한다.
- 자동 증가 `cardId` backlog 는 유지한다. 에셋 정렬 변경만으로 저장·네트워크 ID 가 재할당될 수 있는 위험이다.

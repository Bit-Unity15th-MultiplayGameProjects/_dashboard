# Seven-Shards-ORIGIN — 런타임 안정화 스냅샷

## 1. 주요 변경사항
- Unity 오브젝트 탐색 API를 최신 형태로 정리하고, 일부 `FindFirstObjectByType` 호출을 `FindAnyObjectByType`로 바꿨다.
- 로비 프리팹 베이커에 fresh/saved manifest 비교를 넣어 구조가 같으면 저장을 건너뛰도록 했다.
- `CombatTrainingSceneDriver`, `ArenaJoinPromptHud`, 일시정지 메뉴에서 문자열 기반 참조를 줄이고 명시 참조를 늘렸다.
- 기본 네트워크 씬 경로를 `TitleFlowScenePaths` 상수로 모아 빈 문자열 fallback 을 보강했다.
- UI 런타임 리소스 정책과 `GameNetworkManager` 분리 계획 문서가 추가됐다.

## 2. 코드 품질 리뷰
이번 변경은 전반적으로 “런타임에서 우연히 맞아 떨어지던 참조”를 에디터/프리팹 기준으로 고정하려는 방향이라 좋다. 특히 `PauseMenuRootRefs.optionsContent` 추가와 로비 bake manifest 비교는 프리팹 구조 변경의 감지 가능성을 높인다. 다만 `CombatTrainingSceneDriver`는 `OnValidate`에서 자동 와이어링하고 `Awake`에서는 `Canvas`만 전역 탐색한다. 왜 문제냐면 에디터에서는 연결된 듯 보이지만, 씬 복제·런타임 생성·비활성 루트 조건에서 동작 기준이 달라질 수 있기 때문이다. 개선은 “필수 참조는 serialized, 개발 편의 자동 연결은 editor-only, 런타임 fallback 은 명시 로그”로 경계를 나누는 것이다.

자료구조/복잡도 관점에서는 큰 성능 악화는 보이지 않는다. 오히려 `MidBossPatternSmokeRunner`와 `SimpleProjectilePool`이 `int GetInstanceID()` 키 대신 객체 참조 키를 쓰도록 바뀐 점은 Unity instance id 재사용 가능성에 덜 취약해져 긍정적이다. 단, `FindObjectsByType<T>()` 호출이 여전히 씬 단위 탐색으로 여러 런타임 경로에 남아 있다. 현재 비용은 호출 시점 전체 씬 스캔이고, 대안은 scene registry 또는 serialized refs 다. 네트워크 씬 전환과 late join 이 많은 이 프로젝트에서는 재현 어려운 지연과 순서 의존 버그를 줄이는 의미가 있다.

`StageClearWatcher`의 `[Rpc(SendTo.Server, InvokePermission = Everyone)]` 전환은 NGO 최신화 방향으로 타당하지만, 권한 모델 변경이다. 왜 위험하냐면 종료/로비 복귀는 게임 흐름을 바꾸는 서버 명령이므로 비소유 클라이언트 호출 가능성이 의도인지 실제 2클라에서 확인해야 한다. 버튼 입력 주체, 서버 수신, 씬 전환 결과를 체크리스트로 남겨야 한다.

## 3. 진행도 평가
이전 대비 기술 문서와 UI/로비 안정화는 분명히 전진했다. 특히 Addressables 미정 상태를 정책 문서로 닫은 점은 팀 운영상 의미가 있다. 그러나 핵심 네트워크 오케스트레이터는 아직 코드가 아니라 계획 문서 수준에서만 분리됐다.

진행도는 약 58%로 본다. 플레이 가능한 네트워크 루프를 다듬는 단계에는 들어왔지만, 폰트 정책, 프리팹 bake 검증, `GameNetworkManager` 책임 분리, 콘텐츠 사양서 부족이 제출 가능성을 계속 누르고 있다.

## 4. 다음 권장사항
- `GameNetworkManager`는 transport 또는 spawn 중 하나만 골라 실제 partial/service 단위로 분리한다.
- `StageClearWatcher` RPC 변경은 host+client 2인 수동 스모크로 종료 경로를 검증한다.
- 씬 참조 자동 연결은 에디터 편의 기능과 런타임 fallback 을 분리하고 실패 로그를 남긴다.
- 로비 prefab bake skip 은 CI 또는 에디터 감사 메뉴에서 반복 검증 가능하게 고정한다.
- CJK 폰트는 KR SDF 대리 상태를 끝내고 locale별 실제 자산 기준을 확정한다.

## 5. 문서화 상태
design 점수는 4점이다. 이번 커밋은 운영·기술 정책 문서 중심이고, 게임 경험·레벨·전투·콘텐츠 의사결정 문서는 여전히 약하다. 왜 이 게임이 어떤 플레이 경험을 목표로 하는지 신규 유입자가 문서만으로 잡기 어렵다.

technical 점수는 7점이다. UI 리소스 정책, 네트워크 UI 상태, `GameNetworkManager` 분리 계획은 실무적으로 읽을 가치가 있다. 다만 계획이 코드 구조 다이어그램이나 검증 결과와 아직 연결되지 않았다.

spec 점수는 5점이다. 네트워크 UI 체크리스트류는 늘었지만, 레벨 구성·입력 상태·밸런스 수치·실패 조건 같은 테스트 가능한 사양은 부족하다. 구현 진척을 QA 표로 바꾸려면 기능별 expected result 가 더 필요하다.

## 6. Backlog
- `GameNetworkManager` 싱글톤 책임 집중은 아직 남아 있다. 문서는 생겼지만 디버깅 난이도 자체는 줄지 않았다.
- zh-Hans/zh-Hant/ja 폰트 대리 문제는 변경 근거가 없다. 출시 언어 품질 리스크로 계속 봐야 한다.
- 로비 프리팹 fileID 셔플은 bake skip 장치가 생겼지만 실제 prefab diff 안정화 완료는 아니다.
- 디자인·콘텐츠 사양서는 여전히 기술 구현 속도를 따라가지 못한다.
- Resources 기반 UI 로딩은 정책상 정리됐으나, 메모리와 로드 단위 검증은 장기 과제로 남는다.

## 7. 이전 Backlog 해결
- Resources 기반 UI 프리팹·로케일 대량 로딩의 Addressables 미정 상태는 `UI_RUNTIME_RESOURCE_POLICY.md`에서 “Resources 유지, Addressables 보류”로 운영 결론을 냈다. 다만 백엔드 자체는 그대로라 장기 메모리 검증은 별도 backlog 로 남긴다.

# Seven-Shards-ORIGIN — 런타임 UI 안정화 스냅샷

## 1. 주요 변경사항
- 여러 HUD와 패널에서 Resources 프리팹 누락 시 반복 빌드·반복 로그를 막도록 `_buildFailed` 플래그와 비활성화 처리가 추가됐다.
- `StageClearWatcher`가 클라이언트의 스테이지 클리어 종료 요청을 서버에서 검증하고, 중복 종료 요청을 차단하도록 개선됐다.
- `TMPRuntimeCjkFont`가 ja/zh-Hans/zh-Hant별 OS dynamic 후보군을 분리하고, KR SDF 대리 경로 진입 시 경고와 감사 리포트를 남기게 됐다.
- 런타임 fallback Canvas/EventSystem 자동 생성 경로에 경고 로그가 추가되어 씬 배선 누락을 추적하기 쉬워졌다.
- CJK fallback 문서와 stage clear 수동 체크리스트가 이번 구현 변경을 일부 반영했다.

## 2. 코드 품질 리뷰
이번 커밋은 “조용히 실패하던 UI”를 명시적으로 실패시키는 방향이라 품질 면에서 전진이다. 특히 `HotbarHudBootstrap`, `DownStateHud`, `MerchantShopPanel`, `TransientGameplayMessage`에서 프리팹 누락 후 매 프레임 재시도하거나 null 상태로 계속 흐르던 위험을 줄인 점은 좋다. 왜 중요한가 하면 Resources 기반 UI는 누락이 빌드 후에야 드러나기 쉽고, 반복 로그는 실제 원인을 묻어버린다. 다음 단계는 이 방어 코드를 유지하되, 에디터 감사에서 누락을 빌드 전 실패로 올리는 것이다.

`StageClearWatcher`의 서버 검증은 방향이 맞다. `BossCleared`, 연결 클라이언트, `PlayerStats`, 사망 상태를 확인하므로 “아무 클라이언트나 종료 RPC를 보내는” 위험이 줄었다. 다만 `_serverExitRequestInFlight`가 true가 된 뒤 실제 씬 전환이 실패하면 복구 경로가 없다. 왜 문제인가 하면 실패 한 번이 세션 전체의 버튼을 영구 무시하게 만들 수 있다. 개선하려면 `GameNetworkManager` 실행 함수의 성공/실패 신호를 받거나, 종료 시작 후 타임아웃·씬 전환 이벤트 기준으로 상태를 정리해야 한다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 hot path에 새 LINQ, 반복 정렬, O(n²) 선형 탐색을 추가한 명확한 증거는 없다. 오히려 CJK 폰트는 locale별 `Dictionary` 캐시와 `HashSet` 경고 기록으로 반복 생성과 중복 로그를 줄였다. 다만 `WarnedFallbackLocales`는 locale과 모드를 문자열 결합 키로 관리한다. 현재 비용은 작지만, 장기적으로는 `(locale, mode)` 구조 키나 enum 기반 키가 더 안전하다.

## 3. 진행도 평가
이전 대비 런타임 안정성은 분명히 전진했다. 특히 UI 프리팹 누락, EventSystem 부재, Canvas 자동 fallback 같은 운영 중 흔한 실패가 로그로 드러난다. 네트워크 종료 요청도 서버 권한 모델에 한 걸음 가까워졌다.

다만 프로젝트 완성도는 아직 폴리싱 단계라고 보기 어렵다. UI와 네트워크는 계속 단단해지고 있으나, 중앙 싱글톤 의존·CJK 출시 폰트 정책·문서 사양의 뒤처짐이 남아 있다. 현재 추정 진행도는 67%가 적절하다.

## 4. 다음 권장사항
- `StageClearWatcher` 종료 요청을 호스트·클라이언트·동시 클릭·실패 전환 케이스로 QA하고 체크리스트 결과를 남겨야 한다.
- `GameNetworkManager`에서 포트 설정, 씬 전환, 스폰, 체크포인트 책임을 분리해 네트워크 디버깅 범위를 줄여야 한다.
- JP/SC/TC 전용 TMP SDF를 넣을지, OS dynamic을 허용할지 출시 정책으로 확정해야 한다.
- Resources UI 프리팹 누락은 런타임 로그가 아니라 에디터 감사 실패로 잡도록 자동화해야 한다.
- 디자인·콘텐츠 사양서를 현재 네트워크 로비, 스테이지 클리어, HUD 흐름과 맞춰 갱신해야 한다.

## 5. 문서화 상태
design 점수는 5다. 구현 상태 문서는 늘고 있지만, 코어 루프와 플레이어 경험 목표가 왜 현재 UI·네트워크 구조로 이어지는지 설명하는 설계 문서는 아직 약하다.

technical 점수는 7이다. CJK fallback, 네트워크 UI 상태, 수동 체크리스트는 실무적으로 쓸 수 있다. 다만 `GameNetworkManager` 중심 권한·상태 흐름을 신규 개발자가 한 번에 이해할 아키텍처 문서는 부족하다.

spec 점수는 5다. 체크리스트는 개선됐지만, 레벨·전투·보상·입력·상태 결과 매핑이 `_sample/docs` 수준의 테스트 가능한 사양으로 정리됐다고 보긴 어렵다.

## 6. Backlog
- `GameNetworkManager`가 여전히 여러 책임을 가진 싱글톤이다. 네트워크 장애가 나면 원인 범위가 과도하게 넓다.
- CJK 폰트는 locale별 OS dynamic 후보가 생겼지만, 전용 SDF가 없으면 빌드 환경별 표기 차이를 배제할 수 없다.
- `NetworkMultiplayerLobbyRoot.prefab`의 대량 fileID 셔플은 아직 해결 근거가 없다. 프리팹 diff 신뢰도를 낮춘다.
- 디자인·콘텐츠 사양서는 구현 진척을 따라가지 못한다. 신규 기능 검수 기준으로 쓰기 어렵다.
- Resources 기반 UI 로딩은 방어 코드가 늘었지만 장기 메모리·수명주기 검증은 아직 남아 있다.

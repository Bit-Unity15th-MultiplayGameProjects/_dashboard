# Seven-Shards-ORIGIN — 변경 없음 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위가 동일 SHA에서 동일 SHA로 잡혀 있어 신규 커밋과 실제 diff가 없다. 따라서 코드 전진은 확인되지 않는다.
- 직전 TODO와 Backlog는 대부분 그대로 유효하다. 특히 `GameNetworkManager`, `StageClearWatcher`, CJK 폰트 정책은 해결 근거가 없다.
- 문서 스냅샷에는 네트워크 권한 모델, CJK fallback, UI 리소스 정책, 수동 체크리스트가 존재한다. 기술 문서 기반은 약하지 않다.
- 과거 누락 항목 중 TrapFieldWest 검증, 보스·추격전 스모크, HUD 빌더 중복 축소는 `DASHBOARD_TODO_TRACKER.md`에서 완료 기록이 확인되어 재등장시키지 않는다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 새로 유입된 코드 결함은 없다. 문제는 반대로, 이미 식별된 구조 부채가 줄지 않았다는 점이다. `GameNetworkManager`가 포트, 스폰, 체크포인트, 씬 전환 책임을 계속 보유한다면 네트워크 오류가 발생했을 때 원인 범위가 지나치게 넓어진다. 이는 단순한 취향 문제가 아니라 멀티플레이 디버깅 비용을 폭발시키는 구조다. 포트 설정, 플레이어 스폰, 씬 전환, 체크포인트 복원을 별도 협력 객체로 분리하고, 싱글톤은 orchestration만 맡도록 줄여야 한다.

`StageClearWatcher`는 이전 리포트에서 종료 요청 실패와 중복 처리 QA가 남아 있었다. 문서 체크리스트가 있는 것은 좋지만, 실패 응답·동시 클릭·비소유 클라이언트 요청 같은 경계 조건이 기록으로 닫히지 않으면 실제 시연에서 버튼이 무시되거나 중복 전환되는 문제가 다시 나온다. 개선 기준은 명확하다. 실패 시 UI 상태가 복구되는지, 중복 요청이 한 번만 처리되는지, 서버 권한 검증 로그가 남는지 수동 체크리스트에 결과를 남겨야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 새로 지적할 근거가 없다. hot path의 LINQ, 반복 정렬, O(n²) 탐색, 프레임별 임시 컬렉션 생성이 추가됐다는 증거가 없으므로 추측성 최적화 TODO는 만들지 않는다. 다만 CJK 폰트와 UI 리소스 정책은 장기적으로 캐시·감사 루틴이 중요하므로, 실제 코드 변경이 생긴 시점에 다시 비용을 확인해야 한다.

## 3. 진행도 평가
이 프로젝트는 Netcode 계열 프레임워크를 사용하면서도 로비, 권한 검증, 커스텀 메시지, 런타임 UI 리소스 정책을 직접 설계하는 hybrid 경로로 판단된다. 프레임워크 기반이라 순수 custom-network보다 컨텐츠와 UX 기대치가 높지만, 직접 구현한 경계가 많아 테스트와 문서가 함께 따라와야 한다.

진행도는 67%로 유지한다. 기술 문서와 체크리스트는 쌓였고 수직 슬라이스의 뼈대도 있어 보인다. 그러나 신규 커밋이 없고, 2인 happy path를 깨뜨릴 수 있는 종료 흐름 QA와 `GameNetworkManager` 구조 부채가 그대로다. 이 조합은 medium을 넘어서 high risk로 보는 것이 맞다.

## 4. 다음 권장사항
- `StageClearWatcher` 종료 요청을 실패·중복·비소유 클라이언트 케이스로 QA하고 체크리스트에 결과를 남겨야 한다.
- `GameNetworkManager`를 포트 설정, 스폰, 씬 전환, 체크포인트 복원 단위로 1차 분리해야 한다.
- JP/SC/TC 전용 TMP SDF를 포함할지, OS dynamic fallback을 허용할지 출시 기준으로 확정해야 한다.
- 런타임 UI 프리팹 누락은 로그 정책에서 멈추지 말고 에디터 감사 실패로 승격해야 한다.
- 디자인·콘텐츠 사양서는 현재 네트워크 로비, 스테이지 클리어, HUD 흐름을 기준으로 갱신해야 한다.

## 5. 문서화 상태
design은 6점이다. `DEMO_PLAYER_RACE_OOP_DESIGN.md`, `MID_BOSS_DESIGN.md`, `FINAL_BOSS_DESIGN.md`는 의도와 구조가 비교적 분명하다. 다만 전체 GDD 관점의 코어 루프, 플레이 경험 목표, 콘텐츠 진행 구조가 하나의 기준 문서로 묶이지 않았다.

technical은 8점이다. `NETWORK_MULTIPLAYER_LOBBY_AUTHORITY_MODEL.md`, `GAMENETWORKMANAGER_RESPONSIBILITY_SPLIT_PLAN.md`, `UI_RUNTIME_RESOURCE_POLICY.md`, `PROJECT_CODE_STYLE.md`가 신규 유입자가 읽을 만한 수준이다. 단, 코드가 실제로 그 구조로 분리됐는지는 별도 문제다.

spec은 6점이다. `STAGE_CLEAR_MANUAL_CHECKLIST.md`, `NETWORK_MULTIPLAYER_LOBBY_MANUAL_CHECKLIST.md`, `TMP_CJK_FALLBACK_GUIDE.md`는 테스트 기준으로 쓸 수 있다. 그러나 레벨, 입력, 상태 전이, 보상, 밸런스 수치가 `_sample/docs` 수준의 통합 사양으로 정리되지는 않았다.

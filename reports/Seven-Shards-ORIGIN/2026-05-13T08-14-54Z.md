# Seven-Shards-ORIGIN — 공주 전투 폴리싱 스냅샷

## 1. 주요 변경사항
- `BeastPrincessCombat`, `GiantPrincessCombat`, `SylvanPrincessCombat`가 신규 추가되어 인간 외 3종 공주의 전투 루프가 본격 구현됐다.
- `PlayerAnimator`가 캐릭터별 애니메이션 오버라이드와 공격/차지/다운 상태 우선순위를 통합 처리하기 시작했다.
- `PlayerVFX`에 거인 차지 오라, 차지 폭발, 참격파 라인 VFX가 추가됐다.
- 커밋은 전투 체감 개선 방향으로 분명히 전진했지만, 현재 diff 기준으로 컴파일 중단과 네트워크 권한 문제가 동시에 보인다.

## 2. 코드 품질 리뷰
가장 큰 문제는 `PlayerAnimator.cs`의 `ResolveGiantReplacement()`가 선언되지 않은 `giantChargeHoldClip`를 참조한다는 점이다. 이는 스타일 문제가 아니라 빌드 차단 결함이다. 왜 문제인가 하면, 이 상태에서는 나머지 전투 개선이 실제 플레이 검증 단계로 넘어갈 수 없다. 개선 방향은 단순하다. 필드를 추가할지, 기존 `giantJumpClip`/`giantAttackClip`으로 대체할지 결정하고 컴파일을 먼저 복구해야 한다.

네트워크 권한도 위험하다. `BeastPrincessCombat`와 `GiantPrincessCombat`의 ServerRpc가 공격 박스 중심, 방향, 일부 판정 값을 오너 클라이언트에서 받은 값으로 처리한다. 피해 적용은 서버에서 호출되지만, 판정 위치를 클라이언트가 제출하면 사실상 공격 권한이 클라이언트로 넘어간다. 서버는 플레이어의 현재 transform, facing, 입력 상태, 서버 상수를 기준으로 공격 범위를 재계산해야 한다.

자료구조/복잡도/할당 관점에서는 `SylvanPrincessCombat`가 명확한 hot path 문제를 만든다. 충전/오라 갱신에서 `GetComponentsInChildren<SpriteRenderer>()` 배열을 매 프레임 만들고, 투사체·오라 판정에서 `OverlapCircleAll`/`OverlapBoxAll`을 반복 사용한다. 현재 구조의 비용은 전투 중 GC와 프레임 스파이크다. 렌더러 참조는 생성 시 캐시하고, 물리 질의는 `NonAlloc` 버퍼나 제한된 tick으로 바꾸는 편이 이 프로젝트의 2인 네트워크 전투 안정성에 직접 의미가 있다.

## 3. 진행도 평가
이 프로젝트는 NGO 기반에 로비·권한·런타임 리소스 정책을 직접 얹은 hybrid 경로로 보는 것이 타당하다. 이번 커밋은 컨텐츠 밀도 면에서 전진이 크다. 특히 3종 공주가 각각 다른 전투 정체성을 갖기 시작한 점은 수직 슬라이스 완성도에 도움이 된다.

다만 제출 가능성 기준으로는 감점이 크다. 컴파일 오류 하나가 전체 빌드를 막고, 신규 전투 RPC의 서버 검증이 약하다. 현재 진행도는 64%로 본다. 기능량은 늘었지만, 반복 시연 가능한 2인 빌드로 잠그기 전까지는 안정권이 아니다.

## 4. 다음 권장사항
- `PlayerAnimator`의 미정의 심볼을 먼저 고쳐 빌드 가능한 상태를 복구한다.
- Beast/Giant 공격 RPC는 클라이언트 제출 좌표를 버리고 서버 기준 판정으로 바꾼다.
- 3종 공주 전투를 호스트+클라이언트 2인 기준으로 공격, 스킬, 사망, 재접속까지 확인한다.
- `StageClearWatcher` 종료 요청 실패·중복 요청 케이스는 체크리스트에 실제 결과로 남긴다.
- Sylvan VFX와 물리 판정의 반복 할당은 캐시와 NonAlloc으로 줄인다.

## 5. 문서화 상태
design은 7점이다. `DEMO_PLAYER_RACE_OOP_DESIGN.md`, `FINAL_BOSS_DESIGN.md`, `MID_BOSS_DESIGN.md`는 의도와 구조 판단을 꽤 잘 설명한다. 이번 3종 전투 구현과도 방향은 맞다.

technical은 7점이다. 네트워크 권한 모델, 리소스 정책, GameNetworkManager 분리 계획 등 신규 유입자가 구조를 읽을 문서가 있다. 다만 현재 코드의 신규 종족 전투 RPC 권한 모델은 문서 기준과 다시 대조해야 한다.

spec은 6점이다. 수동 체크리스트와 정책 문서는 있으나, 이번 전투 수치·입력·상태 전이·2인 QA 기대 결과가 표준 사양 수준으로 고정되지는 않았다. 샘플 문서 수준의 데이터 테이블과 판정 기준까지는 아직 부족하다.

## 6. Backlog
- `GameNetworkManager` 싱글톤 책임 집중은 여전히 남아 있다. 네트워크 생명주기와 씬 전환 회귀 범위를 줄이려면 분리가 필요하다.
- `NetworkMultiplayerLobbyRoot.prefab` fileID 셔플은 의미 diff로 정리됐다는 근거가 아직 부족하다.
- `Resources` 기반 UI 로딩은 정책 문서가 있지만 장기 세션 메모리 계측 결과가 없다.
- 디자인·콘텐츠 문서는 늘었지만, 신규 전투 구현의 수치와 QA 기준이 아직 구현 속도를 따라가지 못한다.
- `PlayerAnimator`는 종족별 오버라이드가 빠르게 누적 중이다. 컴파일 복구 뒤 상태 결정과 클립 교체 책임을 분리해야 한다.

## 7. 이전 Backlog 해결
- `zh-Hans/zh-Hant/ja 폰트가 KR SDF 로 대리되어 표기 누락 잠재`는 현재 런타임 정책이 locale 전용 SDF 우선, 없으면 OS dynamic 폰트로 바뀐 근거가 있다. KR SDF 대리 자체를 전제로 한 backlog는 닫는 것이 맞다.

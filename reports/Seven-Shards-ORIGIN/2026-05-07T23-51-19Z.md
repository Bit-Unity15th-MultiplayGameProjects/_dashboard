# Seven-Shards-ORIGIN — UI 현지화 정리 스냅샷

## 1. 주요 변경사항
- `en/ko/ja/zh-Hans/zh-Hant` 로케일 파일에 전투 연습 패널 문자열이 추가됐다. 런타임 UI 하드코딩 문구를 빼낸 점은 명확한 전진이다.
- `CombatTrainingSceneDriver` 의 버튼·토글·상태줄 문구가 localization key 기반으로 바뀌었다. 개발 패널도 다국어 정책에 편입된 점은 좋다.
- `GameNetworkManager`, `ArenaEncounterController` 중심으로 주석·Tooltip·로그 문구가 영어화되고 상수화됐다. 팀 협업 기준으로는 읽기 장벽이 낮아졌다.
- 문서, 폰트 에셋, Addressables, 문제 prefab 정리는 이번 커밋에서 다뤄지지 않았다. 이전 backlog 대부분은 그대로 남아 있다.

## 2. 코드 품질 리뷰
이번 변경의 좋은 점은 문자열 의도가 코드 밖으로 이동했다는 점이다. 특히 `CombatTrainingSceneDriver` 에서 UI 문구를 key 상수로 정리한 것은 이후 번역 수정의 충돌 범위를 줄인다. 다만 `Lf()` 가 `string.Format(L(key), args)` 를 바로 호출하므로 키 누락, placeholder 개수 불일치, 번역 파일의 `{0}` 손상은 런타임 예외로 이어질 수 있다. 다국어 파일을 수동으로 5벌 관리하는 구조라면 key parity와 placeholder parity 검증은 선택이 아니라 필수다.

더 큰 위험은 구조 개선 없이 대형 클래스에 문자열만 더 얹었다는 점이다. `GameNetworkManager` 는 포트 탐색, ILPP 대기, 씬 로드, 스폰, 체크포인트 복귀, 게임오버 흐름을 한 곳에서 가진다. 네트워킹 코드는 실패 재현이 어렵기 때문에 이 결합은 단순한 스타일 문제가 아니라 디버깅 비용 문제다. `CombatTrainingSceneDriver` 역시 UI 생성, 권한 검사, 샌드박스 명령, 적 스폰, 보스 텔레포트가 한 파일에 남아 있어 개발 도구가 커질수록 회귀 위험이 올라간다.

자료구조·복잡도 관점에서는 이번 diff 범위에서 새로 확인되는 명확한 hot path 문제는 없다. 추가된 `string.Format` 은 UI 생성·버튼 이벤트 경로에 가까워 매 프레임 할당 이슈로 단정할 근거가 부족하다. 대신 현재 의미 있는 비용은 선형 탐색 최적화보다 “수동 로케일 테이블 불일치”와 “네트워크 책임 결합”에서 나온다.

## 3. 진행도 평가
진행도는 61%로 본다. 플레이 가능한 네트워크·전투 흐름은 계속 축적되고 있고, 이번 커밋은 현지화와 협업 가독성 면에서 실질적인 정리다. 커밋 메시지도 변경 의도를 잘 설명한다.

하지만 이번 작업은 기능 완성보다 정리 성격이 강하다. 핵심 리스크인 네트워크 매니저 비대화, Resources 로딩 정책, CJK 폰트 검증, 문서 부재는 해소되지 않았다. 브랜치 운용 정보는 없고, 단일 커밋이 넓은 파일을 건드린 만큼 다음 단계에서는 검증 자동화 없이는 안정성을 주장하기 어렵다.

## 4. 다음 권장사항
- 로케일 파일 5종의 key 누락과 placeholder 불일치를 CI 또는 에디터 테스트로 막아야 한다.
- `GameNetworkManager` 에서 포트 시작, 씬 전환, 스폰, 체크포인트 복귀를 테스트 가능한 단위로 분리해야 한다.
- `Resources` 에 쌓인 UI·로케일 로딩 범위를 Addressables 로 옮길지 이번 스프린트 안에 결정해야 한다.
- `CombatTrainingSceneDriver` 는 UI presenter, sandbox command, enemy spawn service 수준으로 나누는 편이 좋다.
- 네트워크 권한 모델과 씬 전환 실패 대응을 `Docs/` 에 기록해 신규 작업자가 재현 가능하게 만들어야 한다.
- CJK 번역이 늘어난 만큼 ja/zh 전용 TMP SDF 검증을 더 미루면 안 된다.

## 5. 문서화 상태
design은 4/10이다. 게임의 플레이 경험과 전투·협동 의도는 코드에서 추정되지만, GDD가 현재 구현을 따라 설명한다는 근거가 없다. 왜 이런 네트워크·전투 흐름을 선택했는지 문서로 확인하기 어렵다.

technical은 4/10이다. 코드 주석은 좋아졌지만 이것은 문서가 아니다. `GameNetworkManager` 의 권한 모델, 씬 전환 순서, 실패 복구 정책을 문서만 보고 이해할 수 있는 상태가 아니다.

spec은 3/10이다. UI 문자열과 개발 패널 기능은 늘었지만 입력, 상태, 결과, 테스트 기준이 사양서로 정리되지 않았다. `_sample/docs` 수준의 체크리스트나 수치 테이블과는 아직 거리가 있다.

## 6. Backlog
- Resources 기반 UI 프리팹·로케일 대량 로딩은 그대로다. 이번 커밋에서 로케일 파일이 더 늘어 Addressables 판단을 늦출수록 전환 비용이 커진다.
- `GameNetworkManager` 의 싱글톤 집중은 계속 위험하다. 네트워크 장애가 나면 포트, 씬, 스폰, 체크포인트 중 어디가 원인인지 좁히기 어렵다.
- zh-Hans/zh-Hant/ja 폰트 대리 문제는 미해결이다. 번역 문자열이 추가된 만큼 글리프 누락 검증의 우선순위는 올라갔다.
- `NetworkMultiplayerLobbyRoot.prefab` 의 fileID 셔플 이슈는 이번 범위에서 해결 근거가 없다. prefab 안정성 검토가 필요하다.
- `Docs/` 는 구현 속도를 따라가지 못한다. 지금 문서화를 미루면 멀티플레이 회귀를 사람 기억에 의존하게 된다.

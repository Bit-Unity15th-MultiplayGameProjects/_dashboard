# Seven-Shards-ORIGIN — 레이스/씬 확장 스냅샷

## 1. 주요 변경사항
- PlayerAnimator 컴파일 차단 이슈는 해결 방향으로 전진했다. RaceAnimatorProfile, IPrincessCombatAnimator, 신규 CharacterData 연결이 추가되어 종족별 애니메이션 기반이 생겼다.
- Sylvan/Giant/Beast 전투 코드와 플레이어 로드아웃 흐름이 확장됐다. 선택 UI 상태가 실제 플레이어 데이터로 이어지는 경로를 만든 점은 좋은 진전이다.
- TrapFieldWest 씬이 8만 줄 이상 변경됐고 대량 fileID/메시 재생성이 섞였다. 이 상태는 기능 리뷰보다 직렬화 잡음이 더 커지는 위험 신호다.
- 외부 에셋, 오디오, 복구 씬, 에디터 유틸이 대량 유입됐다. 컨텐츠 확장은 긍정적이나 제출 산출물 경계가 흐려졌다.

## 2. 코드 품질 리뷰
이번 변경의 좋은 점은 종족별 애니메이션을 ScriptableObject 프로필로 빼려는 방향이다. 공통 PlayerAnimator 안에 모든 클립을 직접 누적하는 방식보다 유지보수성이 높다. 다만 PlayerAnimator 자체는 여전히 상태 우선순위와 종족별 연결 판단을 함께 가진다. 왜 문제인가 하면 새 종족이 추가될 때마다 공통 애니메이터의 회귀 범위가 커지기 때문이다. 개선하려면 상태 결정은 공통 인터페이스로 두고, 종족별 클립/전투 상태는 각 Race 모듈이 소유하도록 더 밀어야 한다.

네트워크 전투는 엄격히 봐야 한다. Beast/Giant 공격 판정이 서버에서 위치·방향·피해 상수를 재계산한다는 근거가 아직 부족하다. 클라이언트가 히트 위치나 피해를 사실상 결정하면 멀티플레이에서 판정 신뢰성이 무너진다. 입력은 요청만 보내고, 서버가 현재 NetworkObject 상태와 서버 상수로 판정을 만드는 구조로 고정해야 한다.

자료구조/복잡도/할당 측면에서는 Sylvan 전투와 VFX 루프가 계속 위험하다. 이전 항목의 `Overlap*All`, `GetComponentsInChildren` 반복 할당 패턴이 명확히 해소됐다는 근거가 없다. 현재 비용은 전투 중 배열·컴포넌트 탐색 할당이 누적되는 구조이고, 개선책은 `Overlap*NonAlloc`, 캐시된 컴포넌트 배열, 필요 시 오브젝트 풀이다. 이 프로젝트는 2인 전투와 오라/투사체가 겹치므로 GC 스파이크가 곧 조작감 문제로 이어진다.

## 3. 진행도 평가
네트워크 경로는 NGO 기반이지만 로비 상태, 캐릭터 선택, 스폰 반영, 룸 메시징을 직접 조합하는 hybrid 에 가깝다. 단순 framework 프로젝트보다 네트워크 설계 난이도는 높지만, 그만큼 권한 모델과 2인 QA 기록이 없으면 제출 안정성을 인정하기 어렵다.

진행도는 64로 본다. 캐릭터 선택과 종족 전투의 수직 슬라이스는 전진했지만, TrapFieldWest 대형 씬 변경, GameNetworkManager 집중, 전투 권한 미검증이 남아 있다. 빌드 차단은 내려갔지만 반복 시연 가능한 2인 happy path 근거는 아직 약하다.

## 4. 다음 권장사항
- Beast/Giant 공격 판정은 클라이언트 좌표 신뢰를 끊고 서버 재계산으로 고정한다.
- Sylvan/Giant/Beast 전투를 호스트+클라이언트 2인 환경에서 같은 체크리스트로 검증한다.
- TrapFieldWest 변경은 의미 배치 변경, 생성 메시/fileID churn, 불필요 복구 산출물로 분리한다.
- StageClearWatcher 종료 실패와 중복 요청은 수동 QA 결과를 문서에 명시한다.
- Sylvan 전투 hot path 의 physics/VFX 할당은 NonAlloc, 캐시, 풀링으로 줄인다.
- 외부 에셋과 `_Recovery` 폴더는 실제 제출·빌드에 필요한 범위만 남긴다.

## 5. 문서화 상태
design 은 7점이다. `CHARACTER_RACE_IMPLEMENTATION_HANDOFF.md`, `DEMO_PLAYER_RACE_OOP_DESIGN.md`, 보스/중간보스 문서가 의도와 현 상태를 잘 설명한다. 다만 Giant/Beast는 아직 수치가 비어 있어 완성형 GDD 수준은 아니다.

technical 은 8점이다. 네트워크 권한 모델, GameNetworkManager 분리 계획, UI 리소스 정책, 코드 스타일 문서가 있어 신규 작업자가 구조를 따라갈 수 있다. 단, 문서가 구현을 대체하는 단계에 머문 항목이 남았다.

spec 은 6점이다. Sylvan 수치와 수동 체크리스트는 실무적으로 쓸 수 있다. 반면 레벨/씬 검수, 종족별 입력·상태·결과 매핑, 2인 QA 판정표는 아직 `_sample/docs` 수준의 표준 사양서에는 못 미친다.

## 6. Backlog
- GameNetworkManager 가 포트·스폰·체크포인트를 한 싱글톤에 보유한다. NGO 생명주기와 씬 흐름이 한 클래스에 묶이면 작은 네트워크 수정도 전체 회귀가 된다.
- NetworkMultiplayerLobbyRoot.prefab 변경은 대량 fileID 셔플 이력이 계속 남아 있다. 프리팹 참조 감사와 의미 diff 분리가 필요하다.
- PlayerAnimator는 프로필 위임에도 상태 우선순위 판단을 중앙에서 계속 가진다. 새 종족이 늘수록 공통 애니메이터가 다시 병목이 된다.
- Resources 기반 UI 로딩은 정책 문서가 생겼지만 장기 메모리 실측과 해제 기준이 없다. 모바일/저사양 환경에서는 반드시 수치로 확인해야 한다.
- 디자인·콘텐츠 문서는 늘었지만 구현 속도를 완전히 따라잡지는 못했다. 특히 Giant/Beast 수치와 씬 QA 기준을 사양서로 닫아야 한다.

# Seven-Shards-ORIGIN — 변경 없음 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 범위가 동일 SHA에서 동일 SHA로 닫혀 있어 신규 커밋과 실제 diff가 없다. 이번 평가는 전진보다 미해결 항목의 회계 재판정이다.
- 직전 TODO 대부분은 해결 근거가 추가되지 않았다. 특히 `GameNetworkManager`, `StageClearWatcher`, CJK 폰트 정책은 그대로 다음 작업으로 남는다.
- 문서 축적은 강점이다. 네트워크 권한 모델, UI 리소스 정책, CJK fallback 가이드, 수동 체크리스트가 존재해 신규 유입자가 맥락을 잡을 수 있다.
- 과거 backlog 중 “네트워크 권한 모델 문서화 공백”과 “Addressable 전환 미정”은 현재 문서 상태 기준으로 더 좁은 이슈로 정리됐다.

## 2. 코드 품질 리뷰
이번 diff에는 신규 코드가 없으므로 새 결함을 특정할 수는 없다. 그러나 이전부터 반복된 구조 부채는 해소되지 않았다. `GameNetworkManager`가 포트 설정, 플레이어 스폰, 체크포인트, 씬 전환을 한 싱글톤에서 계속 가진다면 네트워크 오류 발생 시 원인 범위가 지나치게 넓어진다. 이는 단순 스타일 문제가 아니라 멀티플레이 장애 분석 비용을 키우는 구조 문제다. 포트/세션 설정, 스폰, 씬 전환, 체크포인트 복원을 별도 협력 객체로 나누고 싱글톤은 조율만 맡기는 방향이 맞다.

`StageClearWatcher`는 종료 요청 실패, 중복 클릭, 비소유 클라이언트 요청 같은 경계 조건 검증이 아직 수동 QA 항목으로 남아 있다. 이 부분은 “대체로 된다”로 넘기면 위험하다. 스테이지 종료는 씬 전환과 네트워크 권한이 만나는 지점이므로 한 번의 중복 요청도 로비 복귀 실패나 상태 꼬임으로 이어질 수 있다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 신규 지적이 없다. hot path의 LINQ, 반복 정렬, O(n²) 조회, 프레임별 임시 컬렉션 생성이 새로 확인된 근거가 없으므로 추측성 최적화 TODO는 만들지 않는다. 다만 UI 리소스와 폰트 fallback은 장기적으로 캐시·해제·감사 기준이 중요하므로 실제 코드 변경 시 비용 측정을 함께 붙여야 한다.

## 3. 진행도 평가
구현 경로는 Netcode 계열 프레임워크를 쓰면서 로비, 권한 검증, 커스텀 메시지, 런타임 UI 정책을 직접 설계하는 hybrid로 판단한다. framework 기반이라 연결·RPC 자체의 비용은 낮지만, 프로젝트가 직접 맡은 로비 상태와 씬 전환 책임이 커서 자동 테스트와 수동 재현 기록의 가치가 높다.

진행도는 67로 유지한다. 문서와 체크리스트는 제출 준비에 도움이 되지만, 이번 범위에서 실행 가능한 2인 happy path 안정성이나 구조 분리 전진이 확인되지 않았다. 네트워크 구조 부채가 아직 핵심 경로에 있으므로 risk는 high가 타당하다.

## 4. 다음 권장사항
- `StageClearWatcher` 종료 요청을 실패, 중복, 비소유 클라이언트, 재시도 케이스로 나눠 수동 QA 결과를 남겨야 한다.
- `GameNetworkManager`를 포트 설정, 스폰, 씬 전환, 체크포인트 복원 책임으로 1차 분리해야 한다.
- JP/SC/TC 전용 TMP SDF를 포함할지, OS dynamic fallback을 허용할지 출시 기준으로 확정해야 한다.
- 런타임 UI 프리팹 누락은 로그 경고가 아니라 에디터 감사 실패로 막아야 한다.
- 디자인·콘텐츠 문서는 현재 로비, 스테이지 클리어, HUD 흐름 기준으로 갱신해야 한다.

## 5. 문서화 상태
design은 6점이다. `DEMO_PLAYER_RACE_OOP_DESIGN.md`, `MID_BOSS_DESIGN.md`, `FINAL_BOSS_DESIGN.md`는 설계 의도와 책임 분리를 꽤 잘 설명한다. 다만 전체 GDD, 코어 루프, 플레이어 경험 목표가 하나의 최신 기준 문서로 묶이지 않았다.

technical은 8점이다. `NETWORK_MULTIPLAYER_LOBBY_AUTHORITY_MODEL.md`, `GAMENETWORKMANAGER_RESPONSIBILITY_SPLIT_PLAN.md`, `UI_RUNTIME_RESOURCE_POLICY.md`, `PROJECT_CODE_STYLE.md`는 실무적으로 쓸 수 있다. 문제는 문서가 코드 분리 완료를 보장하지 않는다는 점이다.

spec은 6점이다. `STAGE_CLEAR_MANUAL_CHECKLIST.md`, `NETWORK_MULTIPLAYER_LOBBY_MANUAL_CHECKLIST.md`, `TMP_CJK_FALLBACK_GUIDE.md`는 체크리스트 역할을 한다. 그러나 레벨, 입력, 상태 전이, 보상, 밸런스 수치가 `_sample/docs` 수준의 통합 사양으로 정리되지는 않았다.

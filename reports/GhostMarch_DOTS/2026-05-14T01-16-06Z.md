# GhostMarch_DOTS — 무기 각성 확장 스냅샷

## 1. 주요 변경사항
- 무기 각성 flow가 본격 확장됐다. talisman, horn bow, vajra, jakdu가 추가되고 `WeaponAwakeningApplySystem`과 전용 bridge가 생겼다.
- GPU presentation 쪽은 resource atlas material, 고정 atlas shader, hit effect atlas, tiled background까지 들어와 시각적 수직 슬라이스가 크게 전진했다.
- restart/reset, `PlayerItemState` 초기 레벨 동기화, 아이템 max level 조정처럼 플레이 루프 안정화성 fix가 다수 포함됐다.
- `EnemySeparationSystem`, 신규 fire/hit system들이 추가되며 전투 밀도와 무기별 효과는 늘었지만 검증 표면도 같이 커졌다.
- `dots-resource-asset-map.md`, handoff, rebuild todo가 갱신되어 자산 연결 상태를 추적하려는 방향은 긍정적이다.

## 2. 코드 품질 리뷰
가장 좋은 변화는 presentation과 무기 각성 작업이 더 이상 한 파일에만 쌓이지 않고 system, bridge, material setup으로 분리되기 시작했다는 점이다. 이는 Unity/DOTS 프로젝트에서 책임 경계를 세우는 올바른 방향이다. 다만 이번 범위는 `ItemApplySystem`, 각 무기 fire/hit system, UI bridge, config가 동시에 움직였다. jakdu type 8처럼 int-slot 계약이 늘어난 상태에서 UI preview와 실제 apply가 같은 enum-indexed 정의를 보지 않으면, 레벨업 선택은 보이지만 전투 결과가 다른 위험이 크다.

자료구조/복잡도/할당 관점에서는 `remove player ToEntityArray` 커밋이 명확한 개선이다. 왜냐하면 렌더 hot path에서 매 프레임 임시 EntityArray를 만드는 것은 DOTS 전환의 성능 이득을 직접 깎기 때문이다. 개선 방향은 player만이 아니라 fallback 전체를 snapshot-empty 조건으로 제한하고, 기본 렌더는 snapshot buffer 한 경로로 고정하는 것이다. `EnemySeparationSystem`은 새 hot path 후보이므로 현재 구조가 O(n²)인지 추측으로 단정하지 말고, 500체 이상 기준 계측을 먼저 추가한 뒤 grid/hash 분할이 필요한지 판단해야 한다.

Unity 관점에서는 자산을 `Resources`와 material asset으로 연결한 점은 빠른 수직 슬라이스에는 실용적이다. 그러나 최종 제출 기준에서는 scene, material, atlas row/index, item slot 계약을 smoke test가 검증해야 한다. 현재처럼 `fix`와 `tune` 커밋이 연속되는 구간에서는 수동 Game View 확인만으로는 같은 결함이 반복될 가능성이 높다.

## 3. 진행도 평가
네트워킹 코드는 현재 평가 대상이 아니므로 구현 경로는 로컬 DOTS 수직 슬라이스로 본다. 이 기준에서 무기 각성, atlas 렌더, 배경, 전투 효과가 한 단계 올라와 제출 가능한 플레이 루프에 가까워졌다. 이전 61% 수준에서 68%로 올려 평가한다.

다만 risk는 여전히 high다. 이유는 기능량 증가보다 검증 자동화 증가가 느리기 때문이다. 특히 무기 각성, item level, GPU atlas, restart reset은 시연 중 바로 보이는 핵심 경로다. 이 경로가 Play Mode smoke와 Game View 캡처 기준으로 닫히지 않으면 다음 변경에서 회귀 비용이 크게 늘어난다.

## 4. 다음 권장사항
- 무기 각성 전 경로를 Play Mode 스모크에 고정한다.
- GPU atlas 렌더 기본 경로를 Game View 캡처 기준으로 고정한다.
- Silver Bell orbit 위치·hitbox·owner 조회를 실제 전투 스모크로 닫는다.
- 아이템 선택 계약을 enum-indexed 표와 실제 입력 스모크로 검증한다.
- 신규 무기별 튜닝 값을 `VerticalSliceConfig` 기준으로 일원화한다.
- `EnemySeparationSystem` 고밀도 프레임 비용을 계측 기준으로 잠근다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`와 migration 문서들이 코어 루프와 재구현 방향을 설명하지만, 신규 무기 각성별 의도와 밸런스 목표가 아직 테스트 가능한 수치로 충분히 닫히지 않았다.

technical은 8점이다. assembly boundary, folder layout, verification loop, GPU troubleshooting, resource asset map이 있어 신규 기여자가 구조를 따라갈 수 있다. 이번 asset map 추가도 좋은 신호다.

spec은 7점이다. legacy runtime, weapon/equipment, scene-prefab baseline은 실무 참고가 가능하다. 그러나 atlas row/index, item slot, weapon awakening 조건처럼 이번에 확장된 계약은 자동 검증표 수준까지 더 내려와야 한다.

# GhostMarch_DOTS — DOTS smoke 확장 스냅샷

## 1. 주요 변경사항
- `VerticalSliceSmokeTestRunner`가 추가되어 Play Mode에서 생존, 사격, 명중, 피해, 적 사망, EXP, 레벨업, 아이템 적용, 재시작 초기화를 자동 확인한다.
- `DamageApplySystem`의 체력 clamp 기준이 수정되어 피해가 실제로 반영되는 경로가 복구됐다. 이 변경은 핵심 루프 blocker를 직접 제거했다.
- 적 접촉 피해가 기본 DPS와 몹 타입 배율로 분리됐다. 이전처럼 배율을 초당 피해 필드에 섞는 구조보다 밸런싱 의도가 명확하다.
- Play Mode 종료 시 해제된 DOTS world를 만지는 예외를 방어했고, contact probe enemy가 정상 death/reward 경로를 타도록 `EnemyTypeData`를 추가했다.
- 협업 문서와 TODO 문서가 smoke 검증 상태, reload 운영 규칙, 향후 학습 자료 계획을 반영했다.

## 2. 코드 품질 리뷰
이번 변경의 가장 큰 장점은 “돌아간다”는 주장을 문서가 아니라 자동 smoke로 일부 증명하기 시작했다는 점이다. `DamageApplySystem`의 `ValueRO.Current` 사용 버그를 `ValueRW.Current` 기준 clamp로 고친 것도 정확하다. 기존 코드는 피해를 뺀 뒤 이전 값을 기준으로 다시 clamp해 체력이 줄지 않는 치명적 결함이었고, 이번 수정은 수직 슬라이스의 전투 루프 신뢰도를 크게 올린다.

다만 `VerticalSliceSmokeTestRunner.Run()`이 `ConfigureSceneForSmoke()`를 Play Mode 진입 전에 실행하면서 scene object의 serialized 값을 직접 바꾸고, 없으면 `VerticalSliceItemAutoPickBridge`까지 추가한다. 왜 문제냐면 테스트 보정값이 저장 씬에 묻어 다음 수동 테스트나 커밋에 섞일 수 있기 때문이다. 개선 방향은 Play Mode 진입 후 런타임 인스턴스에만 주입하거나, 테스트 전용 prefab/scene variant를 별도로 두는 것이다. smoke는 검증 도구여야지 씬 상태를 몰래 바꾸는 도구가 되면 안 된다.

자료구조/복잡도 관점에서는 이번 diff의 새 hot path 문제는 크지 않다. `VerticalSliceItemAutoPickBridge`의 `_candidates` 재사용은 적절하고, 후보 수가 8개로 고정되어 선형 순회 비용도 의미 있는 병목이 아니다. 단, 문서에 남아 있는 `VerticalSliceGPURenderBridge`의 `ToEntityArray(Allocator.Temp)` 반복 할당은 고밀도 생존 테스트 전에는 정리해야 한다. 현재 비용은 프레임별 임시 배열 생성이고, 권장 흐름은 chunk iteration 또는 재사용 가능한 render DTO buffer다.

## 3. 진행도 평가
네트워킹 코드가 확인되지 않으므로 구현 경로는 비네트워크 단일 플레이 DOTS 수직 슬라이스로 판정한다. 이번 범위는 기능 추가보다 안정화와 검증 자동화에 가깝지만, 핵심 전투 루프가 자동 smoke로 재현된다는 점에서 제출 가능성은 분명히 전진했다. 진행도는 67로 본다.

위험도는 low가 아니라 medium이다. 이유는 핵심 루프는 좋아졌지만 smoke runner가 씬을 오염시킬 수 있고, 실제 아이템 선택 UI·오디오·HUD·세이브·모바일 화면 검증이 아직 남아 있기 때문이다. 지금 하루 정도 테스트 격리와 검증 범위를 정리하면 이후 디버깅 비용을 크게 줄일 수 있다.

## 4. 다음 권장사항
- `VerticalSliceSmokeTestRunner`의 씬 수정 경로를 Play Mode 런타임 주입 또는 테스트 전용 scene variant로 격리한다.
- 오디오, HUD, 세이브 bridge, 모바일 Game View까지 smoke 범위를 확장해 실제 발표 경로를 검증한다.
- `VerticalSliceItemAutoPickBridge`를 실제 선택 UI로 교체한다. 자동 선택은 smoke 전용 보조 장치로만 남겨야 한다.
- smoke 실패를 `Debug.LogError`에만 의존하지 말고 CI나 batch 실행에서 읽을 수 있는 결과로 남긴다.
- `VerticalSliceAudioBridge`의 SFX 배열 인덱스와 Inspector 길이 검증을 추가해 런타임 배열 오류를 막는다.

## 5. 문서화 상태
design은 7점이다. 플레이 경험 목표와 DOTS 전환 방향이 누적되어 단순 아이디어 수준은 넘었다. 다만 레벨·아이템·적 조합의 의사결정 이유가 `_sample/docs`처럼 표준 사양으로 고정되지는 않았다.

technical은 7점이다. handoff, migration TODO, reload 운영 규칙, smoke 검증 결과가 계속 갱신되는 점은 좋다. 신규 작업자가 흐름을 따라갈 수 있는 수준이나, 시스템별 데이터 흐름도와 자동 테스트 실행 절차는 더 명확해야 한다.

spec은 5점이다. 기능 목록과 TODO는 있으나 입력, 상태, 결과, 밸런싱 수치가 테스트 체크리스트로 바로 쓰일 만큼 정리되지는 않았다. 수직 슬라이스 제출 전에는 적/아이템/무기/피해 공식 표가 필요하다.

## 6. Backlog
- Legacy 이동 후 하드코딩 경로와 serialized 참조 회귀 테스트는 아직 명시적으로 해결되지 않았다.
- Unity 버전·패키지 업그레이드성 변경을 별도 커밋으로 격리하는 운영 원칙은 계속 유지해야 한다.
- TextMesh Pro와 ShaderGraph 리소스 대량 변경의 의도 문서화가 남아 있다.
- Aseprite·AudioGen 대량 자산은 출처와 실제 사용처가 여전히 추적 가능해야 한다.
- `VerticalSliceGPURenderBridge`의 per-frame query allocation 제거는 고밀도 테스트 전까지 남은 성능 부채다.

## 7. 이전 Backlog 해결
- 적 접촉 피해 단위를 기본 DPS와 몹 타입 배율로 분리한 항목은 해결됐다. `PlayerContactDamage`에 `DamageToPlayerMultiplier`가 추가됐고, 적 스폰은 배율을 별도 필드에 기록하며 접촉 피해 시스템이 기본 DPS와 배율을 곱한다.

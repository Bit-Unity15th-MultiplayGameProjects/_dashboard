# GhostMarch_DOTS — 작두 재설계 스냅샷

## 1. 주요 변경사항
- 작두가 이동 투사체에서 플레이어 추적형 근접 slash로 바뀌었다. `LastDirection`, `HasHit`, slash snapshot age/lifetime 추가는 설계 의도가 비교적 명확하다.
- 기본 무기들이 `Level == 0`에서 발사되지 않도록 격리됐다. 초기 플레이가 전체 무기 난사 상태에서 선택 성장 구조로 이동한 점은 긍정적이다.
- GPU 렌더 브릿지에 작두 전용 크기, 종횡비, 진행도 기반 atlas frame 선택이 들어갔다.
- 씬과 mob asset에 극단적인 테스트 수치가 들어갔다. 디버그 의도는 이해되지만 제출용 기본값과 섞인 상태는 위험하다.
- handoff 문서는 최근 작업을 잘 남겼으나, 조사 완료 문서 경로가 diff 통계에 보이지 않아 실제 repo 반영 여부 확인이 필요하다.

## 2. 코드 품질 리뷰
이번 변경의 핵심 전진은 작두의 책임 분리다. `JakduFireSystem`은 생성과 방향 기억을, `JakduHitSystem`은 추적 위치·부채꼴 판정·흡혈·피해 통계를 맡는다. 다만 `spawnPos` 오프셋 `1.0f`, 일반 slash 전방 dot 기준 `0.5f`, GPU 회전 보정 `270f`, 72/42 종횡비 같은 값이 코드 안에 흩어져 있다. 왜 문제냐면 작두 크기·판정·시각 위치를 맞출 때 한쪽만 수정하는 회귀가 쉽다. 개선은 `StrawCutterWeaponState` 또는 `VerticalSliceConfig`에 “시각 오프셋, 판정 arc, sprite aspect”를 명시하고 smoke 판정도 같은 값을 읽게 하는 것이다.

더 심각한 신호는 `DotsVerticalSlice.unity`와 `wisp.asset`의 수치다. `PlayerHealth: 500000000`, `EnemySpawnInterval: 0.001`, `health: 50000`은 실험용 부하/관찰 수치로 보이며, 기본 씬에 남으면 실제 밸런스와 회귀 테스트를 오염시킨다. 이 상태에서는 “게임이 안정적이다”가 아니라 “죽지 않게 만든 실험 환경에서만 보인다”가 된다. 테스트 전용 config나 별도 stress scene으로 분리해야 한다.

자료구조/복잡도 관점에서는 `SilverBellOrbitSystem`이 `Level == 0`인 플레이어마다 모든 orbit projectile을 순회해 owner를 비교한다. 현재는 수량이 작을 수 있으나 고밀도 전투와 여러 owner를 고려하면 플레이어 수×orbit 수의 선형 반복이 된다. 단기 개선은 비활성 전환 이벤트 시 한 번만 정리하거나 owner별 orbit buffer/lookup으로 고정하는 것이다. 그래야 Level=0 가드가 hot path 정리 루프로 변하지 않는다.

## 3. 진행도 평가
네트워킹 코드는 현재 수직 슬라이스 평가에 직접 포함되지 않아 구현 경로는 비네트워크 단일 플레이 DOTS 수직 슬라이스로 본다. 반복 플레이 루프, 무기 성장, GPU presentation, smoke runner가 이미 있어 진행도는 64로 평가한다.

다만 이번 변경은 체감 전투 품질을 올린 동시에 검증 부채를 늘렸다. 작두·무기 해금·GPU atlas·Silver Bell 정리 같은 핵심 변경이 실제 Play Mode smoke로 닫히지 않았고, 기본 씬에 stress 수치가 섞였다. 위험도는 high까지는 아니지만, 지금 하루를 들여 테스트 기준을 고정하지 않으면 다음 변경부터 디버깅 비용이 급증한다.

## 4. 다음 권장사항
- Jakdu slash는 `HasHit`, 120도 arc, grace 이후 판정을 한 smoke로 묶어야 한다.
- Level=0 시작 상태에서 선택 입력으로 무기가 해금되고 발사되는 경로를 고정해야 한다.
- GPU 렌더는 Game View 캡처 기준으로 작두 atlas frame과 fallback 조건을 검증해야 한다.
- Silver Bell은 비활성 정리뿐 아니라 재획득 시 orbit 생성·hitbox 초기화를 같이 확인해야 한다.
- 씬에 들어간 5억 HP, 0.001초 spawn 같은 수치는 stress config로 분리해야 한다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`와 migration 문서들이 코어 방향과 전환 이유를 설명하지만, 최신 무기별 체감 목표와 밸런스 의도는 아직 코드 수치와 완전히 연결되지 않는다.

technical은 8점이다. asmdef 경계, DOTS 전환 전략, GPU presentation troubleshooting, handoff 문서가 실무적으로 유용하다. 신규 유입자가 구조를 읽는 데 충분하지만, 최신 flow-field 조사 문서의 실제 커밋 여부는 확인이 필요하다.

spec은 7점이다. `first-vertical-slice`, `dots-rebuild-todo`, reference baseline 문서가 체크리스트 역할을 한다. 다만 `_sample/docs` 수준의 완성 사양처럼 무기별 데이터 표, 입력-상태-결과 매핑, smoke 판정표가 한곳에 고정되지는 않았다.

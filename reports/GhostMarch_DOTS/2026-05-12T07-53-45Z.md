# GhostMarch_DOTS — DOTS 무기 확장 스냅샷

## 1. 주요 변경사항
- 레벨업 선택 UI가 `VerticalSlicePauseService` 토큰으로 `Time.timeScale`을 관리하도록 바뀌었다. 중첩 pause 조기 해제 위험을 줄인 좋은 방향이다.
- `ItemSelectionBridge`가 선택 이벤트를 쓴 뒤 `ItemApplySystem` 소비를 기다리도록 바뀌었다. 연속 레벨업 후보가 낡은 상태로 계산되던 핵심 위험이 해소됐다.
- `VerticalSliceItemDefinitions`가 추가되어 item id, max level, 업적 key, 수치 preview/apply 계산이 일부 중앙화됐다.
- Silver Bell 궤도 무기가 추가됐고, 기본 spawn 밀도도 0.8초/20마리로 올라갔다. 수직 슬라이스 전투 밀도는 분명히 전진했다.
- 무한 배경 타일은 카메라가 새 tile origin에 들어갈 때만 갱신하고 renderer를 캐싱한다. 이전 성능 지적을 정확히 받아들인 변경이다.

## 2. 코드 품질 리뷰
이번 변경은 이전 지적 중 UI pause 소유권, 연속 선택 apply 순서, 배경 타일 갱신 조건을 실제 코드로 닫았다. 특히 `VerticalSliceInfiniteBackgroundBridge`의 `GetComponent` 반복 제거와 origin 변경 시 갱신은 Unity hot path 관점에서 올바른 개선이다. `ItemApplySystem`과 UI preview가 같은 helper를 보게 된 점도 유지보수성이 좋아졌다.

다만 Silver Bell 구현은 아직 빠른 수직 슬라이스 코드에 가깝다. `SilverBellOrbitSystem`은 orbit projectile을 ECB로 만든 뒤 그 프레임 안에서는 실제 player 기준 위치로 갱신하지 못한다. 생성 엔티티가 `LocalTransform.FromPosition(float3.zero)`로 들어가므로 첫 렌더/첫 hit 판정에서 원점 artifact가 생길 수 있다. 생성 시점에 player 위치와 각도를 넘겨 초기 transform을 확정하거나, 생성 직후 hit/render 대상에서 한 프레임 제외하는 보정이 필요하다.

자료구조/복잡도 측면에서는 `SilverBellOrbitSystem`이 매 프레임 player마다 전체 orbit projectile을 세고 다시 전체 orbit을 순회한다. 현재는 단일 플레이어라 비용이 작지만, orbit 수와 weapon 수가 늘면 O(P×O) 구조가 된다. owner별 buffer 또는 단일 pass에서 owner/count/update를 처리하는 구조로 바꾸는 편이 DOTS 성장 경로에 맞다. 반대로 item id 중앙화는 아직 완성형 table이 아니며 label, level, achievement key switch가 남아 있어 id 변경 취약성이 남는다.

## 3. 진행도 평가
네트워킹 코드는 아직 해당 없음이다. 따라서 이번 평가는 single-player DOTS 수직 슬라이스 기준으로 본다. 전투 밀도, 레벨업 UI, item apply 순서, 신규 무기 1종까지 연결되어 반복 시연 가능한 축은 강화됐다. 진행도는 66으로 본다.

위험도는 medium이다. 핵심 루프는 전진했지만 새 weapon hitbox는 자동 검증 근거가 부족하고, 실제 선택 UI 입력 smoke가 아직 없다. 지금 바로 1일 정도 검증과 초기화 보정을 넣으면 이후 무기별 성장 확장 비용을 크게 줄일 수 있다.

## 4. 다음 권장사항
- `SilverBellOrbitSystem`에서 생성 직후 orbit projectile의 위치와 hitbox를 player 기준으로 초기화한다.
- `VerticalSliceItemSelectionBridge`를 자동 선택 bridge 없이 키보드 1/2/3 또는 클릭 입력으로 Play Mode smoke에 연결한다.
- Silver Bell orbit projectile 조회를 owner별 전체 재검색이 아닌 단일 pass 구조로 정리한다.
- item/weapon 성장 정의를 switch helper에서 enum-indexed 데이터 테이블로 확장해 UI와 system drift를 막는다.
- `WeaponFireSystem`의 spread angle, spawn offset을 상수가 아니라 무기별 tuning config로 옮긴다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`에 DOTS 무기 성장/각성 기준과 회복 아이템 구분이 추가되어 의사결정 근거가 좋아졌다. 다만 무기별 수치 목표와 플레이 감각 기준은 아직 표준 사양서 수준까지는 아니다.

technical은 8점이다. assembly boundary, folder layout, migration strategy, serialized baseline, handoff 문서가 있어 신규 유입자가 구조를 따라갈 수 있다. 이번 handoff 업데이트도 실제 변경과 잘 맞는다.

spec은 7점이다. `dots-rebuild-todo.md`, reference 문서, legacy contract 문서가 테스트 체크리스트처럼 쓸 수 있는 수준이다. 단, 최종 배경 타일링, SFX 배열 검증, asset provenance 같은 release cleanup 사양은 아직 미완료다.

## 6. Backlog

- Legacy 이동 후 serialized baseline은 있으나 하드코딩 경로 회귀를 자동으로 잡는 절차는 계속 유지해야 한다.

- TextMesh Pro와 ShaderGraph 대량 리소스 변경은 의도와 사용처 문서화가 아직 닫히지 않았다.

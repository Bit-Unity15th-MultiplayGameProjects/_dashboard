# GhostMarch_DOTS — FlowField 군중 이동 스냅샷

## 1. 주요 변경사항
- `FlowFieldUpdateSystem`이 추가되어 20×20 격자, 3프레임 주기, 밀도 비용 기반 방향 필드를 생성한다.
- `EnemyChaseSystem`은 flow field 샘플링과 separation velocity 기반 chase-yield를 사용하도록 확장됐다.
- `EnemySeparationSystem`은 위치 보정에서 속도 적분 방식으로 바뀌었고, 플레이어 contact zone hard clamp가 들어갔다.
- 씬 스트레스 값이 `MaxEnemies 5000`, `PlayerCollisionRadius 0.15`로 조정됐고, Unity MCP 패키지와 ProjectSettings 자동 직렬화 변경이 별도 커밋으로 들어갔다.
- 시행착오 문서와 handoff 문서가 보강되어 왜 flow field와 velocity separation으로 갔는지 추적 가능해졌다.

## 2. 코드 품질 리뷰
이번 변경의 좋은 점은 책임 분리가 이전보다 선명해졌다는 점이다. Flow field는 큰 방향, chase는 이동 의지, separation은 겹침 해소를 맡는다. `SeparationVelocity`를 별도 컴포넌트로 둔 결정도 DOTS 관점에서 타당하다. 위치를 매 프레임 직접 튕기는 방식보다 상태가 관측 가능하고 튜닝 노브도 명확하다.

다만 위험 신호가 크다. `EnemyChaseSystem`과 `EnemySeparationSystem` 사이의 실행 순서가 코드로 고정되어 있지 않다. 현재 설계는 chase 이후 separation이 contact clamp를 마무리한다는 전제에 기대는데, 이 전제가 깨지면 플레이어 접촉권 안으로 다시 밀고 들어간 뒤 damage 판정이 실행될 수 있다. 의도한 시스템 순서는 attribute로 명시해야 한다.

자료구조/복잡도 측면에서는 `FlowFieldUpdateSystem`이 3프레임마다 모든 `EnemyTag`를 순회해 density를 누적한다. 300마리 기준 조사 문서는 설득력이 있지만, 이번 씬은 5000마리 스트레스 값으로 올라갔다. 현재 구조는 O(n) 메인스레드 순회가 flow 계산 앞에 붙으므로 Profiler 기준을 먼저 잡고, 필요하면 density write를 Job/버퍼 분할 집계로 옮기는 것이 맞다. 추측성 최적화가 아니라 이번 diff가 직접 만든 비용이다.

## 3. 진행도 평가
네트워킹 코드는 현재 평가 대상이 아니므로 단일 플레이 DOTS 수직 슬라이스 기준으로 본다. 군중 이동의 핵심 난제였던 밀집·진동 대응은 분명히 전진했다. 하지만 “컴파일 통과”와 “5000 enemy에서 반복 시연 가능” 사이에는 아직 거리가 있다.

진행도는 66으로 본다. 낮은 레벨의 수직 슬라이스는 동작 가능한 단계에 들어왔지만, Play Mode smoke와 Game View 캡처, 성능 계측, 디버그 밸런스 분리가 아직 제출 안정성을 막는다. 특히 이번 변경은 시스템 수가 늘어난 만큼 검증 자동화 없이 감으로 튜닝하면 다음 주에 비용이 급증한다.

## 4. 다음 권장사항
- FlowField, Chase, Separation 실행 순서를 명시하고 5000 enemy Play Mode 스모크로 contact clamp와 jitter를 고정한다.
- Jakdu, Silver Bell, Level=0 무기 시작, 아이템 선택 UI를 실제 입력 기반 스모크로 닫는다.
- GPU atlas 기본 렌더는 Game View 캡처와 snapshot-empty 조건으로 검증해 debug fallback 의존을 끊는다.
- `FlowFieldUpdateSystem`의 전체 Enemy 순회 비용을 Profiler 수치로 기록하고 Job 분리 필요 여부를 판정한다.
- 씬에 남은 5억 HP, 0.001초 spawn interval, 5000 enemy stress 값을 테스트 config로 분리한다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`와 migration 문서가 방향성을 제공하지만, 현재 DOTS 수직 슬라이스의 플레이 경험 목표와 밸런스 판정표가 완전히 연결되지는 않았다.

technical은 8점이다. assembly boundary, folder layout, GPU troubleshooting, flow-field investigation이 신규 투입자에게 실제 맥락을 준다. 이번 flow-field 시행착오 문서는 특히 좋다. 실패한 접근과 버린 이유가 남아 있어 반복 실수를 줄인다.

spec은 7점이다. scene/prefab baseline, weapon legacy spec, resource map은 실무적으로 쓸 수 있다. 다만 현재 씬 수치와 테스트 기준이 섞여 있어 “이 값이 시연 기본값인가, 스트레스 조건인가”를 자동 판정하기 어렵다.

# GhostMarch_DOTS — GPU 기본 프레젠테이션 스냅샷

## 1. 주요 변경사항
- `VerticalSliceDebugPresentation`의 fallback quad 기본값을 끄고, GPU bridge를 정상 프레젠테이션 기본 경로로 승격했다.
- `DotsVerticalSlice.unity` 저장값도 같은 방향으로 맞춰 씬 실행 시 debug quad가 기본 렌더를 가리지 않게 했다.
- `VerticalSliceSceneSetup`에 복구 로직을 추가해 setup 실행 시 fallback 설정을 끄고 비활성화된 GPU bridge를 다시 켠다.
- handoff와 presentation guideline 문서도 “fallback은 opt-in debug view”라는 새 운영 기준으로 갱신했다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 좋다. 이전 상태처럼 debug fallback이 기본 경로를 점유하면 실제 GPU 렌더 문제가 숨는다. 기본 시연 경로를 하나로 정한 것은 수직 슬라이스 안정성 측면에서 올바른 결정이다. `ConfigureDebugPresentationDefaults`도 에디터 setup 책임 안에 들어가므로 위치는 자연스럽다.

다만 `FindProperty("renderFallbackQuads")`, `FindProperty("disableGpuBridgeWhenFallbackQuadsRender")`는 serialized field 이름 문자열에 직접 의존한다. 필드 rename 시 조용히 무시될 수 있어 위험하다. 왜 문제냐면 setup repair가 실패해도 컴파일 오류가 나지 않고, 씬 기본값이 다시 debug fallback으로 돌아갈 수 있기 때문이다. 개선하려면 해당 필드명을 상수화하거나, 가능하면 editor 전용 public/internal 설정 메서드로 캡슐화해 rename 안전성을 높이는 편이 낫다.

자료구조/복잡도 관점에서는 이번 diff 자체에 새 hot path나 반복 할당은 없다. 단, 이전 TODO인 `SilverBellOrbitSystem` owner별 orbit 재검색은 여전히 구조적 비용 위험으로 남는다. orbit 수가 늘면 O(P×O) 조회가 프레임 비용으로 누적되므로 owner 기준 단일 패스 캐시나 NativeParallelMultiHashMap 계열로 조회 흐름을 고정하는 것이 맞다.

## 3. 진행도 평가
네트워크 구현은 아직 실제 범위가 아니므로 해당 없음으로 본다. 현재 프로젝트는 DOTS 수직 슬라이스를 제출 가능한 단일 플레이 루프로 만드는 단계이며, 이번 커밋은 렌더 경로 혼선을 줄인 실질적 전진이다. 진행도는 63으로 본다.

다만 이 전진은 기본값 정리에 가깝고, 화면 기준 자동 검증은 아직 부족하다. GPU 렌더가 기본이 됐다는 선언은 Game View 캡처, material slot 누락 처리, 모바일 aspect 스모크까지 연결돼야 제출 안정성으로 인정된다. 리스크는 medium이다.

## 4. 다음 권장사항
- Silver Bell 생성 프레임의 orbit 위치와 hitbox 초기화를 먼저 검증해야 한다.
- 아이템 선택 UI는 자동 선택 bridge 없이 실제 키·클릭 입력 Play Mode 스모크로 막아야 한다.
- GPU 기본 렌더 경로는 Game View 캡처 기반 smoke로 고정해야 한다.
- `SilverBellOrbitSystem`의 owner별 orbit 조회는 단일 패스 구조로 줄여야 한다.
- 아이템·무기 성장 정의와 projectile spread는 데이터 테이블·config로 옮겨야 한다.

spec은 7점이다. `first-vertical-slice.md`, `scene-prefab-serialized-baseline.md`, `weapon-equipment-legacy-spec.md`는 테스트 체크리스트로 쓸 수 있는 수준이다. 그러나 최종 art material, SFX 배열 계약, infinite background 사양은 아직 미완료 항목으로 남아 있다.

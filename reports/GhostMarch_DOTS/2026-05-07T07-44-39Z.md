# GhostMarch_DOTS — DOTS 버티컬 슬라이스 스냅샷

## 1. 주요 변경사항
- `HelloDots` 샘플을 제거하고 플레이어, 적, 투사체, 소울, 경험치 중심의 DOTS 컴포넌트와 시스템을 추가했다.
- `DotsVerticalSlice` 및 베이킹 Host/SubScene 을 추가해 SubScene 기반 authoring 경로를 검증하려는 방향이 보인다.
- `VerticalSliceBootstrap`, 입력·디버그·접촉 Probe 브리지가 추가되어 MonoBehaviour 와 DOTS 월드 사이의 임시 연결이 마련됐다.
- `first-vertical-slice.md` 에 Legacy 기준값 대비 현재 설정값 표가 추가된 점은 긍정적이다. 임시값을 숨기지 않은 태도가 좋다.

## 2. 코드 품질 리뷰
가장 큰 위험은 `VerticalSliceAuthoring.cs` 와 `VerticalSliceBootstrap.cs` 에 같은 MobData 읽기 로직이 두 벌로 생겼다는 점이다. 둘 다 `JsonUtility.ToJson`, reflection field lookup, `collSize` 반지름 변환, exp 올림 처리를 직접 수행한다. 문제는 Baker 와 런타임 bootstrap 이 시간이 지나며 서로 다른 규칙으로 갈라질 가능성이 높다는 데 있다. `MobData -> VerticalSliceMobSpawnEntry/config patch` 변환기를 별도 공용 계층으로 빼고, Baker 와 런타임은 그 결과만 소비하게 하는 편이 맞다.

`DotsVerticalSliceBakingSceneBuilder.cs` 는 Legacy mob asset 경로를 문자열 배열로 직접 들고 있다. 지금은 검증 씬 생성 목적이라 이해할 수 있으나, 이전 Legacy 이동 이슈와 정면으로 충돌한다. 왜 문제냐면 폴더명, 파일명, 공백 포함 asset 명이 바뀌는 순간 메뉴 검증이 런타임이 아니라 에디터에서 조용히 빈 테이블로 약화될 수 있기 때문이다. 검증용 `ScriptableObject` 또는 GUID 기반 reference table 로 고정하고, 누락 시 명시적으로 실패해야 한다.

자료구조/복잡도/할당 관점에서는 `VerticalSliceDebugPresentation.SyncFirstEntity` 와 `VerticalSliceDebugStateBridge.LateUpdate` 의 `ToEntityArray(Allocator.Temp)` 반복이 확인된다. 디버그 전용이라 당장 P0 는 아니지만 매 프레임 여러 Query 배열을 만들고 첫 원소만 읽는 구조는 비용 대비 의미가 낮다. singleton entity 캐시, `TryGetSingleton`, 또는 상태 브리지 한 곳에서 snapshot 을 구성하고 presentation 은 그 값만 렌더링하는 흐름이 더 단순하다. DOTS 전환 중에는 이런 “검증용 코드니까 괜찮다”가 실제 플레이 루프에 남는 경우가 많아 지금 정리해야 한다.

## 3. 진행도 평가
이전보다 전진은 분명하다. 빈 DOTS 샘플 단계에서 벗어나 이동, 스폰, 사격, 피격, 사망, 소울 흡수, 경험치까지 최소 루프의 골격을 세웠다. 특히 SubScene authoring 과 라이브 월드 검증 메뉴를 함께 만든 것은 DOTS 마이그레이션에서 좋은 방향이다.

다만 아직 “게임 기능”이라기보다 “검증 가능한 기술 슬라이스” 단계다. Legacy 데이터와의 parity 는 일부 수치만 연결됐고, 브리지·디버그·에디터 메뉴가 실제 구조보다 앞서 늘어나고 있다. 현재 진행도는 약 34%로 본다. 다음 단계에서 임시값을 사양과 자동 검증으로 고정하지 않으면, DOTS 루프는 돌지만 원작 동작과 맞는지 설명하기 어려워진다.

## 4. 다음 권장사항
- `VerticalSliceAuthoring` 과 `VerticalSliceBootstrap` 의 MobData 반영 로직을 단일 변환기로 묶어라.
- 검증 씬 생성기의 Legacy asset 문자열 경로를 데이터 참조 또는 GUID 표로 교체하라.
- 디버그 표시 계층의 매 프레임 `ToEntityArray` 반복을 snapshot 중심 구조로 줄여라.
- `first-vertical-slice.md` 의 임시 수치 표를 테스트 판정표로 승격하라.
- Baker/LiveWorld 검증 메뉴를 CI 또는 Unity Test Runner 에서 재현 가능한 경로로 연결하라.

## 5. 문서화 상태
design 은 4점이다. 코어 루프와 플레이 경험의 “왜”가 아직 얇고, 이번 변경도 기술 슬라이스 중심이라 디자인 의사결정 문서로 보기는 어렵다.

technical 은 6점이다. `docs/dots.md`, `dots-rebuild-todo.md`, `dots-verification-loop.md` 갱신과 assembly boundary 문서가 있어 신규 개발자가 방향은 잡을 수 있다. 다만 reflection 기반 데이터 브리지와 SubScene 로딩 전략의 책임 경계가 더 명확해야 한다.

spec 은 5점이다. `first-vertical-slice.md` 의 수치 비교표는 좋은 출발이다. 하지만 아직 검증 체크리스트, 합격 기준, Legacy parity 항목이 자동 테스트와 연결되지 않아 사양서라기보다 마이그레이션 노트에 가깝다.

## 6. Backlog
- Unity 버전·패키지 업그레이드성 변경 격리는 이번 diff 에서 해결 근거가 없다.
- TextMesh Pro와 ShaderGraph 리소스 대량 변경 의도 문서화도 이번 범위에서 확인되지 않는다.
- Aseprite·AudioGen 대량 자산의 출처와 실제 사용처는 여전히 별도 추적이 필요하다.
- Legacy 이동 후 하드코딩 경로 회귀 테스트는 오히려 새 검증 씬 빌더의 문자열 경로 때문에 중요도가 올라갔다.
- 업적 코드, 캐릭터 해금, SFX 인덱스 매핑 표는 이번 DOTS 슬라이스 범위 밖으로 남아 있다.

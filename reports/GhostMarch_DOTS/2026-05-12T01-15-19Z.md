# GhostMarch_DOTS — DOTS 수직 슬라이스 안정화 스냅샷

## 1. 주요 변경사항
- DOTS 수직 슬라이스의 카메라가 플레이어를 따라가도록 변경되고, 기본 클램프가 꺼져 infinite-map 방향의 첫 단계가 들어갔다.
- GPU 렌더 브리지에 런타임 fallback material 이 추가되어 최종 art material 이 없어도 화면에 주요 오브젝트가 보이도록 보강됐다.
- Debug presentation 이 단일 enemy/projectile/soul 만 보여주던 구조에서 풀 기반 다중 표시로 개선됐다.
- GameOver, Victory 에 이어 1080x1920 모바일 HUD smoke 테스트가 추가되어 세로 화면 레이아웃 회귀를 잡기 시작했다.
- 문서에는 현재 구현 상태와 남은 GPU/히트 이펙트/무한맵 작업이 이전보다 명확히 정리됐다.

## 2. 코드 품질 리뷰
이번 변경은 “플레이 가능한 DOTS 수직 슬라이스를 끊기지 않게 보이게 한다”는 목표에는 효과적이다. 특히 `VerticalSliceSceneSetup.cs` 가 기존 bootstrap 을 단순 거부하지 않고 누락 bridge 를 복구하는 방향으로 바뀐 점은 좋다. 오래된 씬이 계속 남는 Unity 프로젝트에서는 생성 메뉴보다 repair 메뉴가 훨씬 실전적이다.

다만 `VerticalSliceBootstrap.cs` 의 `EnsureConfigEntities()` 는 게임 설정, 웨이브 밸런스, 성장 곡선을 코드 상수로 직접 생성한다. 지금은 fallback 으로 유용하지만, 이 상태가 오래가면 기획 수치 변경마다 코드 리뷰와 재컴파일이 필요해진다. 개선 방향은 명확하다. 임시 fallback 은 유지하되, 기준 값은 ScriptableObject authoring 또는 baker 입력으로 분리하고, fallback 은 “데이터 누락 시 개발용 기본값”임을 로그와 문서로 분명히 해야 한다.

자료구조/복잡도 관점에서는 `VerticalSliceDebugPresentation.SyncEntityPool()` 이 매 LateUpdate 마다 `query.ToEntityArray(Allocator.Temp)` 를 만든다. 현재 debug fallback 용도라 큰 문제는 아니지만, fallback quads 를 장시간 켜 두는 구조라면 매 프레임 임시 배열 할당 비용이 계속 발생한다. 이미 GPU 경로는 snapshot buffer 기반으로 개선되어 있으므로, debug 경로도 같은 snapshot 을 읽거나 entity count 변화 시에만 풀을 조정하는 흐름이 더 일관되고 비용도 낮다.

## 3. 진행도 평가
네트워킹 코드는 이번 범위에서 확인되지 않으므로 구현 경로 분류는 해당 없음이다. 진행도는 네트워크 보정 없이 단일 플레이 수직 슬라이스 기준으로 본다. Core, save, victory, mobile aspect smoke 가 문서상 통과했고, 화면 표시 fallback 과 카메라 추적도 보강되어 반복 시연 가능성은 이전보다 높아졌다.

다만 최종 GPU material/atlas, infinite-map wrapping, fallback/debug/GPU 경로 정리가 남아 있다. 즉 “보이게 만드는 단계”는 상당히 전진했지만 “최종 경로로 검증된 단계”는 아직 아니다. 제출 가능한 수직 슬라이스 기준 진행도는 68%로 판단한다.

## 4. 다음 권장사항
- GPU fallback 과 debug quad 중 발표·제출에서 사용할 최종 렌더 경로를 하나로 확정한다.
- `VerticalSliceBootstrap` 의 밸런스 기본값을 ScriptableObject 또는 authoring 데이터로 분리한다.
- 모바일 HUD smoke 가 단순 RectTransform 충돌이 아니라 실제 텍스트 가독성까지 보게 보강한다.
- infinite-map 의 ground/background wrapping 규칙을 먼저 문서화한 뒤 구현한다.
- GPU material 과 atlas row/index 계약을 문서와 인스펙터 기본값에 고정한다.

## 5. 문서화 상태
design 점수는 5다. 수직 슬라이스의 방향과 infinite-map 전환 의도는 보이지만, 왜 현재 방식의 카메라 추적과 wrapping 정책을 선택하는지에 대한 디자인 판단은 아직 부족하다.

technical 점수는 7이다. `handoff-state.md` 와 `dots-rebuild-todo.md` 는 실제 구현 상태, smoke 결과, 남은 기술 작업을 잘 따라가고 있다. 신규 작업자가 현재 위험 지점을 파악하는 데 실용적이다.

spec 점수는 5다. 모바일 smoke, GPU material, hit effect 등의 체크 항목은 생겼지만 `_sample/docs` 수준의 데이터 규격, 입력-상태-결과 매핑, 수치 테이블로는 아직 부족하다. 테스트 체크리스트로 쓰기에는 일부만 충분하다.

## 6. Backlog
- Legacy 이동 후 하드코딩 경로와 serialized 참조 회귀 테스트 유지가 아직 확인되지 않았다.
- Unity 버전·패키지 업그레이드성 변경을 별도 커밋으로 격리하는 운용 원칙은 이번 diff 에서 해결 근거가 없다.
- Aseprite·AudioGen 대량 자산의 출처와 실제 사용처 명확화는 여전히 장기 관리 항목이다.
- TextMesh Pro와 ShaderGraph 리소스 대량 변경의 의도 문서화도 이번 범위에서는 해결되지 않았다.

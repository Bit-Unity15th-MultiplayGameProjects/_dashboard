# GhostMarch_DOTS — DOTS 수직 슬라이스 안정화 스냅샷

## 1. 주요 변경사항
- `DamageTextSpawnSystem`이 `DamageResultEvent`를 즉시 비우던 구조를 제거하고, `DamageResultEventClearSystem`을 추가해 데미지 숫자와 피격 이펙트가 모두 소비한 뒤 clear하도록 바꿨다.
- `VerticalSliceGPURenderBridge`는 quad mesh 크기와 matrix scale 이중 적용을 줄이고, 1023개 초과 인스턴스를 여러 배치로 나눠 그리도록 개선했다.
- DOTS 검증 씬에 URP 카메라 데이터와 `PlayerRegenRate` serialized 값이 반영됐다. 씬 저장 결과와 의도 변경이 섞인 diff이므로 후속 검증이 필요하다.
- 협업 문서와 마이그레이션 TODO가 최신 상태로 갱신됐다. 특히 남은 smoke test, GPU 할당, 기획 문서 gap을 명확히 기록한 점은 좋다.

## 2. 코드 품질 리뷰
이번 변경의 가장 좋은 판단은 `DamageResultEvent`의 수명 책임을 별도 시스템으로 분리한 것이다. 이전 구조는 첫 소비자인 `DamageTextSpawnSystem`이 버퍼를 비워 다른 presentation system의 결과가 실행 순서에 따라 사라질 수 있었다. 이는 DOTS에서 매우 위험한 조용한 실패 유형이다. 전용 clear system으로 후처리 위치를 고정한 방향은 맞다. 다만 이 변경은 시스템 순서가 품질의 핵심이므로, 숫자와 이펙트가 같은 프레임에 모두 생성되는지 검증 절차가 문서나 테스트로 남아야 한다.

`VerticalSliceGPURenderBridge`의 batch 분할도 의미 있는 전진이다. 기존 `count < MaxBatch` 조건은 1023개를 넘는 enemy, soul, digit, effect를 조용히 버렸을 가능성이 높다. 생존 장르에서 적과 이펙트 수가 늘어나는 순간 시각 정보가 누락되는 것은 단순 최적화 문제가 아니라 게임 판독성 문제다. batch helper로 분리한 것도 가독성 측면에서 괜찮다.

자료구조/복잡도/할당 관점에서는 아직 분명한 hot path 부채가 남아 있다. `VerticalSliceGPURenderBridge`가 여러 query에서 `ToEntityArray(Allocator.Temp)`를 매 프레임 호출한다. 현재 구조의 비용은 엔티티 목록 복사와 임시 할당 반복이고, 추천 흐름은 chunk iteration 또는 재사용 가능한 render DTO 버퍼다. 이 프로젝트는 대량 적·투사체·이펙트를 전제로 하므로, 이 비용은 실제 모바일 프레임 예산에서 의미 있게 커질 수 있다.

## 3. 진행도 평가
네트워킹 코드는 현재 리뷰 범위에서 해당 없음으로 판단한다. 따라서 `framework/hybrid/custom-network` 보정보다는 싱글플레이 DOTS 수직 슬라이스의 제출 가능성을 기준으로 본다. 컴파일 ready, 짧은 Play Mode 진입, 이벤트 순서 수정, 렌더 batch 절단 수정은 분명한 전진이다.

다만 확장 smoke가 아직 이동, 적 스폰, 명중, 레벨업, 아이템, 오디오/HUD/세이브 bridge, 재시작, 모바일 Game View까지 닫히지 않았다. 실행은 되는 단계로 올라왔지만 반복 시연 가능한 상태라고 단정하기에는 근거가 부족하다. 진행도는 63%가 타당하다.

## 4. 다음 권장사항
- 확장 수직 슬라이스 Play Mode smoke를 자동화하고 결과를 `handoff-state.md`에 남겨야 한다. 지금은 “짧게 켜짐”까지만 검증됐다.
- GPU 렌더 브릿지는 Game View에서 배경, 플레이어, 적, 숫자, 이펙트 크기와 1023개 초과 batch를 실제 화면으로 확인해야 한다.
- `ToEntityArray(Allocator.Temp)` 기반 per-frame 복사를 chunk 순회나 재사용 버퍼로 줄여야 한다. 고밀도 생존 루프에서는 늦게 고치면 비용이 커진다.
- `DamageResultEvent`는 두 presentation consumer가 같은 프레임에 결과를 받는 회귀 테스트 또는 최소 수동 체크리스트가 필요하다.
- 새로 추가한 기획 의도는 시간·DPS·스폰 압력 같은 측정 가능한 밸런스 목표로 내려야 한다.

## 5. 문서화 상태
design은 4/10이다. `game-design-spec.md`에 플레이 경험 목표, 적 역할, 무기 의도가 추가된 점은 좋다. 그러나 아직 “위기 뒤 카타르시스”가 어느 시점, 어떤 수치 변화, 어떤 웨이브 압력으로 구현되는지 부족하다.

technical은 7/10이다. 협업 규칙, handoff, DOTS TODO, deferred findings가 실제 작업 인수인계에 도움이 된다. 특히 씬 diff를 무조건 되돌리지 말라는 규칙은 Unity 협업에서 실용적이다. 다만 새 개발자가 문서만 보고 재현 가능한 smoke를 돌릴 수준은 아니다.

spec은 6/10이다. achievement, 캐릭터 unlock, 오디오/HUD/세이브 bridge 같은 세부 사양은 쌓이고 있다. 반면 실제 아이템 선택 UI, GPU material assignment, SampleScene 설정 절차, 모바일 해상도 QA는 체크리스트 수준까지 내려오지 않았다.

## 6. Backlog
- Legacy 이동 후 경로와 serialized 참조 회귀 테스트는 아직 유지 항목이다. 이번 diff는 DOTS 씬과 문서 중심이며 해당 검증을 해결하지 않았다.
- Unity 버전·패키지 업그레이드성 변경 분리, TMP/ShaderGraph 대량 변경 의도 문서화, Aseprite·AudioGen 출처·사용처 정리는 그대로 남아 있다.
- 실제 아이템 선택 UI 교체와 오디오 SFX 배열 검증은 여전히 high backlog다. 자동 선택과 Inspector 배열 의존은 시연 안정성에 직접 영향을 줄 수 있다.
- 적 접촉 피해 단위 분리는 장기 개선으로 남긴다. 현재 동작보다 밸런싱 가시성 문제에 가깝기 때문에 low로 두는 것이 적절하다.

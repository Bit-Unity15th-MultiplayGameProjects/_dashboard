# GhostMarch_DOTS — 렌더·저장 스모크 확장 스냅샷

## 1. 주요 변경사항
- `VerticalSliceRenderSnapshotSystem` 이 플레이어, 적, 투사체, 소울, 데미지 숫자, 이펙트 렌더 데이터를 ECS 동적 버퍼로 한 번에 스냅샷한다.
- `VerticalSliceGPURenderBridge` 는 더 이상 렌더 패스마다 `ToEntityArray(Allocator.Temp)` 를 호출하지 않고 스냅샷 버퍼를 읽어 그린다.
- 스모크 테스트가 Core, Save GameOver, Save Victory 모드로 확장됐다. 저장 키 백업, 강제 사망, 강제 승리 조건, PlayerPrefs 검증까지 포함한다.
- 문서에는 렌더 할당 제거, HUD/Audio/Save 브리지 존재 확인, 세이브 스모크 실행 경로가 반영됐다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 좋다. `VerticalSliceGPURenderBridge` 에 있던 per-pass 엔티티 배열 생성 책임을 DOTS 시스템으로 옮긴 결정은 명확한 개선이다. 렌더 MonoBehaviour 는 그리기에 집중하고, 데이터 수집은 Simulation 쪽에서 수행하므로 책임 경계도 이전보다 낫다. 특히 적, 투사체, 데미지 숫자처럼 개체 수가 늘어나는 렌더 대상에서 매 프레임 `Allocator.Temp` 배열을 여러 번 만드는 구조를 제거한 점은 실제 병목 가능성을 줄인다.

자료구조·복잡도·할당 관점에서는 이전 backlog 하나가 실제로 해결됐다. 현재 비용 구조는 `ToEntityArray` 반복 할당에서 `DynamicBuffer` 재사용으로 바뀌었고, 이는 이 프로젝트의 고밀도 생존 전투 렌더링에 의미 있는 전진이다. 다만 `VerticalSliceRenderSnapshotSystem` 의 버퍼는 개체 수가 최고점을 갱신할 때 성장 재할당이 발생할 수 있다. 지금 당장 결함은 아니지만, 다음 단계에서는 목표 최대 적·투사체 수 기준으로 `EnsureCapacity` 또는 계측 기준을 두어 고밀도 테스트에서 다시 할당 문제가 생기지 않는지 확인해야 한다.

반대로 `VerticalSliceSmokeTestRunner` 는 기능이 빠르게 늘면서 책임이 무거워지고 있다. 씬 구성 주입, 저장값 백업, ECS 상태 강제 변경, 성공 조건 판정, 로그 요약이 한 클래스에 몰려 있다. 더 큰 문제는 `PlayerPrefs` 키와 `GameObject.Find("HUD_Main")` 류 문자열이 테스트 내부에 직접 박혀 있다는 점이다. 왜 문제냐면 런타임 저장 포맷이나 HUD 오브젝트명이 바뀌어도 테스트가 컴파일 단계에서 깨지지 않고, 플레이 중에야 조용히 실패하거나 더 나쁘게는 잘못된 대상을 검증할 수 있기 때문이다. 개선하려면 저장 키는 SaveBridge와 공유 상수로 올리고, 씬 오브젝트 검증은 별도 EditMode 구성 테스트로 분리하는 편이 안전하다.

## 3. 진행도 평가
이번 diff 범위에는 네트워킹 코드가 없으므로 `framework`, `hybrid`, `custom-network` 중 하나로 보정할 근거는 없다. 진행도는 로컬 DOTS 수직 슬라이스의 제출 가능성 기준으로 산정한다. Core 루프, 재시작, HUD/Audio/Save 브리지 존재, 게임오버 저장, 승리 저장까지 자동 스모크로 확인되기 시작했으므로 반복 시연 가능성은 전보다 뚜렷하게 올라갔다.

진행도는 70%로 본다. 이유는 핵심 루프와 저장 경로가 자동화됐고 커밋도 작고 목적이 분명하기 때문이다. 다만 스모크가 Play Mode에서 브리지를 주입하기 때문에 저장된 씬 자체의 누락을 가릴 수 있고, 모바일 aspect 검증과 고밀도 렌더 계측이 아직 남아 있다. 위험도는 high가 아니라 medium이다. 즉시 시연을 깨는 결함이 보이진 않지만, 지금 테스트 구조를 정리하지 않으면 다음 기능 추가부터 회귀 원인을 찾기 어려워진다.

## 4. 다음 권장사항
- 저장 스모크 실패, 타임아웃, 수동 중단 상황에서도 PlayerPrefs가 반드시 복구되도록 정리 경로를 일원화한다.
- HUD, Audio, Save 브리지가 저장 씬에 실제로 들어 있는지 확인하는 EditMode 또는 씬 구성 검증을 추가한다.
- `EnemyCounts(0)`, `UnlockChrysanthemum`, `UnlockOrchid` 같은 저장 키를 테스트와 런타임이 공유하는 상수로 이동한다.
- 렌더 스냅샷 `DynamicBuffer` 의 capacity 성장 여부를 고밀도 생존 테스트에서 계측하고 필요하면 선확보한다.
- TODO에 남은 모바일 aspect Game View 스모크를 자동 실행 경로에 묶어 발표 환경 리스크를 줄인다.

## 5. 문서화 상태
design 점수는 4점이다. 이번 커밋은 렌더와 테스트 자동화 중심이라 게임 컨셉, 플레이어 경험 목표, 시스템 선택 이유를 설명하는 디자인 문서 갱신은 확인되지 않는다. 현재 상태만 보면 아이디어와 구현은 전진하지만, 왜 이 수직 슬라이스가 목표 경험을 대표하는지 문서 근거가 약하다.

technical 점수는 7점이다. `handoff-state.md` 와 `dots-rebuild-todo.md` 가 최신 검증 결과, 렌더 스냅샷 구조, 저장 스모크 경로를 계속 반영하고 있다. 신규 작업자가 현 상태와 다음 작업을 따라가기에는 실무적으로 유효하다. 다만 아키텍처 다이어그램이나 저장·렌더 데이터 흐름의 독립 문서 수준까지는 아직 아니다.

spec 점수는 5점이다. 스모크 체크 항목과 검증 요약은 사양서의 일부 역할을 하지만, 씬 구성, HUD 객체명, 저장 키, 승리·게임오버 조건이 테스트 코드와 TODO에 흩어져 있다. 테스트 체크리스트로 쓰려면 이 값들을 표준 사양으로 끌어내야 한다.

## 6. Backlog
- Legacy 이동 후 하드코딩 경로와 serialized 참조 회귀 테스트 유지가 아직 남아 있다. 대량 이동 이후 씬·프리팹 참조가 깨지는 문제는 한 번 놓치면 런타임에서만 드러난다.
- Unity 버전·패키지 업그레이드성 변경을 별도 커밋으로 격리하는 규칙도 아직 필요하다. 엔진 변경과 기능 변경이 섞이면 회귀 원인 추적 비용이 커진다.
- Aseprite·AudioGen 대량 자산의 출처와 실제 사용처 문서화가 남아 있다. 사용되지 않는 자산인지 핵심 자산인지 구분되지 않으면 빌드 크기와 라이선스 리스크가 쌓인다.
- TextMesh Pro와 ShaderGraph 리소스 대량 변경의 의도 문서화도 미해결이다. 렌더링 리소스 변경은 눈에 보이는 회귀를 만들기 쉬우므로 근거를 남겨야 한다.

## 7. 이전 Backlog 해결

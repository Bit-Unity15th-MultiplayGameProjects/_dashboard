# GhostMarch_DOTS — 0커밋 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개라 실제 코드 diff와 변경 통계는 없다. 따라서 이번 평가는 신규 구현이 아니라 열린 항목의 현재 위험 재판정이다.
- 직전 high TODO였던 레벨업 선택 UI의 pause 소유권과 연속 선택 후보 계산 문제가 해결됐다는 근거가 없다.
- 문서 스냅샷은 아키텍처와 마이그레이션 기록이 꽤 두껍다. 다만 UI 프리팹, 해상도, 최종 배경 아트 같은 검증 문서는 아직 작업 항목으로 남는다.
- 직전 backlog 5개는 모두 완전 해결 근거가 부족하다. resolved_from_backlog는 비워 두는 것이 맞다.

## 2. 코드 품질 리뷰
이번 diff 자체는 비어 있으므로 새로 나빠진 코드는 없다. 그러나 현재 열린 항목 기준으로 보면 `VerticalSliceItemSelectionBridge.cs`가 여전히 위험 지점이다. 레벨업 UI가 `Time.timeScale`을 직접 소유하면 pause, 결과 화면, 테스트 러너 같은 다른 시스템의 일시정지와 충돌한다. 개선 방향은 선택 UI가 시간을 직접 복구하지 않고, 중앙 pause controller 또는 scoped pause token을 통해 “누가 멈췄는지”를 명시하는 것이다.

자료구조/복잡도 관점에서는 아이템 타입, 라벨, 업적 키가 int/switch/문자열로 흩어진 구조가 가장 먼저 정리돼야 한다. 현재 비용은 새 아이템 추가 시 UI 표시, 저장, 업적 조건이 서로 다른 분기에서 어긋나는 것이다. enum-indexed table이나 ScriptableObject 기반 정의로 `id -> 표시명/업적키/최대레벨`을 한곳에 고정하면 분기 수와 회귀 위험이 동시에 줄어든다.

`VerticalSliceInfiniteBackgroundBridge.cs`의 무한 배경도 hot path 경고가 유효하다. LateUpdate마다 컴포넌트 조회와 머티리얼/위치 갱신을 반복하는 방식은 지금은 작아 보여도 모바일에서는 프레임 예산을 꾸준히 갉아먹는다. Renderer는 캐시하고, 타일 원점이 바뀐 경우에만 좌표와 머티리얼 파라미터를 갱신하도록 흐름을 바꾸는 것이 맞다.

## 3. 진행도 평가
네트워킹 구현 경로는 현재 스냅샷 기준 해당 없음이다. 따라서 진행도는 DOTS 수직 슬라이스의 반복 시연 가능성 기준으로 본다. 아키텍처 문서와 vertical slice 관련 코드·문서는 축적돼 있으나, 레벨업 선택 UI와 배경 브리지처럼 실제 플레이 중 바로 보이는 부분에 high/medium 위험이 남아 있다.

`progress_estimate`는 58로 본다. 낮은 레벨의 수직 슬라이스는 형성됐지만, 이번 범위에 전진 커밋이 없고 직전 핵심 TODO가 해소되지 않았다. 특히 연속 레벨업 선택 버그는 플레이 루프 신뢰도를 직접 깨뜨릴 수 있어 risk는 high가 타당하다.

## 4. 다음 권장사항
- 레벨업 선택 UI의 `timeScale` 제어를 중앙 pause 소유권 모델로 분리한다.
- 연속 레벨업 처리에서 직전 선택 결과가 다음 후보 계산에 즉시 반영되게 고친다.
- 아이템 타입, UI 라벨, 업적 키를 enum 기반 테이블 또는 데이터 자산으로 묶는다.
- 무한 배경은 Renderer 캐시와 타일 원점 변경 감지를 기준으로 갱신 비용을 줄인다.
- 선택 UI 프리팹과 모바일 해상도별 검증 체크리스트를 문서와 smoke 기준에 추가한다.

## 5. 문서화 상태
design은 6점이다. `docs/planning/game-design-spec.md`와 migration 문서가 있어 방향성은 보이지만, 플레이어 경험 목표와 밸런스 수치가 테스트 가능한 표로 충분히 고정됐다고 보기는 어렵다.

technical은 8점이다. `docs/architecture/assembly-boundaries.md`, `folder-and-asmdef-layout.md`, DOTS migration 문서가 신규 유입자가 구조를 따라갈 수 있을 정도로 잘 정리돼 있다. 이 부분은 분명히 강점이다.

spec은 7점이다. `legacy-runtime-spec`, `weapon-equipment-legacy-spec`, `scene-prefab-serialized-baseline` 등은 실무 참고 자료로 쓸 수 있다. 다만 UI 프리팹, 최종 배경 타일링, 해상도별 판정처럼 현재 vertical slice의 완료 기준은 아직 빈칸이 남아 있다.

## 6. Backlog
- Legacy 이동 후 하드코딩 경로와 serialized 참조 회귀 테스트 유지는 계속 이월한다. 기준선 문서만으로는 자동 회귀 방지가 끝났다고 볼 수 없다.
- Unity 버전·패키지 업그레이드성 변경은 앞으로도 기능 커밋과 분리해야 한다. 패키지 변경은 원인 추적 비용이 크다.
- Aseprite·AudioGen 대량 자산은 출처와 실제 사용처를 명확히 해야 한다. 라이선스와 빌드 용량 양쪽 위험이다.
- 런타임 checker 타일은 최종 아트 기반 배경 타일링으로 교체해야 한다. 임시 시각 자산이 플레이 검증 기준으로 굳으면 안 된다.
- TextMesh Pro와 ShaderGraph 대량 리소스 변경의 의도 문서화는 low로 유지한다. 급한 기능 위험은 아니지만 변경 추적성에는 필요하다.

# GhostMarch_DOTS — 빈 diff 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 범위가 동일 SHA 를 가리키며 커밋 수가 0이다. 실제 코드 diff, 문서 diff, 자산 diff 모두 신규 전진 근거가 없다.
- 직전 리포트의 핵심 TODO 는 현재 스냅샷에서 해결 근거가 보이지 않으므로 이월한다. 특히 레벨업 선택 UI와 pause 소유권은 즉시 정리 대상이다.
- 문서는 양이 충분하고 구조도 나쁘지 않다. 다만 문서가 실제 자동 검증과 씬 연결 상태를 대체하지는 못한다.
- validator 회계 대상 backlog 는 해결 증거가 없으므로 모두 이월한다. 같은 계열의 과거 title 도 추적 키로 유지한다.

## 2. 코드 품질 리뷰
이번 diff 자체에는 코드 변경이 없으므로 신규 버그를 만들었다고 판단할 근거는 없다. 그러나 위험이 낮아진 것도 아니다. `VerticalSliceItemSelectionBridge.cs` 계열의 미해결 항목은 UI가 게임 시간, 선택 후보 계산, 업적·저장 계약을 동시에 만지는 구조로 보인다. 이 구조는 작은 UI 수정이 진행도·pause·보상 상태를 함께 흔들 수 있어 위험하다. 개선 방향은 pause 소유권을 별도 서비스나 상태 owner 로 분리하고, 선택 확정 후 다음 후보를 반드시 최신 상태에서 다시 계산하는 것이다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 신규 지적은 없다. 다만 기존 open item 기준으로 `LateUpdate` 배경 갱신과 item type 의 int/string 분기는 계속 비용과 유지보수 리스크를 만든다. 매 프레임 타일 상태를 반복 갱신하는 흐름은 tile origin 변경 이벤트로 제한하고, item 계약은 enum-indexed table 로 묶는 편이 이 프로젝트의 수직 슬라이스 검증에 더 직접적이다.

Unity 관점에서는 `Time.timeScale` 직접 제어가 가장 위험하다. pause, level-up, result, scene transition 이 하나라도 겹치면 UI가 자신이 만들지 않은 정지 상태를 해제할 수 있다. 이건 단순 스타일 문제가 아니라 재현이 어려운 런타임 버그의 전형이다.

## 3. 진행도 평가
네트워킹 구현은 현재 범위에서 해당 없음이다. 진행도는 DOTS 재구현 수직 슬라이스 기준으로 본다. DOTS 구조, migration 문서, reference 문서, smoke runner 흔적은 있어 10-20% 수준의 스캐폴드 단계는 넘었다. 반면 실제 제출 가능한 반복 플레이 루프를 보장하는 UI 프리팹, 해상도 검증, pause 소유권, 자동화 결과 산출은 아직 닫히지 않았다.

따라서 진행도는 58로 본다. 낮은 레벨의 수직 슬라이스는 형성됐지만, 시연 중 레벨업·pause·배경·저장 검증이 흔들릴 여지가 크다. 이번 커밋 범위가 빈 diff라는 점도 전진 속도 측면에서 좋지 않은 신호다.

## 4. 다음 권장사항
- 레벨업 선택 UI의 `timeScale` 복구 책임을 중앙 pause owner 로 옮겨라. UI가 직접 시간을 되돌리는 구조는 더 키우면 안 된다.
- 연속 레벨업 선택은 선택 확정 직후 상태를 반영해 다음 후보를 재계산하라. 중복·초과 선택은 플레이 신뢰도를 바로 깎는다.
- item type, 표시 라벨, 업적 키를 enum 기반 테이블 하나로 고정하라. 문자열 분기는 지금 줄이는 편이 싸다.
- 무한 배경 타일은 매 프레임 갱신하지 말고 타일 원점 변경 시점에만 갱신하라.
- UI 프리팹과 모바일 해상도 체크리스트를 문서가 아니라 실제 검증 절차로 연결하라.

## 5. 문서화 상태
design 은 7점이다. `docs/planning/game-design-spec.md` 와 migration 문서가 게임 방향과 재구현 의도를 설명한다. 다만 샘플 수준처럼 목표 경험, 밸런스 수치, 판정 기준이 촘촘하게 연결된 상태는 아니다.

technical 은 8점이다. `assembly-boundaries.md`, `folder-and-asmdef-layout.md`, `dots-rebuild-strategy.md`, serialized baseline 문서가 신규 참여자가 구조를 파악할 만큼 잘 정리돼 있다. 이 부분은 분명히 강점이다.

spec 은 7점이다. `first-vertical-slice.md`, `legacy-runtime-spec.md`, `weapon-equipment-legacy-spec.md` 는 테스트 체크리스트로 확장 가능한 기반이다. 그러나 UI 프리팹, 해상도, 최종 배경 아트, smoke 결과 같은 실행 판정표가 아직 완전히 닫히지 않았다.

# GhostMarch_DOTS — GPU 프레젠테이션 복구 스냅샷

## 1. 주요 변경사항
- `VerticalSliceGPURenderBridge`가 fallback 크기, 배경 z 위치, fallback material property 처리를 조정해 GPU 기본 표시를 복구했다.
- snapshot buffer가 비어도 player/enemy/soul을 직접 ECS query로 그리는 임시 fallback을 추가했다.
- `VerticalSliceRenderSnapshotSystem`은 `PlayerInput`이 없는 플레이어도 렌더 snapshot에 포함하도록 완화됐다.
- 런타임 checker background tile은 씬에서 꺼졌고, GPU 표시 장애 원인과 재현 절차가 troubleshooting 문서로 남았다.

## 2. 코드 품질 리뷰
이번 변경의 좋은 점은 문제를 콘솔 에러 유무가 아니라 Game View 가시성, hierarchy fallback 오브젝트, material property 경로로 분해했다는 점이다. 특히 `docs/migration/dots-gpu-presentation-troubleshooting.md`는 “무엇을 확인했고 왜 그 결론에 도달했는가”가 남아 있어 다음 장애 대응 비용을 줄인다.

다만 `VerticalSliceGPURenderBridge`는 지금 응급 복구 코드와 기본 렌더 코드가 섞였다. `RenderSouls/RenderEnemies/RenderPlayer`가 snapshot 유무에 따라 direct fallback을 호출한 뒤, 아래에서 `RenderDirectSouls/RenderDirectEnemies/RenderDirectPlayer`를 다시 무조건 호출한다. 왜 문제냐면 snapshot 경로가 살아 있어도 direct query가 덮어 그리므로 회귀가 가려지고, 비어 있는 경우에는 같은 actor가 두 번 그려질 수 있다. 개선은 direct fallback을 “snapshot empty일 때만” 또는 명시적 diagnostic flag일 때만 실행하도록 분리하는 것이다.

자료구조/복잡도/할당 관점에서도 경고가 필요하다. direct fallback은 player/enemy/soul에서 매 프레임 `ToEntityArray(Allocator.Temp)`와 component lookup을 반복한다. 현재 구조의 비용은 렌더 hot path의 O(E+S) 임시 배열 생성이다. 응급 가시성 확보로는 허용되지만 기본 경로로 남기면 DOTS 전환의 이점이 약해진다. snapshot buffer를 신뢰 가능한 단일 소스로 복구하고, 임시 query는 실패 감지용으로 축소해야 한다.

## 3. 진행도 평가
현재 구현 경로는 네트워킹 없는 로컬 DOTS 수직 슬라이스다. 네트워킹 코드는 리뷰 범위에 없어 해당 없음으로 판단한다. 이번 범위는 시연을 막던 GPU 표시 문제를 회복했다는 점에서 분명한 전진이다.

하지만 커밋 6개 중 대부분이 `fix`이고, 핵심 렌더 경로는 자동 Game View smoke가 아니라 수동 확인과 fallback 중첩에 의존한다. 제출 가능한 수직 슬라이스 기준으로는 61% 수준이며, 표시 복구를 검증 자동화로 고정하지 않으면 같은 문제가 다시 발생할 위험이 높다.

## 4. 다음 권장사항
- GPU direct-query fallback을 snapshot-empty 조건으로 제한한다.
- GPU 기본 렌더 경로를 Game View 스모크와 캡처 기준으로 고정한다.
- Silver Bell orbit 위치·hitbox·owner 조회를 한 검증으로 닫는다.
- 아이템 선택 계약을 enum table로 옮기고 실제 입력 스모크를 추가한다.
- Projectile volley spread 값을 무기별 tuning config로 이전한다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`와 migration 문서들이 코어 루프와 재구현 의도를 설명하지만, 최종 아트·맵·밸런스 목표가 테스트 가능한 수치로 모두 닫히지는 않았다.

technical은 8점이다. assembly boundary, folder layout, verification loop, GPU troubleshooting까지 있어 신규 기여자가 구조와 현재 장애 이력을 따라갈 수 있다. 이번 troubleshooting 추가는 특히 긍정적이다.

spec은 7점이다. legacy runtime, weapon/equipment, scene-prefab baseline은 실무 검증에 쓸 수 있다. 다만 GPU 기본 렌더, item growth, Silver Bell 판정처럼 현재 개발 중인 핵심 기능은 아직 자동 판정표가 부족하다.

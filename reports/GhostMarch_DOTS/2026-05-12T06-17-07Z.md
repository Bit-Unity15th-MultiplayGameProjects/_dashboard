# GhostMarch_DOTS — 아이템 선택 UI 개선 스냅샷

## 1. 주요 변경사항
- `WeaponFireSystem`에서 같은 volley 투사체 시작 위치를 수직 오프셋해 다중 발사 가시성을 개선했다.
- `VerticalSliceItemSelectionBridge`가 레벨 표기뿐 아니라 실제 현재값→다음값 수치 preview를 보여주도록 확장됐다.
- DOTS 프레젠테이션 에셋 가이드가 추가되어 머티리얼 슬롯, 셰이더 property, 수동 검증 절차가 문서화됐다.
- `dots-rebuild-todo.md`와 handoff 문서가 이번 구현 상태를 반영하도록 갱신됐다.

## 2. 코드 품질 리뷰
투사체 오프셋은 변경 범위가 좁고 목적도 명확하다. 같은 위치에 겹쳐 생성되던 발사체를 시각적으로 분리한 점은 실제 플레이 피드백 측면에서 좋은 개선이다. 다만 `0.18f` 오프셋은 현재 코드 안에 박힌 튜닝 값이므로, 투사체 크기나 무기별 표현이 늘어나면 `WeaponState` 또는 렌더 설정 쪽으로 옮기는 편이 좋다.

아이템 선택 UI는 사용자가 업그레이드 효과를 이해하기 쉬워졌다는 점에서 분명한 전진이다. 그러나 `VerticalSliceItemSelectionBridge`가 아이템 id, 라벨, 업적 키, preview 계산을 계속 `int`와 `switch`로 직접 들고 있다. 왜 문제냐면 실제 적용 로직인 `ItemApplySystem`과 UI 계산이 따로 변하면 “보여준 수치”와 “적용된 수치”가 조용히 어긋난다. 개선 방향은 아이템 enum과 enum-indexed definition table을 두고 UI preview와 적용 로직이 같은 데이터를 읽게 만드는 것이다.

자료구조/복잡도 관점에서는 이번 diff에 매 프레임 LINQ나 hot path 컬렉션 복사는 추가되지 않았다. `TryGetPlayerState`의 query 생성과 `ToEntityArray`는 선택 UI 이벤트 시점이라 현재 성능 위험은 낮다. 핵심 비용은 런타임 복잡도보다 dense item id를 여러 switch로 흩뿌린 유지보수 비용이다.

## 3. 진행도 평가
네트워킹 구현 경로는 현재 해당 없음으로 본다. DOTS 수직 슬라이스 기준으로는 전투, 렌더링, 아이템 선택, 문서화가 반복 가능한 형태로 모이고 있어 진행도는 64% 수준이다. 이번 변경은 발표 가시성과 UI 이해도를 올렸으므로 순수 전진이다.

다만 레벨업 선택 UI는 아직 pause ownership, 연속 레벨업 반영 순서, 실제 입력 smoke 검증이 남아 있다. 이 셋은 시연 중 “가끔 이상하다”로 보이는 유형이라 지금 정리하지 않으면 후속 기능보다 디버깅 비용이 더 커진다.

## 4. 다음 권장사항
- 레벨업 패널이 `Time.timeScale`을 직접 복구하지 않게 하고 `PauseService`가 중첩 pause를 소유하게 만든다.
- 선택 이벤트를 버퍼에 넣은 직후 다음 패널을 여는 흐름을 끊고, `ItemApplySystem` 반영 이후 후보를 다시 계산한다.
- auto-pick이 아니라 실제 클릭과 키보드 입력으로 `VerticalSliceItemSelectionBridge`를 검증하는 Play Mode smoke를 추가한다.
- 아이템 id, 라벨, 업적 키, preview 계산을 enum 기반 테이블로 묶어 UI와 적용 로직의 중복 계산을 제거한다.
- 무한 배경 타일 갱신은 카메라 원점 타일이 바뀐 순간에만 실행되는지 코드와 smoke로 확인한다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`가 코어 루프와 방향을 담고 있으나, 샘플 사양서 수준의 레벨별 데이터 규격과 측정 목표까지는 아직 부족하다.

technical은 8점이다. assembly boundary, DOTS 전략, 검증 루프, 이번 프레젠테이션 에셋 가이드까지 있어 신규 작업자가 구조를 따라가기 좋다. 다만 일부 항목은 문서 체크리스트와 실제 자동화가 아직 분리돼 있다.

spec은 7점이다. `scene-prefab-serialized-baseline.md`, legacy runtime spec, weapon/equipment spec은 테스트 기준으로 쓸 수 있다. 반면 최종 아트, 오디오 배열, 배경 타일링처럼 Inspector 계약이 필요한 항목은 아직 코드 검증과 연결이 약하다.

- Legacy 이동 후 serialized baseline 문서는 있으나 하드코딩 경로 회귀를 자동으로 잡는 테스트는 아직 부족하다.

# GhostMarch_DOTS — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개라 코드 diff 자체는 없다. 이번 리포트는 새 변경 평가가 아니라 누적 open item 재판정 성격이 강하다.
- 수직 슬라이스의 핵심 위험은 여전히 아이템 선택 UI, 레벨업 pause 소유권, 연속 선택 후보 계산에 집중돼 있다.
- 문서 스냅샷은 DOTS 전환, assembly 경계, 검증 루프, legacy baseline을 꽤 충실히 갖추고 있다.
- 중복 backlog title 일부는 현재 회계 기준에서 정리했다. 같은 이슈의 “불명확함” 표현은 더 구체적인 “문서화/명확화” 항목으로 수렴시키는 것이 맞다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 새로 유입된 코드 품질 저하는 없다. 다만 기존 open item 기준으로 보면 `VerticalSliceItemSelectionBridge.cs`가 여전히 핵심 위험 파일이다. 레벨업 UI가 `timeScale` 복구를 직접 소유하면 pause 메뉴, 컷신, 결과창 같은 다른 정지 사유와 충돌한다. 개선 방향은 UI가 시간을 직접 만지지 않고 중앙 pause service 또는 owner-token 방식으로 “누가 정지시켰는지”를 표현하게 하는 것이다.

자료구조/복잡도 관점에서는 `VerticalSliceItemSelectionBridge.cs`의 int·문자열 기반 아이템 타입, 라벨, 업적 키 분기가 남은 것으로 판정한다. enum/dense id 성격의 계약을 문자열 분기로 흩뜨리면 후보 계산, 표시 텍스트, 저장/업적 키가 서로 어긋나도 컴파일 단계에서 잡히지 않는다. 이 프로젝트에서는 레벨업 선택이 수직 슬라이스의 반복 플레이 루프에 직접 연결되므로 enum-indexed table로 고정하는 편이 비용과 안정성 양쪽에서 의미 있다.

`VerticalSliceInfiniteBackgroundBridge.cs`의 배경 타일 갱신은 이전 항목상 매 프레임 갱신 위험이 남아 있다. 카메라가 같은 타일 원점에 머무는 동안 재배치가 반복되면 모바일 타깃에서 불필요한 Update 비용이 된다. 타일 origin 변경 시점에만 갱신하도록 상태를 캐시하는 방식이 적절하다.

## 3. 진행도 평가
네트워킹 구현은 현재 스냅샷 기준 주요 범위가 아니므로 경로 분류는 해당 없음에 가깝다. 진행도는 DOTS 기반 단일 플레이 수직 슬라이스 기준으로 보정했다. ECS 구조와 검증 문서는 상당히 전진했으나 실제 플레이 루프의 UI 선택, pause, 오디오 검증, 배경 최종화가 아직 막혀 있어 58%로 본다.

변경이 없는 커밋 범위라 전진 증거는 없다. 특히 자동 선택 브릿지를 실제 UI 흐름으로 대체하는 작업이 남아 있으면 “플레이어가 선택하고 결과가 반영되는 루프”가 아직 제출 데모 품질이라고 보기 어렵다. 그래서 risk는 medium이 아니라 high가 타당하다.

## 4. 다음 권장사항
- 아이템 자동 선택 브릿지를 끊고 실제 UI 입력, 선택 결과 반영, 다음 후보 재계산까지 한 번에 검증한다.
- 레벨업 UI의 `timeScale` 직접 제어를 제거하고 pause 소유권을 중앙 서비스로 옮긴다.
- SFX enum 개수와 Inspector 배열 길이를 `OnValidate` 또는 시작 시점 검증으로 고정한다.
- 무한 배경 타일은 매 프레임 재계산하지 말고 타일 원점 변경 시에만 갱신한다.
- UI 프리팹과 모바일 해상도 smoke 체크리스트를 문서와 자동화에 연결한다.

## 5. 문서화 상태
design은 7점이다. `docs/planning/game-design-spec.md`가 게임 방향과 기획 의도를 제공하지만, 샘플 사양서처럼 밸런스 수치와 측정 기준이 모든 시스템에 촘촘히 연결된 수준은 아니다.

technical은 8점이다. `docs/architecture/assembly-boundaries.md`, `folder-and-asmdef-layout.md`, DOTS 전환 문서가 신규 유입자가 구조를 따라갈 수 있을 정도로 구체적이다. 다만 실제 Unity compile/play 검증 결과와 CI 연결은 아직 더 강해져야 한다.

spec은 7점이다. `first-vertical-slice.md`, `dots-rebuild-todo.md`, reference baseline 문서가 테스트 체크리스트로 쓸 수 있는 근거를 제공한다. 그러나 최종 UI 프리팹, 배경 아트 타일링, 오디오 배열 계약처럼 Inspector와 씬 기준의 완료 판정은 아직 빈틈이 있다.

## 6. Backlog
- 실제 아이템 선택 UI 교체는 계속 high backlog다. 자동 선택이 남으면 플레이어 선택 루프 검증이 성립하지 않는다.
- Aseprite·AudioGen 자산은 디렉터리 증거만 있고 출처·사용처 문서 근거가 부족하므로 명확화가 필요하다.
- Legacy 이동 후 serialized baseline 문서는 있으나 회귀 테스트 유지와 하드코딩 경로 전수 검증은 계속 관리 대상이다.
- Unity 버전·패키지 업그레이드성 변경은 다음 기능 변경과 섞이지 않도록 별도 커밋 원칙을 유지해야 한다.
- checker 배경, SFX 배열 검증, TMP·ShaderGraph 변경 의도 문서화는 장기 부채로 남긴다.

## 7. 이전 Backlog 해결
- `Aseprite·AudioGen 대량 자산의 출처와 실제 사용처가 불명확함`은 같은 문제를 더 실행 가능한 `명확화` backlog로 수렴시켰다. 중복 title을 동시에 열어두면 회계만 흐려진다.
- `TextMesh Pro와 ShaderGraph 리소스 대량 변경의 의도가 불명확함`도 `의도 문서화` backlog로 수렴했다. 남은 일은 불명확하다는 관찰이 아니라 문서화 완료 여부다.
- `Unity 버전·패키지 업그레이드성 변경이 기능 커밋에 섞임`은 이번 범위 커밋 수가 0이라 현재 diff에는 재발하지 않았다. 다만 별도 커밋 격리 원칙은 backlog로 유지한다.

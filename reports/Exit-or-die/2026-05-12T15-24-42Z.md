# Exit-or-die — 플레이어 전투 추가 스냅샷

## 1. 주요 변경사항
- 플레이어 `Walk`, `Run`, `Attack` 애니메이션 클립과 컨트롤러가 추가되고 `character_barbarian` 프리팹에 연결됐다.
- `PlayerController`가 클릭 이동 중심 구조에서 키보드 이동, 달리기, 마우스 조준, 서버 공격 판정 구조로 크게 바뀌었다.
- 몬스터 컨트롤러와 몬스터 프리팹들이 함께 수정되어 플레이어 공격·몬스터 피해 흐름을 맞추려는 시도가 보인다.
- `Assets/_Recovery`와 샘플 이미지 일부가 삭제됐다. 저장소 노이즈를 줄인 점은 명확한 개선이다.
- `Test.unity`, `Menu.unity`, `Game.unity`가 함께 변경되어 기능 추가와 씬 실험이 같은 커밋에 섞였다.

## 2. 코드 품질 리뷰
이번 변경의 핵심 위험은 `PlayerController.cs`가 다시 비대해진 점이다. 이동 입력, 서버 이동, 공격 판정, 애니메이션 동기화, 체력, 사망 후 관전 전환이 한 클래스에 몰려 있다. 왜 문제인가 하면, 네트워크 버그가 났을 때 “입력 문제인지, 서버 권한 문제인지, 애니메이션 상태 문제인지”를 분리해서 추적하기 어렵다. 개선하려면 `PlayerMovement`, `PlayerCombat`, `PlayerHealth`, `PlayerAnimationSync` 정도로 책임을 나누고, 서버가 쓰는 상태와 오너 클라이언트가 쓰는 입력 상태를 명시적으로 분리해야 한다.

자료구조·복잡도 관점에서는 새 공격 판정의 `Physics.OverlapSphereNonAlloc` 자체는 버퍼를 재사용하므로 방향은 좋다. 다만 레이어 마스크 없이 주변 콜라이더를 모두 받은 뒤 `HashSet<GameObject>` 중복 제거를 공격마다 생성하는 패턴은 불필요한 판정 범위와 할당을 만든다. 이 프로젝트는 몬스터 수가 방 단위로 늘어나는 구조이므로 `Monster` 레이어 마스크와 고정 `HashSet` 또는 `NetworkObjectId` 기반 중복 체크로 공격 hot path를 고정하는 편이 낫다.

Unity 관점에서도 `Input.GetAxisRaw`, `Input.GetKey`, `Input.GetMouseButtonDown`이 다시 들어온 것은 후퇴다. 이미 Input System 액션 에셋과 UI Input Module을 쓰는 상태이므로, 지금 구 Input API를 추가하면 PC 키보드 기준으로만 기능이 굳고 리바인딩·패드·UI 포커스 정책이 갈라진다.

## 3. 진행도 평가
네트워크 구현 경로는 NGO를 쓰는 `framework` 경로다. 서버 RPC, NetworkVariable, NetworkTransform 기반이 있으므로 네트워크 인프라 난도는 낮게 보정하지만, 그만큼 2인 happy path와 권한 모델 완성도가 더 엄격하게 요구된다.

진척은 있다. 플레이어 전투와 애니메이션까지 수직 슬라이스가 넓어졌다. 그러나 이동 방식을 크게 갈아엎었고, 공격·사망·관전·몬스터 피해가 한 번에 묶여 들어와 회귀 위험이 높다. 현재는 제출 가능한 반복 시연 직전이라기보다 2인 검증을 통해 안정화해야 하는 중간 단계로 본다.

## 4. 다음 권장사항
- `PlayerController`를 이동·전투·체력·애니메이션 동기화 컴포넌트로 분리한다.
- 플레이어 공격은 서버 권위, 대상 레이어, 중복 피해 방지, 쿨다운 기준을 먼저 고정한다.
- 2인 Game 씬에서 이동·공격·사망·관전·탈출 투표를 한 번에 재현 검증한다.
- 몬스터 6종, 계단, 문 프리팹의 NetworkPrefabs 등록과 실제 스폰을 확인한다.
- 구 Input API를 제거하고 Input System 액션으로 이동·달리기·공격을 통일한다.

## 5. 문서화 상태
design은 8점이다. `docs/기획서.md`와 `docs/컨셉.md`에 코어 루프, 입력, 탈출/계단 투표, 게임 흐름이 충분히 정리되어 있어 현재 게임 방향은 읽힌다. 다만 최근 전투 입력 변경이 문서에 즉시 반영됐다는 근거는 부족하다.

technical은 7점이다. `docs/기술설계서.md`와 던전 생성 사양서가 NGO, NetworkVariable, 서버 주도 생성, 씬 생명주기를 설명한다. 그러나 플레이어 이동 권한, 예측·보정, 공격 판정 권한은 코드 변화에 비해 아직 모호하다.

spec은 8점이다. 던전 생성·몬스터·상호작용 사양서가 수치와 QA 체크리스트를 갖추고 있어 테스트 기준으로 쓸 수 있다. 다만 로비 실패, 2인 카메라/스폰 회귀, 입력 액션 매핑은 별도 체크리스트로 더 좁혀야 한다.

## 6. Backlog
- `ClientNetworkTransform` 예측·보정 규칙은 코드와 문서 양쪽에서 여전히 명확하지 않다.
- Exit 계단 프리팹에는 중첩 `NetworkObject` 증거가 남아 NGO 스폰 안정성 위험이 유지된다.
- 네트워크 이동 권한과 클라이언트 보정 정책은 기술 문서의 테스트 가능한 기준으로 닫히지 않았다.
- Input System과 구 Input API가 같이 남아 입력 정책이 더 애매해졌다.
- 로비, 시작계단 방향, 카메라·스폰 회귀 체크리스트와 에셋 정리 부채는 아직 장기 이슈로 남아 있다.

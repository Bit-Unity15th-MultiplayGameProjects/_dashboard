# Exit-or-die — 방 목록 로비 구현 스냅샷

## 1. 주요 변경사항
- Menu 씬의 정적 Join 더미 행을 제거하고, `QuerySessionsAsync` 결과로 `RoomListJoinRow`를 동적 생성하도록 바뀌었다. 실제 로비 데이터로 한 단계 전진한 점은 명확하다.
- 방 생성 시 세션을 public/unlocked 로 만들고, 선택된 세션 id 기준으로 클라이언트 입장을 시도하는 흐름이 추가됐다.
- 게임 시작과 다음 층 이동 전에 호스트 세션을 브라우즈 목록에서 숨기는 `NetworkSessionBrowseBridge`가 추가됐다.
- `LocalMonsterVisionCuller`는 owner 기준으로만 복원하도록 바뀌었고, Game 씬 카메라 near clip 은 `-20`으로 변경됐다. 후자는 명백한 회귀 위험이다.

## 2. 코드 품질 리뷰
방 목록을 더미 텍스트에서 실제 세션 데이터 바인딩으로 바꾼 방향은 좋다. `EnsureUnityServicesReadyAsync`로 초기화와 익명 로그인을 묶은 것도 호출 순서 실수를 줄인다. 다만 `LobbyManager`가 패널 전환, 세션 생성·조회·입장, row 생성, 선택 상태, 레이아웃 rebuild까지 모두 들고 있어 이미 UI 컨트롤러와 네트워크 서비스의 경계가 흐려지고 있다. 다음 단계에서는 세션 조회/입장 서비스와 RoomList 뷰를 분리해야 실패 처리와 테스트가 쉬워진다.

가장 위험한 코드는 `NextStairsInteractable`과 `LobbyUIManager`의 리플렉션 호출 복붙이다. 문자열 assembly/type 조회가 실패하면 아무 일도 안 하고 지나가므로, 게임 시작 후에도 세션이 목록에 남는 문제가 런타임에서만 드러난다. 순환 참조가 문제라면 리플렉션으로 숨기지 말고 공용 assembly에 인터페이스를 두거나 asmdef 의존 방향을 재설계하는 편이 맞다. `Game.unity`의 near clip `-20`은 Unity 카메라 설정으로 부적절하며, 렌더링·컬링·클리핑 문제를 만들 수 있으므로 즉시 양수로 되돌려야 한다.

자료구조/복잡도 관점에서는 `RefreshLobbyList`가 매 갱신마다 기존 row를 전부 `Destroy`하고 다시 `Instantiate`한다. 현재 비용은 O(n) 생성/파괴와 GC이며, 로비 목록을 반복 갱신하면 UI 끊김으로 이어진다. `sessionId -> RoomListJoinRow` 딕셔너리와 비활성 row 풀을 두고, 사라진 세션만 반환하는 구조가 이 프로젝트의 방 목록 UI에 더 맞다. 또한 `RoomListJoinRow`의 TMP 텍스트 설정 리플렉션은 프리팹 계약을 컴파일 타임에 검증하지 못하므로 직렬화 참조나 캐시된 setter로 바꾸는 것이 좋다.

## 3. 진행도 평가
구현 경로는 NGO와 Unity Services Multiplayer를 쓰는 framework 기반이다. 이번 커밋은 방 목록과 선택 Join 흐름을 실제 데이터에 연결했으므로 로비 축에서는 분명한 전진이 있다. 그러나 고수준 프레임워크를 쓰는 만큼 2인 happy path, 씬 전환, 전투 권한, 프리팹 스폰 검증은 더 엄격하게 봐야 한다.

현재 제출 가능한 수직 슬라이스는 약 58%로 본다. 로비는 시연 가능성에 가까워졌지만, Game 씬 카메라 회귀와 리플렉션 기반 네트워크 브리지, 미검증 공격·스폰 흐름이 남아 있어 risk 는 high 다.

## 4. 다음 권장사항
- `Game.unity` near clip 을 양수로 복구하고, 호스트/클라이언트 2인 화면에서 카메라 프레이밍을 즉시 확인한다.
- 세션 숨김 로직은 리플렉션 복붙을 제거하고, 공용 서비스 또는 명시적 의존성으로 컴파일 타임 검증되게 만든다.
- RoomList는 빈 목록, locked/full, Join 실패, 재시도 상태를 UI로 분리하고 row 풀로 갱신 비용을 고정한다.
- 2인 Game 씬에서 이동·공격·사망·관전·시야각을 한 번에 검증하고, 공격 판정은 서버 권위와 레이어 필터로 고정한다.
- NetworkPrefabs 등록, PlayerController 책임 분리, Scrollbar 9-slice 연결은 다음 리포트 전까지 좁은 단위로 끝내야 한다.

## 5. 문서화 상태
design 점수는 8이다. `docs/기획서.md`와 `docs/컨셉.md`는 코어 루프, 씬 흐름, 카메라·상호작용 목표를 충분히 설명한다. 다만 로비 실패 상태와 멀티플레이 예외 경험은 아직 테스트 가능한 시나리오로 좁혀지지 않았다.

technical 점수는 7이다. `docs/기술설계서.md`는 NGO, NetworkVariable, 클라이언트 구조를 설명하지만, `ClientNetworkTransform`의 예측·보정·권한 모델은 현재 코드와 연결된 정책으로 문서화되지 않았다. 이 공백은 네트워크 디버깅 비용을 직접 키운다.

spec 점수는 8이다. 던전 생성·몬스터·상호작용 사양서는 데이터 표, 상태 전이, QA 체크리스트를 갖춰 실무적으로 쓸 수 있다. 반면 RoomList, 입력 정책, 카메라·스폰 수동 회귀 체크리스트는 현재 구현 변화에 비해 사양 갱신이 부족하다.

## 6. Backlog
- `ClientNetworkTransform` 예측·보정 규칙은 코드와 문서 양쪽에서 여전히 명확하지 않다. 이동 흔들림이 생기면 원인 추적이 어렵다.
- Exit 계단 프리팹의 중첩 `NetworkObject` 구조는 아직 스폰 안정성 위험으로 남아 있다.
- 네트워크 이동 권한과 클라이언트 보정 정책은 `docs/기술설계서.md`에 명시해야 한다.
- Input System과 구 `InputManager`가 함께 남아 입력 기준이 흐리다.
- 로비 슬롯, 접속 실패, 카메라 표시 체크리스트는 문서에서 아직 재현 절차로 닫히지 않았다.
- 시작계단 방향 규약과 바닥 교체 기준은 `DungeonGenerator` 기준으로 계속 재검증이 필요하다.
- 카메라·스폰 회귀 수동 체크리스트가 부족해 씬 변경마다 같은 문제가 반복될 가능성이 높다.
- TextMesh Pro 예제와 대량 에셋은 제품 코드와 분리해야 리뷰·빌드 비용이 줄어든다.
- `Deungeon` 폴더명 오탈자와 대소문자 불일치는 Unity 리네임 절차로 한 번에 정리해야 한다.

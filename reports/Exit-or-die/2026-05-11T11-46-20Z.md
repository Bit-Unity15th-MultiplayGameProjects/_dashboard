# Exit-or-die — 오쏘그래픽 카메라 전환 스냅샷

## 1. 주요 변경사항
- Game 씬 카메라가 perspective 에서 orthographic 으로 전환되고 size 7, far 100으로 조정됐다. 시점 방향을 고정하려는 의도는 명확하다.
- `PlayerMainCameraMount.Awake()`에서 `_targetYaw`와 `_currentYaw`를 프리팹 설정값으로 초기화했다. 첫 프레임 회전 스냅을 줄이는 좋은 수정이다.
- `character_barbarian.prefab`에서 `Eod.Network.PlayerController`로 보이는 MonoBehaviour 블록이 제거됐다. 카메라 커밋에 포함되기에는 위험한 변경이다.
- `Wasp.prefab`의 추격 속도가 3.5에서 5로 오르고 `NetworkObject.GlobalObjectIdHash`도 변경됐다. 밸런스와 NGO 스폰 안정성 모두 확인이 필요하다.

## 2. 코드 품질 리뷰
카메라 yaw 초기화는 방향성이 좋다. 프리팹의 `followRigWorldEulerAngles.y`를 런타임 상태의 초기값으로 맞춘 점은, 설정값과 실제 카메라 상태가 분리되어 생기는 첫 프레임 흔들림을 줄인다. 다만 같은 커밋에서 씬 카메라, 플레이어 카메라 리그, 몬스터 프리팹까지 같이 바뀌어 변경 책임이 흐려졌다. 카메라 수정 커밋이라면 카메라 설정과 관련 코드만 남기는 것이 리뷰와 회귀 추적에 맞다.

가장 큰 경고는 `character_barbarian.prefab`의 컨트롤러 제거다. 해당 컴포넌트가 실제 플레이어 입력·이동을 담당한다면 Game 씬의 기본 happy path가 즉시 깨진다. 의도한 중복 제거라면 어떤 컴포넌트가 이동 책임을 대체하는지 커밋 안에서 증명돼야 한다. `Game.unity`의 `near clip plane: -1`도 Unity 카메라 계약에서 벗어나는 값이다. Orthographic 전환으로 보이는 영역을 넓히고 싶었다면 카메라 위치, 회전, orthographic size, 양수 near/far 조합으로 해결해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. 반복 lookup, LINQ 할당, hot path 컬렉션 복사 같은 패턴은 변경된 코드에서 확인되지 않는다. 지금 문제는 성능보다 프리팹 구성 무결성, Unity 카메라 설정 유효성, NGO 프리팹 해시 안정성이다.

## 3. 진행도 평가
이 프로젝트는 Unity Netcode for GameObjects를 사용하는 `framework` 경로로 판단된다. 따라서 네트워크 기반 구현 난이도는 직접 소켓 구현보다 낮게 보되, 그만큼 플레이 루프와 프리팹 안정성 기대치는 높아진다. 이번 커밋은 카메라 사용감 측면에서 전진이 있으나, 플레이어 컨트롤러 삭제 가능성과 NetworkObject 해시 변경이 같이 들어와 시연 안정성을 낮춘다.

진행도는 47로 본다. 반복 시연 가능한 수직 슬라이스가 존재할 가능성은 있지만, 기본 조작과 스폰 매칭이 깨질 수 있는 변경이 핵심 프리팹에 들어갔다. 커밋 메시지도 “게임 카메라 오쏘그래픽 적용”만 설명하고 몬스터 속도·네트워크 해시·플레이어 컴포넌트 변경을 설명하지 못한다.

## 4. 다음 권장사항
- `character_barbarian.prefab`에서 PlayerController 삭제가 의도인지 먼저 확인한다. 의도하지 않았다면 즉시 복구한다.
- `Game.unity`의 near clip을 양수로 되돌리고, orthographic size와 카메라 위치로 화면 구도를 맞춘다.
- `Wasp.prefab`의 `GlobalObjectIdHash` 변경 원인을 확인하고 Network Prefabs 등록과 호스트/클라이언트 스폰을 검증한다.
- 카메라 yaw 기준을 씬, 프리팹, 코드 중 한 곳으로 통일해 첫 프레임 회전 회귀를 막는다.
- 2인 Game 씬 체크리스트를 만들어 카메라, 이동, 몬스터 스폰을 같은 빌드에서 확인한다.

## 5. 문서화 상태
design 점수는 6이다. `docs/기획서.md`와 `docs/컨셉.md`가 던전 익스트랙션, 오쏘그래픽 시점, 방향성 시야, 층별 리스크를 현재 방향에 맞춰 설명한다. 다만 이번 커밋처럼 카메라 시점이나 몬스터 속도가 바뀔 때 어떤 플레이 경험 기준을 만족해야 하는지까지는 닫혀 있지 않다.

technical 점수는 6이다. `docs/기술설계서.md`와 `docs/던전생성.md`가 NGO, Host-Authoritative 모델, 던전 생성 흐름, 시야 시스템을 다룬다. 다만 `ClientNetworkTransform` 예측·보정 규칙과 NetworkObject 프리팹 해시 관리 기준은 코드 수준 이슈를 판단하기엔 아직 부족하다.

spec 점수는 4다. 탈출 손실 공식, 생존 수치, 던전 생성 단계 같은 수치 사양은 있으나 카메라 크기, 몬스터 추격 속도, 입력 정책, 씬 버튼 연결 같은 항목은 테스트 가능한 체크리스트로 정리되어 있지 않다.

# Exit-or-die — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 범위가 동일 커밋 기준이며 커밋 수가 0개다. 이번 기간의 실제 diff, 변경 통계, 커밋 메시지 진전은 없다.
- 직전 TODO와 누적 backlog를 현재 스냅샷 기준으로 재판정했으나, 해결로 볼 코드·문서 근거는 확인되지 않는다.
- NGO 기반 프로젝트인데 이동 권한, 예측·보정, 피해 판정의 소유자가 아직 문서와 코드에서 명확하지 않다.
- 문서는 GDD/TDD 형태를 갖췄지만, 2인 스폰·카메라·체력 동기화를 재현하는 체크리스트는 여전히 부족하다.

## 2. 코드 품질 리뷰
이번 diff가 비어 있으므로 신규 코드 품질 변화는 없다. 다만 기존 위험은 그대로 남아 있다. 특히 `MonsterController`의 피해·상태 전이가 서버 권위로 고정되지 않으면 전투 결과가 클라이언트 상태에 끌려갈 수 있다. 멀티플레이 게임에서 체력, 공격 판정, 몬스터 상태는 신뢰 경계의 핵심이므로 `ServerRpc` 입력 검증과 `NetworkVariable` 표시 책임을 명확히 나눠야 한다.

Unity/NGO 관점에서는 `ClientNetworkTransform`의 의도가 가장 위험하다. 클라이언트 이동을 허용한다면 예측·보정 규칙이 있어야 하고, 서버 권위 이동이라면 클라이언트 입력만 서버로 보내야 한다. 현재 문서에는 그 선택이 없다. 이 상태로 기능을 얹으면 흔들림, 순간이동, 판정 불일치를 디버깅하기 어렵다.

자료구조/복잡도 측면에서 이번 diff 범위의 신규 지적은 없다. 다만 누적 항목의 `MonsterController` FixedUpdate 전수 검색이 실제로 남아 있다면 몬스터·플레이어 수 증가 시 물리 틱마다 O(n) 비용이 반복된다. 스폰·해제 시 타깃 후보를 등록하고 캐시된 컬렉션으로 조회하도록 바꾸는 편이 성능과 책임 분리에 모두 유리하다.

## 3. 진행도 평가
네트워크 구현 경로는 Netcode for GameObjects를 쓰는 `framework` 유형이다. 프레임워크가 연결, RPC, 동기화 기반을 제공하므로 기대치는 단순 접속보다 높다. 현재는 2인 happy path, 몬스터 스폰, 카메라 프레이밍, 서버 권위 전투가 검증되지 않아 제출 가능한 수직 슬라이스로 보기 어렵다.

진행도는 40%로 본다. 던전 생성, 문서, 에셋 배치는 존재하지만 이번 범위에서 전진이 없고, 시연을 깨뜨릴 수 있는 프리팹·권한·체크리스트 문제가 동시에 남아 있다. 리스크는 낮출 근거가 없으므로 high가 맞다.

## 4. 다음 권장사항
- `MonsterController`의 피해·상태 전이를 서버 권위로 고정하고, FixedUpdate 전수 검색은 등록 캐시 기반으로 정리한다.
- 신규 몬스터 6종이 `DefaultNetworkPrefabs.asset`에 등록되어 2인 Game 씬에서 정상 스폰되는지 확인한다.
- `character_barbarian` 프리팹의 입력, 이동, 체력 컴포넌트와 Game 카메라 near clip/yaw 기준을 같은 세션에서 검증한다.
- 몬스터 수치와 행동 타입은 ScriptableObject로 분리하고, 애니메이션 파라미터·상태명 규약을 문서화한다.
- README Quick Start 명령을 실행 가능한 형태로 고친다.

## 5. 문서화 상태
design은 6점이다. `docs/기획서.md`와 `docs/컨셉.md`에 장르, 코어 루프, 리스크 구조는 정리되어 있다. 다만 플레이어 행동이 상태와 결과로 어떻게 이어지는지 수치화가 부족하다.

technical은 5점이다. `docs/기술설계서.md`가 Unity 6, URP, NGO, Host-Authoritative 방향은 말하지만 `ClientNetworkTransform`, 서버 권위 이동, 피해 동기화의 구체 흐름은 없다. 신규 개발자가 이 문서만 보고 구현 경계를 잡기는 어렵다.

spec은 4점이다. `docs/던전생성.md`가 던전 생성 과정을 설명하지만 QA 표, 입력값, 기대 결과, 실패 조건이 부족하다. 현재 문서는 분석 보고서에 가깝고 테스트 체크리스트로 바로 쓰기에는 약하다.

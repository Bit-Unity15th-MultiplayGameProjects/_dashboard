# Exit-or-die — 로비 UI·시야각 스냅샷

## 1. 주요 변경사항
- `Menu.unity`에 Play 버튼, 룸 리스트 패널, Join 버튼 프리팹, Lock/Refresh/RoomList/Button 이미지가 추가됐다. 로비 화면의 외형 작업은 분명히 전진했다.
- `LocalMonsterVisionCuller`, `VisionDarkenShaderGlobals`, 전용 셰이더·머티리얼·URP Renderer 설정이 추가되어 플레이어 시야각 기능의 기반이 들어갔다.
- `PlayerController`, `MonsterController`, 몬스터 프리팹 일부가 시야각 기능에 맞춰 수정됐다. 기능 축은 커졌지만 2인 검증 근거는 아직 부족하다.
- TMP 폰트 에셋 대량 변경과 `Menu.unity` 6천 줄 이상 변경이 함께 들어와 리뷰 노이즈가 크다. UI 변경은 프리팹·스크립트 단위로 쪼개는 편이 낫다.

## 2. 코드 품질 리뷰
로비 UI는 시각적으로 전진했지만, 현재 구조는 위험하다. `Menu.unity`에 `Button - Back`의 `m_OnClick`이 비어 있고, 룸 리스트에는 `Name 11`, `64ms`, `Ping` 같은 정적 TMP 더미가 데이터처럼 배치돼 있다. 왜 문제인가 하면, 이런 결함은 컴파일에서 잡히지 않고 시연 중 버튼이 눌리지 않는 형태로만 드러난다. 개선하려면 `LobbyRoomListPresenter` 같은 스크립트가 룸 데이터 수신, row 생성, Join 클릭 위임을 맡고 씬에는 프리팹 템플릿과 컨테이너 참조만 남겨야 한다.

시야각 기능은 별도 컴포넌트와 셰이더 자산으로 분리한 점은 좋다. 다만 전역 셰이더 상태를 건드리는 구조라면 로컬 소유자, 사망, 관전, 씬 전환에서 반드시 상태 초기화를 검증해야 한다. 그렇지 않으면 한 클라이언트의 시야 처리나 이전 씬 상태가 다른 표시 흐름에 남는 문제가 생긴다.

자료구조·복잡도 관점에서는 이번 diff 안에서 확정적으로 지적할 hot path LINQ나 반복 할당 근거는 제한적이다. 다만 룸 리스트는 정적 오브젝트 나열로 끝내면 다음 단계에서 자식 선형 순회와 반복 Instantiate로 흐르기 쉽다. `lobbyId -> row` 매핑과 row 풀을 먼저 잡으면 UI 갱신 비용과 코드 복잡도를 동시에 줄일 수 있다.

## 3. 진행도 평가
네트워크 경로는 NGO 기반 `framework` 프로젝트로 분류한다. 따라서 연결·RPC 자체보다 게임 루프, UI/UX, 2인 happy path, 권한 모델의 명료성이 더 엄격하게 요구된다. 이번 변경은 로비 화면과 시야각 표현에서 전진이 있지만, 시연 가능한 수직 슬라이스를 안정화했다는 증거는 부족하다.

진행도는 53으로 본다. 문서와 콘텐츠 사양은 꽤 쌓였고 화면 구성도 늘었지만, 버튼 배선 공백, 더미 룸 리스트, 공격·체력·이동 결합, NetworkObject 중첩, ClientNetworkTransform 정책 미정이 모두 남아 있다. 현재 리스크는 기능 부족보다 통합 검증 부재 쪽이 더 크다.

## 4. 다음 권장사항
- `Menu.unity`의 Back·Join·Create·Refresh 클릭 경로를 실제 `LobbyManager` 기능에 맞게 재배선하고 PlayMode에서 클릭 검증한다.
- 룸 리스트는 정적 TMP 더미를 제거하고, 로비 id 기준 row 재사용 구조로 데이터 바인딩한다.
- 시야각 기능 추가 후 2인 Game 씬에서 이동, 공격, 사망, 관전, 탈출 투표까지 한 번에 회귀 확인한다.
- `PlayerController`는 이동·공격·체력·UI 책임을 분리해 네트워크 권한과 테스트 범위를 좁힌다.
- 몬스터와 계단·문 프리팹은 `NetworkPrefabs` 등록 후 Host+Client 스폰 로그로 검증한다.

## 5. 문서화 상태
design은 8점이다. `docs/기획서.md`, `docs/컨셉.md`에 코어 루프, 씬 흐름, 상호작용 목표가 충분히 정리돼 있다. 다만 이번에 들어온 로비 룸 리스트 UX와 실패 상태까지는 아직 사양으로 닫히지 않았다.

technical은 7점이다. `docs/기술설계서.md`와 각 사양서가 NGO, 던전 생성, 몬스터 구조를 설명하지만, `ClientNetworkTransform`의 예측·보정·권한 정책은 여전히 코드와 문서 양쪽에서 명확하지 않다.

spec은 8점이다. 던전 생성·몬스터·상호작용 사양서는 표와 QA 체크리스트를 갖춘 실무 문서 수준이다. 그러나 로비 슬롯, 접속 실패, 카메라·스폰 수동 회귀표가 비어 있어 실제 테스트 지시서로는 한 단계 모자라다.

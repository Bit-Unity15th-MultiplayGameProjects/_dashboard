# Exit-or-die — 관전 시야 보강 스냅샷

## 1. 주요 변경사항
- 사망 후 관전 대상의 위치와 조준 방향을 기준으로 화면 마스크와 몬스터 가림을 유지하도록 `LocalMonsterVisionCuller`와 `SpectatorFollower`가 확장됐다.
- Netcode 씬 로드 후 카메라 참조를 다시 찾는 흐름과 관전용 시야 스냅샷이 추가됐다. 층 전환 버그를 직접 겨냥한 변경이라는 점은 좋다.
- UGS 세션에 `eod_listing` 속성을 두고 인게임 세션을 방 목록에서 제외하는 필터가 들어갔다.
- GDD/TDD/컨셉/README와 신규 `docs/시야_및_관전.md`가 현재 조작 체계와 관전 시야 규칙에 맞게 갱신됐다.
- Menu 씬 Scrollbar 스프라이트 연결은 진행됐으나, 실제 해상도별 9-slice 검증 근거는 아직 부족하다.

## 2. 코드 품질 리뷰
관전 시야 문제를 코드와 문서 양쪽에서 맞춘 점은 분명한 전진이다. 특히 `SpectatorVisionOverlaySnapshot`으로 관전용 파라미터를 묶은 결정은 인스펙터 값 드리프트를 줄이는 방향이다. 다만 `PlayerMainCameraMount.cs`가 카메라 마운트, 관전 팔로워, 풀스크린 마스크, 몬스터 렌더러 가림, DTO까지 품으면서 책임이 과도하게 커졌다. 이 구조는 사망·데스폰·씬 재로드가 겹칠 때 디버깅 표면을 넓힌다. `SpectatorFollower`와 시야 판정 유틸은 별도 파일·서비스로 빼는 편이 맞다.

자료구조/복잡도 측면에서는 명확한 개선 필요 지점이 있다. `SpectatorFollower.RefreshSpectatedMonsterCache()`가 0.12초마다 `FindObjectsByType(Type)`로 씬 전체를 훑고, 각 몬스터에서 `GetComponentsInChildren<Renderer>(true)` 배열을 다시 만든다. 현재 비용은 “관전 중 주기적 전체 탐색 + 배열 재수집”이며, 몬스터 수와 씬 전환 횟수에 비례해 불안정해진다. `MonsterController` 등록 리스트와 캐시된 Renderer 묶음을 공통 시야 타깃으로 제공하면 비용도 줄고 생존/관전 가림 로직도 하나로 맞출 수 있다.

네트워킹 쪽은 리플렉션 호출 일부를 직접 호출로 바꾼 점은 좋지만, `NetworkSessionBrowseBridge`를 `LobbyManager.cs` 하단 정적 클래스로 합친 것은 임시 처방에 가깝다. 왜 문제냐면 로비 관리, 세션 공개 상태, 게임 시작 전환 책임이 한 파일에 다시 뭉치기 때문이다. 명시적 서비스와 asmdef 의존 방향을 고정하고, 실패 시 UI 상태까지 분리해야 한다.

## 3. 진행도 평가
구현 경로는 NGO와 Unity Multiplayer Service를 쓰는 framework 기반에, 세션 필터와 권한 규칙 일부를 직접 얹은 hybrid에 가깝다. 절차적 던전, 몬스터, HP/사망, 투표, 관전까지 수직 슬라이스는 보인다. 그래서 진행도는 65로 본다.

다만 위험도는 high다. 이유는 관전·씬 재로드·세션 숨김·몬스터 가림이 모두 네트워크 생명주기와 맞물려 있고, 아직 2인 회귀 검증 근거가 없다. 프리팹 등록, 공격 권위, 이동 예측·보정 문서화도 남아 있어 시연 중 재현 어려운 문제가 나올 가능성이 높다.

## 4. 다음 권장사항
- 관전 몬스터 가림은 `FindObjectsByType` 기반 임시 탐색을 끝내고, `MonsterController` 등록 캐시와 공통 원뿔 판정으로 통합한다.
- 2인 Game 씬에서 near clip, 이동, 공격, 사망, 관전, 다음 층 전환을 한 번에 재현하는 수동 회귀를 먼저 돈다.
- 세션 숨김은 `LobbyManager` 내부 정적 클래스가 아니라 명시적 세션 표시 서비스로 분리하고, `NextStairs` 호출 경로까지 정리한다.
- RoomList는 인게임/잠금/빈 목록/조인 실패를 각각 다른 UI 상태와 로그로 보여준다.
- 공격 판정, NetworkPrefabs 등록, `PlayerController` 책임 분리는 네트워크 시연 안정성에 직접 영향을 주므로 다음 순위로 묶어 처리한다.

## 5. 문서화 상태
design은 8점이다. `docs/기획서.md`와 `docs/컨셉.md`가 WASD 이동, 마우스 조준, 관전 시야까지 현재 구현 방향을 따라갔다. 다만 로비 실패와 회귀 검증 표는 아직 약하다.

technical은 7점이다. `docs/기술설계서.md`는 클래스 구조와 사망·관전 흐름을 꽤 잘 설명한다. 그러나 `ClientNetworkTransform`의 예측·보정 정책과 이동 권한 모델은 여전히 문서에서 명시적으로 닫히지 않았다.

spec은 8점이다. 던전 생성, 몬스터, 상호작용, 시야 문서가 실제 작업 기준으로 쓸 수 있는 수준이다. 단, 카메라·스폰·로비 슬롯의 QA 체크리스트가 테스트 절차로 충분히 쪼개지지 않았다.

## 6. Backlog
- `ClientNetworkTransform` 예측·보정 규칙은 코드와 문서 모두에서 아직 기준이 불명확하다.
- Exit 계단 프리팹의 중첩 `NetworkObject` 구조는 현재 증거상 계속 남아 있다.
- 네트워크 이동 권한과 클라이언트 보정 정책은 TDD에서 명시적으로 닫아야 한다.
- Input System 파일과 구 `InputManager`가 동시에 남아 입력 정책이 혼재한다.
- 로비 슬롯, 접속 실패, 카메라 표시 체크리스트는 문서상 아직 부족하다.
- 시작계단 방향과 바닥 교체 기준은 `DungeonGenerator` 기준으로 재검증이 필요하다.
- 카메라·스폰 회귀는 수동 절차가 없어 다음 변경 때 다시 깨질 위험이 있다.
- TextMesh Pro 예제와 대량 에셋은 제품 자산과 계속 섞여 있다.
- `Assets/asset/Deungeon` 오탈자와 대소문자 문제는 유지보수 비용으로 남아 있다.

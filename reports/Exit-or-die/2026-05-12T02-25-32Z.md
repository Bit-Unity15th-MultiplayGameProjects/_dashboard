# Exit-or-die — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개이며, 실제 diff도 비어 있다. 따라서 이번 스냅샷은 신규 구현 평가가 아니라 이전 위험 항목의 미해결 여부 재검토에 가깝다.
- 직전 TODO였던 `character_barbarian`, Game 카메라 near clip, Wasp NetworkObject 해시는 해결 근거가 없다. 모두 2인 Game 씬 happy path를 직접 깨뜨릴 수 있는 항목이다.
- 현재 문서는 GDD, TDD, 던전 생성 분석 문서가 존재해 방향성은 잡혀 있다. 다만 테스트 가능한 씬·입력·상태·결과 표는 여전히 부족하다.
- 직전 backlog 중 완전히 해결됐다고 볼 근거가 없으므로 `resolved_from_backlog`는 비워 두고 모두 이월한다.

## 2. 코드 품질 리뷰
이번 diff가 비어 있어 신규 코드 스타일, 중복, 예외 처리 악화는 확인되지 않는다. 그러나 미해결 항목의 성격은 가볍지 않다. 플레이어 프리팹에서 입력·이동 책임이 사라졌거나 대체 컴포넌트 기준이 불명확하면 씬은 열려도 실제 플레이가 멈춘다. 이 문제는 “프리팹에서 보이는 컴포넌트가 현재 권한 모델의 진실인가”를 확인해야 하며, 삭제 의도와 대체 경로를 명확히 기록해야 한다.

Unity 관점에서는 음수 near clip이 특히 위험하다. 카메라 프레이밍을 near plane으로 우회하면 렌더링·컬링 계약 밖의 값에 의존하게 되며, 환경이나 Unity 버전 차이에서 증상이 흔들릴 수 있다. 위치, rotation, orthographic size를 기준으로 다시 맞추는 것이 맞다. Wasp와 계단 프리팹의 NetworkObject 해시·중첩 구조도 런타임 스폰 실패로만 드러날 가능성이 높아, Prefab 등록표와 실제 prefab 구조를 한 번에 대조해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서는 자료구조/복잡도 관련 명확한 지적 없음. 다만 과거 ledger에 있던 `FixedUpdate` 전수 검색, LateUpdate 탐색, 던전 배치 검증 이슈는 현재 diff로 해소된 근거가 없으므로 다음 코드 변경 때 성능 계측과 함께 재확인해야 한다.

## 3. 진행도 평가
네트워크 구현 경로는 Netcode for GameObjects 기반이므로 `framework`로 분류한다. 이 경로는 연결·RPC·직렬화 비용을 프레임워크가 대신해 주는 만큼, 제출 가능한 수직 슬라이스에서는 2인 입장, 스폰, 이동, 카메라, 전투 기본 흐름이 반복 재현돼야 한다.

현재는 문서와 뼈대는 있으나, 카메라·플레이어 프리팹·NetworkObject 해시·권한 정책이 동시에 미확정이다. 변경 없는 스냅샷이므로 전진도도 인정하기 어렵다. 제출 가능성 기준 진행도는 42로 본다.

## 4. 다음 권장사항
- `character_barbarian` 프리팹에서 플레이어 입력·이동 책임이 어디에 있는지 확인하고, 삭제된 `PlayerController`가 의도인지 복구 대상인지 결정한다.
- Game 카메라 near clip을 양수로 되돌리고, 카메라 위치와 orthographic size로 같은 화면 구도를 재현한다.
- Wasp 프리팹의 NetworkObject 해시 변경 원인과 `DefaultNetworkPrefabs` 등록 상태를 비교해 2인 스폰 실패 가능성을 제거한다.
- 카메라 yaw 기본값을 코드, 씬, 플레이어 리그 중 하나의 기준으로 고정해 첫 프레임 스냅 회귀를 막는다.
- 2인 Game 씬에서 카메라, 이동, 스폰을 한 번에 확인하는 수동 체크리스트를 문서에 남기고 실제 녹화로 검증한다.

## 5. 문서화 상태
design은 6점이다. `docs/기획서.md`와 `docs/컨셉.md`에 장르, 코어 루프, 리스크 구조, 시야 컨셉이 정리돼 있어 방향성은 읽힌다. 다만 플레이 상태 전이와 실패 조건이 아직 기획 검증표 수준은 아니다.

technical은 4점이다. `docs/기술설계서.md`가 Unity 6, URP, NGO, Host-Authoritative 방향을 적고 있으나, 실제 `ClientNetworkTransform`, Lobby, Scene 전환, 권한 주체를 코드 단위로 연결하지 못한다. 신규 개발자가 문서만 보고 구조를 따라가기에는 부족하다.

spec은 3점이다. `docs/던전생성.md`는 알고리즘 분석 문서로 유용하지만, 입력값, 기대 출력, 검증 케이스, QA 체크리스트가 사양서 형태로 고정돼 있지 않다. `_sample/docs` 기준의 테스트 가능한 테이블과 상태 매핑이 필요하다.

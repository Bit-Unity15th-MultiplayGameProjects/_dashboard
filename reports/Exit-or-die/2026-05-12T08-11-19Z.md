# Exit-or-die — 문서 사양화 스냅샷

## 1. 주요 변경사항
- 이번 커밋은 코드보다 문서 전진이 핵심이다. 던전 생성, 몬스터, 상호작용 시스템 사양서가 추가되고 GDD/TDD가 크게 확장됐다.
- `docs/specs/사양서_던전생성.md`는 생성 단계, RoomConfig, 몬스터 스폰, 네트워크 Spawn 흐름, QA 항목까지 구체화했다.
- `docs/specs/사양서_몬스터.md`는 6종 몬스터, 상태 전이, 애니메이션 상태명, NetworkVariable 동기화 원칙을 정리했다.
- 다만 이번 diff는 문서 중심이라 실제 프리팹 등록, 계단 NetworkObject 중첩, 2인 Game 씬 회귀 검증은 아직 닫혔다고 볼 근거가 부족하다.

## 2. 코드 품질 리뷰
문서 품질은 분명히 좋아졌다. 특히 던전 생성 사양서가 서버 주도 생성, `NetworkObject.Spawn(true)`, 층 전환 플로우, QA 체크리스트를 명시한 점은 이후 구현 판단 기준으로 쓸 수 있다. 이는 단순 아이디어 문서가 아니라 개발자가 “무엇을 구현했고 무엇을 검증해야 하는지”를 대조할 수 있는 수준이다.

위험 신호도 명확하다. 문서가 좋아진 것과 런타임 안정성이 확보된 것은 별개다. 현재 증거상 `stairs_Lock.prefab`에는 `NetworkObject`가 두 번 등장한다. NGO 프리팹에서 중첩 NetworkObject는 동적 스폰 실패나 소유권 추적 오류를 런타임까지 숨길 수 있으므로, 프리팹 루트를 하나의 NetworkObject 기준으로 정리해야 한다.

자료구조/복잡도 관점에서는 `MonsterController`의 플레이어 조회가 아직 완전히 안심할 단계가 아니다. 문서는 0.3초 캐시와 `FindGameObjectsWithTag` 미사용을 말하지만, 이전 코드 이슈의 핵심은 플레이어 생명주기와 연결된 등록 캐시가 아니라는 점이다. 전역 태그 조회나 주기 갱신은 플레이어 수가 적어도 네트워크 스폰/디스폰 타이밍에서 누락을 만들 수 있으므로, Spawn/Despawn 등록 리스트로 고정하는 편이 더 단순하고 검증 가능하다.

## 3. 진행도 평가
네트워크 구현 경로는 NGO를 쓰는 `framework` 프로젝트로 본다. 프레임워크가 연결·동기화 기반을 담당하므로, 평가 기준은 문서 작성보다 2인 happy path, 프리팹 등록, 씬 전환, 상호작용 투표 검증 쪽에 더 엄격해야 한다.

진행도는 58%로 본다. 문서와 사양은 한 단계 크게 전진했지만, 제출 가능한 수직 슬라이스 관점에서는 NetworkPrefabs 등록, 중첩 NetworkObject, 입력 정책 혼재, Game 씬 수동 회귀가 여전히 시연 리스크다. 그래서 진행도는 중간 이상으로 올리되 risk는 high를 유지한다.

## 4. 다음 권장사항
- `DefaultNetworkPrefabs.asset`에 몬스터 6종, 계단, 문 프리팹을 등록하고 호스트+클라이언트 2인 스폰을 확인한다.
- `stairs_Lock.prefab`, `stairs_wide_Lock.prefab`의 NetworkObject를 루트 1개 구조로 정리한다.
- Game 씬에서 카메라 yaw, 관전 전환, Exit/NextStairs 투표 표시와 씬 이동을 한 번에 회귀 검증한다.
- InteractionButton의 TMP 라벨, 표시 조건, 실제 클릭 입력을 씬 기준으로 확인한다.
- 클릭 이동 입력을 Input System 액션으로 통일하고 구 `Input` 호출 잔존 여부를 제거한다.

## 5. 문서화 상태
design은 8점이다. `docs/기획서.md`가 코어 루프, 입력, 탈출/계단 투표, 밸런스 수치까지 확장되어 현재 프로젝트 방향을 이해하기 충분하다. 다만 일부 수치는 “플레이테스트로 확정”에 머물러 최종 기준표로는 더 고정해야 한다.

technical은 7점이다. `docs/기술설계서.md`와 던전 생성 사양서가 서버 생성, 네트워크 Spawn, 씬 생명주기를 다루지만, 클라이언트 이동 예측·보정과 권한 모델은 아직 코드와 연결된 결정 문서로 부족하다.

spec은 9점이다. 던전 생성, 몬스터, 상호작용 사양서가 `_sample/docs`에 가까운 구조를 갖췄고, 상태·수치·QA 체크리스트가 실제 테스트 기준으로 쓸 수 있다. 이번 커밋의 가장 큰 성과다.

## 6. Backlog
- `ClientNetworkTransform` 예측·보정 규칙은 아직 코드와 문서 사이에서 판정 기준이 불명확하다.
- 계단 프리팹의 중첩 `NetworkObject`는 NGO 스폰 안정성을 직접 위협하므로 장기 부채가 아니라 즉시 줄여야 할 리스크다.
- Input System과 구 InputManager가 함께 남아 있어 입력 정책이 흔들린다.
- 로비 슬롯, 접속 실패, 카메라·스폰 회귀 체크리스트는 아직 문서화가 부족하다.
- Recovery 씬, 대량 예제 에셋, `Deungeon` 오탈자는 빌드와 리뷰 비용을 계속 키운다.

## 7. 이전 Backlog 해결
- “문서가 컨셉 중심이며 테스트 가능한 수치·상태 사양이 부족함”은 해결로 본다. 현재 `docs/specs`의 던전 생성·몬스터 사양서와 확장된 GDD가 상태표, 수치, QA 체크리스트를 포함해 테스트 가능한 문서 수준으로 올라왔다.

# Exit-or-die — 상호작용·전투 스냅샷

## 1. 주요 변경사항
- `MonsterController`가 `NetworkBehaviour`로 전환되고 서버 전용 AI, 애니메이션 `NetworkVariable`, 몬스터 정적 리스트 캐시가 추가됐다.
- 문, 문게이트, 다음 계단, 탈출구 상호작용과 투표 흐름이 새 컴포넌트로 분리됐다. 기능 축은 크게 전진했다.
- `PlayerController`에 HP 감소, 사망 애니메이션, 관전 전환, 전원 사망 시 메뉴 복귀 흐름이 추가됐다.
- `Game` 씬에 상호작용 버튼과 상태 텍스트 UI가 들어갔고, `character_barbarian` 프리팹에 `PlayerInteraction`이 붙었다.
- GDD/TDD/던전 문서가 실제 구현 기준으로 갱신됐다. 다만 검증표와 네트워크 정책 문서화는 아직 부족하다.

## 2. 코드 품질 리뷰
이번 변경은 “플레이 가능한 루프” 쪽으로 분명히 전진했다. 특히 `Interactable` 기반으로 문·계단·탈출구를 같은 추상화 아래 둔 점, 투표자 목록을 `HashSet<ulong>`으로 관리한 점, 몬스터 간 분리 계산에서 매번 `FindObjectsByType`를 돌지 않고 `AllMonsters`를 쓰도록 바꾼 점은 좋은 판단이다. 중복 비용을 줄이면서 코드 의도도 더 명확해졌다.

반대로 `PlayerInteraction`은 `InteractionButton`, `ExitStatusText` 같은 씬 오브젝트 이름 문자열과 `FindObjectsByType`에 의존한다. 왜 문제냐면 UI 이름 변경이나 씬 복제만으로 상호작용이 조용히 비활성화될 수 있기 때문이다. 개선하려면 Game 씬 전용 UI 바인더를 두고, 프리팹 또는 씬 참조를 인스펙터에서 명시적으로 주입하는 방식이 낫다. 현재 diff상 버튼에는 이미지 컴포넌트만 보이고 TMP 라벨 근거가 약하므로 실제 버튼 표시도 반드시 확인해야 한다.

자료구조·복잡도 관점에서는 개선과 잔여 문제가 같이 있다. `AllMonsters`와 `HashSet`은 적절하지만, `MonsterController.RefreshPlayerCache()`는 여전히 0.3초마다 `FindGameObjectsWithTag("Player")`로 전역 조회를 반복한다. 플레이어 수가 적어 당장 치명적이지는 않지만 서버 AI hot path에 남아 있다. 플레이어 `OnNetworkSpawn/OnNetworkDespawn` 등록 리스트로 바꾸면 비용과 실패 지점을 동시에 줄일 수 있다.

## 3. 진행도 평가
네트워크 구현 경로는 NGO와 Unity Multiplayer Service를 쓰는 `framework` 경로다. 연결·RPC·동기화 비용을 프레임워크가 덜어주는 만큼, 평가는 “2인 반복 시연 가능한 수직 슬라이스” 기준으로 봐야 한다. 이번 범위에서 전투, HP, 사망, 관전, 상호작용, 투표가 붙어 60% 수준까지 올라왔다.

다만 `NetworkPrefabs` 등록 근거가 약하고, 계단 프리팹의 중첩 `NetworkObject`가 남아 있으며, 이동 입력 정책과 클라이언트 보정 문서가 여전히 흔들린다. 기능은 늘었지만 네트워크 회귀 테스트가 부족해 리스크는 `high`로 유지한다.

## 4. 다음 권장사항
- 몬스터 6종과 계단·문 프리팹을 NetworkPrefabs에 등록하고 2인 스폰 검증.
- Exit 계단 프리팹의 중첩 NetworkObject를 루트 1개 구조로 정리.
- 2인 Game 씬에서 카메라 yaw·관전·탈출 투표 회귀 검증.
- Game 씬 InteractionButton에 실제 라벨 TMP와 클릭 동작 검증.
- 클릭 이동 입력을 Input System 액션으로 이전하고 구 Input 호출 제거.
- MonsterController 플레이어 조회를 태그 검색 주기 캐시에서 등록 캐시로 교체.
- 몬스터별 스탯과 행동 타입을 ScriptableObject 데이터로 분리.
- 몬스터 애니메이션 상태명과 NetworkVariable byte 매핑 규약 문서화.

## 5. 문서화 상태
`design`은 7점이다. `docs/기획서.md`가 코어 루프, 조작, HP, 몬스터, 투표, 관전까지 현재 구현을 꽤 잘 반영한다. 다만 “왜 이 밸런스인가”와 플레이 경험 목표는 아직 얕다.

`technical`은 7점이다. `docs/기술설계서.md`는 컴포넌트 계층, 네트워크 동기화 대상, 씬 전환, 최적화 표를 제공해 신규 유입자가 구조를 파악할 수 있다. 그러나 이동 권한, 예측·보정, 실패 처리 정책은 아직 명시가 부족하다.

`spec`은 6점이다. 몬스터 스탯, 입력, 상태 흐름, 던전 생성 단계는 있으나 `_sample/docs` 수준의 테스트 가능한 체크리스트, 데이터 스키마, 씬별 기대 결과표는 부족하다.

# Exit-or-die — 체력바 UI 스냅샷

## 1. 주요 변경사항
- `character_barbarian.prefab` 에 월드 스페이스 `Slider - HP` 가 추가됐고, 기본 비활성 상태에서 Game 씬 진입 시 켜지도록 연결됐다.
- `PlayerController` 가 Game 씬 판별, HP 슬라이더 탐색, 월드 캔버스 카메라 빌보드 회전을 담당하도록 확장됐다.
- `LobbyPlayerState` 는 Game 씬에서 READY 텍스트를 숨기도록 바뀌었고, 씬 로드 이벤트 구독 해제가 추가됐다.
- 캐릭터 프리팹의 `NetworkObject.GlobalObjectIdHash` 값이 변경됐다. NGO 스폰 호환성 검증이 반드시 필요하다.

## 2. 코드 품질 리뷰
이번 커밋은 “머리 위 체력바를 보이게 한다”는 목표 자체는 명확하다. 특히 `sceneLoaded` 이벤트를 중복 구독하지 않도록 먼저 해제한 점, Game 씬에서 로비 READY 텍스트를 숨긴 점은 방향이 좋다. 다만 구현 위치가 좋지 않다. `PlayerController` 가 이동, 씬 복구, HP UI 표시, 월드 캔버스 빌보드까지 떠안고 있어 책임이 섞였다. 이후 피격, 사망, 부활, 관전 UI가 붙으면 플레이어 컨트롤러가 디버깅하기 어려운 중심 병목이 된다. HP 바는 `PlayerOverheadHud` 같은 별도 컴포넌트로 분리하고, 컨트롤러는 체력 상태 변경 이벤트만 전달하는 편이 낫다.

하드코딩 문자열도 위험하다. `"Slider - HP"`, `"Canvas"`, `"Game"` 에 의존하면 프리팹 이름 변경이나 씬명 변경이 런타임 무표시 버그로 이어진다. 특히 `FindDeepChildByName` 은 한 번만 도는 구조라 비용은 제한적이지만, 실패해도 조용히 넘어간다. 직렬화된 `Slider`, `Canvas` 참조를 명시하고 누락 시 경고를 남겨야 한다.

자료구조/복잡도 관점에서는 `LateUpdate()` 의 카메라 탐색이 핵심이다. `Camera.main` 이 없을 때 모든 플레이어 인스턴스가 매 프레임 `FindObjectsByType<Camera>` 로 전체 검색을 수행할 수 있다. 현재 비용은 플레이어 수와 카메라 수에 비례해 반복된다. 씬 로드 시 활성 카메라를 1회 캐시하거나 카메라 매니저에서 참조를 주입하면 비용과 실패 지점을 동시에 줄일 수 있다.

## 3. 진행도 평가
이 프로젝트는 NGO 기반이므로 구현 경로는 `framework` 로 분류된다. 네트워크 연결·RPC·동기화 기반 비용은 프레임워크가 줄여주지만, 그만큼 실제 게임 루프, UI 상태, 전투/체력 규칙의 완성도가 더 중요하다. 이번 변경은 시각적 전진은 있으나 체력 값이 서버 권위 상태와 연결되지 않아 기능 완성으로 보기는 어렵다.

진행도는 49로 본다. 로비와 Game 씬 전환 UI가 조금 정리됐고 캐릭터 HUD도 시작됐지만, 남은 핵심 리스크가 네트워크 권한, 프리팹 스폰 안정성, 문서화 부재에 몰려 있다. 특히 `GlobalObjectIdHash` 변경은 2인 접속 시 런타임에서만 터질 수 있으므로 위험도는 high 다.

## 4. 다음 권장사항
- 체력바를 서버 권위 체력 상태와 `NetworkVariable` 표시로 연결한다.
- `NetworkObject.GlobalObjectIdHash` 변경 뒤 호스트·클라이언트 2인 스폰 회귀 테스트를 수행한다.
- `PlayerController` 의 HP UI·빌보드 책임을 별도 컴포넌트로 분리한다.
- `LateUpdate` 카메라 탐색을 씬 로드 시 캐시하는 흐름으로 바꾼다.
- Game 씬 UI 표시 규칙을 공통 헬퍼와 문서로 정리한다.

## 5. 문서화 상태
design 점수는 4다. 컨셉과 방향성은 존재하는 것으로 보이나, 이번 커밋의 체력·피격·사망 UX가 어떤 플레이 경험을 목표로 하는지 문서상 근거가 보이지 않는다.

technical 점수는 3이다. NGO 권한 모델, 씬 전환 시 UI 생명주기, 체력 상태의 소유자가 서버인지 클라이언트인지 설명이 부족하다. 신규 팀원이 이 문서만 보고 안전하게 네트워크 HUD를 붙이기는 어렵다.

spec 점수는 2다. 체력 최대값, 피해량, 회복, 사망 조건, HP 바 표시/숨김 조건 같은 테스트 가능한 수치와 상태표가 없다. `_sample/docs` 수준의 체크리스트형 사양과는 아직 거리가 크다.

## 6. Backlog
- NGO 중첩 프리팹 구조와 캐릭터 프리팹 해시 변경은 스폰 안정성 검증 전까지 계속 위험하다.
- 네트워크 이동 권한, 예측·보정 정책, Game HUD 상태 소유자가 문서화되지 않았다.
- 시작계단 방향 규약, Menu Setting 버튼 오연결, 입력 시스템 이중화는 아직 해결 근거가 없다.
- 문서는 여전히 컨셉 중심이고 체력·레벨·네트워크 상태를 테스트할 수 있는 사양이 부족하다.
- Recovery 씬, 대량 예제 에셋, `Deungeon` 폴더명 오탈자는 변경 노이즈와 유지보수 비용을 계속 만든다.

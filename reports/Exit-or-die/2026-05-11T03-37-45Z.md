# Exit-or-die — 네트워크 이동 보강 스냅샷

## 1. 주요 변경사항
- 로비에 Ready/Start 버튼, 준비 상태 텍스트, `LobbyPlayerState` 네트워크 변수가 추가됐다.
- 플레이어 이동이 키보드 직접 이동에서 우클릭 목적지 요청, 서버 검증, 서버 이동 방식으로 바뀌었다.
- 던전 시작 계단 주변 스폰 위치 계산, 보행 가능 셀 그리드, A* 경로 탐색 코드가 추가됐다.
- 로컬 플레이어 카메라를 캐릭터에 붙이고, 고정 yaw와 벽 가림 보정 로직을 넣었다.
- 바닥과 벽 프리팹에 레이어와 콜라이더가 대량 추가되어 이동 충돌 기반이 마련됐다.

## 2. 코드 품질 리뷰
이번 변경의 방향 자체는 좋다. 벽을 뚫는 직접 `transform.position` 이동에서 벗어나 서버가 목적지를 검증하고 이동을 수행하려는 선택은 NGO 프로젝트에서 필요한 전진이다. 또한 바닥/벽 레이어를 나누고 던전 입구 스폰을 서버에서 처리하려는 점도 플레이 가능한 수직 슬라이스에 가까워지는 변화다.

다만 현재 구현은 위험 신호가 분명하다. `character_barbarian.prefab`에 `PlayerController`가 두 번 붙어 있다. 이 상태에서는 owner 입력, `ServerRpc`, 서버 `FixedUpdate` 이동이 중복 실행될 수 있다. 이는 단순 정리 문제가 아니라 멀티플레이 이동 판정이 흔들리는 원인이다. 또한 `PlayerController`는 서버 이동을 수행하는데 프리팹에는 `ClientNetworkTransform`이 남아 있어 권한 모델이 코드만 보고 해석되지 않는다. "누가 위치의 최종 권위자인가"가 불분명하면 이후 보정, 순간이동, 재접속 버그를 추적하기 어렵다.

자료구조/복잡도 관점에서는 `LobbyUIManager.CheckCanStart()`가 host의 `Update()`마다 `FindObjectsByType<Eod.UI.LobbyPlayerState>()`를 호출한다. 현재 비용은 매 프레임 씬 전체 검색과 배열 할당이다. 추천 흐름은 접속/스폰 시 플레이어 상태 목록을 캐시하고, `NetworkVariable.OnValueChanged` 또는 접속 이벤트에서 버튼 상태만 갱신하는 방식이다. 이 프로젝트는 네트워크 UI가 계속 늘어날 가능성이 높으므로 지금 이벤트 기반으로 바꾸는 편이 유지보수와 성능 모두에 의미가 있다.

## 3. 진행도 평가
구현 경로는 NGO를 사용하는 `framework` 경로다. 연결, RPC, 동기화 기반은 프레임워크가 담당하므로 컨텐츠와 플레이 루프 안정성 기대치를 높게 잡아야 한다. 이번 커밋은 로비 준비, 씬 전환, 카메라, 클릭 이동까지 이어지는 낮은 수준의 수직 슬라이스를 만들었다.

다만 2인 happy path를 반복 시연하기에는 아직 불안하다. 중복 `PlayerController`, 이동 권한 혼재, A* 미연결, 매 프레임 UI 검색이 동시에 존재한다. 진행도는 46으로 본다. 전진은 확실하지만, 다음 단계는 기능 추가보다 이동/동기화 안정화가 우선이다.

## 4. 다음 권장사항
- `character_barbarian`에서 중복 `PlayerController`를 즉시 제거하고 프리팹 NetworkBehaviour 순서를 확인한다.
- 서버 권위 이동을 유지할지, owner authority 이동을 유지할지 결정하고 `ClientNetworkTransform` 정책을 문서와 코드에 맞춘다.
- `DungeonGenerator.TryFindFloorNavPath()`를 실제 서버 이동 검증에 연결하거나, 쓰지 않을 코드라면 제거한다.
- `LobbyUIManager`의 준비 상태 체크를 `Update()` 검색이 아니라 이벤트 기반 캐시로 바꾼다.
- 우클릭 입력을 포함해 Input System과 구 Input API 중 하나로 입력 정책을 통일한다.

## 5. 문서화 상태
design 점수는 4다. 컨셉과 큰 방향은 있는 것으로 보이나, 이번 변경처럼 로비 준비와 던전 입장 흐름이 추가됐을 때 플레이어 경험 목표와 실패 조건이 문서로 따라오지 않는다.

technical 점수는 3이다. NGO를 쓰는 프로젝트인데 이동 권한, `NetworkTransform` 권위, 씬 전환 후 플레이어 재배치 흐름이 설명되어 있지 않다. 신규 개발자는 코드와 프리팹을 동시에 열어 추론해야 한다.

spec 점수는 2다. 입력, 상태, 결과 매핑이 테스트 체크리스트 수준으로 정리되어 있지 않다. 예를 들어 Ready 버튼 상태, Start 가능 조건, 우클릭 이동 실패 조건, 벽/바닥 레이어 규칙이 표로 고정되어야 한다.

## 6. Backlog
- NetworkObject 중첩 프리팹 구조 문제는 이번 범위에서 Exit 계단 프리팹 변경 근거가 없어 유지한다.
- 네트워크 이동 권한과 예측·보정 정책은 코드가 더 늘었지만 문서화되지 않았다.
- 시작계단 주변 스폰은 보강됐으나 방향 규약과 바닥 교체 기준의 최종 정합성은 아직 불명확하다.
- Menu 씬 Setting 버튼 연결 문제는 제공 diff만으로 해결을 확인할 수 없다.
- 문서는 여전히 테스트 가능한 수치와 상태 사양이 부족하다.
- Input System과 구 InputManager 공존 문제는 `Input.GetMouseButton` 추가로 더 명확히 남았다.
- Recovery 씬, 대량 예제 에셋, `Deungeon` 폴더명 오탈자는 저장소 구조 부채로 계속 남아 있다.

# Exit-or-die — 몬스터 AI 도입 스냅샷

## 1. 주요 변경사항
- 벌·쥐·뱀·개구리·거미 계열 몬스터 에셋과 Wasp 프리팹, 애니메이션 컨트롤러가 추가되었다.
- `MonsterController` 가 신규 추가되어 감지, 배회, 추격, 공격, 도주, 끼임 상태를 한 클래스에서 처리한다.
- `DefaultNetworkPrefabs.asset` 와 Wasp 프리팹에 NGO `NetworkObject` 연결이 들어갔다.
- `DungeonGenerator` 와 전용 에디터에 시작 계단 배치 관련 경고 표시가 추가된 것으로 보인다.
- 플레이어 카메라 방향 전환 입력이 `PlayerMainCameraMount` 쪽에 추가되었다.

## 2. 코드 품질 리뷰
몬스터 기능을 실제 씬에 올릴 수 있는 수준까지 밀어 넣은 점은 분명한 전진이다. 다만 `MonsterController.cs` 하나가 상태머신, 물리 이동, 타겟 탐색, 분리 처리, 공격, 애니메이션 제어, 에디터 검증까지 모두 담당한다. 지금은 기능 추가 속도에는 유리하지만, 다음 단계에서 몬스터 종류가 늘면 수정 범위가 계속 커진다. 상태 전이와 이동 정책, 전투 판정은 최소한 별도 메서드군이 아니라 별도 책임 단위로 분리해야 한다.

자료구조/복잡도 관점에서는 명확한 위험이 있다. `FindGameObjectsWithTag("Player")` 와 `FindObjectsByType<MonsterController>` 가 감지·분리 흐름에서 반복 호출된다. 몬스터 수가 늘면 FixedUpdate 계열 hot path 에서 전체 씬 검색과 배열 할당 비용이 누적된다. 플레이어와 몬스터를 등록·해제하는 registry 또는 manager 캐시를 두고, 근거리 판정만 고정 배열/리스트로 순회하는 구조가 맞다.

또한 `PerformAttack()` 이 플레이어를 실제 피해 시스템으로 처리하지 않고 로그로 끝난다. 이는 조용히 실패하는 게임플레이 결함이다. 왜 문제냐면 공격 애니메이션은 보이는데 룰은 적용되지 않아 QA 가 “동작하는 것처럼 보이는 미구현”을 놓치기 쉽기 때문이다. 개선하려면 `IDamageable` 같은 명시적 계약을 만들고 서버 권한에서만 피해를 확정해야 한다.

## 3. 진행도 평가
이 프로젝트는 NGO를 쓰는 `framework` 경로로 분류된다. 프레임워크가 스폰과 동기화 기반을 제공하는 만큼, 현재 평가는 네트워크 기반 구축보다 게임 루프와 권한 일관성, 콘텐츠 통합도를 더 엄격하게 본다. 몬스터 콘텐츠 추가는 수직 슬라이스의 밀도를 높였지만, 공격·이동 권한이 서버 기준으로 정리되지 않아 멀티플레이 시연 안정성은 아직 낮다.

진행도는 47으로 본다. 던전, 플레이어, 네트워크 프리팹, 몬스터가 같은 방향으로 붙고 있으나 반복 시연 가능한 2인 플레이 루프라고 보기에는 권한 모델, 전투 결과, 성능 hot path, 문서 사양이 모두 부족하다. 방향은 전진이지만 리스크는 높다.

## 4. 다음 권장사항
- `MonsterController` 이동·공격·상태 변경을 서버 권한 기준으로 제한한다.
- `FindGameObjectsWithTag`, `FindObjectsByType` 반복 호출을 registry 캐시로 대체한다.
- 공격 로그를 제거하고 실제 플레이어 체력/피해 인터페이스와 연결한다.
- 몬스터별 이동 속도, 감지 거리, 행동 타입을 ScriptableObject 데이터로 분리한다.
- Blender/OBJ/FBX 원본과 런타임 프리팹 보관 기준을 정하고 불필요한 중복을 줄인다.

## 5. 문서화 상태
design 점수는 4다. 컨셉과 방향성은 있는 것으로 보이나 이번 변경의 몬스터별 역할, 등장 조건, 플레이 경험 목표가 문서로 확인되지 않는다. 몬스터가 왜 필요한지보다 “추가했다”에 머문 상태다.

technical 점수는 3이다. NGO를 사용하는 프로젝트인데 이동 권한, 몬스터 AI 실행 주체, 피해 확정 주체가 문서화되지 않았다. 신규 유입자가 코드를 보기 전에는 서버와 클라이언트 중 누가 상태를 소유하는지 판단하기 어렵다.

spec 점수는 2다. 몬스터 감지 범위, 공격 쿨다운, 피해량 같은 수치는 코드와 프리팹에만 있고 테스트 가능한 표준 사양으로 정리되지 않았다. `_sample/docs` 수준의 QA 체크리스트와 데이터 스키마가 필요하다.

## 6. Backlog
- `NetworkObject` 중첩 프리팹 문제는 이번 변경에서 해결 근거가 없다.
- 네트워크 이동 권한과 예측·보정 정책 문서 부재가 계속 핵심 리스크다.
- Menu 씬 Setting 버튼 연결 오류는 관련 수정이 확인되지 않아 유지한다.
- 시작계단 경고는 추가됐지만 방향 규약과 바닥 교체 기준 해결까지는 확인되지 않는다.
- 문서, 입력 정책, Recovery 씬, 대량 에셋 혼재, `Deungeon` 폴더명 문제는 그대로 남아 있다.

# Exit-or-die — 로비 네트워크 표시 스냅샷

## 1. 주요 변경사항
- `Player.prefab` 이 삭제되고 `character_barbarian` 이 NGO PlayerPrefab 으로 교체되었다.
- Menu 씬에 대기 패널, CharacterCamera, P1~P4 스폰 포인트, 로비 스포너가 추가되었다.
- `ClientNetworkTransform` 이 클라이언트 권위에서 서버 권위로 변경되었다.
- 로비 캐릭터 표시를 위해 `LobbyPlayerSpawner`, `LobbySpawnPointsLayout` 이 새로 추가되었다.
- Player 레이어, 품질 설정, UnityConnect, runInBackground 등 프로젝트 설정도 함께 변경되었다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 “대기방에서 플레이어가 보이게 한다”는 목표가 명확하다. 패널을 분리하고 전용 CharacterCamera 를 둔 점은 로비 UI와 캐릭터 표시를 분리하려는 시도로 볼 수 있다. 그러나 구현은 아직 안정 단계가 아니다. `LobbySpawnPointsLayout` 하나가 카메라 계산, 스폰 포인트 배치, 더미 아바타 생성, 네트워크 플레이어 숨김, 컴포넌트 제거까지 담당한다. 이 구조는 수정 지점이 한 파일에 몰려 회귀가 크게 난다. 레이아웃 계산, 로컬 표시, 네트워크 슬롯 관리를 클래스로 분리해야 한다.

네트워크 권한 모델은 더 위험하다. `ClientNetworkTransform` 은 서버 권위로 바뀌었지만 `PlayerController` 는 여전히 오너 클라이언트가 `transform.position` 을 직접 바꾼다. 이 조합은 비호스트 클라이언트에서 입력은 되는 것처럼 보이다가 서버 동기화로 되돌아가는 흔들림 또는 이동 불가를 만들 수 있다. 서버 권위라면 입력을 ServerRpc 로 보내고 서버가 이동을 확정해야 하며, 클라이언트 예측을 쓸 경우 보정 규칙을 명시해야 한다.

자료구조/복잡도 관점에서도 명확한 지적이 있다. `LobbySpawnPointsLayout.LateUpdate()` 는 매 프레임 `FindObjectsByType<NetworkObject>`, 새 `List`, `HashSet`, 정렬을 수행한다. 현재 구조의 비용은 프레임당 전체 NetworkObject 검색과 GC 할당이다. 추천 흐름은 접속/해제/스폰 이벤트에서 플레이어 목록을 캐시하고, 변경 시에만 슬롯 표시를 갱신하는 방식이다. 로비는 네트워크 상태 변화가 이벤트성이라 매 프레임 전수 검색이 필요 없다.

## 3. 진행도 평가
이전 대비 로비 화면과 캐릭터 표시에는 분명한 전진이 있다. 특히 바로 Game 씬으로 넘기지 않고 대기 패널을 보여주는 흐름은 멀티플레이 UX의 기반이다. 다만 커밋 대부분이 “현재 호스트에서만 원활함”, “화면 비율 보정”, “Y로테이션 수정”처럼 증상 대응에 가깝다.

완성도는 약 34%로 본다. 핵심 로비 표시 작업은 시작됐지만 PlayerObject 생성 조건, 서버 권위 이동, 재접속 슬롯, 문서화가 아직 고정되지 않았다. 지금 이 상태에서 기능을 더 얹으면 네트워크 디버깅 비용이 급격히 증가한다.

## 4. 다음 권장사항
- 서버 권위 이동 정책과 예측·보정 문서를 먼저 고정해야 한다.
- AutoSpawn 비활성 상태에서 PlayerObject 를 누가, 언제, 어떻게 생성하는지 구현해야 한다.
- `LobbySpawnPointsLayout` 을 레이아웃·카메라·표시 책임으로 분리해야 한다.
- `LateUpdate` 의 NetworkObject 전수 검색과 임시 컬렉션 할당을 이벤트 캐시로 바꿔야 한다.
- 연결 해제 후 빈 슬롯을 재사용하도록 슬롯 배정 규칙을 수정해야 한다.

## 5. 문서화 상태
design 점수는 6점이다. `docs/기획서.md`, `docs/컨셉.md`, `docs/던전생성.md`가 던전 로그라이크 전환, 코어 루프, 던전 생성 방향을 설명하고 있어 단순 placeholder 상태는 아니다. 다만 이번 로비 UX와 멀티플레이 대기 흐름이 디자인 목표로 연결되는 설명은 아직 약하다.

technical 점수는 6점이다. `docs/기술설계서.md`에 Unity 6, URP, NGO Host-Authoritative 모델, 던전/시야/세션 보호 구조가 정리돼 있다. 다만 `ClientNetworkTransform`의 예측·보정, PlayerObject 생성, 로컬 더미 표시 같은 이번 구현 세부는 문서와 코드가 아직 맞물리지 않는다.

spec 점수는 4점이다. 던전 생성과 생존 메커닉의 일부 수식·상태는 문서에 있지만, 입력, 로비 슬롯, 접속/해제, 카메라 비율별 표시 기준은 테스트 가능한 표나 체크리스트로 정리되지 않았다.

## 6. Backlog
- NetworkObject 중첩 프리팹 구조는 이번 범위에서 검증 또는 수정 근거가 없다.
- `ClientNetworkTransform`의 서버 권위 전환은 문서의 큰 방향과 맞지만, 예측·보정 규칙과 PlayerController 입력 경로는 아직 구체적으로 연결되지 않았다.
- 시작계단 방향 규약과 바닥 교체 기준은 변경 대상이 아니었다.
- Menu 씬 Setting 버튼 연결 문제는 diff 상 해결 근거가 확인되지 않는다.
- 로비 슬롯, 접속 실패, 카메라 표시 체크리스트는 아직 부족하다.
- Input System 과 구 InputManager 병존은 `Input.GetAxisRaw` 사용으로 계속 남아 있다.
- Unity Recovery 씬, 대량 에셋 혼재, `Deungeon` 폴더명 문제도 그대로 이월한다.

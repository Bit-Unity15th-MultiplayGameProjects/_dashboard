# GhostMarch_2 — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개이고 diff도 비어 있다. 따라서 이번 리포트는 신규 구현 평가가 아니라 이전 open item의 재판정이다.
- 문서 스냅샷에는 `architecture.md`, `asmdef-structure.md`, `tdd.md`, `game-design.md` 등 핵심 문서가 존재한다. 문서 기반은 이전보다 꽤 정리되어 있다.
- 반대로 NGO 패키지 해결, Boot-Lobby 전환, Lobby 권한 모델, UI asmdef 의존성 축소는 해결 근거가 없다.
- 콘텐츠 목록은 있으나 ScriptableObject 필드와 실제 수치 자산으로 이어졌다는 근거가 부족하다.

## 2. 코드 품질 리뷰
이번 diff가 비어 있으므로 새로 유입된 코드 품질 문제는 없다. 그러나 이것은 좋은 신호가 아니라 검증 부채가 그대로 남았다는 뜻이다. 특히 `LobbyManager`, `BootInitializer`, `Packages/manifest.json`은 이전 리포트에서 high로 분류된 항목인데, 빌드 로그나 씬 전환 로그가 없으면 “동작한다”가 아니라 “아직 확인되지 않았다”로 봐야 한다.

가장 위험한 구조적 문제는 UI 계층 의존성이다. 문서상 UI는 EventBus 구독만 해야 하지만, 이전 backlog는 `GhostMarch2.UI.asmdef`가 거의 모든 게임 모듈을 참조한다고 지적했다. 이 상태가 유지되면 UI가 상위 오케스트레이터가 되어 게임 로직을 직접 만지기 쉬워진다. 개선 방향은 UI가 `Core`와 얇은 네트워크 이벤트 계약만 참조하고, Lobby/HUD 동작은 EventBus 또는 전용 DTO로 받게 만드는 것이다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적이 없다. 코드 변경이 없고 hot path의 `Update`, 네트워크 sync, UI refresh 루프가 제시되지 않았기 때문에 LINQ 할당이나 선형 탐색 비용을 추정으로 지적하지 않는다. 다만 적 200~500마리 동기화 설계를 문서화한 만큼, 실제 구현 단계에서는 배열 기반 snapshot buffer와 object pool 검증이 필수다.

## 3. 진행도 평가
네트워크 경로는 NGO·Relay·Lobby를 쓰는 framework 기반이지만, 적 동기화는 커스텀 배치 스냅샷을 계획하므로 hybrid에 가깝다. 이 경로에서는 연결·RPC 자체는 프레임워크가 덜어주지만, 권한 모델과 스냅샷 계약은 직접 검증해야 한다.

현재는 문서 진척은 좋으나 제출 가능한 수직 슬라이스 근거가 약하다. Boot-Lobby 전환, 2인 이동 동기화, 패키지 의존성 해결, 빈 gameplay 모듈 탈출이 확인되지 않았다. 따라서 진행도는 28%로 본다. 문서는 프로젝트 방향을 설명하지만, 플레이 가능한 빌드를 대신하지 못한다.

## 4. 다음 권장사항
- Unity 에디터에서 NGO 포함 전체 import와 Boot-Lobby 전환을 실행하고 콘솔 로그를 남긴다.
- `LobbyManager`의 서버 권한 흐름을 `IsServer`, `IsHost`, RPC 진입점 기준으로 명시한다.
- `GhostMarch2.UI.asmdef`를 축소하고 UI가 게임 모듈을 직접 참조하지 못하게 막는다.
- Game 씬에서 2인 플레이어 이동 동기화 happy path를 먼저 고정한다.
- `content-list.md`의 항목을 SO 필드, 최소 Wave/Mob/Item 자산으로 연결한다.

## 5. 문서화 상태
design은 8점이다. `concept.md`, `game-design.md`, `from-ghost-march-1.md`가 게임 방향, 코어 루프, 전작 계승 지점을 설명하고 있어 팀원이 의도를 읽을 수 있는 수준이다.

technical은 7점이다. `architecture.md`, `asmdef-structure.md`, `dev-structure.md`, `tdd.md`가 권한 모델과 모듈 경계를 제시한다. 다만 UI asmdef 실제 상태와 문서 원칙이 충돌하고, 패키지 fork 배경과 빌드 검증 기록이 빠져 실전 온보딩 문서로는 아직 불안하다.

spec은 5점이다. `content-list.md`는 존재하지만 `_sample/docs`처럼 필드 스키마, 수치 테이블, QA 기준, 자산 ID 규칙까지 내려오지는 않는다. 기획 문서에서 런타임 데이터로 넘어가는 다리가 아직 약하다.

## 6. Backlog
- UI.asmdef가 거의 모든 모듈을 참조하는 문제는 계속 남는다. 왜 문제냐면 레이어 역전이 발생한다. Core 이벤트 계약으로 좁혀야 한다.
- NGO 전이 의존성은 실제 Unity import와 빌드로 확인해야 한다. 문서상 가능하다는 설명만으로는 부족하다.
- content-list는 아직 SO 필드와 수치 자산으로 연결되지 않았다. 다음 단계는 문서가 아니라 데이터화다.
- Combat·Enemy·Wave·Pickup·Resonance는 구현 경계가 비어 있다. 최소 루프 하나를 완성해야 위험이 줄어든다.
- SaintsField·JSAM git URL 변경 배경은 문서에 남겨야 한다. 패키지 출처가 불명확하면 재현성이 깨진다.
- CLAUDE 문서 링크와 대형 TMP·폰트 Resources 적재 정책은 계속 검증 대상이다.

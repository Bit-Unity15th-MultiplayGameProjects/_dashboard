# GhostMarch_2 — 무변경 재점검 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개이고 실제 diff도 비어 있다. 코드·씬·자산 측면의 신규 전진은 확인되지 않는다.
- 현재 문서 스냅샷에는 `architecture.md`, `asmdef-structure.md`, `tdd.md`, `rules.md`, `todo.md`가 존재한다. CLAUDE 문서 링크 누락 계열 backlog는 해결로 본다.
- 네트워크 경로는 NGO + Relay + Lobby인 framework 경로다. 그러나 패키지 import, Boot-Lobby 전환, 2인 이동 happy path 로그가 아직 없다.
- 콘텐츠 문서는 양이 늘었지만 무기·적·웨이브 수치가 ScriptableObject 필드와 실제 자산으로 이어지는 증거는 부족하다.

## 2. 코드 품질 리뷰
이번 범위에는 실제 코드 diff가 없으므로 새로 들어온 가독성, 중복, 예외 처리 결함은 특정할 수 없다. 대신 이전 ledger의 위험은 그대로 남아 있다. 특히 `GhostMarch2.UI.asmdef`가 거의 모든 게임 모듈을 참조하는 구조는 UI를 상위 계층으로 만들 위험이 크다. 왜 문제냐면 UI 변경이 전투·웨이브·네트워크 구현 세부에 직접 묶여 테스트와 교체 비용을 키우기 때문이다. 개선은 UI가 `Core`의 EventBus와 네트워크 이벤트 계약만 바라보게 줄이는 방향이 맞다.

네트워킹 쪽은 더 직설적으로 위험하다. `LobbyManager`의 서버 권한 기준과 RPC 호출 경로가 코드 가드로 고정되지 않으면, 에디터에서는 버튼이 눌려도 실제 Host/Client 권한에서는 런타임 실패가 난다. 권한 주체를 메서드명, `IsServer`/`IsHost` guard, ServerRpc 진입점으로 명확히 분리해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. `Update`, 네트워크 sync, UI refresh의 LINQ·임시 컬렉션·선형 탐색 패턴을 확인할 변경이 없으므로 추측성 최적화 항목은 만들지 않는다.

## 3. 진행도 평가
이 프로젝트는 NGO가 연결, RPC, 동기화 기반을 제공하는 framework 경로다. 따라서 직접 소켓을 구현한 프로젝트보다 게임 루프, 씬 흐름, UI, 콘텐츠 밀도에 대한 기대치가 높다. 현재는 문서와 구조 계획은 꽤 있으나, 제출 가능한 수직 슬라이스의 핵심인 2인 접속 후 같은 Game 씬에서 이동 동기화가 검증되지 않았다.

진행도는 24로 본다. Boot, Lobby, Player, UI의 방향은 잡혀 있지만 패키지 해결 로그와 happy path 재현 기록이 없어 시연 안정성을 말할 단계가 아니다. 네트워크 설계 부채와 콘텐츠 공백이 동시에 남아 risk는 high가 타당하다.

## 4. 다음 권장사항
- Unity 에디터에서 NGO/Relay/Lobby 패키지를 실제 import하고 Boot-Lobby 전환 로그를 남겨라.
- `LobbyManager`에 서버 권한 guard와 RPC 진입 경로를 고정해 런타임 실패를 조기에 터뜨려라.
- UI asmdef 참조를 `Core`와 네트워크 이벤트 계약 중심으로 축소하라.
- Game 씬에서 Host/Client 2인 이동 동기화 happy path를 먼저 닫아라.
- content-list, SceneBuilder, Combat/Enemy/Wave, TMP 폰트 정책은 수직 루프 검증 뒤 좁혀 처리하라.

## 5. 문서화 상태
design은 8점이다. `concept.md`, `game-design.md`, `from-ghost-march-1.md`가 컨셉, 코어 루프, 전작 계승 맥락을 충분히 설명한다. 다만 개별 시스템이 실제 데이터 표로 내려가는 깊이는 아직 부족하다.

technical은 8점이다. `architecture.md`, `asmdef-structure.md`, `dev-structure.md`, `tdd.md`가 네트워크 권한 모델, 어셈블리 방향, 개발 규칙을 설명한다. 단, 문서의 의존성 원칙과 실제 UI asmdef backlog가 충돌하므로 문서만으로 완료 판정은 불가하다.

spec은 5점이다. `content-list.md`는 존재하지만 무기·적·웨이브의 필드명, 수치 범위, SO 자산 세트, QA 기준까지 내려가지 않는다. `_sample/docs` 수준의 테스트 가능한 사양서로 보려면 아직 한 단계 부족하다.

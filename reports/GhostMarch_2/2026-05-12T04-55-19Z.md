# GhostMarch_2 — 변경 없음 재검토

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개라 실제 코드 diff, 변경 통계, 커밋 메시지 개선은 없다.
- 이전 리포트의 핵심 위험인 NGO 패키지 resolve, Boot-Lobby 전환, 2인 이동 동기화가 검증 자료 없이 유지된다.
- 문서 스냅샷 기준으로 GDD, 아키텍처, asmdef 구조 문서는 존재하며 기술 방향은 꽤 명확하다.
- 반대로 content-list 는 목록 수준에 머물러 SO 필드, 수치 자산, 테스트 가능한 사양으로 연결되지 않았다.
- validator 회계 대상 backlog 7건은 현재 증거만으로 완전 해결을 확인할 수 없어 모두 이월한다.

## 2. 코드 품질 리뷰
이번 범위에는 실제 diff 가 없으므로 신규 코드 품질 저하는 확인되지 않는다. 다만 변경 없음 자체가 문제다. `Packages/manifest.json` 은 NGO 2.11.0 을 선언하지만 `transport/services.core` 전이 의존성이 Unity 에디터에서 실제 resolve 됐는지 로그가 없다. 네트워킹 프로젝트에서 패키지 resolve 는 코드보다 앞선 실행 조건이다. 이 단계가 확인되지 않으면 이후 Lobby, Relay, Scene 전환 실패가 코드 버그인지 패키지 문제인지 구분할 수 없다.

아키텍처 문서에는 UI 단방향 원칙과 EventBus 경유 원칙이 적혀 있지만, `asmdef-structure.md` 는 `GhostMarch2.UI` 참조를 “전체”로 설명한다. 이 결정은 UI가 상위 계층처럼 게임 모듈을 직접 호출하는 길을 열기 때문에 위험하다. 개선하려면 UI는 `Core` 의 이벤트 계약과 필요한 네트워크 표시 계약만 참조하고, 게임 로직 호출은 EventBus 이벤트로 역전시켜야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. 실제 변경 코드가 없고 `Update`, 네트워크 sync, UI refresh hot path 의 LINQ·임시 컬렉션·선형 탐색 패턴을 확인할 근거가 없다. 추측성 최적화 대신, 다음 구현 때 Player 이동 동기화와 Enemy/Wave 루프에서 per-frame 할당을 프로파일링 기준으로 검토해야 한다.

## 3. 진행도 평가
구현 경로는 Unity NGO, Relay, Lobby 를 쓰는 `framework` 경로다. 이 경로는 연결·RPC·동기화 기반 비용이 낮은 대신, 최소 게임 루프와 2인 happy path 검증 기대치가 높다. 현재는 문서와 패키지 선언은 있으나 에디터 resolve, Boot-Lobby 전환 로그, Game 씬 2인 이동 동기화가 확인되지 않아 제출 가능한 수직 슬라이스 이전 단계로 본다.

진행도는 28%로 판단한다. 기술 방향 문서는 기반을 제공하지만, Combat·Enemy·Wave 중 실제 플레이 루프가 비어 있고 네트워크 권한 가드도 검증되지 않았다. 위험도는 high 다. 이유는 네트워킹 미검증과 어셈블리 경계 부채가 함께 남아, 이후 디버깅 비용이 급격히 커질 가능성이 높기 때문이다.

## 4. 다음 권장사항
- Unity 에디터에서 `manifest.json` resolve, 스크립트 컴파일, Boot-Lobby 씬 전환 로그를 먼저 확보한다.
- `LobbyManager` 에서 Host/Server 권한 기준, ServerRpc 호출 가능 조건, 실패 로그를 코드로 고정한다.
- `GhostMarch2.UI.asmdef` 참조를 축소하고 UI가 게임 모듈을 직접 호출하지 못하게 구조를 잠근다.
- Game 씬에 Host 1명, Client 1명 이동 동기화 happy path 를 만들고 체크리스트로 반복 검증한다.
- content-list 의 무기·적·웨이브 표를 SO 필드 정의와 최소 `.asset` 세트로 연결한다.

## 5. 문서화 상태
design 은 7점이다. `game-design.md`, `concept.md`, `from-ghost-march-1.md` 로 게임 방향과 코어 루프는 읽을 수 있다. 다만 개별 시스템의 성공/실패 조건이 테스트 기준까지 내려오지는 않았다.

technical 은 8점이다. `architecture.md`, `dev-structure.md`, `asmdef-structure.md` 가 네트워크 권한 모델, 부트 순서, 어셈블리 구조를 설명한다. 다만 실제 asmdef 설계와 문서 원칙이 충돌하는 부분은 빨리 정리해야 한다.

spec 은 4점이다. `content-list.md` 는 무기·몹·공명 항목과 일부 수치를 제공하지만, `_sample/docs` 수준의 스키마, 필수 필드, 밸런싱 기준, QA 체크리스트가 없다. 파일 존재만으로 사양서 역할을 한다고 보기는 어렵다.

- SaintsField·JSAM git URL 변경 배경, content-list 와 SO 자산 연결, 대형 TMP·폰트 Resources 정책도 아직 해결되지 않았다.

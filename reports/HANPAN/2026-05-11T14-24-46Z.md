# HANPAN — WebGL 배포 스냅샷

## 1. 주요 변경사항
- `.gitignore`에 Unity 자동 복구 씬 경로가 추가됐다. 불필요한 복구 산출물이 커밋되는 위험을 줄인 점은 적절하다.
- `ProjectSettings.asset`에 WebGL 그래픽 API 설정이 추가됐다. WebGL 빌드 대상이 실제 배포 경로로 편입됐다는 신호다.
- Work_Breakdown에서 WebGL 빌드, 빌드 용량 기록, Itch.io 배포가 완료 상태로 갱신됐다.
- 단, P2-3의 완료 기준이 “방 생성·방 코드 표시”에서 “로비 화면 표시”로 낮아졌다. 이는 검증 범위 축소이므로 명확히 경고해야 한다.

## 2. 코드 품질 리뷰
이번 diff는 런타임 C# 코드 변경이 없고, 설정 파일과 진행 문서 갱신이 중심이다. `.gitignore`에 `Assets/_Recovery/`를 제외한 결정은 Unity 작업 중 자동 복구 씬이 섞여 들어오는 문제를 차단하므로 타당하다. 다만 WebGL 빌드 설정 변경은 재현성이 중요하다. `ProjectSettings.asset`만 보면 왜 WebGLSupport가 자동 API로 설정됐는지, Chrome ANGLE D3D11 변경이 필수 조건인지 선택 조건인지 알 수 없다. 환경 의존 해결책은 문서에 조건과 재현 절차가 같이 남아야 다음 작업자가 같은 결과를 낼 수 있다.

가장 큰 문제는 Work_Breakdown의 완료 기준 후퇴다. 기존 P2-3은 Boot → 로비 → 방 생성 → 방 코드 표시였는데, 이번 변경은 Itch.io → Boot → 로비 화면 표시로 끝난다. 왜 문제냐면 로비 렌더링은 WebGL 실행 확인일 뿐이고, 멀티플레이 수직 슬라이스의 핵심인 방 생성·접속·상태 동기화는 검증하지 않는다. 개선하려면 P2-3은 “싱글 브라우저 렌더링 확인”으로 이름을 바꾸고, 방 생성과 방 코드 표시는 P2-4 또는 별도 항목으로 명확히 이동해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. `Update`, 네트워크 동기화, UI refresh, LINQ 할당 같은 코드 패턴이 변경되지 않았으므로 추측성 최적화 항목을 만들 근거가 없다.

## 3. 진행도 평가
구현 경로는 이번 diff만으로 확정할 수 없으나, README가 외부 URL 2인 접속과 방 생성·게임 완주를 완료 기준으로 두고 있으므로 멀티플레이 웹 배포 수직 슬라이스 단계로 판단한다. WebGL 빌드와 Itch.io 배포가 완료된 것은 제출 가능성 측면에서 분명한 전진이다.

진행도는 72로 본다. 단일 브라우저 실행과 배포 URL 확보에 더해 `BlackAndWhite_Spec.md`와 `QA_Checklist.md`가 실제 테스트 기준으로 정리되어 제출 전 검증 기반은 충분히 올라왔다. 다만 2인 크로스 브라우저 1판 완주와 외부 URL 2인 접속 증적이 아직 비어 있다. 멀티플레이 프로젝트에서 이 항목은 선택 검증이 아니라 제출 안정성의 핵심이다. 따라서 risk는 low가 아니라 medium이다.

## 4. 다음 권장사항
- 외부 URL 기준으로 Chrome/Edge 또는 Chrome/Firefox 2인 접속 후 방 생성, 참가, 1판 완주를 기록해야 한다.
- WebGL 빌드 명령, Unity 버전, 브라우저 설정, 압축 방식, 업로드 절차를 재현 가능한 문서로 고정해야 한다.
- P2-3의 완료 기준을 화면 표시와 방 생성 검증으로 분리해야 한다. 현재 표기는 진행률을 실제보다 좋게 보이게 만든다.
- `QA_Checklist.md`의 WebGL 빌드 결과 표에 실제 브라우저, 외부 URL, 빌드 용량, 2인 완주 결과를 증적으로 남겨야 한다.

## 5. 문서화 상태
design은 6점이다. 프로젝트 진행표와 게임 흐름 단서는 있고, 흑과 백의 룰 목표도 `BlackAndWhite_Spec.md`로 이어진다. 다만 플레이 경험 목표와 전체 게임 묶음의 기획 의도가 `_sample/docs` 수준으로 정리됐다고 보긴 어렵다.

technical은 8점이다. `Docs/SceneArchitecture.md`, `README/4. Network_Authority.md`, `README/3. Work_Breakdown.md`가 Additive 씬 흐름, NGO 상태머신, 호스트 마이그레이션, WebGL 진행 상태를 꽤 구체적으로 다룬다. 남은 약점은 WebGL 빌드 절차와 브라우저 환경값을 재현 가능한 단계로 더 고정하는 것이다.

spec은 8점이다. `README/5. BlackAndWhite_Spec.md`와 `README/6. QA_Checklist.md`가 입력, 상태 전이, HUD, 승패 조건, 2인 수동 QA 절차를 테스트 가능한 수준으로 정리한다. 아직 비어 있는 것은 실제 외부 URL 2인 완주 결과와 WebGL 빌드 증적이다.

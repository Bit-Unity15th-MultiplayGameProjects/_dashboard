# HANPAN — 스테일 리스폰 복귀 수정 스냅샷

## 1. 주요 변경사항
- `GameFlowManager.HandleStaleGameRespawn()`에서 `ReturnToRoom = true`를 세팅하여 Matching 씬 재로드 후 `Start()`가 Lobby로 되돌리는 문제를 막았다.
- `BlackAndWhiteController`의 서버 stale respawn 경로가 게임 씬 시각 요소를 숨기고 Room 복귀를 호출하도록 바뀌었다. 마이그레이션 후 P2 로비 잔류를 겨냥한 수정이다.
- 게임 씬 숨김 대상에 `EventSystem`이 추가되어 Matching 복귀 시 2개 EventSystem 경고를 줄였다.
- Work Breakdown과 QA Checklist에 재게임 초기화, 재접속, 브라우저별 검증 표가 추가됐다. 단, 다수 항목은 아직 예정 또는 미검증 상태다.

## 2. 코드 품질 리뷰
이번 수정의 방향은 맞다. stale respawn을 서버와 클라이언트 양쪽에서 “구 게임 씬 은닉 + Room 복귀”로 맞춘 점은 상태 불일치를 줄인다. 특히 `ReturnToRoom`을 static 플래그로 둔 선택은 `HANPAN.Core`와 `HANPAN.Network` 순환참조를 피하려는 현실적 타협이다. 다만 이 플래그는 암묵 상태이므로, 나중에 복귀 경로가 늘어나면 “누가 언제 true로 만들었는가”가 흐려진다. 장기적으로는 전환 요청 객체나 `GameManager` 소유 상태로 모으는 편이 안전하다.

위험 신호는 디버그 코드다. `GameFlowManager.GoToLobby()`와 `GoToRoom()`에서 호출 때마다 `new System.Diagnostics.StackTrace()`를 두 번 생성한다. 현재 구조의 비용은 전환 hot path에서 불필요한 스택 캡처와 문자열 할당이 발생하는 것이다. 추천 흐름은 `const bool EnableFlowTrace` 또는 빌드 심볼 조건부 로그로 감싸고, 호출자 문자열은 한 번만 계산하는 방식이다. 이 프로젝트는 WebGL 배포가 목표라 로그·할당이 곧 프레임 지연과 브라우저 콘솔 노이즈로 이어지므로 의미 있는 지적이다.

자료구조/복잡도 측면에서는 이번 diff에 `List.Find`, LINQ, 반복 검색 같은 컬렉션 선택 문제는 보이지 않는다. 대신 명확한 할당 문제는 `StackTrace` 생성이다. 이는 자료구조 교체보다 진단 로그 수명 관리 문제로 보는 것이 정확하다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 세션 상태머신과 복귀 정책을 직접 설계하는 `hybrid`에 가깝다. 2인 수직 슬라이스와 방 복귀 루프는 상당히 진전됐고, 이번 커밋은 마이그레이션 후 로비 잔류라는 실제 시연 리스크를 좁혔다.

진행도는 72로 본다. Editor 기준 happy path는 가까워졌지만, 배포 URL P3-2가 아직 미완료이고 InRoom disconnect 헬스체크가 장기 대기실에서 끊김을 놓칠 수 있다. 커밋 14개 중 상당수가 fix/debug/merge인 점도 순수 기능 전진보다 안정화에 시간을 쓰고 있음을 보여준다.

## 4. 다음 권장사항
- `SessionManager`의 InRoom 헬스체크를 진입 시 1회 예약이 아니라 disconnect 이벤트 기준 재예약 구조로 바꿔야 한다.
- `GameFlowManager`의 StackTrace 로그는 즉시 조건부 진단 플래그 뒤로 넣어야 한다. 지금 형태로 WebGL에 싣는 것은 좋지 않다.
- 준비 초기화 실패는 재입장 안내만으로 끝내지 말고, 실패 원인 로그와 재시도 버튼까지 연결해야 한다.
- `UnlockSessionAsync`는 첫 입장 `OnEnable` 공통 경로가 아니라 Returning 완료 경로에서만 호출되도록 분리해야 한다.
- P3-2, P3-3, P3-4는 문서 표 추가가 아니라 실제 배포 URL 결과 입력까지 끝나야 완료로 인정된다.

## 5. 문서화 상태
design은 8점이다. `README/1. Concept_Template.md`, `README/5. BlackAndWhite_Spec.md`가 게임 컨셉과 흑과 백 규칙을 꽤 구체적으로 담고 있다. 다만 장기 컨텐츠 확장 문서는 아직 계획 수준이 섞여 있다.

technical은 8점이다. `Docs/SceneArchitecture.md`와 `README/4. Network_Authority.md`가 씬 수명, NGO 전환, 권한 모델, 호스트 마이그레이션을 신규 유입자가 따라갈 수 있을 정도로 설명한다. 단, 실제 코드가 빠르게 바뀌는 영역이라 stale 문서 위험이 계속 있다.

spec도 8점이다. `README/5. BlackAndWhite_Spec.md`와 `README/6. QA_Checklist.md`는 상태, 입력, QA 절차가 테스트 기준으로 쓸 수 있는 수준이다. 이번에 표가 늘어난 것은 좋지만, 체크박스와 예정값이 많아 증적 문서로는 아직 부족하다.

# HANPAN — 재게임 안정화 스냅샷

## 1. 주요 변경사항
- 재게임 시작 시 `BlackAndWhiteGameUI`가 히스토리, 결과 오버레이, 항복 UI를 초기화하도록 보강했다.
- `BoardLayout3D`는 새 게임 진입 조건을 `AwaitingBothPlayers + Round 1 + non-tiebreaker`로 잡아 이전 게임 타일 잔존을 지운다.
- `NetworkSceneTransition`은 호스트 마이그레이션 후 `_loadedGameScene`이 비어도 실제 로드 씬을 보완해 기존 언로드 경로를 재사용한다.
- `RoomUI`는 ready 초기화 완료 전 준비·시작 버튼을 잠그도록 바뀌어 잔존 ready 자동 시작 위험을 낮췄다.
- 개발 계획 문서는 `Base_01_Matching` 통합 구조와 2026년 푸터로 갱신됐다.

## 2. 코드 품질 리뷰
이번 변경은 재게임과 방 재진입의 실제 장애를 정면으로 줄였다는 점에서 좋다. 특히 `BoardLayout3D` 주석은 왜 적 손패를 `ResetBoard()`에서 지우면 안 되는지 설명하고 있어, NetworkVariable 이벤트와 ClientRpc 도착 순서 문제를 코드 옆에서 바로 이해할 수 있다. 이는 네트워크 버그 재발 방지에 도움이 된다.

다만 `RoomUI.OnEnable()`에서 `UnlockSessionAsync()`가 여전히 fire-and-forget으로 호출된다. 왜 문제냐면 첫 입장, 게임 복귀, 잠금 해제 완료, ready 초기화 완료가 한 메서드 안에서 섞이면 버튼 잠금의 원인을 로그만 보고 분리하기 어렵다. 개선하려면 Returning 상태에서만 unlock을 시작하고, 완료 이벤트 또는 명시적 결과를 받은 뒤 `RefreshUI()`와 버튼 해제를 한 지점에서 수행해야 한다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 `Update()` hot path의 LINQ, 반복 검색, O(n²) lookup 같은 명확한 지적은 없다. `ResetGameUI()`의 `Destroy` 루프와 `ResetBoard()`의 Renderer 순회는 재게임 시작 이벤트에서만 실행되므로 지금 당장 자료구조 교체 대상은 아니다. 다만 히스토리 항목 수가 늘거나 재시작을 빠르게 반복하는 QA에서 프레임 끊김이 보이면 풀링으로 전환할 근거가 생긴다.

## 3. 진행도 평가
구현 경로는 NGO, UGS Lobby/Relay, WebSocket Transport를 쓰되 세션 상태, 호스트 마이그레이션, 씬 복귀 정책을 직접 설계한 `hybrid`에 가깝다. 2인 수직 슬라이스와 재게임 안정화는 상당히 전진했지만, 외부 배포 URL 기준 P3-2 검증이 아직 문서상 예정 상태라 제출 안정성은 확정되지 않았다.

진행도는 74%로 본다. 로컬 또는 제한 환경의 happy path는 제출권에 들어왔지만, 최근 커밋 대부분이 재시작·잔존 상태·NRE 수정이라는 점은 위험 신호다. 네트워크 상태 전이가 계속 고쳐지고 있으므로 risk는 `high`가 맞다.

## 4. 다음 권장사항
- ready 초기화 실패 시 단순 재입장 안내가 아니라 재시도 버튼, 실패 원인 로그, 퇴장 fallback을 분리한다.
- `UnlockSessionAsync()`는 Returning 경로에서만 실행하고 완료 이벤트 기준으로 UI 잠금을 해제한다.
- WebGL 재빌드 후 Itch.io URL에서 코드 입장 2인 1판 완주를 실제 브라우저와 URL까지 기록한다.
- 재게임 시작 시 UI 초기화, 보드 초기화, 손패 RPC 순서를 2클라이언트 회귀 체크리스트로 고정한다.
- `NetworkSceneTransition` 종료·복귀·호스트 마이그레이션 반복 시나리오를 QA 문서에 추가한다.

## 5. 문서화 상태
design은 7점이다. `Unity_Dev_Plan`과 컨셉 문서가 현재 씬 구조와 개발 범위를 설명하지만, 플레이 경험 목표와 선택 이유는 샘플 문서 수준까지 촘촘하지는 않다.

technical은 8점이다. `Docs/SceneArchitecture.md`와 `README/4. Network_Authority.md`가 Additive 씬 흐름, 세션 상태, 권한 모델을 실무적으로 추적 가능하게 적고 있다. 다만 코드 변경 때마다 상태표가 최신인지 검증하는 장치가 부족하다.

spec은 8점이다. `README/5. BlackAndWhite_Spec.md`와 QA 체크리스트가 게임 규칙, 상태, 테스트 절차로 활용 가능한 수준이다. 남은 약점은 P3-2 외부 URL 검증, 브라우저별 흑화면 조건, 재접속 QA 결과가 아직 실제 결과로 충분히 채워지지 않았다는 점이다.

## 6. Backlog
- `Base_01_Matching` UI는 여전히 씬과 빌더 중심이다. Prefab·데이터 기반 분리 전까지 씬 파일 리뷰 비용이 계속 발생한다.
- `Game_01_BlackAndWhite`도 생성물 의존이 남아 있다. 재빌드 결과가 커밋에 크게 섞이면 로직 리뷰 신뢰도가 떨어진다.
- P2-4는 완료됐지만 P3-2 외부 URL 2인 완주 증적은 아직 예정 상태다.
- WebGL 흑화면은 브라우저별 지원 기준 문서가 생겼지만, 실제 지원 범위 검증으로 닫히지는 않았다.
- 씬 재빌드 시 파일 ID 대량 변경 문제는 구조적으로 해소되지 않았다.

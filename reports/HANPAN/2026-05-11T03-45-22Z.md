# HANPAN — M5 QA 문서화 스냅샷

## 1. 주요 변경사항
- `Work_Breakdown.md`의 M5 잔여 작업을 Phase 1~3으로 재구성했다. 문서 정비, WebGL 검증, 배포 순서가 이전보다 명확해졌다.
- 신규 `README/6. QA_Checklist.md`가 추가됐다. LobbyUI, RoomUI 상태 전이와 2인 수동 QA 흐름이 테스트 문서로 분리됐다.
- `BlackAndWhite_Spec.md` v1.3에 HUD 수치, 승패 조건, 정보 은닉 검증표가 들어갔다. 테스터가 확인할 기준이 전보다 훨씬 구체적이다.
- WebGL 빌드, 로컬 서버 검증, 2인 크로스 브라우저 테스트, 배포 확인은 모두 아직 미완료 상태로 남아 있다.

## 2. 코드 품질 리뷰
이번 커밋 범위는 문서 3개만 변경했으므로 런타임 코드의 가독성, 구조, 에러 처리에 대한 직접 평가는 제한적이다. 다만 문서 품질은 분명히 개선됐다. 특히 상태 전이 표를 “초기화 중, 목록 표시, 빈 목록, 실패, 마이그레이션”으로 나눈 것은 UI 디버깅 기준으로 실무 가치가 있다. 왜 중요한가 하면, 멀티플레이 UI 문제는 재현 순간의 상태명이 없으면 로그와 화면을 연결하기 어렵기 때문이다. 다음 단계에서는 이 표의 상태명이 실제 코드 이벤트명, `StatusText`, 버튼 interactable 변경 지점과 일치하는지 대조해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. 코드 변경이 없기 때문에 `Update` hot path, LINQ 할당, 선형 탐색, 임시 컬렉션 생성 여부를 새로 판정할 근거가 없다. 추측으로 최적화 TODO를 만드는 것은 오히려 리뷰 품질을 떨어뜨린다. 현재 필요한 것은 성능 개선보다 WebGL 빌드와 2인 QA에서 실제 병목이나 오류를 관측하는 일이다.

Unity 관점에서는 이전부터 이어진 씬 YAML·Editor Builder 의존이 여전히 위험하다. 문서에 “M6 기술 부채”로 인지한 점은 좋지만, 아직 구조가 바뀐 것은 아니다. UI를 Prefab과 ScriptableObject 테마로 분리하지 않으면 작은 색상·레이아웃 수정도 씬 파일 대량 diff로 이어져 리뷰 신뢰도가 계속 떨어진다.

## 3. 진행도 평가
네트워크 구현 경로는 NGO/UGS 계열을 활용하면서 세션, 방, 준비 상태, 복귀 흐름을 직접 조립하는 `hybrid`로 보는 것이 타당하다. 프레임워크가 RPC와 동기화 기반을 제공하지만, 실제 제출 리스크는 방 생성, 준비, 씬 전환, 호스트 이탈, 게임 복귀 같은 프로젝트 고유 흐름에 몰려 있다.

진행도는 66%로 평가한다. 문서화는 크게 전진했고 수동 QA 기준도 생겼다. 그러나 제출 가능성의 핵심인 WebGL 빌드 성공, 로컬 서버 실행, 2인 크로스 브라우저 1판 완주가 아직 체크되지 않았다. 이 상태는 “무엇을 검증할지 안다” 단계이지, “반복 시연 가능하다” 단계는 아니다.

## 4. 다음 권장사항
- WebGL 빌드를 먼저 실행하고 산출물 크기, 경고 수, 주요 경고를 `QA_Checklist.md` §5에 기록해야 한다.
- 2인 크로스 브라우저 테스트를 문서 기준으로 수행해야 한다. 체크리스트가 있어도 실제 결과가 없으면 리스크는 줄지 않는다.
- 게임 오버 후 방 복귀, 준비 초기화, 재시작을 최소 2회 반복 검증해야 한다. 이 흐름은 시연 중 가장 자주 깨지는 구간이다.
- LobbyUI와 RoomUI 상태 전이 표를 코드 이벤트명과 대조해야 한다. 문서 상태명과 실제 분기가 어긋나면 QA 로그가 무의미해진다.
- 흑과 백 QA에는 입력 조작 매핑과 실패 케이스를 추가해야 한다. “무엇을 눌렀을 때 막혀야 하는가”가 아직 약하다.

## 5. 문서화 상태
design 점수는 6점이다. 게임 컨셉과 개별 게임 사양은 존재하지만, 이번 변경은 플레이 경험 목표나 디자인 의사결정을 깊게 보강한 작업은 아니다. “왜 이 규칙인가”보다 “무엇을 확인할 것인가”에 초점이 있다.

technical 점수는 7점이다. SceneArchitecture, Network_Authority, Work_Breakdown 참조 흐름이 유지되고, QA 문서가 세션 상태를 확인 가능한 형태로 보완했다. 다만 빌드 결과와 실제 네트워크 실패 로그가 아직 비어 있어 신규 유입자가 안정성을 판단하기에는 부족하다.

spec 점수는 8점이다. HUD 수치, 승패 조건, 정보 은닉, 상태 전이, 2인 QA 절차가 테스트 체크리스트로 쓸 수 있는 수준까지 올라왔다. 단, 입력 매핑과 씬별 세부 사양이 아직 충분히 구조화되지 않아 9점 이상으로 보기는 어렵다.

## 6. Backlog
- 레벨/씬별 상세 사양서 입력 매핑은 아직 남아 있다. 상태 전이와 HUD 수치는 개선됐지만, 입력·씬별 판정 기준은 더 구체화해야 한다.
- `Game_01_BlackAndWhite.unity` 생성물 diff 문제는 이번 커밋에서 해결되지 않았다. Builder와 씬 산출물 변경 범위를 계속 분리해야 한다.
- `Base_02_Lobby.unity`와 `LobbyUIBuilder.cs`의 UI 하드코딩 문제도 유지된다. M6로 미루더라도 Prefab 분리 계획은 실제 작업으로 전환해야 한다.
- `Base_01_Matching.unity`의 파일 ID 대량 변경 리스크도 남아 있다. 씬 재빌드가 필요한 작업은 커밋 단위를 엄격히 쪼개야 한다.

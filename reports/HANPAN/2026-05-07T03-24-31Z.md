# HANPAN — 룸 전환 안정화 스냅샷

## 1. 주요 변경사항
- `SessionManager`에 NGO disconnect suppress 로그, Join/Leave 추적 로그, 호출자 추적이 추가됐다.
- `RoomUI.OnEnable`의 세션 없음 즉시 로비 복귀 로직이 제거되고, 세션 이벤트 구독은 `sm != null` 조건으로 감싸졌다.
- `LobbyUI`와 `GameFlowManager`에 방 입장, 룸 전환, 로비 전환 추적 로그가 들어갔다.
- `BoardLayout3D`는 슬롯 머티리얼 색 변경 방식에서 배치 타일 `Cube` 생성 방식으로 바뀌었다.
- `HandTile3D`는 URP `_BaseColor`와 Built-in `_Color`를 함께 설정하도록 수정됐다.

## 2. 코드 품질 리뷰
이번 변경은 기능 완성보다 장애 원인 추적에 가깝다. 특히 `SessionManager`의 300초 suppress는 현재 문제를 숨기는 시간창이며, 상태 전이의 주인을 명확히 하지 않는다. 왜 문제냐면 UGS 세션, NGO disconnect, UI 활성화가 서로 다른 타이밍에 발생하는데 이를 시간으로 덮으면 재현 환경마다 다른 결론이 나온다. 개선 방향은 `Lobby`, `Joining`, `Room`, `LoadingGame`, `InGame`, `Returning`, `Disconnected` 같은 명시적 상태머신으로 전이를 고정하는 것이다.

`RoomUI.OnEnable`에서 세션 없음 로비 복귀를 제거한 결정은 회귀를 막기 위한 응급처치로는 이해된다. 그러나 입장 중 세션 준비 전 상태와 실제 세션 이탈 상태를 구분하지 않으면 같은 문제가 다시 발생한다. `OnEnable`에서 네트워크 복구, ready 초기화, session unlock, 이벤트 구독까지 수행하는 것도 책임이 과하다. UI는 현재 상태를 표시하고 입력을 전달해야 하며, 세션 생명주기 제어는 `SessionManager` 쪽으로 모아야 한다.

자료구조/복잡도 관점에서 선형 탐색이나 O(n²) 패턴은 이번 diff 범위에서 뚜렷하지 않다. 다만 할당 측면에서는 `BoardLayout3D.SpawnTile`이 턴 제출과 게임오버 재구성 때마다 `CreatePrimitive`와 `Destroy`를 수행한다. 현재 보드 규모에서는 치명적 hot path는 아니지만, 모바일 URP와 네트워크 턴 피드백에서는 작은 히치도 체감된다. 슬롯별 타일 오브젝트를 미리 만들고 active/material만 바꾸는 구조가 더 단순하고 안전하다.

## 3. 진행도 평가
전진은 있다. URP 색상 미적용 문제와 보드 타일 표시 방식은 실제 플레이 감각을 개선하는 방향이다. 그러나 6개 커밋 대부분이 `fix`, `debug`, `refactor`이며, 순수 기능 확장보다 룸 입장 회귀 추적에 집중돼 있다. 이는 네트워킹 핵심 경로가 아직 안정화되지 않았다는 강한 신호다.

현재 진행도는 38%로 본다. 핵심 게임 화면은 일부 동작하는 단계로 보이나, 멀티플레이 룸 복귀와 입장 플로우가 흔들리면 플레이 가능한 빌드라고 보기 어렵다. 다음 스냅샷 전에는 로그 추가보다 상태 전이 설계를 정리해야 한다.

## 4. 다음 권장사항
- `Room/Session` 전환을 명시적 상태머신으로 고정하고 300초 suppress 의존을 제거한다.
- 세션 ID와 전환 경로 로그는 조건부 진단 플래그 뒤로 이동해 기본 빌드 노출을 막는다.
- `RoomUI.OnEnable`에서 입장 중 상태와 실제 연결 끊김 상태를 분리해 처리한다.
- `BoardLayout3D`의 배치 타일은 `Primitive/Destroy` 대신 슬롯별 캐시나 프리팹으로 교체한다.
- 기술 문서에 네트워크 권한 모델, 룸 상태 전이, disconnect 처리 기준을 추가한다.

## 5. 문서화 상태
design 점수는 3점이다. 게임의 목표 경험과 코어 루프를 설명하는 문서가 충분히 확인되지 않으며, 현재 변경도 디자인 의사결정 갱신과 연결되지 않는다.

technical 점수는 2점이다. 지금 가장 위험한 영역이 네트워크 상태 전이인데, 코드에는 임시 suppress와 로그가 늘고 문서에는 권한 모델과 상태 흐름이 보이지 않는다. 신규 작업자가 이 문서만 보고 안전하게 수정하기 어렵다.

spec 점수는 2점이다. 이전 backlog의 레벨/씬별 입력 매핑, UI 상태 전이, HUD 수치 부재가 이번 범위에서 해소되지 않았다. 테스트 체크리스트로 사용할 수 있는 사양서 수준에는 아직 못 미친다.

## 6. Backlog
- 레벨/씬별 상세 사양서 입력 매핑·UI 상태 전이·HUD 수치 부재가 남아 있다. 플레이 검증 기준이 문서가 아니라 기억과 씬 상태에 의존한다.
- `Game_01_BlackAndWhite` 씬 생성물 커밋 문제는 해결 근거가 없다. 씬 diff 리뷰 비용이 계속 높게 유지된다.
- `01_Lobby` UI 구성이 씬 파일에 하드코딩된 문제도 그대로다. UI 반복 수정 시 씬 충돌 가능성이 크다.
- URP 템플릿 튜토리얼과 Mobile 렌더러 잔존도 정리되지 않았다. 불필요 자산은 빌드·리뷰 잡음을 만든다.

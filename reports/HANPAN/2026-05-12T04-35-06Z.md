# HANPAN — 종료 안정화와 씬 재빌드 스냅샷

## 1. 주요 변경사항
- `NetworkSceneManager.Dispose` 계열 NullReferenceException 수정 커밋이 들어왔다. 종료·복귀 경로의 즉시 크래시 위험을 줄인 점은 명확한 전진이다.
- `IGameSceneController`가 새로 추가됐고, 빌더 통합 및 Matching/BlackAndWhite 씬 재빌드가 수행됐다.
- `Base_01_Matching.unity`와 `Game_01_BlackAndWhite.unity`가 각각 대량 변경됐다. 기능 변경과 생성물 변경이 섞여 리뷰 비용이 매우 높다.
- `BlackAndWhite3DSceneBuilder`, `LobbyUIBuilder`, `RoomUIBuilder`, `MatchingSceneBuilder`가 같이 움직였다. 빌더 중심으로 정리하려는 방향은 좋지만 결과 검증이 아직 부족하다.

## 2. 코드 품질 리뷰
이번 변경의 좋은 점은 종료 시 NRE를 직접 겨냥했다는 점이다. 네트워크 씬 전환 계층에서 Dispose 타이밍 예외가 남으면 Editor 종료, WebGL 탭 종료, Room 복귀 같은 정상 사용자 동작이 모두 오류처럼 보인다. 이런 결함은 로그를 오염시키고 실제 세션 이탈 버그와 구분하기 어렵기 때문에 반드시 회귀 시나리오로 잠가야 한다.

반대로 현재 diff의 가장 큰 문제는 씬 재빌드가 코드 변경을 덮어버릴 정도로 크다는 점이다. `Game_01_BlackAndWhite.unity`만 2만 라인 이상 움직였고, `Base_01_Matching.unity`도 수천 라인 단위로 바뀌었다. 왜 문제냐면 실제 로직 회귀인지 Unity 직렬화 재배치인지 리뷰어가 구분하기 어렵기 때문이다. 개선하려면 빌더 변경, 씬 재생성, 런타임 버그 수정 커밋을 분리하고, 가능하면 반복 UI와 보드 생성물을 Prefab 또는 데이터 소스로 더 밀어내야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 hot path 선형 탐색, 매 프레임 LINQ, 반복 컬렉션 할당 지적은 확인되지 않는다. 지금 더 중요한 비용은 런타임 CPU가 아니라 생성 씬 대량 직렬화로 인한 변경 추적 비용이다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 세션 상태머신·재접속·씬 복귀 정책을 직접 설계하는 `hybrid`에 가깝다. 로컬 2인 흐름과 문서화는 상당히 올라왔고, 종료 예외 수정도 실전 안정성에 의미가 있다.

다만 제출 가능한 수직 슬라이스 기준으로는 배포 URL 2인 완주, 코드 입장 재검증, ready 초기화 실패 처리, Returning 잠금 전이가 아직 닫히지 않았다. 진행도는 72%로 본다. 기능은 데모권에 들어왔지만, 네트워크 복귀와 배포 검증이 남아 risk는 high다.

## 4. 다음 권장사항
- ResetReadyOnRoomEnter 실패를 UI 차단과 재시도로 연결한다. 실패가 경고로 끝나면 stale ready가 게임 시작 조건을 오염시킨다.
- RoomUI의 잠금 해제는 Returning 완료와 세션 Changed 이벤트에서만 확정한다. OnEnable 쓰기는 상태 전이 원인을 흐린다.
- WebGL 재빌드 후 Itch.io 배포 URL에서 코드 입장 2인 완주를 다시 기록한다.
- Dispose NRE 수정은 Editor 정지, 탭 종료, 게임 종료 후 Room 복귀 반복으로 회귀 체크한다.
- `IGameSceneController` 도입 후에도 컨트롤러가 `GameFlowManager`를 직접 호출하지 않는지 정리한다.

## 5. 문서화 상태
design은 8점이다. `Concept_Template`, `Unity_Dev_Plan`, `BlackAndWhite_Spec`이 현재 게임의 방향과 핵심 루프를 설명하고 있어 팀원이 의도를 따라가기 충분하다.

technical은 8점이다. `Docs/SceneArchitecture.md`와 `README/4. Network_Authority.md`가 씬 수명, 세션 상태, 권한 모델을 꽤 구체적으로 다룬다. 다만 `Unity_Dev_Plan`에는 옛 `Base_01_Lobby/Base_02_Room` 구조와 푸터 연도 오기가 남아 최신성과 충돌한다.

spec은 8점이다. `BlackAndWhite_Spec`과 `QA_Checklist`가 테스트 체크리스트로 쓸 수 있는 수준이다. 부족한 부분은 P3-2 배포 URL 실측 결과와 브라우저별 WebGL 실패 범위다.

## 6. Backlog
- `Base_01_Matching` UI는 아직 씬 직렬화와 Builder 결과에 강하게 묶여 있다. Prefab·데이터 기반 분리가 필요하다.
- `Game_01_BlackAndWhite` 생성물 커밋은 계속 리뷰 비용을 키운다. 빌더 코드 변경과 씬 산출물 커밋을 분리해야 한다.

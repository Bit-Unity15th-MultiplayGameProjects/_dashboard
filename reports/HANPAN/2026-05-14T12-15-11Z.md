# HANPAN — 05.14 재경기 안정화 스냅샷

## 1. 주요 변경사항
- `RetryButton` interactable 초기화 버그를 수정하고 2클라 회귀 검증 RT1~RT3 통과 기록을 추가했다.
- 호스트 이전 후 재경기 관련 버그 3종을 `BlackAndWhiteController`, `NetworkSceneTransition`, 씬 빌더 쪽에서 보정했다.
- WebGL 05.14 검증 행과 QA 체크리스트 v3.0 갱신이 들어가, 배포 URL 기준 검증 추적성이 일부 좋아졌다.
- `Game_01_BlackAndWhite.unity`가 25,432줄 규모로 재배치됐다. 기능 수정보다 생성물 churn이 리뷰 비용을 압도한다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 맞다. 재경기·호스트 이전·RetryButton처럼 실제 플레이 중 끊기는 지점을 고친 점은 분명한 전진이다. 특히 실패 후 버튼 상태를 복구하는 수정은 UI가 “막힌 것처럼 보이는” 치명적인 체감 버그를 줄인다. 다만 `RoomUI.cs`와 씬 직렬화 참조가 함께 움직이는 구조라서, 코드 수정만으로는 안전하다고 보기 어렵다. Inspector 참조와 빌더 산출물이 일치하는지 확인하지 않으면 다음 재빌드에서 같은 문제가 다시 나온다.

가장 큰 위험은 여전히 씬 생성물이다. `Game_01_BlackAndWhite.unity` 한 파일에서 1.2만 줄 이상 삽입·삭제가 발생했는데, 이 상태에서는 실제 의미 변경과 Unity 파일 ID 재배치가 섞여 리뷰가 거의 불가능하다. 왜 문제냐면 네트워크 씬, UI 참조, TMP, 버튼, 오버레이가 직렬화 참조에 의존하는데 파일 ID churn은 null 참조 회귀를 숨기기 때문이다. 개선하려면 빌더 재실행 후 객체 수, 핵심 컴포넌트 참조, 버튼 OnClick, NetworkObject 배치만 비교하는 검증 기준을 별도로 둬야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 hot path 지적은 없다. 다만 생성 씬 안에 반복 오브젝트와 머티리얼 인스턴스가 계속 직렬화되는 구조는 런타임 복잡도보다 리뷰·검증 복잡도를 키우는 문제다. `List`/`Dictionary` 선택보다 지금은 Prefab·데이터 테이블·빌더 검증으로 생성 단위를 안정화하는 것이 더 의미 있다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 세션 상태, 호스트 이전, stale scene 정리, Matching 복귀를 직접 제어하는 hybrid에 가깝다. WebGL 2클라 검증과 재경기 수정이 누적되어 제출 가능한 수직 슬라이스는 상당히 올라왔다. 현재 진행도는 82로 본다.

다만 risk는 high다. 이유는 완성도가 낮아서가 아니라, 네트워크 씬 수명과 Unity 씬 직렬화 부채가 겹쳐 있기 때문이다. 지금 상태는 “동작한다”와 “다음 수정에도 깨지지 않는다” 사이에 아직 간극이 있다.

## 4. 다음 권장사항
- `RoomUIBuilder`에 이번 `Base_01_Matching` 흑백 UI 변경을 즉시 반영해야 한다.
- Ready·Start·Retry 버튼의 SerializedField 참조를 실제 Inspector 기준으로 재검증해야 한다.
- `Base_01_Matching` 재빌드 결과를 현재 씬과 대조해 생성 노이즈와 의미 변경을 분리해야 한다.
- 호스트 이전 후 재경기 검증은 1회 통과가 아니라 반복 실행 로그로 남겨야 한다.
- `BlackAndWhite3DSceneBuilder` 재생성 전후 의미 diff 기준을 추가해야 한다.

# HANPAN — 재입장 안정화 스냅샷

## 1. 주요 변경사항
- `GameOver` 상태의 late rejoin 보정이 `BlackAndWhiteController` 직접 `GoToRoom()` 호출에서 `GameFlowManager.HandleStaleGameRespawn()`으로 이동했다. 책임 분리는 이전보다 낫다.
- 세 Join 계열 메서드의 ready 초기화 try-catch가 `ResetReadyAfterJoinAsync()`로 공통화됐다. 중복 제거는 명확한 개선이다.
- `RoomUI.RefreshUI()`가 unlock 이후 비호스트 버튼을 다시 활성화하도록 보정됐다. Player2 준비 버튼 비활성화 버그의 직접 원인은 줄었다.
- `BlackAndWhite_Spec`에 Boot 씬, 네트워크 오류, 재입장 시나리오가 추가됐다. 문서가 QA 기준에 가까워진 점은 좋다.

## 2. 코드 품질 리뷰
이번 변경의 가장 좋은 점은 책임 방향을 바로잡은 것이다. 게임 컨트롤러가 Room 전환 판단까지 직접 들고 있던 구조는 게임이 늘어날수록 각 컨트롤러에 복귀 보정이 흩어지는 문제가 있었다. `GameFlowManager.HandleStaleGameRespawn()`으로 이동한 것은 적절하다. 다만 여전히 `GameFlowManager.Instance`가 없으면 로그만 남기고 복구가 끝난다. Matching 씬 로드 보장과 Room 전환 실패 처리가 분리돼 있지 않으므로, 이 경로는 시연 중 재입장 실패를 조용한 UI 정지로 만들 수 있다.

`ResetReadyAfterJoinAsync()` 공통화도 올바른 방향이다. 그러나 현재 구현은 `SessionNotFound`류만 분류하고, 나머지 오류는 경고 후 계속 진행한다. 왜 문제인지 분명하다. ready 값이 이전 세션에서 남으면 방장이 시작 조건을 잘못 판단해 자동 시작 또는 버튼 상태 꼬임이 재발할 수 있다. 개선 방향은 오류를 "무시 가능 / 사용자 재시도 필요 / 입장 차단"으로 나누고, `RoomUI`에 준비 상태 불확정 표시와 재시도 버튼을 연결하는 것이다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적은 없다. 새로 추가된 코드는 Join 직후와 UI refresh 분기 위주이며, hot path에 LINQ, 임시 컬렉션, 반복 선형 탐색을 추가하지 않았다. 최적화보다 상태 전이의 관측 가능성과 실패 정책을 먼저 고정해야 하는 변경이다.

## 3. 진행도 평가
구현 경로는 NGO, UGS Session/Lobby/Relay를 쓰되 세션 상태, ready 정책, 씬 복귀 흐름을 직접 설계하는 `hybrid`에 가깝다. 에디터 2인 재검증과 재입장 보정이 확인됐으므로 제출 가능한 수직 슬라이스는 상당히 올라왔다.

다만 최종 제출 경로인 WebGL 배포 URL에서 재빌드 후 2인 완주가 아직 비어 있다. 네트워킹 게임에서 로컬/에디터 검증과 브라우저 배포 검증은 동급이 아니다. 진행도는 82로 본다. 기능은 거의 제출권에 들어왔지만, 배포 환경과 ready 실패 처리 때문에 risk는 여전히 high다.

## 4. 다음 권장사항
- WebGL을 다시 빌드하고 배포 URL에서 코드 입장 기준 2인 완주를 먼저 검증한다. 이 항목이 제출 가능 여부를 가른다.
- `RoomUI._suppressLockFeedback`를 첫 refresh 억제가 아니라 `UnlockSessionAsync` 완료 이벤트나 명시 상태로 교체한다.
- `ResetReadyAfterJoinAsync()` 실패를 로그로만 끝내지 말고 UI 차단, 수동 재시도, 입장 허용 기준으로 분리한다.
- 코드 입장뿐 아니라 ID 재입장과 빠른입장 재접속 결과를 `QA_Checklist`에 브라우저·URL·빌드 기준으로 기록한다.

## 5. 문서화 상태
design은 8점이다. `Concept_Template`, 개발 계획, 작업 분해 문서가 프로젝트 목표와 마일스톤을 충분히 설명한다. 다만 일부 계획 문서에는 과거 씬명과 현재 `Base_01_Matching` 구조가 섞여 있어 정본 관리가 더 필요하다.

technical은 9점이다. `Docs/SceneArchitecture.md`와 `README/4. Network_Authority.md`가 씬 수명, NGO 경계, 세션 상태, 권한 모델을 신규 유입자가 따라갈 수 있는 수준으로 정리한다. 현재 프로젝트의 강점이다.

spec은 8점이다. `README/5. BlackAndWhite_Spec.md`가 입력, 상태, 결과, 네트워크 오류, 재입장 시나리오까지 확장됐다. 다만 QA 결과가 시나리오 표와 실제 실행 증적으로 완전히 연결되지는 않았다.

## 6. Backlog
- `Base_01_Matching` UI는 아직 Prefab·데이터 기반으로 분리되지 않았다. Builder와 씬 직렬화가 계속 결합되면 작은 UI 수정도 큰 diff로 번진다.
- `Game_01_BlackAndWhite` 씬 생성물 비대화는 남아 있다. 리뷰자는 기능 변경과 Unity 직렬화 노이즈를 계속 같이 읽어야 한다.
- 씬 재빌드 시 파일 ID 대량 변경 문제도 유지된다. Prefab 분리나 빌더 안정화 없이는 회귀 추적 신뢰도가 낮다.
- WebGL 흑화면 조건은 브라우저별 지원 범위로 아직 닫히지 않았다. 배포 URL 검증 때 브라우저, 압축, 메모리 조건을 함께 기록해야 한다.

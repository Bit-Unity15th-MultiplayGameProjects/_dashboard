# HANPAN — v2.8 문서 갱신 스냅샷

## 1. 주요 변경사항
- `README/3. Work_Breakdown.md`가 v2.8로 갱신되며 `GameFlowManager` StackTrace 진단 로그 격리 항목이 추가됐다.
- 변경은 문서 1개에 한정된다. 실제 `GameFlowManager.cs` 코드의 StackTrace 생성은 이번 커밋에서 제거되지 않았다.
- M6 부채 목록에도 StackTrace 격리가 추가되어 문제 인식은 명확해졌으나, 배포 전 처리 기준은 아직 실행되지 않았다.
- 이전부터 남아 있던 InRoom 헬스체크, Returning 잠금 해제, 배포 URL QA 항목은 현재 스냅샷 기준 계속 미해결이다.

## 2. 코드 품질 리뷰
이번 커밋은 코드 변경 없이 문서 TODO를 추가한 작업이다. 문제를 문서화한 점은 좋다. 특히 `GoToLobby()`·`GoToRoom()` 전환 경로에서 `new System.Diagnostics.StackTrace()`가 반복 생성된다는 지적은 정확하다. 다만 이 항목은 이미 런타임 hot path에 있는 성능·로그 품질 문제이므로 문서 TODO로만 남기면 위험하다. WebGL에서는 스택 캡처와 문자열 할당이 브라우저 콘솔 노이즈, 프레임 지연, 원인 분석 비용으로 바로 연결된다.

자료구조/복잡도/할당 관점에서는 이번 diff 자체에 컬렉션 선택이나 탐색 복잡도 문제는 없다. 대신 문서가 지목한 StackTrace는 명확한 할당 문제다. 현재 비용은 전환 호출마다 스택 객체와 호출자 문자열을 새로 만드는 구조이고, 개선 방향은 `#if UNITY_EDITOR` 또는 `EnableFlowTrace` 같은 정적 진단 플래그 뒤에서만 1회 계산하는 것이다. 이 프로젝트는 Lobby↔Room 전환을 GC 없는 SetActive 경로로 설계했으므로, 그 경로에 진단 할당이 남아 있으면 설계 의도와 충돌한다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰는 `hybrid`에 가깝다. 2인 매칭, 룸, 흑과 백 1판, 복귀 구조는 상당히 갖춰졌고 문서도 현행 구조를 잘 따라가고 있다. 그래서 수직 슬라이스 진행도는 72로 본다.

다만 위험도는 high다. 이유는 남은 문제가 단순 폴리싱이 아니라 네트워크 상태 전이, WebGL 배포 QA, 복귀·잠금 해제 경로에 걸려 있기 때문이다. 특히 P3-2 배포 URL 2인 완주와 브라우저별 검증이 비어 있으면 제출 시연 안정성을 확정했다고 보기 어렵다.

## 4. 다음 권장사항
- `GameFlowManager` StackTrace 로그를 즉시 진단 플래그 뒤로 옮겨야 한다. 문서화가 아니라 코드 제거가 완료 기준이다.
- InRoom disconnect 헬스체크는 disconnect 이벤트마다 기존 예약을 취소하고 새 지연 판정을 예약하는 구조로 바꿔야 한다.
- 준비 초기화 실패는 사용자 재시도 버튼, 원인 로그, 준비 버튼 차단 상태가 같은 실패 모델을 보도록 연결해야 한다.
- `UnlockSessionAsync`는 첫 입장 공통 경로가 아니라 Returning 완료 경로에서만 실행되게 분리해야 한다.
- P3-2, 재게임 초기화, ID·빠른매칭 재접속 QA는 배포 URL과 브라우저명을 실제 값으로 채워야 한다.

## 5. 문서화 상태
design은 8점이다. `README/1. Concept_Template.md`, 개발 계획, Work Breakdown이 프로젝트 목표와 마일스톤을 충분히 설명한다. 다만 `_sample/docs`처럼 플레이 경험 목표와 선택 이유가 일관된 데이터 규격까지 내려오지는 않는다.

technical은 8점이다. `Docs/SceneArchitecture.md`와 `README/4. Network_Authority.md`가 씬 수명, NGO 전환, 권한 모델, 복귀 흐름을 신규 인원이 따라갈 수 있을 정도로 정리한다. 단, 일부 항목은 문서상 완료와 코드상 미완료 TODO가 섞여 있어 검증 표기가 더 엄격해야 한다.

spec은 8점이다. `README/5. BlackAndWhite_Spec.md`와 QA 체크리스트가 상태, 입력, 승패, 테스트 흐름을 실제 체크리스트 수준으로 제공한다. 아직 배포 URL 실측 결과와 브라우저별 지원 범위가 빈칸이라 9점 이상은 아니다.

## 6. Backlog
- `Base_01_Matching`은 여전히 씬 직렬화와 Editor Builder 중심이다. Prefab·데이터 기반 계약으로 분리하지 않으면 UI 수정 때마다 리뷰 범위가 커진다.
- `Game_01_BlackAndWhite` 씬 생성물 문제는 해소 근거가 부족하다. 생성물 커밋이 계속 리뷰 비용을 키우는 구조로 본다.
- P2-4는 완료됐지만 P3-2 배포 URL 2인 완주 증적은 아직 비어 있다. 외부 URL 기준 검증 없이는 제출 안정성을 주장하기 어렵다.
- WebGL 흑화면 조건은 Chrome, Edge, Safari 등 브라우저별 지원/차단 기준으로 확정해야 한다.
- 씬 재빌드 시 파일 ID 대량 변경 문제는 Prefab 기준 재생성 흐름이 잡힐 때까지 계속 리뷰 신뢰도를 낮춘다.

# HANPAN — 05.14 QA·씬 재빌드 스냅샷

## 1. 주요 변경사항
- `.gitattributes`가 추가되어 Unity 씬 파일 diff 노이즈를 줄이려는 방향은 확인된다. 문제를 인식하고 도구 레벨에서 완화하려 한 점은 좋다.
- `Base_01_Matching.unity`가 5천 줄 이상 재배치되며 Room UI의 흑백 톤, 슬롯, 버튼, 텍스트 계층이 크게 바뀌었다.
- `README/6. QA_Checklist.md`에 05.14 회귀 테스트 통과 기록이 추가된 것으로 보인다.
- 반면 이번 범위에는 `RoomUIBuilder.cs` 또는 `RoomUI.cs` 변경이 없다. 현재 프로젝트 원칙인 “빌더 단일 소스”와 충돌한다.

## 2. 코드 품질 리뷰
가장 큰 위험은 씬 파일을 직접 바꾼 흔적이다. `README/0. Unity_Dev_Plan.md`는 `Base_01_Matching` UI를 빌더 메뉴로만 생성·수정한다고 못박고 있는데, 이번 diff는 씬 YAML만 대량 변경되고 `RoomUIBuilder.cs`가 따라오지 않았다. 왜 문제냐면 다음에 빌더를 재실행하는 순간 이번 UI 조정, 버튼 계층, 직렬화 참조가 조용히 사라질 수 있기 때문이다. 개선은 단순하다. 이번 씬 결과를 `RoomUIBuilder`에 역반영하고, 재생성 후 diff가 같은지 확인하는 절차를 QA 항목으로 고정해야 한다.

`fd25e7f` 메시지는 NVar 핸들러·재시도 버튼·UI 흑백 검증을 말하지만, 코드 diff가 아니라 씬과 문서 중심이다. 검증 기록은 필요하지만, 런타임 결함을 막는 최종 방어선은 코드와 직렬화 참조다. 특히 Ready, Start, Retry 계열 버튼은 null이어도 일부 경로에서만 드러나는 UI 버그가 되기 쉽다. Inspector 참조 스냅샷 또는 빌더 재생성 검증을 남겨야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음. C# hot path, LINQ, 반복 탐색, 컬렉션 재할당 변경이 보이지 않는다. 이번 리뷰의 성능 이슈는 알고리즘 비용이 아니라 생성물 관리 비용과 리뷰 가능성 저하에 있다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 세션 상태와 권한 흐름을 직접 설계한 `hybrid`로 보는 것이 맞다. 05.14 회귀 테스트 통과 기록과 UI 검증은 수직 슬라이스 완성도를 올리는 신호이며, 문서도 현행 아키텍처를 상당히 잘 설명한다.

다만 제출 가능성 관점에서는 위험이 높다. 네트워크 자체보다 “씬 생성 정본이 무엇인가”가 흔들리면 QA 통과 상태를 재현하지 못한다. 현재 진행도는 76으로 보되, 빌더-씬 불일치를 방치하면 다시 60대 중반으로 떨어질 수 있다.

## 4. 다음 권장사항
- `RoomUIBuilder`에 이번 흑백 UI 변경을 역반영하라. 씬 파일만 맞는 상태는 유지보수 가능한 완료가 아니다.
- Ready, Start, Retry 등 `RoomUI` 핵심 버튼 참조를 Inspector 기준으로 재검증하라.
- `Base_01_Matching` 재빌드 산출물과 현재 씬 diff를 대조한 기록을 남겨라.
- 05.14 QA 통과 행에 로그, 스크린샷, 브라우저 페어, 빌드 ID를 붙여라.
- WebGL 배포 URL에서 2클라 코드 입장 1판을 다시 검증하라.

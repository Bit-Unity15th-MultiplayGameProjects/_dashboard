# HANPAN — 2026-05-13 후속 수정 스냅샷

## 1. 주요 변경사항
- `BlackAndWhiteGameUI`가 `BindController` 진입 시 기존 컨트롤러 이벤트를 해제하고, `OnDisable`/`OnDestroy`에서도 같은 해제 경로를 타도록 정리됐다.
- `RoomUI.ResetReadyOnEnterAsync`에 준비 초기화 실패 시 2초 후 자동 재시도 1회가 추가됐다.
- QA 문서에 호스트 마이그레이션 후 로비 목록 재노출 확인, WebGL 빌드 시각, ZIP 크기, 배포 URL 기준 검증 기록이 보강됐다.
- 두 Unity scene 파일에 색상, TMP 스타일, RectTransform, Canvas 채널 변경이 함께 들어왔다. UI 개선 의도는 보이나 리뷰 노이즈도 같이 증가했다.

## 2. 코드 품질 리뷰
`BlackAndWhiteGameUI.UnbindController()`로 컨트롤러 이벤트 해제를 한곳에 모은 점은 좋은 방향이다. 재바인딩 전에 기존 구독을 해제하는 선택도 맞다. 다만 현재 구현은 `_state.RoundNumber.OnValueChanged += (_, v) => ...`, `_state.Phase.OnValueChanged += (_, v) => ...` 람다 구독을 해제하지 못한다. 이 상태에서 `OnDisable` 후 재바인딩되면 이전 `MatchState`가 비활성 UI 인스턴스를 계속 참조할 수 있다. 왜 문제냐면 씬 잔존·재게임·late rejoin 경로에서 HUD 갱신이 중복되거나 파괴된 객체로 콜백이 날아갈 수 있기 때문이다. `OnRoundNumberChanged`, `OnPhaseChanged` 같은 명명 핸들러로 바꾸고, `_boundState`를 보관해 `UnbindController`에서 NetworkVariable 구독까지 풀어야 한다.

`RoomUI`의 자동 재시도는 이전보다 낫지만, 실패 후 상태 회복 UX는 아직 약하다. 2초 재시도까지 실패하면 사용자는 여전히 나갔다 들어오는 것밖에 할 수 없다. UGS/Lobby 계열 실패는 일시적일 수 있으므로 같은 화면에서 재시도 버튼을 제공하고, 성공 시 `SetInteractable(true)`와 `RefreshUI()`로 복구되는 경로를 명시해야 한다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 `Update` hot path의 LINQ, 반복 선형 탐색, 임시 컬렉션 생성 같은 명확한 지적은 없다. 다만 람다 이벤트 구독은 구조상 해제가 불가능해 생명주기 비용을 만든다. 이는 성능 최적화 문제가 아니라 재바인딩 안정성 문제로 봐야 한다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 세션 상태, 호스트 마이그레이션, 씬 복귀 정책을 직접 엮는 `hybrid`에 가깝다. 배포 URL 2인 완주와 QA 표 갱신이 있어 제출 가능한 수직 슬라이스는 상당히 올라왔다.

진행도는 83으로 본다. 핵심 플레이 루프와 WebGL 배포 증적은 있으나, UI 이벤트 수명과 준비 초기화 실패 복구가 아직 시연 중 랜덤 실패로 이어질 수 있다. scene 직렬화 부채도 다음 변경의 리뷰 신뢰도를 계속 갉아먹는다.

## 4. 다음 권장사항
- `BlackAndWhiteGameUI`의 `RoundNumber`/`Phase` 구독을 명명 핸들러로 바꾸고 `UnbindController`에서 정확히 해제한다.
- 준비 초기화 실패 화면에 수동 재시도 버튼을 추가하고, 성공 시 버튼 잠금과 상태 문구가 정상 복구되는지 검증한다.
- `BoardLayout3D.OnDestroy`에 `OnTiebreakerReset` 해제가 실제 구현돼 있는지 코드로 확인하고 문서와 맞춘다.
- QA 통과 표에 콘솔 로그, 스크린샷, 녹화 또는 실행 로그 파일명을 남겨 재현 가능한 증거로 만든다.
- scene 변경은 Builder 재생성 결과와 대조해 의도한 UI 변경만 남기는 절차를 추가한다.

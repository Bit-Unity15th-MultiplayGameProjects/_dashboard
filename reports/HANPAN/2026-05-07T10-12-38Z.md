# HANPAN — 네트워크 씬 수명 안정화 스냅샷

## 1. 주요 변경사항
- `BlackAndWhiteController`가 전역 `FindFirstObjectByType` 대신 같은 씬 내부에서 UI·보드 컴포넌트를 찾도록 변경됐다.
- `Phase == GameOver` 상태의 구 게임 씬 재스폰을 서버와 클라이언트로 나눠 처리하여 호스트 마이그레이션 후 Room UI가 사라지는 문제를 막았다.
- 재시작 전 `MatchState.Phase`를 `Idle`로 되돌려 새 게임을 late rejoin으로 오판하는 경우를 줄였다.
- `BlackAndWhiteGameUI`에서 게임 씬의 `Camera`까지 비활성화하여 Matching 씬 위로 3D 보드가 비치는 문제를 직접 차단했다.
- 클라이언트 경로에서 `SceneLoader.AllGameScenes`를 순회해 잔존 게임 씬을 정리하는 보강이 추가됐다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 타당하다. 전역 검색으로 인해 이전 씬의 컴포넌트에 바인딩되는 문제를 같은 씬 루트 기준 검색으로 제한한 것은 Unity 씬 병존 구조에서 반드시 필요한 수정이다. 특히 `BlackAndWhiteController.OnNetworkSpawn`에서 서버의 구 씬 재스폰과 클라이언트 late rejoin을 분리한 판단은 버그 원인을 코드에 비교적 명확히 남긴 좋은 결정이다.

다만 위험 신호도 분명하다. 씬 내부 검색 코드가 `BlackAndWhiteController`, `BlackAndWhiteGameUI`, `BoardLayout3D`, `BoardView3D`에 반복된다. 왜 문제냐면 씬 수명 정책이 바뀔 때 네 곳을 함께 고쳐야 하고, 한 곳만 빠져도 다시 크로스씬 바인딩 회귀가 난다. `SceneScopedLookup.FindInSameScene<T>(Component owner)` 같은 공용 헬퍼로 통합하고, “전역 검색 금지”라는 정책을 코드 구조로 강제하는 편이 낫다.

자료구조/복잡도 관점에서는 `SceneLoader.AllGameScenes` 선형 순회가 현재 등록 씬 수 기준으로 작고 세션 시작 전 1회 수행되므로 성능 문제는 아니다. 다만 비용보다 유지보수성이 문제다. 게임 씬 추가 시 문자열 배열을 수동 갱신해야 하므로 누락되면 클라이언트 잔존 씬 정리가 실패한다. 빌드 세팅, Addressables, 또는 게임 모드 정의 데이터에서 씬 목록을 생성하도록 바꾸면 누락 비용을 줄일 수 있다. 또한 `NetworkSceneTransition`은 `_loadedGameScene`을 null로 만든 뒤 로그에 출력해 실제 씬명이 비는 버그가 있다. 사소해 보여도 네트워크 씬 버그 추적에서는 로그 신뢰도가 곧 디버깅 속도다.

## 3. 진행도 평가
이번 범위는 기능 확장보다 재시작·퇴장·호스트 마이그레이션 회귀를 막는 안정화 작업이다. 플레이 가능성에는 의미 있는 전진이지만, 커밋 3개가 모두 `fix`인 점은 핵심 루프 구현보다 씬 수명 부채 상환에 시간을 쓰고 있다는 신호다.

현재 완성도는 약 45%로 본다. BlackAndWhite 한 종목의 네트워크 플레이는 점차 안정화되고 있으나, 여러 게임 씬을 공통 규칙으로 운영하는 구조와 문서화가 부족하다. 이 상태에서 게임 종류를 늘리면 버그가 선형이 아니라 조합적으로 늘어날 가능성이 높다.

## 4. 다음 권장사항
- 게임 씬 수명과 네트워크 세션 소유권을 단일 상태표로 정리해야 한다. 서버, 호스트 마이그레이션, 클라이언트 재접속의 책임 경계를 먼저 고정해야 한다.
- 재시작·퇴장·호스트 마이그레이션 통합 재현 체크리스트를 작성해야 한다. 이번 버그군은 자동화 전이라도 수동 시나리오가 없으면 반복된다.
- 씬 내부 컴포넌트 검색 로직을 공용 헬퍼로 통합해야 한다. 같은 정책을 복붙으로 유지하는 것은 다음 회귀의 씨앗이다.
- `NetworkSceneTransition`의 로그와 null 처리 순서를 고쳐야 한다. 네트워크 이슈에서 부정확한 로그는 원인 분석을 직접 방해한다.
- BlackAndWhite 입력, UI 상태, HUD 수치 사양을 문서화해야 한다. 구현이 있어도 판정 기준이 없으면 QA가 경험담에 의존한다.

## 5. 문서화 상태
design 점수는 4점이다. 게임의 큰 방향과 씬 흐름은 일부 드러나지만, 플레이어 경험 목표와 각 게임 모드가 왜 그렇게 설계됐는지까지 설명하는 문서는 확인되지 않는다.

technical 점수는 3점이다. 이번 diff만 봐도 네트워크 세션, additive scene, late rejoin, host migration이 얽혀 있는데 이를 설명하는 권한 모델과 상태 전이 문서가 없다. 신규 개발자는 코드를 실행해 보며 역추적해야 한다.

spec 점수는 2점이다. 이전 backlog의 입력 매핑, UI 상태 전이, HUD 수치 부재가 그대로 남아 있다. 테스트 체크리스트로 쓸 수 있는 사양서 수준에는 아직 도달하지 못했다.

## 6. Backlog
- 레벨/씬별 상세 사양서 입력 매핑·UI 상태 전이·HUD 수치 부재는 계속 남아 있다. 특히 네트워크 재접속 시 UI 판정 기준이 문서화되지 않았다.
- `Game_01_BlackAndWhite.unity` 생성물 커밋 문제는 이번 범위에서 해소 근거가 없다. 씬 변경 리뷰 비용을 낮추는 구조가 필요하다.
- `Base_02_Lobby.unity` UI 하드코딩 문제도 유지된다. Lobby는 재접속·퇴장 흐름의 중심이라 프리팹화 우선순위가 높다.
- URP 튜토리얼·Mobile 렌더러 잔존 이슈는 낮은 우선순위지만 정리되지 않았다. 제출 전 불필요 자산은 빌드와 리뷰 노이즈를 만든다.

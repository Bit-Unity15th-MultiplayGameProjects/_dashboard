# HANPAN — 연장전 안정화와 WebGL 검증 스냅샷

## 1. 주요 변경사항
- 연장전 완전 동점 시 A 강제 승리 대신 재연장전으로 진입하도록 수정했다. 이는 게임 규칙의 공정성 측면에서 명확한 개선이다.
- 연장전 진입 때 `Score`, `HasSubmitted`, `PendingColor`, `RoundNumber`를 초기화하고 양쪽에 `{1,2,3}` 패를 다시 지급하도록 정리했다.
- 재게임 시 이전 NGO 세션 씬을 plain unload로 처리하는 `_sceneFromCurrentSession` 플래그가 추가되어 SceneHandle LogError를 줄였다.
- Itch.io 2인 완주, Chrome Default, Firefox 결과가 문서에 기록되어 WebGL 시연 리스크가 크게 낮아졌다.
- TMP 왕관 글리프 경고를 제거했으나, 아이콘 자체를 빈 텍스트로 대체한 결정은 UI 의도 전달 측면에서 후속 보완이 필요하다.

## 2. 코드 품질 리뷰
`BlackAndWhiteController`의 연장전 수정은 방향이 좋다. 특히 `HasSubmitted`를 초기화하지 않아 다음 제출이 막히던 원인을 주석으로 남긴 점은 디버깅 가능한 코드에 가깝다. 다만 `OnTiebreakerReset`을 `BlackAndWhiteGameUI`와 `BoardLayout3D`에서 구독만 하고 해제·중복 방지 패턴이 보이지 않는다. `BindController`가 재호출되거나 씬 잔존 후 재바인딩되면 같은 리셋이 여러 번 실행될 수 있다. `OnDisable`/`OnDestroy`에서 해제하거나, 구독 직전에 `-=` 후 `+=` 하는 방식으로 방어해야 한다.

`NetworkSceneTransition`의 `_sceneFromCurrentSession`은 “현재 NGO가 아는 씬”과 “이전 세션 잔존 씬”을 구분한다는 점에서 타당하다. 왜 문제였는지 코드 주석도 충분하다. 단, 이 클래스의 정적 상태가 계속 늘고 있으므로 회귀 테스트 없이 손대면 재접속·재게임·abort 흐름이 다시 얽힐 위험이 있다.

자료구조/복잡도 관점에서는 이번 diff 범위에서 명확한 hot path 문제는 없다. `_playerStates` 순회와 `new byte[] { 1, 2, 3 }` 생성은 연장전 진입 시점의 저빈도 작업이라 최적화 대상이 아니다. 지금은 Dictionary나 풀링보다 이벤트 수명 관리가 더 중요한 비용이다.

## 3. 진행도 평가
구현 경로는 NGO, Relay, Lobby를 쓰되 매칭·권한·씬 복구 규칙을 직접 설계한 hybrid에 가깝다. 외부 URL 2인 완주와 브라우저별 WebGL 결과가 기록되면서 제출 가능한 수직 슬라이스는 80%대에 진입했다.

다만 최근 커밋이 대부분 QA 보강과 결함 수정이라는 점은 경고 신호다. 큰 방향은 맞지만, 네트워크 씬 전환과 연장전 이벤트가 정적 상태·ClientRpc·Unity 이벤트에 걸쳐 있어 작은 누락이 런타임에서만 드러난다. 현 시점 위험도는 high까지는 아니지만 medium으로 보는 것이 맞다.

## 4. 다음 권장사항
- `OnTiebreakerReset 구독 해제와 중복 방지 추가`: 재연장전·재게임 반복에서 HUD와 보드 초기화가 중복 호출되지 않게 막아야 한다.
- `9라운드 점수차 종료 QA 항목 복구 후 2클라 재검증`: 동점 연장전만 검증하면 기본 승패 종료 경로의 회귀를 놓친다.
- `준비 초기화 실패를 재시도 버튼과 원인 로그로 연결`: 네트워크 실패는 사용자 복구 행동과 개발자 로그가 같이 있어야 한다.
- `호스트 마이그레이션 후 로비 목록 재노출을 외부 URL로 검증`: 방장 전환 성공과 로비 목록 반영 성공은 같은 문제가 아니다.
- `재게임·재입장 QA 증적을 배포 URL 기준으로 정정 기입`: 에디터 통과와 배포 URL 통과를 섞어 쓰면 시연 안정성 판단이 흐려진다.

## 5. 문서화 상태
design은 8점이다. `Unity_Dev_Plan`과 컨셉 문서가 프로젝트 목표, 씬 구성, 마일스톤을 충분히 설명한다. 다만 일부 완료 표기가 실제 QA 근거와 섞여 있어 의사결정 기록으로는 약간 흔들린다.

technical은 9점이다. `Docs/SceneArchitecture.md`와 `Network_Authority.md`가 씬 수명, 권한 모델, NGO/Plain SceneManager 경계를 신규 참여자가 따라갈 수 있을 정도로 설명한다.

spec은 8점이다. `BlackAndWhite_Spec.md`와 `QA_Checklist.md`는 테스트 체크리스트로 쓸 수 있는 수준이다. 그러나 이번에 W2 비동점 종료 항목이 제거된 것처럼 핵심 규칙의 검증 행이 사라지는 관리 리스크가 남아 있다.

# YJ — 서버 검증 보강 스냅샷

## 1. 주요 변경사항
- `TurnMachine.SyncStatus`가 생명값뿐 아니라 `currentTurn`까지 서버 상태로 반영하도록 바뀐 점은 분명한 전진이다.
- 서버 `GameRoom`에 `PieceOwners`, `PromotedPieces`, `ValidateMove`가 추가되어 이동 검증을 서버 쪽으로 옮기려는 방향이 보인다.
- `C_MoveHandler`가 포획, 승급, 입궁 승리, 턴 전환 후 `S_SyncStatus` 브로드캐스트까지 처리하도록 확장됐다.
- `C_DropHandler`도 서버 검증을 시작했지만, 포로 보유 여부와 기존 보드 배치 여부를 확인하지 않아 아직 위험하다.

## 2. 코드 품질 리뷰
이번 커밋의 좋은 점은 서버 권위 구조로 실제 이동 검증을 옮기기 시작했다는 점이다. 이전처럼 클라이언트가 보낸 이동 결과를 거의 그대로 믿는 구조보다는 훨씬 낫다. 다만 `CheckMovePattern`은 현재 `Jang`, `Sang`, 승급 기물을 모두 8방향 1칸 이동 가능으로 처리한다. 이는 GDD의 장/상/후 이동 규칙과 다르며, 승패 판정 자체가 다른 게임이 될 수 있다. 기물별 허용 이동 벡터를 테이블로 분리하고, `PromotedPieces`는 별도 타입처럼 검증해야 한다.

가장 큰 경고는 `C_DropHandler`다. 현재는 `pieceId`가 실제 포로 목록에 있는지, 이미 보드 위에 있는 기물인지, 좌표가 보드 안인지 충분히 검증하지 않는다. 이 상태에서는 클라이언트가 임의 ID를 보내 기물을 생성하거나, 자기 기물을 드랍 패킷으로 순간이동시키는 식의 서버 권위 붕괴가 가능하다. 서버는 `CapturedPiecesByTeam` 같은 명시적 보유 목록을 두고, 드랍 성공 시에만 목록에서 제거해야 한다.

자료구조/복잡도 측면에서는 `GetPieceAt`과 `IsOccupied`가 매 요청마다 `PiecePositions`를 선형 순회한다. 4x3 보드라 비용 자체는 작지만, 검증 코드가 여러 곳에 중복되어 버그가 커진다. `int[4,3]` 점유표나 좌표→pieceId 인덱스를 `PiecePositions`와 함께 갱신하면 조회 비용과 검증 흐름이 동시에 단순해진다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷, 룸, 턴, 보드 상태를 직접 구현하고 있으므로 컨텐츠 양보다는 패킷 계약과 서버 권위 안정성을 더 크게 본다. 이번 커밋은 이동 검증과 턴 동기화에서 의미 있는 전진이 있어 진행도는 52로 본다.

하지만 제출 가능한 2인 수직 슬라이스로 보기에는 아직 불안정하다. Drop 권한 구멍, 씬 로드 중 `S_SyncStatus` 유실 가능성, TCP partial packet 처리 미확정, 실패 응답 부재가 모두 시연 중 랜덤한 오작동으로 이어질 수 있다. 그래서 위험도는 여전히 high다.

## 4. 다음 권장사항
- `C_Drop`에 포로 보유 목록, 보드 범위, 기존 배치 여부 검증을 추가해야 한다.
- `CheckMovePattern`을 문자열 분기 대신 기물별 이동 벡터 테이블로 고정해야 한다.
- `S_Login`과 `S_SyncStatus`는 씬 로드 완료 후 메인 스레드 상태 큐에서 처리해야 한다.
- TCP 수신 경로에 Size 헤더 기준 누적 파서와 partial packet 테스트를 붙여야 한다.
- TDD에 `C_Move`/`C_Drop` 권한, 실패 응답, 상태 소유자 표를 현재 코드 기준으로 갱신해야 한다.
- 보드 좌표 조회는 선형 순회 대신 4x3 점유표나 좌표 인덱스로 분리하는 편이 낫다.

## 5. 문서화 상태
design은 7점이다. `docs/concept.md`와 `docs/gdd.md`에 코어 루프, 포로, 승급, 승리 조건이 상당히 구체적으로 적혀 있다. 다만 placeholder와 빈 KPI, 룰 출처 불명확성이 남아 최종 GDD로 보기엔 부족하다.

technical은 5점이다. `docs/tdd.md`의 분량은 충분하지만 현재 서버 권위 구현, 실패 응답, 패킷 프레이밍, 재접속 상태 스냅샷과 정확히 맞지 않는다. 신규 인원이 문서만 보고 네트워크 구조를 안전하게 수정하기 어렵다.

spec은 6점이다. 기물 이동, 포로, 승급, 입궁 같은 기능 사양은 표와 플로우로 정리되어 있다. 그러나 Drop 금지 규칙과 실제 코드가 어긋나고, 2인 happy path 검증표가 아직 제출용 QA 체크리스트 수준은 아니다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족: 서버 검증 확장분이 TDD의 소유권·실패 응답 표와 맞지 않는다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재: 프레이밍, 실패 응답, 재접속 계약이 아직 고정되지 않았다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가: 현재 턴 수신은 시작됐지만 상태 유실 대응은 없다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요: 문서를 QA 기준으로 쓰기엔 빈칸과 미확정 항목이 남아 있다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정: 현재 경로 기준으로 registry 적용 여부를 다시 확인해야 한다.
- 포로 재배치 시 '상대 진영 첫 줄' 금지 규칙은 일본 쇼기와 다름 — 룰 출처 재확인 필요: 문서와 코드의 제한 범위가 다르게 읽힌다.

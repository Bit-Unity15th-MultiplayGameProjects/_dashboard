# YJ — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수는 0개다. 이번 스냅샷에서는 코드·문서 변경으로 확인되는 전진이 없다.
- 이전에 남았던 서버 권위, 턴 상태, 패킷 프레이밍 위험은 해결 근거가 없어 그대로 유지된다.
- 문서는 `concept.md`, `gdd.md`, `tdd.md`가 존재하고 게임 규칙 표도 일부 갖췄다. 이 점은 분명한 기반이다.
- 다만 문서 안에 placeholder, 빈 표, 검증 공백이 남아 있어 제출용 사양서로 보기에는 아직 부족하다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 신규 코드 결함을 추가로 지적할 수는 없다. 그러나 누적 이슈 기준으로 보면 위험한 축은 명확하다. 서버가 이동·드랍의 최종 판정을 소유해야 하는데, 현재 항목들은 클라이언트 요청과 로컬 `TurnMachine` 상태가 결과처럼 취급될 가능성을 계속 가리킨다. 이 구조는 지연, 재전송, 재접속 상황에서 양쪽 보드가 갈라지는 원인이 된다. 개선 방향은 단순하다. 클라이언트는 의도만 보내고, 서버 검증 결과 패킷만 화면 상태에 반영해야 한다.

자료구조/복잡도 관점에서는 `Assets/Scripts/Controller/PlayerController.cs`의 `pieceId` 조회가 `FindObjectsOfType` 선형 탐색으로 남았다는 누적 항목이 해결 근거 없이 유지된다. 턴 처리나 동기화 중 id 조회가 반복되면 오브젝트 수에 비례해 비용이 증가하고 Unity 오브젝트 열거 비용도 붙는다. 기물 생성·제거 시점에 `pieceId -> Piece` registry를 갱신하면 비용이 O(1)에 가까워지고, 네트워크 패킷 적용 코드도 더 명확해진다.

커스텀 TCP 스택을 직접 들고 가는 선택은 학습 가치가 높지만, 패킷 경계와 실패 응답이 확정되지 않으면 디버깅 비용이 폭발한다. 지금은 기능을 더 얹기보다 수신 버퍼, 패킷 크기 헤더, 실패 로그, 서버 권위 상태를 먼저 고정해야 한다.

## 3. 진행도 평가
구현 경로는 `custom-network`로 판단한다. 서버 프로세스, 패킷 정의, 세션·룸, TCP 수신 처리를 직접 다루는 경로이므로 단순 콘텐츠 양보다 네트워크 안정성과 end-to-end 재현성이 더 중요하다. 이 난이도는 일부 보정하지만, 패킷 프레이밍과 서버 권위 검증이 미확정이면 2인 시연이 랜덤하게 깨질 수 있어 진행도는 42%로 본다.

이번 범위에는 커밋이 없어 이전 대비 실질 전진은 없다. 수직 슬라이스의 다음 기준은 “두 클라이언트가 같은 서버 결과만 보고 이동·드랍·턴 전환을 반복 재현할 수 있는가”다.

## 4. 다음 권장사항
- `docs/tdd.md`에 상태 소유자, 패킷 방향, 실패 응답을 한 표로 고정해야 한다.
- `GameRoom`의 이동·드랍 처리는 서버 검증 성공 후에만 상태를 바꾸도록 정리해야 한다.
- `Session` 수신 경로는 partial packet을 누적 파서로 조립하고 경계 테스트를 붙여야 한다.
- `TurnMachine`은 로컬 턴 계산이 아니라 서버 턴 상태 수신 결과를 표시해야 한다.
- 포로 Drop 룰 출처, 2인 happy path, 실패 로그 기준을 GDD/TDD 검증표로 묶어야 한다.

## 5. 문서화 상태
design은 6점이다. 컨셉 문서에 코어 루프, 기물 이동, 포로, 승급, 승리 조건이 표와 흐름도로 정리되어 있어 설계 뼈대는 좋다. 다만 작성 가이드 문장과 빈 메타 항목이 남아 실제 확정 문서로는 덜 닫혔다.

technical은 4점이다. `tdd.md` 파일 크기는 충분하지만, 현재 open item 기준으로 권한 모델, 패킷 계약, 서버 검증 코드와의 동기화가 아직 증명되지 않는다. 신규 개발자가 문서만 보고 네트워크 흐름을 안전하게 수정하기 어렵다.

spec은 5점이다. 기물 이동, 포로 재배치, 승급, 승리 조건 수치와 조건은 일부 구체적이다. 그러나 Drop 금지 규칙의 출처, 2인 재현 체크리스트, 실패 로그 기준이 비어 있어 테스트 사양으로 바로 쓰기에는 부족하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족은 계속 남는다.
- 클라이언트 로컬 `TurnMachine`이 서버 상태와 분리된 구조는 재접속·지연 대응을 막는다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재는 높은 위험으로 유지된다.
- 포로 재배치의 “상대 진영 첫 줄” 금지 규칙은 룰 출처를 재확인해야 한다.
- GDD/TDD의 빈 검증 계획과 메타 정보 공백은 제출 문서 신뢰도를 낮춘다.
- `pieceId` 선형 탐색은 registry 기반 조회로 바꿔 턴 처리 비용과 책임을 줄여야 한다.

# YJ — 실행 설정 보강 스냅샷

## 1. 주요 변경사항
- `NetworkManager`의 기본 서버 IP가 `192.168.0.18`에서 `127.0.0.1`로 변경됐다. 로컬 테스트 기본값으로는 이전보다 안전하다.
- `-ip`, `-port` 커맨드라인 인자를 파싱해 실행 시 접속 대상을 덮어쓸 수 있게 됐다. 빌드 배포 환경을 고려한 방향은 맞다.
- `GameScene.unity`의 직렬화된 `_serverIp`도 `127.0.0.1`로 맞춰졌다.
- 다만 `NetworkManager.cs`에 깨진 한글 주석이 추가됐고, 인자 검증 실패 로그가 부족하다. 작은 변경이지만 품질 신호는 좋지 않다.

## 2. 코드 품질 리뷰
이번 변경의 방향은 명확하다. 서버 주소를 개인 LAN IP에 묶어두던 문제를 줄이고, 실행 인자로 접속 대상을 바꿀 수 있게 한 점은 실전 시연에서 의미 있는 전진이다. 특히 `DontDestroyOnLoad` 싱글톤이 유지되는 구조에서는 최초 `Awake()`에서 한 번 설정을 확정하는 방식이 현재 구조와 충돌하지 않는다.

그러나 구현 완성도는 아직 낮다. `"-ip"`, `"-port"`가 매직 스트링으로 직접 박혀 있고 `ToLower()` 비교를 사용한다. 이 코드는 hot path는 아니므로 성능 문제는 작지만, `StringComparison.OrdinalIgnoreCase`와 상수로 바꾸는 편이 의도가 선명하다. `-port abc` 같은 실패 입력은 조용히 무시되며, 잘못된 IP 형식도 검증하지 않는다. 시연 현장에서 접속 실패가 나면 원인을 로그만 보고 추적하기 어렵다. 깨진 주석은 즉시 수정해야 한다. 인코딩이 깨진 주석은 코드 이해를 돕지 못하고, 협업자에게 저장소 관리가 느슨하다는 신호를 준다.

자료구조/복잡도 관점에서는 이번 diff 범위에 명확한 hot path 문제는 없다. 커맨드라인 인자 배열을 한 번 순회하는 O(n) 비용은 실행 초기화 경로라 문제가 아니다. 다만 기존 `PlayerController.FindPieceById()`의 `FindObjectsOfType(true)` 선형 탐색은 여전히 남아 있어, 네트워크 패킷 적용 경로에서는 별도 registry로 고정해야 한다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷, 세션, 룸, TCP 수신 경계를 직접 책임지는 구조이므로 단순 컨텐츠 양보다 네트워크 계약과 실패 처리의 안정성이 중요하다. 이번 커밋은 실행 환경 설정을 개선했지만, 서버 권위 검증·턴 상태·partial packet 테스트 같은 핵심 리스크는 그대로다.

진행도는 53으로 본다. 2인 플레이 수직 슬라이스의 뼈대는 있으나, 반복 시연을 안정적으로 보장하기에는 아직 위험하다. 특히 종료 처리, 드랍 검증, 씬 로드 후 상태 적용, TCP 경계 테스트가 남아 있어 `risk_level`은 high가 타당하다.

## 4. 다음 권장사항
- GameRoom의 이동·드랍·퇴장 상태 변경을 하나의 큐 또는 동일 lock 범위로 묶어라.
- C_Drop 서버 검증에 보드 범위, 점유 칸, 포로 소유권, 상대 진영 금지를 모두 넣어라.
- S_GameOver 수신 즉시 입력·타이머·패킷 송신을 멈추고, 종료 사유 필드를 패킷에 포함하라.
- S_Login/S_SyncStatus는 씬 로드 완료 후 적용되는 큐로 바꿔 초기 상태 유실을 막아라.
- NetworkManager 실행 인자는 상수화, 형식 검증, 실패 로그, LobbyScene 적용 여부까지 확인하라.

## 5. 문서화 상태
design은 7점이다. `docs/gdd.md`와 `docs/concept.md`에 코어 루프, 기물 규칙, 포로·승급·승리 조건이 꽤 구체적으로 정리돼 있다. 다만 placeholder와 빈 표가 남아 있어 확정 문서로 보기에는 부족하다.

technical은 6점이다. `docs/tdd.md`에 전용 서버 구조, 패킷 개념, 동기화 흐름이 존재한다. 하지만 실제 코드의 실패 응답, 재접속 스냅샷, partial packet 처리, 권한 검증 범위와 완전히 맞물리지는 않는다.

spec은 6점이다. 기물 이동·드랍 제한·승리 조건 수치는 테스트 케이스로 전환 가능한 수준까지 일부 내려와 있다. 반면 QA 체크리스트와 happy path 재현 절차가 빈약해, 문서만 보고 검증을 수행하기에는 아직 부족하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족: 문서의 서버 권위 설명과 C_Drop·실패 응답 구현 범위가 아직 맞지 않는다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재: 직접 구현 경로인데 프레이밍, 실패 응답, 재접속 계약이 테스트로 고정되지 않았다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가: 턴의 최종 소유자가 코드에서 일관되게 드러나지 않는다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요: 현재 문서는 쓸 만해졌지만 제출용 확정 문서 수준은 아니다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정: 패킷 적용 경로에서 씬 전체 탐색을 반복하는 구조가 남아 있다.

# YJ — 서버 턴 동기화 스냅샷

## 1. 주요 변경사항
- 서버 `CheckMovePattern`이 왕·장·상·자·후 이동 규칙을 분기해 실제 기물별 이동 판정에 가까워졌다. 이 부분은 이전보다 명확한 전진이다.
- `S_SyncStatus`에 입궁 대기 상태가 추가되고, 클라이언트 `TurnMachine`의 주요 setter가 서버 동기화 경로로 묶였다.
- 클라이언트는 이동·드랍 후 로컬 `EndTurn()`을 호출하지 않고 서버 패킷을 기다리도록 바뀌었다. 서버 권위 방향 자체는 맞다.
- 반대로 `C_DropHandler`는 여전히 포로 보유 여부와 보드 범위를 검증하지 않는다. 이 상태는 규칙 버그가 아니라 권한 모델 결함이다.
- `NetworkManager` 기본 IP가 `127.0.0.1`에서 사설망 주소로 바뀌어 다른 환경에서 바로 접속 실패할 가능성이 커졌다.

## 2. 코드 품질 리뷰
이번 커밋의 방향은 “클라이언트 즉시 판정 축소, 서버 승인 반영 확대”라서 올바르다. 특히 이동 패턴 판정과 `S_SyncStatus` 확장은 이전보다 책임 경계가 분명해졌다. 그러나 `PacketHandler.S_SyncStatusHandler`가 `SyncFromServer()`를 직접 호출한 뒤 `OnSyncStatus()`에서도 같은 동기화를 반복한다. 기능상 큰 비용은 아니지만 상태 갱신 위치가 두 곳으로 보이기 때문에 이후 디버깅 때 “어디서 턴이 바뀌었는가”를 추적하기 어려워진다. 한 메서드만 상태 변경을 소유하도록 정리해야 한다.

가장 위험한 부분은 서버 `C_DropHandler`다. 현재는 턴만 맞으면 임의 `pieceId`를 임의 좌표에 넣을 수 있는 구조에 가깝다. `PieceOwners[pieceId] == session.MyTeam`, 현재 보드 밖 포로 상태인지, `IsWithinBoard()`인지, 목적지가 비어 있는지를 모두 같은 검증 함수에서 확정해야 한다. 지금처럼 검증이 흩어지면 클라이언트가 잘못된 패킷 하나만 보내도 서버 상태가 오염된다.

자료구조/복잡도 측면에서는 `PlayerController.FindPieceById()`가 네트워크 패킷 처리 경로에서 `FindObjectsOfType<Piece>(true)`와 `Array.Find`를 호출한다. 현재 비용은 패킷마다 씬 전체 검색 O(n)과 배열 할당이다. 기물 수가 작아도 네트워크 동기화 경로에서는 의미 있는 나쁜 습관이다. `pieceId -> Piece` 딕셔너리를 초기화 시 구성하고, 생성·포획·드랍 때 갱신하는 방식이 더 단순하고 안전하다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷 직렬화, 룸, 세션, 턴 동기화를 직접 구현하고 있으므로 같은 컨텐츠 양의 프레임워크 프로젝트보다 네트워크 기반 작업의 난도가 높다. 이번 커밋은 서버 권위 턴 흐름과 이동 검증에서 실제 전진이 있다.

다만 제출 가능한 2인 수직 슬라이스 기준으로는 아직 high risk다. Drop 검증, partial packet 테스트, 씬 로드 중 상태 패킷 유실이 모두 시연 중 랜덤 실패로 이어질 수 있다. 현재 진행도는 54로 본다. “돌아가는 것처럼 보이는 단계”에서 “반복 시연 가능한 단계”로 넘어가려면 네트워크 경계 테스트와 서버 검증을 먼저 닫아야 한다.

## 4. 다음 권장사항
- `C_Drop`은 `ValidateDrop()`으로 묶고, 포로 보유·보드 범위·점유 칸·소유권을 서버에서 한 번에 판정해야 한다.
- `S_Login`과 `S_SyncStatus`는 `PlayerController.Instance`가 없을 때 버리지 말고 씬 준비 후 적용할 큐에 넣어야 한다.
- TCP Size 헤더 분할 수신 테스트를 작성해 패킷이 여러 번에 나뉘어 와도 같은 결과가 나오는지 고정해야 한다.
- 포로 재배치 룰은 GDD, TDD, 클라이언트 하이라이트, 서버 검증이 같은 문장을 바라보도록 정리해야 한다.
- 서버 보드 점유 조회는 4x3 점유표 또는 좌표 인덱스로 고정해 이동·드랍 검증의 기준점을 하나로 줄여야 한다.

## 5. 문서화 상태
design은 7점이다. `concept.md`와 `gdd.md`에 코어 루프, 포로, 승급, 입궁 승리 등 주요 경험과 시스템 관계가 꽤 구체적으로 적혀 있다. 다만 문서 이력 placeholder와 일부 빈 표가 남아 최종 기획 문서로 보기에는 정리가 부족하다.

technical은 6점이다. `tdd.md`는 패킷 구조, 서버 권위, 요청 검증 표를 갖추고 있고 이번 커밋에서 `C_Drop` 설명도 추가됐다. 그러나 설명과 코드가 일치하지 않는 부분이 있어 신규 개발자가 문서만 보고 구현 기준을 확정하기 어렵다.

spec은 6점이다. 기물 이동, 포획, 승급, 승리 조건은 표와 플로우로 읽을 수 있다. 하지만 포로 배치 제한과 검증 체크리스트가 아직 테스트 케이스 수준으로 닫히지 않아 QA 기준으로 바로 쓰기에는 부족하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족: `C_Drop` 문서와 서버 코드의 검증 범위가 맞지 않는다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재: 실패 응답, 재접속 스냅샷, partial packet 계약이 아직 고정되지 않았다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가: 동기화 함수는 생겼지만 상태 유실 방어는 없다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요: 현재 문서는 구현 참고용은 가능하나 제출용 정합성은 부족하다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정: 패킷 처리 경로에서 씬 전체 검색을 계속 수행한다.

# YJ — 수신버퍼 추가 및 문서 정리 스냅샷

## 1. 주요 변경사항
- `docs/concept.md` 의 닫히지 않은 코드블록이 수정됐고, 핵심 테마 문장이 실제 내용으로 일부 채워졌다.
- 클라이언트와 서버 양쪽에 동일한 `RecvBuffer` 클래스가 추가됐다. 다만 이번 diff 안에서는 실제 수신 루프에 연결된 흔적이 없다.
- `PlayerController`, `NetworkManager`, 서버 세션 코드에서 깨진 인코딩 주석과 불필요 주석이 제거됐다.
- `docs/gdd.md` 는 프로젝트명, 관련 문서, 피치, 장르·타겟 일부가 채워졌지만 작성 가이드 문구가 여전히 남아 있다.

## 2. 코드 품질 리뷰
주석 정리는 긍정적이다. 깨진 인코딩 주석과 `[cite]` 흔적은 유지보수자가 코드 의도를 오해하게 만들고, 실제 로직보다 노이즈를 키운다. 이번 커밋에서 이를 걷어낸 것은 방향이 맞다. 다만 `PlayerController` 의 핵심 로직은 주석만 제거됐을 뿐 구조 개선은 없다. 왕 포획 시 `isGameStarted = false` 로 흐름을 끊는 방식은 여전히 로컬 상태에 강하게 의존한다. 서버 종료 통지와 클라이언트 이동 처리의 책임 경계가 명확하지 않아, 지연·중복 패킷·재접속 상황에서 상태 불일치가 발생할 수 있다.

`RecvBuffer` 추가는 커스텀 TCP 구현에서 필요한 단계다. TCP 는 메시지 단위가 아니라 바이트 스트림이므로, 수신 버퍼와 패킷 길이 기반 프레이밍 없이는 패킷이 쪼개지거나 합쳐질 때 랜덤하게 실패한다. 그러나 이번 diff 기준으로 `Session` 수신 루프가 `RecvBuffer` 를 사용하도록 바뀌지 않았다. 즉 위험을 해결한 것이 아니라 해결할 재료만 추가한 상태다. 서버 쪽 `RecvBuffer` 도 namespace 없이 전역 클래스라 프로젝트가 커질수록 충돌 가능성이 생긴다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 `List.Find`, LINQ, 매 프레임 할당 같은 명확한 hot path 지적은 없다. 대신 동일한 `RecvBuffer` 파일이 클라이언트와 서버에 복붙되어 있어 유지보수 비용이 생긴다. 현재 비용은 성능보다 변경 불일치 위험이다. 공통 패킷/버퍼 라이브러리로 분리하거나 최소한 동일 소스 공유 전략을 정하는 편이 맞다.

## 3. 진행도 평가
이 프로젝트는 명백히 `custom-network` 경로다. 서버 프로세스, 세션, 룸, 패킷, 바이트 직렬화, 수신 버퍼까지 직접 구현하고 있다. 따라서 콘텐츠 양이 적은 것은 일부 보정할 수 있지만, 대신 패킷 경계·권한 모델·실패 처리의 완성도가 진행도의 핵심이다.

현재는 2인 플레이 수직 슬라이스의 골격은 보이나, 네트워크 안정성을 반복 시연 가능하다고 보기 어렵다. 특히 버퍼가 추가됐지만 연결되지 않았고, 턴 상태도 클라이언트 로컬과 서버 상태가 분리된 정황이 남아 있다. 진행도는 44%로 평가한다. 방향은 맞지만 아직 “데모 중 운 좋게 동작” 단계에 가깝다.

## 4. 다음 권장사항
- `RecvBuffer` 를 `Session`/클라이언트 수신 루프에 실제 연결하고, 길이 부족·복수 패킷 수신 테스트를 작성해야 한다.
- 턴의 주인을 서버로 고정하고, 클라이언트 `TurnMachine` 은 표시·입력 잠금 역할로 낮춰야 한다.
- `docs/tdd.md` 에 패킷 ID, payload, 권한 주체, 실패 응답, 재접속 방식을 표로 작성해야 한다.
- `docs/gdd.md` 의 작성 가이드 문장과 placeholder 를 제거하고 실제 플레이 사양으로 채워야 한다.
- 포로 재배치 금지 규칙은 십이장기 원 룰 기준으로 다시 확인하고 코드·문서에 출처를 남겨야 한다.

## 5. 문서화 상태
design 은 5점이다. `concept.md` 와 `gdd.md` 에 게임 정체성, 코어 루프, 핵심 테마가 일부 들어갔고 이번 커밋에서 실제 문장도 늘었다. 그러나 GDD 안에 작성 가이드가 그대로 남아 있어 최종 설계 문서로 보기에는 아직 부족하다.

technical 은 2점이다. 커스텀 네트워크를 직접 구현하는 프로젝트인데 권한 모델, 패킷 프레이밍, 세션 복구, 에러 처리 표가 보이지 않는다. 신규 개발자가 문서만 보고 네트워크 구조를 이해하기 어렵다.

spec 은 3점이다. 턴 제한 시간 같은 일부 수치는 정리됐지만, 입력·상태·결과 매핑과 테스트 체크리스트가 부족하다. 보드게임은 규칙의 모호함이 곧 버그가 되므로 룰 사양을 더 엄격히 고정해야 한다.

## 6. Backlog
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재는 아직 남아 있다. 코드보다 먼저 계약이 없으면 디버깅 비용이 급격히 커진다.
- 클라이언트 로컬 `TurnMachine` 과 서버 상태 분리 문제도 해결되지 않았다. 재접속·지연·중복 수신 상황에서 가장 먼저 깨질 영역이다.
- `docs/gdd.md`, `docs/tdd.md` 의 placeholder 문제는 일부 완화됐지만 완료는 아니다. 작성 가이드 문장이 남아 있다.
- 포로 재배치 금지 규칙은 코드 변경 없이 주석만 정리됐다. 룰 출처 재확인이 계속 필요하다.

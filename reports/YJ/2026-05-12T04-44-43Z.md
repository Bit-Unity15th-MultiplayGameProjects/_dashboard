# YJ — 무변경 재검토 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 커밋 수가 0개이고 diff도 비어 있다. 코드·문서 양쪽에서 이번 구간의 실질 전진은 확인되지 않는다.
- 이전부터 남아 있던 서버 권위, 패킷 프레이밍, TurnMachine 동기화 문제가 그대로 핵심 리스크다.
- `docs/concept.md`, `docs/gdd.md`, `docs/tdd.md`는 분량과 구조는 갖췄으나 placeholder와 검증 공백이 여전히 보인다.
- 필수 backlog 회계 항목 중 현재 스냅샷만으로 완전 해결을 증명할 수 있는 항목은 없다.

## 2. 코드 품질 리뷰
이번 diff 범위에는 신규 코드가 없어 새로 유입된 가독성·구조 문제를 특정할 수 없다. 다만 누적 상태 기준으로 보면 위험 신호는 명확하다. 서버가 이동·드랍의 최종 판정자가 되어야 하는데, 문서와 코드의 권한 모델이 아직 같은 언어로 고정되지 않았다. 이 상태에서 기능을 더 얹으면 버그가 “클라이언트 표시 문제”인지 “서버 검증 문제”인지 분리하기 어렵다.

커스텀 TCP 스택을 직접 구현하는 선택은 학습 가치는 크지만, 수신 버퍼·패킷 경계·실패 응답이 모호하면 2인 플레이가 간헐적으로 깨지는 쇼스토퍼가 된다. 지금은 `Session`, `RecvBuffer`, `PacketHandler`, `GameRoom`의 책임 경계가 먼저 확정돼야 한다. 요청 패킷을 곧바로 결과처럼 다루는 흐름은 불법 수, 재전송, 중복 적용을 막지 못한다.

자료구조/복잡도 관점에서는 이번 diff 범위의 신규 근거는 없다. 단, 누적 backlog의 `FindObjectsOfType` 기반 `pieceId` 조회는 현재 참조 파일이 사라져 완전 해결을 증명하지 못하므로 회계상 유지한다. 실제 경로가 바뀌었다면 새 `PlayerController` 위치에서 `pieceId -> Piece` registry를 확인하고, 없으면 턴 처리 hot path의 선형 탐색을 제거해야 한다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷 정의, 바이트 직렬화, 세션·룸 관리를 직접 들고 가므로 같은 분량의 콘텐츠를 프레임워크 프로젝트처럼 요구할 수는 없다. 그 대신 패킷 계약, 프레이밍, 권한 모델, 실패 처리의 완성도가 진행도의 핵심 기준이다.

현재는 보드게임 규칙과 문서 골격은 어느 정도 잡혔지만, 반복 가능한 2인 happy path를 보장할 증거가 부족하다. 특히 TCP partial packet, 서버 검증 결과 반영, 로컬 TurnMachine 분리 문제가 남아 있어 제출 가능한 수직 슬라이스는 약 40% 수준으로 본다. 위험도는 높다.

## 4. 다음 권장사항
- TDD에 권한 모델·상태 소유권·패킷 실패 응답 표를 동기화한다.
- GameRoom 이동·드랍 처리에 서버 검증 결과만 반영하도록 고정한다.
- TCP 수신 버퍼와 패킷 프레이밍 경로를 누적 파서로 확정한다.
- TurnMachine을 서버 턴 상태 수신 기준으로 재정렬한다.
- 포로 Drop 룰 출처와 2인 happy path 검증표를 GDD/TDD에 고정한다.

## 5. 문서화 상태
design은 7점이다. `concept.md`와 `gdd.md`에 코어 루프, 기물 규칙, 포로·승급·승리 조건이 표와 플로우로 정리돼 있어 기획 의도는 읽힌다. 다만 `YYYY.MM.DD`, 빈 작성자·승인자, 일부 가이드 문장이 남아 완성 문서로 보기엔 부족하다.

technical은 5점이다. `tdd.md`가 존재하고 네트워크·Unity 작업 기준을 다루는 흔적은 있으나, 서버 권위 상태 소유자, 패킷 실패 응답, 재접속·지연 처리 기준이 코드와 맞물려 고정됐다는 근거가 약하다.

spec은 6점이다. GDD의 기물별 이동, 포로 재배치, 승급, 승리 조건은 테스트 케이스로 전환 가능한 수준이다. 반면 포로 Drop 룰 출처, 2인 happy path 체크리스트, 실패 로그 기준이 부족해 회귀 검증 문서로는 아직 헐겁다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족은 계속 미해결이다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재도 유지된다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가한 위험이 남아 있다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요 항목은 아직 닫을 수 없다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정했는지 확인되지 않았다.
- 포로 재배치 시 '상대 진영 첫 줄' 금지 규칙은 일본 쇼기와 다름 — 룰 출처 재확인 필요 항목은 문서 기준으로도 남아 있다.

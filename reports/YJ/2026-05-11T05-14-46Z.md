# YJ — 네트워크 문서 보강 스냅샷

## 1. 주요 변경사항
- `LobbyUI`, `PlayerController`, `TurnMachine`이 `Assets/Scripts/Game/` 하위로 이동했다. Unity `.meta`는 대부분 유지되어 참조 손상 위험은 낮아 보인다.
- 서버 `GameRoom.cs`에서 `session.Send(loginPkt.Write().Array)`를 `session.Send(loginPkt.Write())`로 바꿨다. 패킷 전송 타입 불일치 문제를 줄인 변경으로 보인다.
- 클라이언트와 서버의 `RecvBuffer.cs`가 삭제됐다. TCP 스트림 기반 프로젝트에서는 매우 큰 구조 변경이므로 수신 조립 경로 재확인이 필요하다.
- `docs/tdd.md`가 성능, 보안, 개발 환경, 리스크, 프로토타입 범위까지 크게 보강됐다. 반면 `docs/gdd.md`는 가이드 문구 제거 중심이며 빈 섹션이 여전히 많다.

## 2. 코드 품질 리뷰
이번 코드 변경은 구조 이동과 소폭 수정이 중심이다. `Assets/Scripts/Game/`로 게임 로직을 모은 방향은 좋다. 루트 `Assets/Scripts/`에 UI, 턴, 컨트롤러가 흩어져 있던 상태보다 탐색성이 좋아지고, 이후 asmdef나 네임스페이스 분리로 확장하기 쉽다. 다만 폴더 이동만으로 아키텍처가 정리된 것은 아니다. `LobbyUI`의 깨진 문자열처럼 인코딩 문제가 그대로 남아 있으면 사용자-facing UI 품질과 협업 안정성을 동시에 해친다.

가장 위험한 부분은 `RecvBuffer` 삭제다. TDD는 TCP 패킷 뭉침/끊김 방지와 버퍼 재사용을 중요 항목으로 적고 있는데, 실제 diff에서는 그 기반 클래스가 사라졌다. TCP는 메시지 단위가 아니라 바이트 스트림이므로, 수신 루프가 헤더 크기만큼 누적하고 전체 패킷 길이를 만족할 때만 처리하는 구조가 반드시 필요하다. 이 경로가 다른 코드로 대체되지 않았다면 2인 플레이가 네트워크 상황에 따라 랜덤하게 깨진다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위만으로 `List.Find`, LINQ, 매 프레임 할당 같은 명확한 hot path 문제는 확인되지 않는다. 오히려 현재 확인되는 할당 리스크는 최적화 문제가 아니라 설계 일관성 문제다. 문서의 “버퍼 재사용” 목표와 코드의 버퍼 삭제가 충돌하므로, 성능 이전에 패킷 프레이밍의 정확성을 먼저 복구해야 한다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷, 세션, 룸, TCP 송수신을 직접 다루므로 같은 보드게임이라도 Mirror나 NGO 기반 프로젝트보다 네트워크 기반 비용이 크다. 그 점을 감안하면 문서와 서버 뼈대가 있는 것은 진척으로 인정된다.

다만 수직 슬라이스 기준으로는 아직 46% 수준이다. 2인 happy path가 반복 시연 가능한지 판단하려면 패킷 조립, 서버 권위 검증, 턴 상태 동기화가 코드로 확인돼야 한다. 현재 커밋은 문서 진전이 크지만, 핵심 런타임 위험을 제거했다기보다 위험 목록을 더 잘 적은 단계에 가깝다.

## 4. 다음 권장사항
- 삭제된 `RecvBuffer`를 대체하는 수신 조립 루프를 확인하고, 없으면 즉시 복구한다. TCP 프레이밍은 선택 사항이 아니다.
- TDD의 서버 권위 모델을 `GameRoom` 처리 흐름에 연결한다. 이동·드랍 패킷은 서버 검증 후 승인 패킷으로만 확정돼야 한다.
- `TurnMachine`은 로컬 턴 증가가 아니라 서버 승인 상태를 반영하도록 정리한다. 지연과 재접속 대응의 출발점이다.
- GDD의 빈 내러티브, UI/UX, 사운드, 밸런스 섹션은 “해당 없음”인지 “추후 작성”인지 명확히 닫아야 한다.
- 포로 재배치 금지 규칙은 룰 출처와 테스트 케이스로 고정한다. 애매한 룰은 멀티플레이 디싱크의 원인이 된다.

## 5. 문서화 상태
design 점수는 5점이다. 핵심 경험, 코어 루프, 보드 구조는 작성돼 있으나 UI/UX, 사운드, 밸런스, 내러티브가 비어 있어 GDD만 보고 구현 기준을 잡기에는 부족하다. placeholder 제거도 아직 끝나지 않았다.

technical 점수는 6점이다. TDD는 이번 커밋에서 가장 많이 전진했다. 성능 목표, 보안, 개발 환경, 리스크가 추가되어 신규 참여자가 방향을 이해하는 데 도움이 된다. 그러나 “서버 권위”, “AES-256”, “GameRuleValidator” 같은 항목이 현재 코드와 어느 정도 연결됐는지 불명확하다.

spec 점수는 4점이다. 패킷 헤더 구조와 프로토타입 범위는 있으나 패킷별 필드, 실패 응답, 상태 전이, 테스트 체크리스트가 부족하다. `_sample/docs` 수준의 사양서처럼 바로 QA 항목으로 쓰기에는 아직 추상적이다.

## 6. Backlog
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재는 아직 남아 있다. TDD가 보강됐지만 패킷별 주인, 검증 주체, 실패 응답이 표로 닫히지 않았다.
- 클라이언트 로컬 `TurnMachine`과 서버 상태 분리 문제는 파일 이동 외 해결 근거가 없다. 승인 기반 턴 동기화가 필요하다.
- GDD/TDD placeholder 문제는 부분 개선이다. 특히 GDD 하단의 프로젝트명, 작성일, 빈 섹션은 다음 리뷰에서도 검출될 가능성이 높다.
- 포로 재배치의 “상대 진영 첫 줄” 금지 규칙은 코드 변경이 없으므로 유지된다. 룰 출처와 테스트 케이스를 함께 고정해야 한다.

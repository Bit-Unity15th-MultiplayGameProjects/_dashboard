# YJ — 문서 정리 스냅샷

## 1. 주요 변경사항
- `docs/gdd.md`와 `docs/tdd.md`에서 대량으로 붙어 있던 범용 템플릿 placeholder 블록이 삭제되었다. 이 점은 명확한 전진이다.
- 여러 Markdown 표가 헤더와 구분선을 갖도록 정리되었다. 문서 가독성은 이전보다 좋아졌다.
- 반대로 실제 설계 내용 추가는 적다. 삭제와 표 정리가 대부분이며, 네트워크·턴 동기화의 핵심 불확실성은 코드와 문서 양쪽에 남아 있다.
- 일부 표는 여전히 깨져 있다. 특히 `docs/tdd.md`의 `UI/UX` 행은 선행 `|`가 빠져 있고, 셀 내부 `|` 문자도 이스케이프되지 않았다.

## 2. 코드 품질 리뷰
이번 diff는 코드 변경 없이 문서만 수정한다. 따라서 C# 구조, Unity 생명주기, `GetComponent` 캐싱, `Instantiate/Destroy`, 싱글톤 의존 같은 코드 품질 항목은 새로 악화되거나 개선됐다고 판단할 근거가 없다. 다만 문서 변경의 품질만 보면 “표 형태로 보이게 만든 것”과 “검증 가능한 사양으로 만든 것”은 다르다. 현재 TDD의 패킷 구조표는 헤더 크기 정도만 있고, 패킷별 필드, 방향, 실패 응답, 서버 검증 책임이 아직 부족하다.

자료구조/복잡도/할당 관점에서는 이번 diff 범위에서 명확한 지적 없음이다. 코드 hot path, 컬렉션 조회, LINQ 할당, `Update` 내부 탐색 패턴이 변경되지 않았기 때문이다. 근거 없이 `Dictionary`나 풀링을 요구하는 것은 부적절하다. 대신 다음 리뷰에서는 `TurnMachine`, `PacketHandler`, 서버 룸/세션 관리 쪽에서 턴 대상 조회와 패킷 라우팅이 선형 탐색으로 반복되는지 확인해야 한다.

문서 자체의 품질 문제는 남아 있다. `docs/gdd.md`의 `10-3`, `10-4`는 제목만 있고 내용이 없다. `docs/tdd.md`는 보안 전략을 문장으로는 적었지만 실제 프로토콜 계약과 실패 시퀀스가 없다. 커스텀 네트워크 경로에서는 이 공백이 단순 문서 미비가 아니라 디버깅 비용 폭증으로 이어진다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로로 보는 것이 맞다. TCP 소켓, 수동 패킷 직렬화, 서버 세션, 브로드캐스팅을 직접 다루는 구조이므로 컨텐츠 양보다 패킷 계약, 권한 모델, 실패 처리, 2인 end-to-end 재현성이 더 중요하다.

이번 변경은 문서 위생 개선으로는 의미가 있지만, 제출 가능한 수직 슬라이스를 직접 끌어올린 변경은 아니다. placeholder 제거는 인정하되, 서버 권위 턴 진행과 클라이언트 표시 상태의 경계가 아직 확인되지 않으므로 진행도는 42로 유지한다. risk는 high가 타당하다.

## 4. 다음 권장사항
- TDD에 서버가 소유하는 상태와 클라이언트가 표시만 하는 상태를 표로 분리해야 한다.
- `TurnMachine`이 로컬 기준으로 턴을 진행한다면 서버 상태 수신 기준으로 재정렬해야 한다.
- 포로 Drop 규칙은 일본 쇼기 원전과 프로젝트 변형을 구분해 GDD와 코드 주석에 남겨야 한다.
- GDD/TDD의 깨진 표, 빈 검증 계획, 날짜·버전 메타 정보를 한 번에 정리해야 한다.
- 2인 접속부터 이동, 포획, Drop, 게임 종료까지 happy path 체크리스트를 만들어야 한다.

## 5. 문서화 상태
design은 5점이다. 십이장기 콘셉트, 기물, 보드, 승리 조건은 어느 정도 보이지만 코어 루프의 의도와 플레이 경험 목표가 아직 얕다. 빈 성장·검증 섹션도 감점 요인이다.

technical은 5점이다. 기술 스택, 패킷 헤더, JobQueue, 보안 방향은 적혀 있다. 그러나 커스텀 네트워크에서 가장 중요한 권한 모델, 패킷별 계약, 재접속·지연·실패 응답 시퀀스가 부족해 신규 개발자가 구현 기준으로 쓰기 어렵다.

spec은 4점이다. 기물과 보드 일부 수치는 있으나 룰 예외, Drop 가능 칸, 승급, 입궁, 게임 종료 조건이 테스트 케이스 수준으로 정리되지 않았다. `_sample/docs` 수준의 사양서처럼 자동·수동 QA 체크리스트로 바로 쓰기는 어렵다.

## 6. Backlog

- 클라이언트 로컬 `TurnMachine`과 서버 상태 분리 문제도 미해결이다. 지연, 재접속, 패킷 재전송 상황에서 바로 깨질 수 있다.
- 포로 재배치 금지 규칙은 원전과 다를 가능성이 있으므로 의도한 변형인지 확인해야 한다.
- GDD/TDD의 큰 placeholder 덩어리는 제거됐지만, 빈 섹션과 깨진 표는 다음 정리 대상으로 남긴다.

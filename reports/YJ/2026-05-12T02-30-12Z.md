# YJ — 변경 없음 재점검 스냅샷

## 1. 주요 변경사항
- 리뷰 대상 범위의 커밋 수가 0개라서 코드·문서 diff 기준의 신규 구현 진척은 없다.
- 현재 문서는 `docs/concept.md`, `docs/gdd.md`, `docs/tdd.md` 세 축이 존재하며, 룰·시스템 설명은 이전보다 형식은 갖춘 상태다.
- 다만 이전 TODO와 Backlog의 핵심인 서버 권한 모델, TurnMachine 동기화, TCP 프레이밍은 해결 근거가 없다.
- 누적 ledger에서 사라졌던 RecvBuffer, 서버 검증, pieceId 조회 이슈도 해결 기록이 없어 재등장 대상으로 본다.

## 2. 코드 품질 리뷰
이번 diff는 비어 있으므로 새 코드의 가독성·구조 저하는 평가할 수 없다. 그러나 변경 없음은 중립이 아니다. 직전 리포트에서 high로 분류된 네트워크 권한 모델, 서버 검증 연결, TurnMachine 분리 문제가 그대로 남아 있다면 디버깅 비용은 계속 증가한다. 특히 커스텀 TCP 서버를 선택한 프로젝트에서 “어느 상태가 서버 소유이고 어느 상태가 클라이언트 표시용인지”가 문서와 코드 양쪽에 고정되지 않으면, 이동·드랍·턴 종료가 모두 재현 어려운 동기화 버그로 이어진다.

자료구조/복잡도 관점에서는 이번 diff 범위에 신규 근거는 없지만, 누적 항목의 `FindObjectsOfType` 기반 pieceId 조회는 해결 근거가 없다. 이 패턴이 턴 처리나 네트워크 반영 경로에 남아 있으면 보드 크기가 작아도 유지보수성이 나쁘고, 오브젝트 비활성화·씬 재로드 때 조회 누락이 생긴다. pieceId→Piece registry로 고정하면 비용보다도 상태 추적 기준이 명확해지는 점이 이 프로젝트에 더 중요하다.

Unity 관점에서는 `PlayerController`가 입력, 룰, 네트워크 송신, 결과 UI를 함께 들고 있는 누적 징후가 계속 보인다. 이 구조는 작은 프로젝트에서는 빠르게 보이지만, 멀티플레이 검증이 붙는 순간 실패 원인을 분리하기 어렵다. 최소한 룰 판정, 네트워크 요청, 시각 반영을 별도 경계로 나누어야 한다.

## 3. 진행도 평가
이 프로젝트는 `custom-network` 경로다. 서버 프로세스, 패킷, 세션, 룸, 수신 버퍼를 직접 다루므로 컨텐츠 양은 일부 보정할 수 있다. 현재 문서와 구조상 수직 슬라이스의 뼈대는 있다고 보지만, 2인 happy path와 실패 로그 기준이 명확하지 않아 반복 시연 안정성은 아직 낮다.

진행도는 47로 판단한다. 어려운 구현 경로를 택한 점은 인정하지만, 권한 모델·프레이밍·턴 상태 중 하나만 흔들려도 실제 플레이가 랜덤하게 깨진다. risk는 high가 맞다.

## 4. 다음 권장사항
- `docs/tdd.md`에 서버 소유 상태와 클라이언트 표시 상태를 표로 고정한다.
- `GameRoom` 이동·드랍 처리를 서버 검증 결과만 브로드캐스트하는 흐름으로 묶는다.
- `Session`/`RecvBuffer`의 TCP partial packet 처리 기준을 누적 파서로 확정한다.
- `TurnMachine`은 로컬 선전환이 아니라 서버 턴 패킷 수신 기준으로만 갱신한다.
- 포로 Drop 금지 구역은 원전 룰, 프로젝트 변형, 테스트 케이스를 함께 문서화한다.

## 5. 문서화 상태
design은 7점이다. `concept.md`와 `gdd.md`에 코어 루프, 기물 시스템, 포로, 승급, 승리 조건이 비교적 구체적으로 정리돼 있다. 다만 `YYYY.MM.DD`, 승인자, KPI, 빈 표가 남아 있어 제출 문서로는 정리가 더 필요하다.

technical은 5점이다. `tdd.md`가 존재하고 분량도 있지만, 권한 모델·패킷 계약·실패 응답·상태 소유자 표가 코드와 맞물려 검증되는 수준이라는 근거가 부족하다. 커스텀 네트워크 프로젝트에서는 이 부분이 문서 점수의 핵심이다.

spec은 6점이다. 기물 이동, 포로, 승급, 승리 조건의 규칙 표와 플로우는 테스트 케이스의 출발점으로 쓸 수 있다. 그러나 포로 Drop 룰 출처, 2인 재현 체크리스트, 실패 로그 기준이 비어 있어 QA 문서로는 아직 불완전하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족은 계속 high 이슈다.
- 클라이언트 로컬 TurnMachine 분리는 재접속·지연 상황에서 직접적인 위험이다.
- 커스텀 네트워크 스택의 프로토콜 사양 부재는 사라진 과거 항목이 아니라 미해결 부채다.
- 포로 재배치 금지 규칙은 “십이장기 공식 룰” 표기만으로는 출처 검증이 끝나지 않는다.
- GDD/TDD의 빈 검증 계획과 메타 정보는 제출 품질을 낮추므로 계속 이월한다.
- pieceId 조회 registry 전환은 성능보다 상태 식별 안정성을 위해 남겨야 한다.

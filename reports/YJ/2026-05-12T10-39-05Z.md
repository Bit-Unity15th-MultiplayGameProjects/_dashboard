# YJ — GDD/TDD 문서 갱신 스냅샷

## 1. 주요 변경사항
- `docs/gdd.md`에 십이장기 규칙 출처와 일본 쇼기와의 차이를 추가했다. 출처를 명문화한 점은 좋다.
- `docs/tdd.md`가 v1.1, v1.2로 갱신되며 서버 권위 이동 검증, `PieceOwners`, `IsTrying` 구조를 설명하기 시작했다.
- 기존 네트워크 패킷 통신 설명 일부가 서버 검증 중심 설명으로 교체됐다. 방향은 맞지만 실제 코드 검증 근거가 따라오지 않는다.
- 이번 커밋은 문서 변경만 포함한다. 이전 핵심 TODO였던 드랍 검증, 이동 패턴, partial packet 테스트는 코드상 해결됐다고 볼 근거가 없다.

## 2. 코드 품질 리뷰
이번 diff는 코드가 아니라 문서 변경이다. 따라서 함수 구조나 예외 처리 자체의 신규 결함은 없다. 다만 TDD가 “서버 권위 기반 PieceOwners 및 IsTrying 독립 판정 시스템 명세 동기화 완료”라고 선언한 것은 위험하다. 실제 서버 구현과 테스트가 같이 움직이지 않으면 문서가 상태를 설명하는 것이 아니라 희망 상태를 주장하게 된다. 특히 `C_Drop`과 `CheckMovePattern`은 이전 리포트에서 쇼스토퍼로 분류된 영역이므로, 문서 문장보다 실패 케이스를 재현하는 서버 테스트가 먼저 필요하다.

자료구조/복잡도 관점에서는 현재 스냅샷 기준으로 `Assets/Scripts/Game/PlayerController.cs`의 `FindPieceById`가 `FindObjectsOfType<Piece>(true)` 선형 탐색에 남아 있다. 작은 보드라도 네트워크 동기화마다 씬 전체 검색을 수행하는 구조는 비용보다 결정성 문제가 더 크다. `pieceId -> Piece` registry를 보드 상태 갱신 시 함께 유지하면 조회 비용이 O(1)이 되고, 서버 패킷 반영 순서도 추적하기 쉬워진다. `GameRoom`의 좌표 조회도 4x3 점유표로 고정하는 편이 검증 중복을 줄인다.

## 3. 진행도 평가
이 프로젝트는 명백한 `custom-network` 경로다. 서버 프로세스, 패킷, 세션, 수신 버퍼, 룸 상태를 직접 다루므로 문서화 진척은 의미가 있지만, 제출 가능한 수직 슬라이스에서는 네트워크 안정성과 권한 검증이 더 큰 비중을 가진다.

진행도는 48으로 본다. GDD/TDD가 이전보다 구체화된 점은 전진이지만, 2인 happy path를 안정적으로 보장할 핵심 구현과 테스트가 남아 있다. 특히 partial packet, 실패 응답, 씬 전환 중 상태 유실은 시연 중 랜덤 장애로 드러날 가능성이 높아 `risk_level`은 여전히 high다.

## 4. 다음 권장사항
- `C_Drop`은 포로 보유 여부, 좌표 범위, 점유 칸, 턴 권한을 서버에서 한 번에 검증해야 한다.
- `CheckMovePattern`은 문서의 왕·장·상·자·후 이동 벡터를 코드 테이블로 고정해야 한다.
- `S_Login`과 `S_SyncStatus`는 씬 로드 완료 후 큐에서 적용해 첫 동기화 유실을 막아야 한다.
- TCP `Size` 헤더 기준 partial packet 테스트를 먼저 만들고 수신 루프를 그 테스트에 맞춰 고쳐야 한다.
- 포로 재배치 규칙은 GDD의 출처 설명, 기존 본문 표, 클라이언트 금지 로그, 서버 검증을 하나로 맞춰야 한다.
- `GameRoom` 좌표 조회는 4x3 점유표나 좌표 인덱스로 분리해 검증 흐름을 단순화해야 한다.

## 5. 문서화 상태
design은 6점이다. `docs/gdd.md`는 코어 루프, 이동·포로·승급·승리 조건을 갖추고 있으나 KPI, UI 비주얼, 일부 표에 placeholder와 빈 행이 남아 있다.

technical은 5점이다. `docs/tdd.md`가 서버 권위와 패킷 검증 표를 추가한 점은 좋지만, 패킷 프레이밍, 실패 응답, 재접속 스냅샷, 테스트 기준이 아직 구현 확인 가능한 수준은 아니다.

spec은 5점이다. 이동 규칙과 포로 시스템 표는 사양서로 활용 가능해졌지만, 포로 재배치 금지 여부가 문서 내부와 코드 사이에서 충돌한다. QA 체크리스트로 쓰려면 입력, 상태, 실패 응답, 기대 로그가 더 구체적이어야 한다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족: TDD 선언이 실제 `PacketHandler`와 `GameRoom` 검증으로 확인되지 않는다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재: 프레이밍, 실패 응답, 재접속 계약이 아직 테스트 가능한 형태가 아니다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가: 씬 전환과 상태 재수신 기준이 약하다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요: 문서가 좋아졌지만 QA 기준으로는 아직 빈칸이 많다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정: 동기화 반영 경로에서 씬 검색을 제거해야 한다.

## 7. 이전 Backlog 해결
- 포로 재배치 규칙 출처 재확인은 `docs/gdd.md`의 규칙 출처 섹션 추가로 해결로 판정한다. 다만 본문 표와 클라이언트 코드가 아직 금지 규칙을 말하므로, 남은 문제는 “규칙 정합성 통일” TODO로 재분리했다.

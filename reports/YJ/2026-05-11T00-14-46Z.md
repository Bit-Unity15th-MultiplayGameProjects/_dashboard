# YJ — 로비 네트워킹 스냅샷

## 1. 주요 변경사항
- `LobbyScene`에 방 생성, 자동 매칭, 새로고침, 방 목록 스크롤 UI가 추가되었다. `RoomItem.prefab`도 새로 만들어져 로비 화면의 실제 상호작용 단위가 생겼다.
- 클라이언트와 서버 양쪽 `Packet.cs`에 `C_RoomRequest`, `C_RefreshRoomList`, `S_RoomList`가 추가되었다. 로비 기능을 네트워크 프로토콜로 확장한 점은 명확한 전진이다.
- `S_Login`에 `currentTurn`이 포함되어 입장 시 서버 턴 상태를 클라이언트에 전달하기 시작했다. 이전보다 서버 상태 동기화 의식이 좋아졌다.
- 서버 `PacketHandler`가 방 생성·입장·자동매칭을 처리하도록 확장되었다. 다만 실패 응답과 검증 경로가 아직 얕다.

## 2. 코드 품질 리뷰
가장 큰 위험은 `NetworkManager.OnReceiveCompleted`의 수신 모델이다. 현재 코드는 `BeginReceive` 한 번의 결과를 곧바로 하나의 완전한 패킷으로 해석한다. TCP는 메시지 경계가 없으므로 한 패킷이 쪼개지거나 여러 패킷이 합쳐질 수 있다. 이 구조는 로컬 테스트에서는 통과해도 실제 지연 환경에서 랜덤하게 깨진다. 수신 누적 버퍼를 두고 `size`만큼 완성된 패킷을 반복 파싱하는 방식으로 바꿔야 한다.

서버 권한 검증은 일부 개선됐지만 일관성이 부족하다. `HandleMove`는 턴 검사를 하지만, 드랍 처리에서는 좌표 범위 검사가 제거되었고 기물 소유권·점유 칸·합법 드랍 규칙이 충분히 보이지 않는다. 왜 문제냐면 클라이언트가 임의 좌표와 기물 ID를 보내도 서버가 규칙 엔진 역할을 끝까지 보장하지 못하기 때문이다. 이동·드랍 모두 서버의 단일 룰 검증 함수로 통합하는 편이 좋다.

자료구조/복잡도 관점에서는 `RoomManager.Instance.GetRoomList().Find(...)`, `FindRoom(targetRoomId)`, 로비 목록 생성이 선형 탐색 기반으로 남아 있다. 지금 규모에서는 병목이 아니지만, 방 요청마다 O(n) 조회가 반복되는 구조다. `roomId -> GameRoom`은 동적 key 조회이므로 `Dictionary<int, GameRoom>`로 관리하고, 자동매칭 후보만 별도 큐나 리스트로 분리하면 코드 의도와 비용이 모두 안정된다.

## 3. 진행도 평가
이번 커밋은 “로비 생성 서로 상호작용 완료”라는 메시지에 맞게 UI와 서버 방 처리까지 묶어낸 점에서 분명한 전진이다. 이전의 단일 게임 진입 흐름에서 벗어나 매칭 전 단계가 생겼고, 빌드 설정에도 로비 씬이 반영된 것으로 보인다.

다만 완성도는 아직 낮게 봐야 한다. 네트워크 프레이밍, 실패 응답, 재접속, 룰 검증, 문서화가 동시에 비어 있다. 이 상태로 기능을 더 얹으면 디버깅 가능한 경계를 잃기 쉽다. 현재 진행도는 약 34%로, 핵심 루프 일부와 로비 골격은 있으나 안정적인 2인 플레이 빌드 전 단계다.

## 4. 다음 권장사항
- TCP 수신부를 누적 버퍼 기반 패킷 파서로 재작성해야 한다. 이 작업은 모든 네트워크 기능의 기반이므로 최우선이다.
- 방 입장 실패, 정원 초과, 존재하지 않는 방, 서버 연결 실패를 명시 응답 패킷과 UI 상태로 처리해야 한다.
- 이동과 드랍 검증을 서버 룰 엔진으로 통합해야 한다. 클라이언트 입력은 요청일 뿐, 판정 주체는 서버여야 한다.
- `NetworkManager`의 싱글톤 상태, 씬 전환, `MyTeam/CurrentTurn` 초기화를 세션 상태머신으로 정리해야 한다.
- `RoomManager`는 `Dictionary<int, GameRoom>` 기반 조회로 바꾸고 자동매칭 후보 관리 책임을 분리해야 한다.
- `docs/tdd.md`의 프로토콜 표와 권한 모델을 실제 `Packet.cs` / 서버 검증 코드와 동기화해야 한다.

## 5. 문서화 상태
design 점수는 7점이다. `docs/concept.md`와 `docs/gdd.md`가 십이장기 콘셉트, 타겟, 핵심 경험, 코어 루프, 시스템 구성을 꽤 상세히 담고 있다. 다만 이번 로비·매칭 흐름이 플레이어 경험과 어떻게 연결되는지는 아직 별도 갱신이 필요하다.

technical 점수는 6점이다. `docs/tdd.md`가 클라이언트/서버 구조와 네트워크 설계를 설명하는 기반은 갖췄다. 다만 이번에 추가된 로비 패킷, 실패 응답, TCP 프레이밍, 서버 검증 규칙이 실제 코드와 같은 수준으로 동기화되지는 않았다.

spec 점수는 5점이다. GDD/TDD는 게임 규칙과 시스템 단위를 어느 정도 설명하지만, 방 생성·자동매칭·새로고침·입장 실패 같은 네트워크 상태와 결과 매핑은 테스트 체크리스트로 바로 쓰기엔 부족하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화가 부족하다. 패킷은 늘었지만 “누가 어떤 상태를 소유하고 서버가 어디까지 판정하는가”가 코드와 문서에서 함께 고정돼야 한다.
- 클라이언트 로컬 `TurnMachine`과 서버 턴 상태 분리는 여전히 위험하다. `S_Login.currentTurn` 추가는 좋은 시작이지만 재접속·지연 보정 설계는 아니다.
- 포로 재배치 규칙 출처와 `docs/concept.md` 코드블록 문제는 이번 diff 범위에서 해결 흔적이 없다.

## 7. 이전 Backlog 해결
- 씬 분리 문제는 일부 해결로 본다. `LobbyScene.unity`에 로비 UI와 `NetworkManager`가 배치되었고, `EditorBuildSettings.asset` 변경이 포함되어 매칭/대기실 진입 구조가 단일 SampleScene 상태에서 벗어났다.

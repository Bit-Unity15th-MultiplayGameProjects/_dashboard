# YJ — 게임 종료·드랍 검증 스냅샷

## 1. 주요 변경사항
- 서버에 `CapturedPieces`가 추가되어 포로 소유 여부를 기준으로 `C_Drop`을 거부할 수 있게 됐다. 이전보다 서버 권위 모델에 한 발 가까워진 점은 분명한 개선이다.
- 연결 끊김 시 상대 승리 처리와 빈 방 정리가 들어갔다. 방이 계속 남는 문제를 정리하려는 방향은 좋다.
- 결과 패널과 로비 복귀 버튼이 추가되어 게임 종료 후 사용자 흐름이 생겼다.
- 로비 방 목록 갱신에서 자식 삭제 루프와 버튼 리스너 초기화가 개선됐다.
- 반대로 씬에 서버 IP `192.168.0.18`이 직렬화되어 다른 실행 환경에서 접속 실패가 날 가능성이 커졌다.

## 2. 코드 품질 리뷰
이번 변경은 “서버에서 드랍을 검증한다”는 중요한 방향을 잡았지만, 검증이 아직 완결되지 않았다. `PacketHandler.HandleDrop`은 포로 보유 여부와 점유 칸은 본다. 그러나 `IsWithinBoard` 검사가 없어 음수나 보드 밖 좌표가 `PiecePositions`에 들어갈 수 있고, 문서의 “상대 진영 배치 금지”도 `Ja`에만 적용된다. 이 상태는 서버 권위처럼 보이지만 실제로는 일부 불법 상태를 서버가 확정하는 구조다.

게임 종료 처리도 위험하다. `PlayerController.OnServerGameOver`에서 기존 `isGameStarted = false`와 타이머 중지가 빠졌기 때문에 결과 UI가 떠도 입력·타이머·패킷 송신 경로가 살아 있을 수 있다. 종료는 UI 표시가 아니라 상태 전이로 처리해야 한다. `S_GameOver`에 종료 사유를 싣지 않고 클라이언트가 “양쪽 왕이 살아 있으면 기권승”으로 추론하는 방식도 취약하다. 동기화 지연이나 오브젝트 비활성 상태가 섞이면 결과 메시지가 틀릴 수 있다.

자료구조/복잡도 관점에서는 두 지점이 확인된다. 서버는 `GetPieceAt`으로 좌표 조회 때마다 `PiecePositions`를 순회한다. 보드는 4x3이라 비용 자체는 작지만, 검증의 기준 자료구조가 흩어져 좌표 무결성을 보장하기 어렵다. `occupant[4,3]` 또는 좌표 인덱스를 두면 범위·점유·포획 판정을 같은 경로로 묶을 수 있다. 클라이언트는 `FindPieceById`가 패킷마다 `FindObjectsOfType<Piece>(true)`를 호출한다. 이는 오브젝트 검색과 배열 할당을 반복하므로 `pieceId -> Piece` registry로 고정해야 한다.

## 3. 진행도 평가
이 프로젝트는 명백한 `custom-network` 경로다. 서버 프로세스, 룸, 세션, 패킷 직렬화, 보드 검증을 직접 들고 가고 있으므로 컨텐츠 양만으로 평가하면 안 된다. 이번 범위에서 포로 소유 검증, 방 정리, 종료 UI가 추가되어 수직 슬라이스는 전진했다.

다만 제출 가능한 반복 시연 기준으로는 아직 위험이 높다. 종료 후 상태 정지, 드랍 범위 검증, TCP partial packet 테스트, 씬 전환 중 상태 큐가 남아 있다. 현재 진행도는 58%로 본다. 낮은 수준의 2인 플레이 루프는 보이지만, 커스텀 네트워크의 실패 경계가 테스트로 고정되지 않아 시연 안정성은 아직 부족하다.

## 4. 다음 권장사항
- 먼저 `OnServerGameOver`에서 입력, 타이머, 추가 송신을 닫아라. 결과 UI는 그 다음 문제다.
- `GameRoom`의 이동·드랍·퇴장 상태 변경을 같은 lock 범위나 룸 큐에서 처리하라. 지금처럼 일부만 잠그면 더 위험하다.
- `C_Drop` 검증을 보드 범위, 점유, 포로 소유, 전체 기물 상대 진영 금지까지 하나의 서버 함수로 묶어라.
- `S_GameOver`에 `reason`을 추가해 왕 포획, 입궁, 타임아웃, 연결 끊김을 명시하라.
- TCP Size 헤더 분할 수신 테스트와 씬 로드 후 상태 큐를 다음 리포트 전까지 반드시 고정하라.

## 5. 문서화 상태
design 점수는 7점이다. `docs/concept.md`와 `docs/gdd.md`에 십이장기 컨셉, 코어 루프, 기물·포로·승리 시스템이 비교적 충실히 정리되어 있다. 다만 placeholder와 빈 표가 남아 문서 완성도를 깎는다.

technical 점수는 6점이다. `docs/tdd.md`에 소켓 기반 구조, 패킷 포맷, 서버 권위 방향이 있다. 그러나 실제 코드의 드랍 검증, 실패 응답, 재접속 스냅샷, 수신 경계 테스트와 맞물려 있지 않다.

spec 점수는 6점이다. 이동·포로·승리 조건의 테이블은 테스트 케이스로 전환 가능한 수준에 가깝다. 하지만 포로 배치 금지 룰이 서버 코드와 다르고, 검증 계획이 아직 QA 체크리스트로 쓰기엔 부족하다.

## 6. Backlog
- 문서화된 권한 모델과 실제 서버 검증 코드의 동기화 부족: 드랍 범위와 상대 진영 룰이 문서와 코드에서 갈린다.
- 커스텀 네트워크 스택의 권한 모델·프로토콜 사양 부재: partial packet, 실패 응답, 재접속 계약이 아직 고정되지 않았다.
- 클라이언트 로컬 TurnMachine이 서버 상태와 분리되어 재접속·지연 대응 불가: 서버 동기화가 있어도 씬 전환 중 유실 가능성이 남아 있다.
- GDD/TDD에 남은 빈 검증 계획과 문서 메타 정보 보강 필요: 문서가 많아졌지만 빈 칸과 placeholder가 QA 신뢰도를 낮춘다.
- pieceId 조회를 FindObjectsOfType 선형 탐색 대신 registry로 고정: 패킷 처리마다 씬 전체 검색을 반복하는 구조는 제거해야 한다.
